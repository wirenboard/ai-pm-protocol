### Review typology

**Single source for "the review TYPES, each with its own cadence, depth, and scope."** The protocol reviews every change **by diff** and audits **compliance** periodically — but review is a **layered system**, not one undifferentiated pass. This subsection is the one home for the review-type registry: the enum, each type's cadence/depth/scope, and the per-type **deterministic-half / AI-half** split. Every consumer (`pm-audit.md` now, the per-diff review loop, future architectural / functional / criticality slices later) **references this `### Review typology` subsection by name** ("`### Review typology` in `workflow/review-typology.md`") and **re-encodes neither the type list nor the cadence rule** — mirroring how `### Project kind` and `### Decision authority` are single-sourced. This is an EPIC: **slice 1** lays the framework and builds the first, lightest type (**smell / hygiene**); the heavier three types are **registered here as named later slices, not built**.

**The five review types.** Each names its **cadence · depth · scope · deterministic-half · AI-half**:

- **per-diff** *(built — exists today)* — **cadence:** every change. **depth:** low/medium AI. **scope:** the change's diff. **deterministic-half:** lint/format on the diff. **AI-half:** semantic review of the diff (the `/pm-plan` review-loop `code-review` pass). Catches: correctness of *this change*.
- **smell / hygiene** *(built — THIS slice)* — **cadence:** frequent / light (run from `/pm-audit`'s `## Technical quality` sweep). **depth:** selectable (low→ultra, proportionate to scope — see `pm-audit.md`). **scope:** the **proportional new-code-gated** surface (below). **deterministic-half:** smell **detection** — an enumerable static-analysis catalog (dead code, duplication, high cognitive complexity, long methods, magic numbers, over-complexity) a hook/linter *should* run every time; **named here as a downstream/future path, NOT built in this slice** (this markdown-prose repo has no linter to host it — see the det-vs-AI split below). **AI-half:** **prioritization / root-cause** of which smells matter — the `code-review` skill at the selected depth. Catches: functionality-preserving warning signs of deeper weakness (smell ≠ bug — the change still works).
- **architectural** *(later slice — not built)* — **cadence:** heavier / periodic, scoped to changed + hotspot. **depth:** high AI. **scope:** changed code + churn/complexity hotspots. **deterministic-half:** layering / dependency-direction / cycle **rules** (ArchUnit / dependency-cruiser / fitness functions — on a code project; for a markdown-prose repo, AI-judgement over the doc/agent structure). **AI-half:** "is this the right approach / does it adhere to our conventions" — convention/approach judgement over the deterministic rule output. Catches: design-level coupling/cohesion, layering violations, dependency cycles, duplicated abstractions, convention drift.
- **functional / integration** *(later slice — not built)* — **cadence:** rare / manual / heaviest. **depth:** ultra AI. **scope:** cross-component seams. **deterministic-half:** contract / integration / e2e / characterization **tests** — *testable by running*. **AI-half:** reasoning about cross-component behavior from reading the seams tests don't cover. Catches: whether components **interact** without problems. *Note: this type is **more testable than reviewable** — prefer pointing at integration/contract tests; the AI sweep adds value only where tests are absent (legacy, never-tested seams).*
- **criticality-prioritization** *(later slice — not built)* — **a lens over all the types, not a standalone pass.** It orders findings critical→cosmetic by blast-radius, derived from existing artifacts: a feature's Product Contract `## Must work` / `## Must not break` invariants (what is critical), `docs/threat-model.md` (security-critical surfaces), and git churn/hotspot (defect-prone areas). No new tagging — criticality is computable from contracts + threat-model + churn.

**Proportional new-code-gating (the shared cadence machinery — research-validated "Clean as You Code").** The dominant cadence rule across mature tools is **review NEW/changed code, not already-clean code**. The cost of a review is proportional to the **changed** surface: an **incremental** sweep reviews **changed-since-last** (`git diff <last-sweep-sha>..HEAD`); **legacy / never-swept code is covered by the first-run full sweep and the periodic full re-sweep** — not a separate per-incremental detector. A tree already swept-clean since the last sweep is **not** re-reviewed end-to-end. Each type records a **last-sweep marker** (date + commit SHA) so the next sweep can scope `git diff <last-sweep-sha>..HEAD`; when nothing un-swept remains, the sweep is **silent** ("swept-clean since `<date>`"). This is the single concept; its concrete realization for the smell type — **a `## Quality sweep` line in the audit report, no new file** — lives in `pm-audit.md`, which references this principle by name. **Marker↔audit-report coupling caveat:** because the last-sweep marker lives in the `audit-*.md` report, it exists **only where an audit-report does** — safe while `/pm-audit`'s `## Technical quality` sweep is the sole sweep trigger. A future review type or slice whose sweep has a **non-audit trigger** (a standalone sweep not driven by `/pm-audit`) needs a **dedicated marker home**, not the audit-report line — a constraint to honor before adding such a trigger.

**Engine selection (the single source for "which review engine runs which type").** Two engines exist: the built-in `/code-review` skill (lightweight, single-pass, depth-selectable low→ultra) and `wb-development:code-review-orchestrator` (the multi-aspect whole-project reviewer that fans out one subagent per dimension). The rule, stated once here, by scope:

- **per-diff** → always the built-in `/code-review`. The protocol **never auto-routes a routine per-diff review to the orchestrator** — a single diff does not need a whole-project multi-aspect pass, and the per-diff loop must stay cheap. This is **protocol routing discipline** (this file + the per-diff loop in the pipeline), not a hook block: a human may still invoke the orchestrator on a diff whenever they choose.
- **whole-codebase** types (**smell / hygiene** built; **architectural** / **functional** registered-later) → `wb-development:code-review-orchestrator` is the **preferred engine when available**, because the multi-aspect whole-project review is exactly what these types call for. When the orchestrator is **unavailable** (not installed), fall back to the built-in `/code-review` at the selected depth (`ultra` for a deep / first / legacy sweep). The environment switch **`WB_REVIEW_ORCHESTRATOR=off`** forces the built-in fallback **even when the orchestrator is available** (cost control / staying on the built-in engine); default (unset) = prefer the orchestrator when available. This switch governs only the engine preference (read by the sweep prose in `pm-audit.md`); it is not a hook and blocks nothing.

This **reframes** `deny-review-orchestrator` (v2.25.1): the hard per-diff **deny is removed** (the orchestrator is a legitimate engine, no longer a denied skill), and the per-diff → built-in guarantee now rests on **protocol routing discipline** (this rule + the pipeline's per-diff loop) plus the existing soft `UserPromptSubmit` route-reminder — not a `PreToolUse` deny.

**The per-type deterministic-vs-AI split (the concrete application of backlog #211 — "Deterministic-enforceable vs AI-evaluated check boundary").** Each type above names a **deterministic half** (structural / repeatable checks a hook/linter/test runs every time — cheap, reliable) and an **AI half** (semantic judgement — root-cause, prioritization, convention/approach adherence, cross-component reasoning — the `code-review` skill at low→ultra depth). This registry is **where that #211 boundary gets applied concretely**, type by type. For the **smell type** specifically: the deterministic detection half is **named as a downstream/future path, not built in slice 1** — this markdown-prose protocol repo has no linter to host smell-*detection*, so the deterministic half is genuinely aspirational-for-downstream-code-projects, a documented boundary (per #211), not an undelivered feature; slice 1 ships **only** the AI half (the `code-review` smell sweep).

**Boundary — what this typology is distinct from.** The review typology is **not** the compliance audit (`/pm-audit`'s other dimensions — plan/review/contract/docs artifact presence and currency; those check *whether the protocol was followed*, this checks *code/structure quality*). It is **not** the per-diff `code-review` pass alone (that is one type — `per-diff` — in this registry). And it is **not** the cross-model-review backlog item (a *different model* on individual high-risk diffs, to close the shared-LLM-blind-spot gap — orthogonal to this whole-tree single-model layering). All four coexist; this subsection states the boundary so they stay distinct.
