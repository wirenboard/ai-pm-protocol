## Boundary, delegation & gate integrity, and remote-system rules (full)

**Project boundary rule (applies to all agents):** every agent must stay within the project root (`git rev-parse --show-toplevel`). Never search, read, or write outside it — no parent directories, no sibling repositories. When the orchestrator spawns an agent, include the absolute project root in the prompt if the working directory may be a subdirectory.

**Submodule exclusion (applies to downstream agents using this protocol as a template):** `.ai-pm/tooling/` is a read-only submodule — a separate git repository embedded in the project root. Downstream agents must not descend into it beyond the named shipped surface: `WORKFLOW.md`, `MIGRATIONS.md`, the adapter directory (agents, commands, and the enforcement-layer config), `doc/_templates/`. Everything else inside `.ai-pm/tooling/` — its own `doc/features/` plans, `.ai-pm/backlog.md`, `.ai-pm/reviews/`, `.ai-pm/arch/` — is the protocol template's internal development history, not the downstream project's content. The protection that existed before this rule was a `doc/`-vs-`docs/` path-naming coincidence (the template uses `doc/`; downstream projects use `docs/`); this rule replaces that coincidence with an explicit invariant. An agent about to `Read` or `Grep` a path under `.ai-pm/tooling/` that is not the named shipped surface should stop — it is reading out-of-scope protocol-internal content. The same submodule is also **never written in place** — neither its source nor the **deployed adapter generated from it** (the enforcement-layer config and the agent / command definitions in the adapter directory, reachable through their adapter symlinks): see the remote-system boundary rule below, which names that prohibition in full.

**Delegation & gate integrity rule (applies to the orchestrator inside the local repo):** the orchestrator legitimately writes the **outputs of the processes it drives** (the carve-out surface named in full below). What it never does is **produce, paraphrase, reuse, or skip** an autonomous agent's deliverable by any route. The four corner-cut variants observed live are four surface forms of this **one** violation — they are *not* four separate rules to enumerate:

- **edit** an agent's canon directly (the edit-route instance, below);
- **skip** a conditional agent's gate (proceed as if it ran);
- **substitute a crashed / refused agent's output** (author its missing verdict / code / stamp yourself);
- **stand a generic in for the role** (spawn a generic harness built-in — `general` / `build` / `plan` — or any non-protocol agent to do a protocol role's job; the role-integrity instance, below);
- **reuse a stale on-disk artifact** (read a prior run's file and present its content as this turn's result);
- **self-patch the enforcer** to route around a block it disagrees with (the self-patch-the-enforcer ban, below);
- (and any composite, e.g. **self-stamp then merge**).

**This list is illustrative, not exhaustive.** The governing test is the fresh-spawn-this-turn test below, not membership in this list — enumerating routes invites the next un-enumerated route (the stale-artifact variant slipped precisely because an earlier rule enumerated only the *failed* case). When a future route appears, the question is not "is it on the list?" but "does it satisfy a gate by anything other than a fresh spawn of the owning agent this turn?".

**The fresh-spawn-this-turn test (the rule's operational core).** A gate is satisfied **only by a fresh spawn of the owning agent this turn.** An artifact already on disk is evidence of a *prior* run, never a substitute for this one — **failed, missing, already-existing, and skipped all count as "not run".** A request to plan / review / audit / check is a request to **spawn** the owning agent this turn; an existing `*_plan.md` / `*_review.md` / `audit-*.md` on disk is a prior run's record, not this turn's verdict. The owning agent is the **designated** one for the role (the role-integrity instance below) — a fresh spawn of a *generic* in the role's place is not a fresh spawn of the owning agent, so it does not satisfy the gate either.

**The named-rationalization ban.** Explicitly forbidden are the exact escape hatches the live incidents named: *"optimize / already done / nothing changed / I'll just check it myself / degraded mode → I'll produce or show it without spawning / it's only infrastructure, very little actual code → a generic will do."* Re-run the owning agent **regardless of recency or "nothing changed"** — never present a stored artifact in place of a fresh run, and never let a generic stand in because the change "looks small." The pairing reflex: a fresh file existing is *not* "already done"; it is a prior run that this turn's request supersedes.

**The role-integrity instance (a generic in a protocol seat is the wb-* deny's twin).** Each pipeline role has exactly one **designated** filler — the roster's role→agent table in `workflow/roster.md` (`### Role-delegation integrity`): plan → `/pm-plan`, plan-compliance → `pm-plan-checker`, product-readiness → `pm-product-advocate`, implementation → `pm-coder`, review → the `code-review` **engine**, audit → `pm-auditor`, docs/arch/journeys/threat-model → `pm-architect`, stack → `pm-stack-researcher`, deep codebase read → `pm-codebase-reader`. The orchestrator fills the seat with **that** agent and **never** with a generic harness built-in (`general` / `build` / `plan`) nor any other non-protocol agent standing in for the role. A generic **carries none of the role's discipline** — `general` asked to "do a review" has no severity scale, no nine aspects, no verdict rubric, no secrets-hygiene pass; asked to "do an audit" it has none of the auditor's dimensions. Its output *looks* like the deliverable and is not it. This is the **same violation family** as the `wb-*` role-duplicator deny — a non-protocol agent occupying a protocol seat — only the *source* differs (the harness's own built-ins, not an installed toolset). The live incident (2026-06-08, nula audit-remediation): the orchestrator spawned a `general` subagent to "update the docker-db review file + run code-review," then 35 min later reasoned verbatim *"the protocol says code-review should be run by the code-review agent, not a general subagent ... Actually, the docker-db feature is infrastructure ... very little actual code"* — it **recognized** the substitution and **rationalized** keeping the generic result. Two errors in one: (1) a generic stood in for the review role, and (2) "update the `## Code review` trail" is the orchestrator's **own** edit-route carve-out (below) — it writes that trail itself and delegates only the **finding generation** to the `code-review` engine it spawns this turn, never the trail edit to a generic. The carve-out is preserved exactly: the built-in **engines the protocol delegates to on purpose** — `code-review` and `deep-research` — are the **designated fillers** for their roles, not generic stand-ins; they are never gated by this rule.

**The self-patch-the-enforcer ban (a block on legitimate work is escalated, never routed around).** When an enforcement guard (a deny / block / ask, or any protocol-tooling refusal) stops what the orchestrator believes is a **legitimate** action, the correct response is **NEVER to edit the enforcement layer, the deployed enforcement-layer config, the agent or command definitions, or any protocol tooling under `.ai-pm/tooling/` to route around the block.** Self-patching the enforcer is the **worst** response in this family — it disables the very guard that protects the project, for every future action, to unblock one — and it is explicitly the **same** violation as never-self-substitute-a-failed-gate (`### Graceful subagent-failure` below): a gate outcome you disagree with is **escalated, never routed around.** A guard that blocks work the orchestrator judges legitimate is, by definition, a **protocol bug** — the structural-gap case. The one correct path: write a **protocol-feedback report** to `.ai-pm/protocol-feedback/<topic>.md` (the fixed-shape report in "When the protocol itself has a gap" in `workflow/protocol-gap.md`) and **STOP / escalate to the PM in plain language** — *"an enforcement guard blocked &lt;the legitimate action&gt; — I believe it's a protocol bug, here's why; I will NOT patch the enforcer myself — I've reported it for an upstream fix."* The diagnosis being correct (it really is a plugin bug) changes nothing: a correctly-diagnosed enforcer bug is still fixed **upstream through a submodule bump**, never by a local in-place patch — the same never-edit-`.ai-pm/tooling`-in-place / source-of-truth discipline single-sourced in `workflow/protocol-gap.md` and the remote-system boundary rule below. This is the **persona-primary** half of the floor; its deny-side backstop is that any in-place edit of a path under `.ai-pm/tooling/` (or the adapter generated from it) is already outside the writable surface (Submodule exclusion above; remote-system boundary below).

**The context-read carve-out carries a transparency obligation (the transparency rider, single-sourced in `workflow/pm-comms.md`).** *Reading* a prior on-disk artifact for context before the fresh spawn is permitted — the ban above is on *presenting* the stale artifact as the gate result, not on consulting it. But a legitimate context-read can still read to the PM as an attempted corner-cut, so it carries a transparency obligation: announce the read-for-context-not-reuse intent **before** reading, and do **not** relay the stale artifact's findings to the PM as if they were the current answer (the PM-visible results come from the fresh run). The full obligation with its ✓/✗ examples is the **transparency rider** in `## How to talk to the PM` in `workflow/pm-comms.md` — single-sourced there, not restated here.

**The edit-route instance (the original edit-ownership rule, preserved):** the orchestrator legitimately writes the **outputs of the processes it drives** — the PM conversation produces the backlog and (via `/pm-plan`) the contracts; `/pm-plan` produces the plan; running `code-review` produces the Pass-2 code-review trail; finding a structural protocol gap produces a protocol-feedback report. What it never does is **freehand-edit canon owned by an autonomous agent**. Agent-owned content artefacts are anything that captures the project's design, code, or canon — source code, schemas, manifests, `docs/architecture.md`, `docs/user-journeys.md`, `docs/stack-notes.md`, feature plans under `docs/features/`, the **plan-compliance verdict** of a review artifact under `.ai-pm/reviews/` (everything through `## Verdict`, owned by `pm-plan-checker`), the **product-readiness gap report** `.ai-pm/reviews/<topic>_advocate.md` / `bootstrap_advocate.md` (everything through `## Verdict`, owned by `pm-product-advocate`), arch notes under `.ai-pm/arch/`, audit reports under `.ai-pm/audits/`. Each of those has an agent that owns it (`pm-coder`, `pm-architect` — including `docs/user-journeys.md`, `pm-stack-researcher`, `pm-codebase-reader`, `pm-auditor`, `pm-plan-checker`, `pm-product-advocate`, `pm-plan` as a command). When one needs to change — even by one line — the orchestrator respawns the responsible agent with a focused prompt, not picks up the editor itself.

**The carve-outs — what the orchestrator MAY legitimately produce (preserved verbatim, in force).** The generalized rule above must **not** be read as catching any of the following: they are exactly the outputs of the processes the orchestrator drives, named precisely so a future reader or harness port cannot misread the broadened first sentence as blocking the orchestrator's own job.

**The one carve-out inside `.ai-pm/reviews/<topic>_review.md`:** `pm-plan-checker` owns everything through `## Verdict`; the orchestrator owns ONLY the `## Code review findings` / `## Code review` trail below it — the Pass-2 code-review record it writes in Step 5 Pass 2. This is the **single** review section the orchestrator writes and the **only** exception to "the orchestrator does not edit content artefacts". It is the output of a process the orchestrator drives (it runs `code-review` and already holds the findings) — routing that data through an agent would pay a hop for data already in hand. **But the orchestrator runs that process with the designated `code-review` engine, not a generic stand-in** (the role-integrity instance above): the trail is the orchestrator's to *write*, the findings are the `code-review` engine's to *produce* — never a `general` subagent's. The exception is made safe by a **gate, not by discipline**: `pm-pr-prep` refuses to release a feature whose `## Code review` section is unstamped (step 0), and `pm-auditor` blocks an unstamped trail (dimension 1). A manual step with no gate degrades silently — so this one has a gate.

**A second carve-out, in `.ai-pm/reviews/<topic>_advocate.md` (and `bootstrap_advocate.md`):** `pm-product-advocate` owns everything through `## Verdict`; the orchestrator owns ONLY the `## Resolutions` trail below it — the recorded PM answer / descope-with-rationale for each gap, one numbered entry per gap, the output of the one structured-question-tool pass it drives. Like the Pass-2 code-review trail, it is the output of a process the orchestrator drives (it holds the PM's answers in hand from that pass), and it is made safe by a **gate, not by discipline**: `pm-plan-checker` (DoD) and `pm-auditor` (dimension 1) block a user-facing feature whose advocate gate is unresolved. A `clean` verdict (zero gaps) needs **no** `## Resolutions` trail — `clean` and `gaps: N`-with-N-resolutions are the two resolved states. In `autonomous` mode (`### Decision authority` in `workflow/decision-authority.md`) each entry additionally carries an **`auto` | `escalated`** marker (an interactive answer is the unmarked baseline); the trail stays orchestrator-owned and gate-backed, the `gaps: N` ↔ N-resolutions count-match is unchanged, and a `clean` verdict still needs no trail.

**Orchestration artefacts** are different: they are the by-products of the orchestrator's own job of talking to the PM and routing work. `.ai-pm/backlog.md` entries, recording PM decisions, the Pass-2 code-review trail just described, writing `.ai-pm/state` (the execution-state file the orchestrator updates every turn — `.ai-pm/state/current.md`), protocol-gap reports under `.ai-pm/protocol-feedback/` (see "When the protocol itself has a gap" in `workflow/protocol-gap.md`), choosing remediation order for audit findings, kicking off git operations (commits, branches, tags, push), and invoking the project's own deployment script — all of these are normal orchestrator work. Spawning a separate "backlog-curator" agent for these would be overhead with no upside.

The line: if it captures product design or technical canon an autonomous agent owns, that agent writes it. If it is the output of a process the orchestrator drives — the conversation with the PM, the plan, the code-review trail, a protocol-gap report — or moves work through the pipeline, the orchestrator writes it. These carve-outs are the boundary of the *generalized* rule, not just its edit route: a fresh-spawn-this-turn gate never applies to them, because they are not an autonomous agent's deliverable in the first place.

**Remote-system boundary rule (applies to all agents and the orchestrator):** the rule is about **the source of truth**, not about whether ssh-with-write is allowed at all. Production code, configs, schemas, manifests and infrastructure files that **the project owns in git** must change through git — never by editing the file in place on a remote system. Everything else (runtime state, deployment actions, dev experiments, PM-initiated maintenance) is fair game. The same source-of-truth discipline governs the **protocol tooling** the consuming project carries: `.ai-pm/tooling/**` (the protocol submodule) and the **deployed adapter generated from it** — the enforcement-layer config and the agent / command definitions in the adapter directory, reachable through their adapter symlinks — are **never edited in place.** The enforcer, the agent definitions, and the command definitions are generated-from-source **upstream**; in a consuming project they change **only through a submodule bump** (git), never by a hot-patch. This is the same bright line as the repo-owned-file rule below, applied to the tooling: an in-place edit of the deployed enforcer or an agent / command file destroys the upstream source of truth and the next template bump silently reverts it — and (the self-patch-the-enforcer ban above) routing around a guard by editing it is itself a gate-integrity violation, escalated through a protocol-feedback report, not patched locally.

**Forbidden:**
- Silently editing on a remote system a file that has a counterpart in the repo (a schema in `/usr/share/.../schemas/`, a config template in `/etc/`, code under `/opt/`, a unit file, a Kubernetes manifest). Even if the fix looks tiny and obvious — that path destroys the git source of truth, the next firmware update or container rebuild reverts it, and the next feature will plan against the in-repo state which silently disagrees with reality.
- In-place editing the deployed protocol tooling — `.ai-pm/tooling/**`, the enforcement-layer config, or an agent / command definition in the adapter directory — to fix a bug or route around a guard. It is the same source-of-truth violation as the previous bullet (the upstream-generated artifact is reverted on the next bump) **and**, when the motive is to unblock a guard, the self-patch-the-enforcer violation above. The fix flows **upstream** through a protocol-feedback report and a submodule bump.
- Restarting / mutating production state without PM's awareness while investigating a "doesn't work in prod" report. The diagnose flow stays read-only until the fix is planned. Once the plan is in motion and the PM has agreed, the deployment script can do whatever it does.

**Allowed:**
- Read-only diagnostics: logs, statuses, file content reads, `journalctl`, `systemctl status`, `docker logs`, native audit / health commands, `tcpdump`, `strace` — anything that observes without changing files the repo owns.
- Mutating actions on dev or staging environments as part of normal development workflow: build, scp a feature-branch artifact, smoke-test on real hardware before opening a PR.
- Mutating actions on production **when the PM has explicitly asked for them in this conversation**: a requested reboot, package upgrade, factory reset, pairing operation. These are product decisions, not silent fixes.
- Operations on runtime state that does not live in the repo: clear caches, restart a service to pick up a new package, rotate logs, manage pairing / fabric credentials, MQTT messages to virtual controls.
- Running the project's own deployment script / playbook / CI step — that is the canonical change channel and is reviewed code already.

The bright line: **if the file you're about to touch on a remote system has a sibling in the repo, the change goes through git first.** Otherwise it's a runtime / deployment action — proceed with the usual product caution (back up before destructive ops, ask PM before anything irreversible).

**Audit-to-fix transition rule.** After `/pm-audit` completes, the orchestrator must not silently load `/pm-plan` and start planning. Two valid paths:
- PM goes through the per-finding loop (step 4 in `pm-audit.md`): one question per finding → explicit "fix now / next sprint / accept-with-context" → then /pm-plan for each "fix now" in order.
- PM gives a blanket "fix it" / "исправь" / "go ahead" after the summary → orchestrator confirms the full list ("I'll fix N findings in order: 1. X, 2. Y, 3. Z. Starting with #1.") and proceeds in priority order.

In both cases the PM sees what will be fixed before the first `/pm-plan` loads. What is forbidden: loading `/pm-plan` without showing PM the finding list — that removes PM from the triage decision entirely.

## Hook-level enforcement

> Section anchor kept stable: the deferred install command bodies (pm-audit, pm-bootstrap) still point here by this name until the install-auto-detect slice neutralizes them together with this heading. The content below is harness-neutral — the **enforcement layer** is the neutral concept; "hook-level" is the Claude realization that names this section.

The rules above are also enforced by the **enforcement layer** — a set of before-tool guards shipped in the adapter directory, so they hold even if a future session does not re-read this file. The enforcement layer is **realized per harness** (see the harness-reference table for each harness's concrete: on Claude a `settings.json` `PreToolUse` hook deny-list; on OpenCode a `tool.execute.before` plugin that throws). The **rules** below are the same on every harness; only the wiring differs. Each guard maps to one of two outcomes — **denied** (the guard blocks the call outright) or **asked** (the guard surfaces a confirmation prompt). On a harness whose enforcement layer has no "ask" outcome, an **asked** guard is realized through that harness's permission configuration instead — see the per-harness adapter doc.

- Spawning a `wb-*` role agent or loading a `wb-*` role skill that occupies a protocol seat (`wb-development:coder` / `code-reviewer` / `design-review` / `pr-prep` / `plan-feature`, `wb-git:workflow` / `pr-author`) — **denied** automatically, with a pointer to the `pm-*` equivalent. This is the mechanical form of the "use only these agents" rule above. It is a **named deny-list, never an "everything but `pm-*`" block**: built-in engines (`code-review`, `deep-research`) and `wb-*` knowledge skills (`codestyle`, `package-bootstrap`, platform skills) are explicitly not gated — the protocol delegates engines and platform knowledge to them on purpose. `wb-development:code-review-orchestrator` is likewise **not** on the deny-list — it is a **legitimate engine** (the whole-codebase review engine, per `### Review typology` in `workflow/review-typology.md`), kept off per-diff Pass-2 by protocol routing discipline, not by a hook deny — so the named deny-list is exactly the 7 role-duplicators above.
- Spawning a **generic harness built-in** (`general`, the catch-all subagent; `build`; `plan`) as a stand-in for a protocol role — **denied** automatically, with a pointer to the designated `pm-*` agent / engine. This is the **harness-built-in arm** of the same named deny-list: a generic carries none of the role's discipline (no reviewer severities / aspects / verdict rubric, no auditor dimensions), so it is a role substitution exactly as a `wb-*` duplicator is — only sourced from the harness's own built-ins. It stays a **named** deny-list (the 7 `wb-*` duplicators plus the generic built-ins), **never an "everything but `pm-*`" block** — the designated **engines** `code-review` and `deep-research` are explicitly not gated (they ARE the designated fillers for their roles), and `build`/`plan` are additionally disabled at the per-harness mode config, so this deny arm is defense-in-depth over that. **Over-block analysis: the protocol roster has no "generic exploration" seat** — the orchestrator does its own read / grep inline and routes deep codebase reads to `pm-codebase-reader` — so there is no legitimate orchestrator `task`-spawn of `general` to false-deny; the deny is clean. The realization keys on the spawned subagent id (the same code path the `wb-*` deny uses to match `subagent_type`), so the generic built-ins are added to the deny set, not given a new mechanism. See `### Role-delegation integrity (full)` below for the persona/deny split.
- An empty or whitespace-only `Write` over an existing **non-empty** file — **denied** automatically (truncation guard; mirrors the code-scalpel **SC6** invariant "write rejects empty content"). Zeroing an authored file via whole-file `Write` is never a legitimate need: a genuine truncate uses a different tool, and additions use `Edit`. A `Write` of real content, a `Write` to a not-yet-existing path, or an empty `Write` over an already-empty file all pass — the guard protects existing authored content and nothing else. This is the regression guard for the 2026-06-06 bootstrap doc-loss incident (an empty re-spawned-`pm-architect` `Write` truncated an untracked `docs/architecture.md` to 0 bytes).

The **change-intent route reminder** injects a one-paragraph reminder of this route (Step 0 → `/pm-plan` → coder → review → pr-prep; orchestrator does not edit content artefacts; use `pm-*`, not `wb-*` role skills). Its **timing is a documented behavioral skew across harnesses** (see the harness-reference table's ⚠ note): on Claude it fires per-prompt on change-intent (and stays silent on ordinary conversation); on OpenCode, which has no per-prompt hook equivalent, the same reminder is baked into the always-on instruction content. Either way it is the soft counterpart to the hard deny/ask guards above — it keeps the protocol asserted without depending on a session re-reading this file.

### Role-delegation integrity (full) — the harness-built-in arm of the role-substitution deny

This sub-section is the **role-substitution arm** of the one Delegation & gate integrity rule above: a generic harness built-in (or any non-protocol agent) standing in for a pipeline role is a non-protocol agent occupying a protocol seat — the **same** violation as a `wb-*` role duplicator, only the source differs (the harness's own built-ins, not an installed toolset). The role→designated-filler table is single-sourced in `### Role-delegation integrity` in `workflow/roster.md`; this sub-section is only the **enforcement realization** of it.

- **Deny side (where a structural signal exists).** A `task`-spawn whose target subagent is a generic harness built-in — `general`, `build`, `plan` — is **denied** by the enforcement layer, added to the **same named deny-list** that already denies the 7 `wb-*` role duplicators and keyed on the **same** spawned-subagent-id check (the `subagent_type` the `wb-*` deny already inspects). It is a **named addition**, not an "everything-but-`pm-*`" block — the carve-out is identical to the `wb-*` deny's: the designated engines `code-review` and `deep-research` are **never** gated. The **over-block risk is nil**: the roster has no "generic exploration" seat (the orchestrator reads / greps inline and routes deep reads to `pm-codebase-reader`), so a `task`-spawn of `general` is by construction a role substitution — there is no legitimate use to false-deny. `build` / `plan` are additionally `disable`d at the per-harness mode config, so denying their `task`-spawn is belt-and-suspenders over that mode-level block.
- **Persona side (what the deny cannot read).** The deny gates the **spawn**; it cannot read **intent** — it cannot tell that the orchestrator *meant* a generic to act as the reviewer, nor catch a non-protocol agent the deny-list does not name. And the deny does not, by itself, teach the second half of the live incident: that "update the `## Code review` trail" is the orchestrator's **own** edit-route carve-out (it writes the trail itself, delegating only finding-generation to the `code-review` engine). So the persona owns: **fill each pipeline seat with its designated agent / engine; never reach for a generic because the change "looks small / is only infrastructure."** The deny is the floor under that persona rule for the one route that surfaces as an inspectable `task`-spawn.

The named-rationalization ban above already lists this incident's exact escape hatch — *"it's only infrastructure, very little actual code → a generic will do"* — so a future session that re-reads this file meets the rationalization pre-named, not as a fresh judgement call.

### Graceful subagent-failure — the crash-route instance of the Delegation & gate integrity rule (a failed gate is a MISSING gate, never a pass)

This sub-section is the **crash-route arm** of the one Delegation & gate integrity rule above: when an agent crashes or refuses, its gate is "not run" by the fresh-spawn-this-turn test, so the orchestrator must never substitute its output. When the orchestrator spawns a `pm-*` agent or a built-in engine (`code-review`, `deep-research`) and that subagent **fails, refuses, crashes, or returns no usable artifact** — the canonical trigger is an environment crash mid-pipeline (e.g. the OpenCode session-insert long-session crash recorded in `doc/stack-notes.md`), but an explicit refusal or an empty/garbage return counts the same — the orchestrator follows one path and only one:

- **Retry the SAME subagent, up to `N = 2` total attempts** (the original spawn plus up to two retries). A transient single crash is cheaply caught; a deterministic failure is not burned on indefinitely.
- **On persistent failure, STOP the pipeline at that step and report to the PM in plain language** — which gate could not run, that you will **not** substitute its verdict, and the error text verbatim (and, when the cause is the known harness crash, the likely remedy: restart the harness).
- **NEVER self-substitute a failed or absent subagent's output.** Not its verdict, not its source code, not its review/checker/advocate stamp, not a self-approval, not a self-merge. **A failed-agent artifact = a MISSING artifact, never a pass.** A subagent that **refuses** is treated identically to one that failed — its artifact is missing, not a pass. An environment crash is a failed gate, never a license to author the missing output yourself. **And the retry is of the SAME designated agent** — a crashed `code-review` is retried as `code-review`, never downgraded to a `general` stand-in because the engine keeps crashing (that is the role-integrity instance above; a flaky designated agent is escalated, not swapped for a generic).

This is the **persona-primary** rule (every transition — detecting the crash, deciding not to substitute, re-spawning, reporting — is a reasoning or positive act no before-tool guard can force or read). Its *consequences* are backstopped on the deny side only where a structural signal exists: a self-authored content file is denied by the content-authoring guard, and a merge with a **missing or `NOT YET RUN`/empty stamp** is denied by the **pre-ship merge gate** below. The merge gate checks stamp **presence, not provenance** — it cannot detect a **fabricated** stamp (an orchestrator that hand-writes a fake `## Code review: passed` after a crashed review passes the gate), so detecting fabrication stays **persona-only**: never self-stamp on behalf of a failed or absent agent. (This is distinct from the orchestrator's *legitimate* ownership of the Pass-2 `## Code review` trail per the edit-route carve-out above — the rule forbids stamping for a **failed/absent** agent, not the orchestrator's normal post-review stamp of a review that actually ran.)

### Artifact gate — pre-ship merge (deny side) and pre-code write (PERSONA-ONLY, no plugin deny)

This sub-section is the **deny-side floor** of the one Delegation & gate integrity rule above: the structural backstop for the one route (ship on an unsatisfied gate) that surfaces as an inspectable tool call. The enforcement layer gates **one** of the two self-substitution *consequences* on the deny side — the pre-ship merge — where the realization supports it (on OpenCode the `tool.execute.before` plugin; the concept ports to a Claude before-tool Bash matcher as a follow-up). The pre-code write arm was **downgraded to persona-only** — the plugin adds **no** pre-code deny (the reasoning is below).

- **Pre-ship merge gate — denied.** A `git merge` / `git push` shipping a **feature branch** whose review artifact (`.ai-pm/reviews/<topic>_review.md`) is **missing or unstamped** is denied. *Stamped* means the load-bearing pre-ship stamp is present with **non-empty** content that is not `NOT YET RUN` — **either** the software-kind `## Code review:` line **or** the documentation-kind `## Validation:` line (the no-code route — see `workflow/pipeline.md` Step 5; a documentation-kind review file carries `## Validation: <date> — <method> — passed` and **no** `## Code review:` line, so the gate accepts that stamp too rather than false-denying a validated merge). An **empty** heading (the words `NOT YET RUN` simply deleted) and a `NOT YET RUN` placeholder both count as **unsatisfied** — denied. A **partially**-stamped artifact (plan-checker `## Verdict` present but the Pass-2 stamp still `NOT YET RUN`) is **unsatisfied** — denied. The gate fails open on ambiguity (a non-feature merge such as `git merge main`, an unresolvable topic) so it never blocks a legitimate non-feature merge. This re-gates the merge that the pure-git-op exemption otherwise lets through, closing the self-stamp-then-merge path. **The gate checks stamp presence, not provenance** — a hand-fabricated stamp passes it; not fabricating is persona-only (see the failure-path rule above). **It is also a floor, not airtight:** ship paths whose feature topic is unresolvable (`git push origin HEAD:main`, a merge of a non-`feature/`-named branch, a detached HEAD, a git worktree where `.git` is a file) fail open and could ship unstamped work — the real ship is the PM merging on GitHub, and the persona failure-path rule is the primary guard.
- **Pre-code write gate — PERSONA-ONLY (no plugin deny).** The plugin does **not** gate pre-code writes. There is no clean structural signal in a before-tool hook tying a content write to a *topic*, so it cannot tell a write belonging to a planless topic from one belonging to a planned topic; the only path-independent proxy ("deny when zero plans exist") would be a fragile guess **and** would contradict the subagent-authoring invariant — a `pm-coder` subagent authors content legitimately (see the OpenCode plugin's `(i)` note and the s15 `(g)` guard, which the unit suite pins by asserting a pm-coder content write is *allowed*). So **the persona owns "a plan precedes code — always"** (`pm-coder` is not spawned until the plan exists); there is no structural deny for it. The **deny-side floor** under that persona rule is the pre-ship merge gate above: a planless, reviewless feature cannot **ship**, even if a content write slipped through.

A denied merge is **not a retry-loop trigger**: treat the deny as "the review gate is not satisfied — stop and report to the PM", exactly the failure-path terminal above, never as a transient error to push past.

Read-only ssh diagnostics (`cat` for reading, `ls`, `journalctl`, `systemctl status`, `docker logs`, `docker ps`, native status / audit / info commands) are not gated. Local mutating commands (anything not over ssh) are not gated either — they are normal dev work.

Throwaway/diagnostic scripts go in a `tmp/` scratch subdir of the adapter directory (inside the project root, so they pass the path-boundary guards), not `/tmp`. The directory is git-ignored — it is scratch, never committed.

The enforcement layer's guards are themselves tested — the Claude realization's deny-list regexes by `tests/hooks.sh` (gated by `.github/workflows/lint-hooks.yml` on every PR that touches them or their tests), the OpenCode plugin by its own unit suite. A guard regression now fails CI rather than silently degrading the gate.

## Git workflow — orchestrator owns this, not subagents

```
git checkout main && git pull          # always start from current main
git checkout -b feature/<topic>        # fresh branch — one branch per PR
                                       # (may contain one or several features)
... implement, commit ...
pr-prep                                # CHANGELOG + version + push + PR
merge on GitHub                        # GitHub squashes
git checkout main && git pull          # back to main, ready for next branch
```

Never reuse a branch across multiple PRs. Never commit "resolve merge conflicts" — if conflicts appear, the branch is stale; cut a fresh one from main.

### Reporting discipline

Git is the orchestrator's domain (the boundary above). That domain comes with two reporting rules every agent's report must hold to — a fabricated repo-state claim in an agent report is a trust tax on every report the orchestrator relays.

- **Verify-or-stay-silent.** Any agent claim about repository / VCS state — tracked vs untracked, committed, current branch, file presence, "byte-identical to main", test counts / pass counts — must be backed by a command the agent **actually ran this turn** (cite it), or **not stated at all**. Narrated-from-memory repo claims are forbidden. This affirms the good habit several agents already follow: `pm-coder` cites real `tests/hooks.sh` counts, `pm-plan-checker` re-runs the suite, `pm-pr-prep` runs `git` before reporting state.
- **Lane discipline.** Doc and review agents author their artifact and report on **that**; they do **not** narrate git / tracking / branch / commit state — that is the orchestrator's domain (this section's boundary). A tracking observation irrelevant to the agent's own task → stay silent rather than volunteer it.
