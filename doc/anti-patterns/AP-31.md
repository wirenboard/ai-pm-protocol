# AP-31. Spec staleness (code drift'нул от spec'а)

**Что нельзя:**

- Сохранять feature spec в состоянии «merged + acceptance: ok» когда code на код-путях этой фичи был touched после `last_modified:` / `spec_approved:` дольше пороговой window'ы (default 30 дней) без явного operator-acknowledgement.
- Игнорировать staleness warning через AI rationalization («spec не обязательно reflect'ит latest behavior» / «code refactor cosmetic»). Решение «accept staleness» — operator-driven, не AI-derived.
- Промалкивать staleness когда spec'а никогда не было updated после initial approval, но имплементация многократно изменялась.

**Почему:** spec — это PM-facing contract. Когда он stale, PM теряет ground truth для:
- review trail на следующие фичи (Layer 1/2/3 anti-drift checks работают по stale spec'у → false sense of compliance);
- onboarding нового dev'а / нового AI session (читает «текущую правду» из spec'а, который не описывает actual behavior);
- AP-33 cross-feature invariant check'а (extract'ит invariants из stale spec → false positive contradictions).

Brownfield (В1/В2) retrofit'нутые spec'и **тоже** подпадают под AP-31 — они начинают жизнь in-sync с code, но code refactor / bug fix без spec update создаёт ту же staleness.

**Failure mode common pattern:** feature merge'нулась, bug fix через 2 месяца добавил behavior на existing endpoint, тест прошёл, ничего не зафлагилось, никто не обновил spec. Через 6 месяцев новый dev читает spec, видит «endpoint X делает Y», не знает что после bugfix endpoint делает Y + Z. Quietly диверrgent ground truth.

**Mode:** *warning, не fail*. Regex-based detection через `git log` — false positive rate non-zero (cosmetic refactor / comment-only change может triggernuть). Operator решает per-case: update spec / mark `deprecated:` / acknowledge через override marker.

**Как поступать вместо:**

1. **Linter detection** через `check-spec-discipline.sh` family `cross-feature-bounded`, check `spec-staleness`:
   - Read `staleness_days:` из `.ai-pm/.bootstrap-state.md` (default 30).
   - Для каждого `*_spec.md` extract frontmatter `last_modified:` (fallback `spec_approved:` если нет).
   - `git log --since=<spec-date> -- <implied-code-paths>` где implied paths — из `affects_features:` frontmatter ИЛИ heuristic «files matching topic name».
   - Если code touched после spec date > `staleness_days` AND spec не updated → warn `spec-staleness: <spec> — code touched <commit-sha>, threshold exceeded`.

2. **Operator actions** при staleness warning:
   - **Update spec:** добавить bug fix note + bump `last_modified:` date.
   - **Mark deprecated:** если фича действительно retired — frontmatter `deprecated: <date>` + reason + replacement.
   - **Acknowledge через override:** для cosmetic refactor — commit body `[spec-staleness-override: <reason>]`.

3. **Configurable threshold:** `staleness_days: 30` в `.ai-pm/.bootstrap-state.md`. Per-product tuning (daily-deploy: 7; quarterly-cadence: 90+) без code edit.

**Override:** `[spec-staleness-override: <reason>]` в HEAD commit body downgrade'ит warning'и до `log_ok`. Pattern identical AP-25 `[source-bounded-override:]` — operator знаком с механикой.

**Hard floor** — override **не** освобождает от:
- Security path features (auth / crypto / PII / payments) — spec update mandatory даже при cosmetic refactor, audit trail критичен.
- Behaviour-changing modification без spec update — override marker не legitimizes diverging ground truth, surface на reviewer как fork.

**Use case examples:**

- *Legitimate staleness warning:* feature `auth_spec.md` merged 2026-01-01. Bug fix в `src/auth/session.ts` 2026-04-05 (90 days, > 30 threshold) — без spec update. Linter warns: «spec-staleness: auth_spec.md — code touched 2026-04-05 > 30 days from last_modified». Operator updates spec с `## Bug fixes` log entry + `last_modified: 2026-04-05`.
- *Cosmetic refactor:* `src/auth/session.ts` reformatted (lint auto-fix). Linter triggers warning. Operator commit body: `[spec-staleness-override: cosmetic lint reformat, no behaviour change]`.
- *Brownfield В2:* retrofit'нутый `billing_spec.md` extract'нутый 2026-02-01. Через 3 месяца code refactor billing flow без touch spec'а. Linter warns. Operator решает: refactor было descriptive change → update spec; deeper change → fork-justification на следующей feature touching billing.

**Relationship:**

- **AP-33** — sibling Layer 3 check (cross-feature contradiction). Staleness может maskировать contradiction (extract invariants из stale spec → false positives).
- **AP-21** (rework versioning) — staleness threshold пройден означает spec нужен `.v2` если behavior-changing.
- **AP-25** (source-bounded) — Layer 1 sibling: AP-25 catches drift at creation, AP-31 catches drift over time.
- **AP-23** (test assertion weakening) — similar declared-trade-off pattern (override marker required).
