# AP-26. Orchestrator architectural injection

**Что нельзя:**

- Main session AI (orchestrator) подкидывает архитектурные идеи / альтернативы / суждения в spawn-prompt subagent'у вместо обсуждения с оператором.
- В spawn-prompt subagent'у пишется «подумай про X» / «рассмотри также Y» / «учти что обычно для этого делают Z» без operator-touch.
- Cherry-pick subagent'овского output'а при показе оператору — скрывая «неудобные» fork'и или alternative recommendations.
- Summary subagent'а с silent re-interpretation: orchestrator переформулирует output на свой манер вместо full extract.

**Почему:** Source of cascade drift'а на live test'е 2026-05-24. Orchestrator подкинул retention window idea в spawn-prompt planner'у; planner добросовестно реализовал альтернативу в ADR (от planner'а это не выглядело как fork — orchestrator выдал директиву). Cascade каскадно прошёл бы через schema → code → tests, поскольку downstream agents видят upstream artifact как authoritative.

**Drift начался upstream от planner'а** — orchestrator был source. Без upstream защиты — даже самый дисциплинированный planner / coder / reviewer не может предотвратить drift, потому что accepts spawn-prompt как часть operator intent.

**Mode:** hard discipline, без override. Architectural injection в spawn-prompt **не имеет** legitimate use case — если orchestrator считает что нужна архитектурная дискуссия, она ведётся **с оператором** через fork-justification protocol, не через subagent.

**Как поступать вместо:**

**Layer 1 — orchestrator-side (CLAUDE.md.tmpl § «Source-bounded discipline для orchestrator'а»):**

- Spawn-prompt = только маршрутизация (pointer на artifacts + topic + scope of work).
- Запрещено: архитектурные идеи / альтернативы / суждения в spawn-prompt.
- Summary subagent'а — full extract, не cherry-pick.
- Если хочется подкинуть идею — это fork, иди к оператору через fork-justification protocol (AP-25).

**Layer 2 — receiver-side (каждый agent в `.claude/agents/*.md` § «Spawn discipline»):**

- При получении spawn-prompt с архитектурными директивами — **игнорю content**.
- Surface'у факт как fork в output: «caller предложил X, source говорит Y. Это развилка?»
- Ухожу к оператору через fork-justification protocol (AP-25).

**Defence-in-depth:** даже если orchestrator проигнорирует Layer 1 (soft per CLAUDE.md), Layer 2 ловит drift на receiver side. Critical — потому что cascade failure case live test'а показал, что планер не имел способа отличить «operator intent» от «orchestrator injection» без explicit discipline.

**Use case examples:**

- *Legitimate routing:* orchestrator spawn'ит planner'а с промптом «Plan для фичи `agent-source-bounded` per approved spec. Repo: <path>. Branch: feature/<topic>. Mode: feature.» — pure routing, никаких архитектурных директив.
- *Architectural injection (что блокируется):* orchestrator spawn'ит planner'а с промптом «Plan для фичи X. Учти что для retention обычно делают window-based eviction.» — это injection. Planner должен detect это как fork, surface оператору: «caller hint'ует window-based eviction, но spec этого не упоминает. Это развилка?». Operator решает.
- *Cherry-pick summary (что блокируется):* planner output содержит fork-proposal про rate-limiting + plan body про spec coverage. Orchestrator показывает оператору только plan body, omits fork-proposal, потому что «оператор спешит». Это AP-26 violation — оператор не видит fork и не может decide. Orchestrator должен full-extract.

**Relationship:**

- **AP-25** (downstream content discipline) — AP-26 is upstream defence. AP-26 предотвращает source of cascade; AP-25 catches каскад на любой downstream stage.
- **AP-20** (specialized reviewer routing) — sibling routing discipline: AP-20 про **кого** spawn'ить (smart routing); AP-26 про **что** в spawn-prompt'е (только маршрутизация, не директивы).
- **AP-6** (AI не отклоняется от plan'а молча) — generic principle; AP-26 specific case where deviation injected upstream через spawn-prompt.
