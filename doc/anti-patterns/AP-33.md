# AP-33. Cross-feature invariant contradiction (Layer 3)

**Что нельзя:**

- Merge'ить feature spec F-N, который нарушает invariant установленный другим feature spec F-M (или foundational doc — vision / positioning / mvp-scope / threat-model) без явного operator-touch'а на fork.
- Принимать «independent feature» framing без cross-feature pass когда новая F-N имеет `topology_impact: yes` / `threat_impact: yes` / touches shared component.
- Драфтить spec'у который dilutes existing invariant (e.g. F-01 «encrypted client-side» → F-07 «server-side temp storage for export» — invariant erosion, не silent break, но change of promise).

**Почему:** Layer 1 (AP-25/26 source-bounded) ловит drift в одном artifact'е. Layer 2 (AP-27/28/29 cross-doc-bounded) ловит drift в одной фиче (между spec/plan/ADR одной фичи). Остаётся Layer 3: **cross-feature contradiction**.

Live example pattern: F-01 устанавливает invariant «server не имеет plaintext access к user-data». Через 3 месяца F-07 (export to CSV) добавляет server-side temp storage для async export — invariant нарушен. F-07 spec не упоминает F-01 (independent feature). PM не помнит invariant'ы F-01 (3 месяца / 10 features назад). Silent break invariant'а который был договорённостью с пользователями / threat-model'ом / regulatory.

Без Layer 3 — PM holds в голове growing list cross-feature invariants. С 10+ features это превышает working memory; invariant erosion становится unavoidable.

**Failure mode common pattern:** AI extract'ит invariants из spec F-N (read-pass на foundational + spec creation), reviewer проходит per-spec review (AP-25/26/27/28/29), но **cross-feature reasoning** требует читать все existing specs одновременно — этот шаг AI часто skip'ает («новый spec независим, mature engineering»). PM-оператор подсознательно trust'ит «independent feature» framing.

**Mode:** *critical hard fail* по умолчанию для linter check'а; *operator-gate* для reviewer Layer 3 pass.

Critical because: contradicting invariant — не cosmetic, это change of promise. PM **должен** touch'нуть решение (preserve invariant + redesign F-N, или promote invariant change через AP-21 rework F-01).

**Как поступать вместо:**

1. **Linter detection** через `check-spec-discipline.sh` family `cross-feature-bounded`, check `cross-feature-invariant`:
   - Extract «invariant statements» из каждого spec'а через regex baseline (patterns: `всегда|никогда|never|ни при каких|обязательно|запрещено|MUST|MUST NOT`).
   - Scope: `## Invariants extracted` section ИЛИ весь body если spec имеет `topology_impact: yes`.
   - Для новой / changed F-N spec'и cross-check invariants against other specs + foundational docs.
   - Match contradiction → hard fail.

2. **Reviewer Step 2.5 Layer 3 pass** (cross-feature reviewer pass) — invoked при:
   - new F-N spec creation;
   - `topology_impact: yes` flag в frontmatter;
   - foundational doc change в same PR;
   - `affects_features:` list populated (explicit cross-feature scope).

3. **Fork-justification на cross-feature contradiction** (AP-25 pattern extended):
   - **Останавливаюсь.** Не merge'ю F-N spec.
   - **Surface через AskUserQuestion в business terms:**
     - **Existing invariant (F-M):** «<точная цитата>» (`<spec>:<line>`)
     - **F-N предлагает:** «<что меняется invariant'е>»
     - **Это:** change of promise / acceptable extension / mis-classification?
     - **Что выбираем?** Preserve invariant + redesign F-N / promote invariant change (AP-21 rework F-M) / mis-classification — F-N не нарушает.
   - **Жду ответ оператора.**
   - **После approval** — codify decision: либо F-N redesigned, либо F-M.v2 created (AP-21 rework), либо `[cross-feature-override: <reason>]` marker.

4. **Hard cross-feature triggers** (mandatory operator-gate):
   - F-N меняет foundational doc (vision / positioning / mvp-scope / threat-model).
   - F-N имеет component shared с другим F-M (отслеживается через `affects_features:` frontmatter list).
   - Invariant overlap >= 2 between F-N spec и foundational / другие spec'и.

**Override:** `[cross-feature-override: <reason>]` marker в HEAD commit body downgrade'ит fail → warn для linter checks. Pattern identical AP-25 `[source-bounded-override:]`.

**Hard floor** — override **не** освобождает от:
- Foundational doc invariant violation (vision / threat-model) — operator-touch mandatory, override не replaces.
- Security-touching invariant change (encryption / auth / PII handling) — fork-justification protocol full mandatory.

**Use case examples:**

- *Legitimate cross-feature contradiction (what gets flagged):* F-01 `auth_spec.md` invariant «sessions expire after 24h». F-07 `long-running-export_spec.md` introduces background job на 72h без session refresh. Layer 3 linter catches: invariant overlap, contradiction. Reviewer escalates: «F-01 promises 24h session expiry, F-07 adds 72h job. Change of promise or job uses service account?» Operator clarifies → F-07 redesigned (service account, not user session).
- *Acceptable extension:* F-01 invariant «user-data encrypted at rest». F-N adds full-text search via encrypted index (CryptDB pattern). Linter flags overlap. Reviewer surfaces: «F-N adds index on encrypted data — does this expose plaintext?» Operator confirms: index is searchable encryption, no plaintext. Override marker: `[cross-feature-override: searchable encryption, no plaintext exposure]`.
- *Foundational invariant violation (hard floor):* vision.md states «client-side encryption only». F-N adds server-side storage. Override **не** allowed без AP-21 vision rework. Reviewer hard-blocks, escalates: «vision invariant requires AP-21 rework before F-N can merge».

**Relationship:**

- **AP-31** — sibling Layer 3 check (staleness). AP-31 about spec ↔ code drift, AP-33 about spec ↔ spec contradiction.
- **AP-25/AP-26** (Layer 1, source-bounded) — Layer 3 builds on top: catches drift Layer 1 misses (cross-artifact reasoning).
- **AP-27/AP-28/AP-29** (Layer 2, cross-doc-bounded) — Layer 2 within-feature, Layer 3 across-features.
- **AP-21** (rework versioning) — when contradiction is operator-approved as change of promise, AP-21 path opens (.v2 spec для F-M).
- **AP-1** (proactive ADR) — Layer 3 violation often surfaces ADR candidate (architectural decision: which invariant prevails).
