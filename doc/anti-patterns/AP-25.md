# AP-25. AI artifact extends beyond source

**Что нельзя:**

- AI agent (любой из 5 subagent'ов после v0.7.0 consolidation: project-bootstrap / planner / coder / reviewer / release-helper + main session AI / orchestrator) пишет artifact с поведением / decision / behavior, не подтверждённым source artifact'ами этого agent'а (см. per-agent specifics в `.claude/agents/*.md` § «Source contract»).
- Planner создаёт ADR с alternatives, которых нет в spec'е («для полноты»).
- Coder добавляет input validation rules / API fields / DB columns / retry logic, не упомянутые в spec/plan'е («just in case»).
- Reviewer surface'ит findings без `diff_reference:` / `spec_reference:` («код выглядит подозрительно где-то ещё»).
- Project-bootstrap записывает в `.bootstrap-state.md` значение, которое не из operator-confirmation и не из auto-extraction («reasonable default»).
- Release-helper добавляет invented impact descriptions / migration steps не из actual commits.

**Почему:** на live test'е 2026-05-24 planner создал ADR с retention window альтернативой, которой не было в spec'е; кейс caught оператором благодаря domain expertise, в слабом домене drift'нул бы каскадом в schema → code → tests на месяцы. Без enforcement этого паттерна — template **бесполезен**: оператор не может trust AI-output, потому что в любой точке цепочки AI мог тихо расширить scope. Review trail / advisor / fitness functions работают только если их **input** — настоящая спека, а не AI-расширенная версия.

**Failure mode common pattern**: AI agent видит «логичное расширение» source artifact'а — недостающая alternative, очевидная validation, helpful default — и **молча реализует** в своём output вместо surface'нуть оператору как fork. PM-оператор (не читает код) не имеет mechanism'а это поймать post-hoc.

**Mode:** *requires-operator-touch (declared decision)*, не hard block. Расширение beyond source разрешено, но **требует** fork-justification protocol с operator-approved результатом.

**Как поступать вместо:**

1. **Каждый agent имеет explicit Source contract** в своём `.claude/agents/<agent>.md` файле (секции `## Source contract` / `## Fork-justification protocol` / `## Spawn discipline`) — что есть ground truth, что считается fork'ом, какой output check применяется.

2. **Универсальный fork-justification protocol** (формат фиксирован для всех agents — единый pattern для PM-оператора):
   - **Останавливаюсь.** Не пишу artifact / не реализую расширение.
   - **Формулирую structured proposal** через AskUserQuestion:
     - **Source говорит:** «<точная цитата>» (`<file>:<line-range>`)
     - **Я предлагаю по-другому:** `<что меняется>`
     - **Почему:** `<конкретный аргумент>`
     - **Что выбираем?**
   - **Жду ответ оператора.** Никаких параллельных действий.
   - **Только после ответа** — кодифицирую решение с reference на source + operator_approved timestamp.

3. **Linter enforcement** через `check-spec-discipline.sh` family `source-bounded`:
   - `adr-spec-reference` — каждый ADR имеет frontmatter `spec_reference:` + `operator_approved:`.
   - `plan-spec-reference` — каждый `<topic>_plan.md` имеет frontmatter `spec_reference:`.
   - Future-extensible: добавление source-bounded check для нового artifact type (release-helper output / bootstrap-state values / etc.) = новая функция `check_source_bounded_<artifact>` с тем же override marker pattern.

**Override:** `[source-bounded-override: <reason>]` marker в HEAD commit body downgrade'ит fail → warn для linter checks. Pattern идентичен existing `[test-modify-override: <reason>]` (AP-23) и `[review-override: <reason>]` (AP-16). Используется для legacy migration / bootstrap edge cases.

**Hard floor** — override **не освобождает** от:
- Fork-justification protocol для security-touching artifact'ов (auth / crypto / PII / payments). Operator-touch обязателен даже при override marker'е, потому что security regressions через silent extension критичны.
- ADR creation без `operator_approved:` поле — этот marker сам по себе **доказательство** operator-touch'а; override marker не replaces его.

**Use case examples:**

- *Legitimate fork:* planner драфтит plan, видит что spec не упоминает rate limiting на endpoint, но domain (payment processing) делает это critical. Stop'ается, AskUserQuestion: «Source говорит: endpoints в spec'е без rate-limit constraints. Я предлагаю: добавить rate-limit invariant в plan. Почему: payment-touching, AP-15 backend guide рекомендует. Что выбираем?» Operator approve'ит — plan extended с reference на operator-touch.
- *Silent drift (что блокируется):* planner драфтит ADR с 4 alternatives, последняя — invented «middle-ground» (retention window не упомянут в spec'е) добавлен «для thoroughness». Frontmatter `spec_reference:` присутствует, но содержимое ADR содержит alternative не traceable к spec'е. Linter не catches содержание (это design intent, не frontmatter check), но reviewer Step 7 в Mandatory baseline section находит ADR alternative без spec citation и flag'ует. Coder при follow-up implementing должен detect mismatch и trigger fork-justification на свою сторону.
- *Trivial override:* legacy product repo адаптируется на template v0.6, existing ADRs не имеют frontmatter полей. Migration commit body: `chore: backfill ADR frontmatter [source-bounded-override: template-v0.6-legacy-migration]`. Linter downgrade'ит fail → warn, audit trail preserved.

**Relationship:**

- **AP-1** (proactive ADR) closes timing gap; **AP-25** closes content gap. AP-1: ADR не пишется до plan'а. AP-25: ADR (или любой artifact) не extends beyond source без operator-touch.
- **AP-24** (ADR auto-extraction для > 50 LOC arch content в spec'е) — sibling pattern: AP-24 про **где** жить architectural content, AP-25 про **откуда брать** content.
- **AP-26** (orchestrator architectural injection) — upstream defence для AP-25. AP-26 prevents orchestrator от подбрасывания идей в spawn-prompt; AP-25 catches drift даже если AP-26 не сработал (defence-in-depth).
- **AP-6** (AI не отклоняется от plan'а молча) — generic principle, AP-25 — concrete enforcement через source contract + fork protocol + linter.

См. также `development-protocol.md` § 9.5 (Source-bounded contract).
