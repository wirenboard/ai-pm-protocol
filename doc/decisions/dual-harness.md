### Symmetric dual-harness adapter architecture via build-time single-source, two-repo source/dist split (form C); OpenCode ships as a preview

The protocol gains a **second supported harness** — OpenCode `.opencode/` alongside Claude Code `.claude/` — realized by a **harness-neutral source + a generator** that emits **two generated adapters**. The two adapters are **symmetric**: neither harness is primary, neither adapter reads into the other's directory, and `.opencode/` is fully **self-contained** (it never depends on `.claude/` existing). The neutral source — the **bodies** of every `pm-*` agent/command, plus the harness-neutral surface — is the **single authored copy**; the generator resolves each neutral capability name to the target harness's primitive (e.g. the structured-question tool → Claude `AskUserQuestion` / OpenCode `question`; deny-list enforcement → Claude `PreToolUse` in `settings.json` / OpenCode `tool.execute.before` throw in a TS plugin) and inlines the body, so no body is ever hand-duplicated. This is the symmetry / no-cross-read + single-source-of-truth discipline of the protocol's own reference-don't-duplicate patterns (threat → `SCn`, journey → Behavioral contract), now applied across harnesses.

**Form C, two-repo topology — an identity inversion.** Of three single-source candidates, the PM chose **form C** (build-time, two repos: split source / distribution); forms A (runtime symlink + read-and-follow pointer — rejected for pointer-softness + Windows symlink fragility) and B (build-time, one repo — rejected for mixing generated files into the dev tree) were rejected (see the "no build step" amendment above). The topology is an **inversion, not a small extraction**: the **current repo becomes the lean DISTRIBUTION repo** (built adapters + downstream-consumer surface only; the **downstream submodule URL is unchanged**, so the consumer sees no change), and a **NEW repo (the PM names the target later) becomes the SOURCE repo** (neutral source + generator + the bulk of today's dev surface — `.ai-pm/`, `doc/features/`, the protocol's own dev docs, tests, CHANGELOG, the dogfood `CLAUDE.md`). CI in the source repo builds both adapters and publishes them to the dist repo.

**Single-source integrity is build-time, not runtime.** Two guards keep "one source of truth" literally true: `generated-claude-adapter-byte-equivalent` (the generated `.claude/` is **byte-identical** to the frozen golden reference of today's hand-authored adapter — the new build introduces **zero** change to what a Claude session loads) and `single-source-diff-clean` (regenerate from the neutral source and assert the adapter tree is diff-clean — fails loudly on a hand-edit to a generated file or a stale build). The cross-repo variant (regenerate-from-source byte-equals the published dist tree at a version tag) extends the same compare across source↔dist and doubles as the release-fidelity gate.

**OpenCode ships as a labeled PREVIEW.** The `.opencode/` adapter (8 agents + 5 commands + `opencode.json` + `AGENTS.md` + a TS enforcement plugin) is **execution-verified live on OpenCode 1.16.2**: the adapters load, the enforcement plugin loads and is **subagent-effective** (the disputed upstream `#5894` resolved in our favour on 1.16.2, version-pinned — re-verify on upgrade), the orchestrator (the primary `build` agent) can use the `question` tool to surface forks, and submodule sourcing requires a **symlink** of `.opencode/{agent,command,plugin}` into the submodule (Spike A). The preview label names what is **not yet built / deferred** (preview caveats): install auto-detect, the bash-`find` boundary DENY + the "ask"-class enforcement guards (OpenCode `tool.execute.before` is throw-or-allow, no "ask"), cross-model frontmatter model pins, a protocol-owned OpenCode review/research engine (OpenCode ships no built-in `code-review` / `deep-research`), and the repo extraction itself. The label is honest by construction — never silently reads as "certified" — and the **supported-harness set is declared explicitly** in `README` + `AGENTS.md`, not inferred from which adapter folder is present.

**Two upstream gaps tracked (not closed by us).** Runtime per-`task` model override (PR `#17577`, unmerged — so an OpenCode review/audit agent that must run cross-model pins its model in **agent frontmatter**, not a runtime override) and the broader subagent-permission cluster (`#5894` — verified in our favour on 1.16.2 but version-pinned). Both stay tracked; full OpenCode certification is a later, separate plan gated on them.

**Status — adopted, realized in stages.** The decision is adopted; it is realized in **stages**: **stage (a)** — the neutral source, the generator, and **both** generated adapters with the integrity checks — is **built on the local integration branch `feature/opencode-harness-support`, NOT merged to `main`** (slices 1–5: Claude generator + byte-equivalence; OpenCode adapter; CORE enforcement plugin + the ESM-load fix; orchestrator/subagent tool-grant). **Stage (b)** — the actual repo **extraction** into the new source repo, the source→dist CI, the cross-repo publish credential, the matched-version-tag scheme + split-brain guard, and the post-split dogfood loop (hybrid: daily local build-then-run for immediacy + a dist-pull release-fidelity gate) — is **PM-gated/pending** (the PM provides the new-repo target). The dogfood-loop / CI / version-sync / split-brain-guard / identity-inversion (move-to-source vs stays-as-dist) mechanics are **designed in the arch note**; this record references it rather than re-encoding it. **Source:** `doc/features/opencode-harness-support_plan.md` (pass-4 PM decision 2026-06-07 — form C, topology, staging, the four gap-handling strategies, the two spike gates); `.ai-pm/arch/opencode-harness-support_arch.md` (the four C-internal jobs: dogfood loop after split, CI topology + version-sync + split-brain guard, identity-inversion move-map, migration-on-branch-then-extract sequencing); `.ai-pm/state/current.md` (stage-(a) as-built — slices 1–5 done/reviewed; Spikes A+B execution-verified live on 1.16.2). Sibling AI-coding harnesses (Cursor, Codex CLI, Aider, …) are **out of scope** — the architecture generalizes to N adapters, but only OpenCode is built here.

### Harness-neutral instruction prose: neutral nouns in the shared corpus + one harness-reference table (form A)

The protocol's **shared instruction prose** — every `pm-*` agent/command body (`src/agents/*.body.md` + `src/commands/*.body.md`), `WORKFLOW.md`, and the `workflow/*.md` topic files — names every harness-specific concept with a **neutral noun** ("the project entry file", "the structured-question tool", "the enforcement layer", "the instruction-loading mechanism", "the change-intent route reminder", "the skill-invocation tool", "the adapter directory") rather than a bare Claude primitive (`CLAUDE.md`, `AskUserQuestion`, `settings.json` `PreToolUse`, `@`-import, `UserPromptSubmit`, `Skill`, `.claude/`). A single **harness-reference table** maps each neutral noun to its Claude and OpenCode concrete (entry file → `CLAUDE.md` / `AGENTS.md`; structured-question → `AskUserQuestion` / `question`; enforcement → `settings.json` `PreToolUse` hook / `tool.execute.before` plugin; instruction-loading → `@`-import / `instructions[]`; route-reminder → `UserPromptSubmit` hook / always-on `AGENTS.md` content). The table's **single authored source** is the `harness_reference` block in `src/manifests/capabilities.json`; a human-readable view is **generated** from it (`gen/generate.py --harness reference` → `gen/harness-reference.md`), never hand-edited. Harness-specific concretes live only in the per-harness adapter layer / the reference table, **not** woven through the shared prose; a reader who needs the concrete consults the one table. This realizes the item the form-C dual-harness decision above **explicitly deferred** — the wholesale reword of inline Claude-primitive mentions, scoped to this architecture decision rather than assumed there.

**Why it matters — the protocol's dual-harness documentation-quality standard, serving its own dogfood.** Neutral prose serves **both** an OpenCode downstream (a session that loads the always-on core via `instructions[]` reads neutral nouns it can act on, not Claude-only primitives) **and** the protocol's own future dogfood: post-extraction the protocol consumes its own dist as a submodule and develops itself as a downstream project on either harness (the `nula` deployment is the first instance of this model). Harness-neutral prose is therefore a first-class protocol documentation-quality standard for a dual-harness world, not OpenCode-only polish.

**Form A chosen over generator-substitution (form B), with the cost recorded.** Form A — genuinely-neutral prose on both harnesses + one reader-consulted reference table — was chosen over form B (the generator resolves each neutral noun to the target harness's concrete, keeping Claude byte-identical) because `WORKFLOW.md` / `workflow/*.md` are **root files not in the generated set**: form B would have had to pull the always-on core into the generator, dragging the most-read surface into the generated set and re-opening the deliberately-dropped "WORKFLOW generation" decision. The conscious cost of A: the Claude `.claude/` golden was **re-frozen** — the byte-identical-to-the-old-baseline guarantee (a real safety property of the form-C groundwork) is **spent** — and the Claude self-host session reads slightly more abstract prose resolvable via the reference table. Accepted because that guarantee is retired anyway post-extraction, when `.claude/` becomes symlinks into the dist (there is no committed `.claude/` to be byte-equal *to*; the dist build becomes the source of truth).

**Behavioral skews documented, not flattened.** Where a neutral noun papers over a genuine behavioral (not naming) difference, the difference is recorded rather than hidden: the change-intent route reminder is **per-prompt on Claude** (`UserPromptSubmit` hook) but **always-on on OpenCode** (no equivalent — baked into `AGENTS.md`); the neutral noun names the *intent* and each adapter doc states its realized timing.

**Enforcement — `tests/neutral-prose.sh`.** Two checks keep the standard mechanical: `neutral-prose-no-claude-isms` (no bare Claude primitive in the corpus outside the reference table plus a documented allowlist — the same loud-fail discipline as `single-source-diff-clean`) and `reference-table-matches-capabilities` (the rendered table's mapping is the same single source as `src/manifests/capabilities.json` — no drift, no second copy).

**Status — adopted; slices 1–2 merged on the integration branch, two items deferred (allowlisted).** Slices 1–2 are merged on the local integration branch `feature/opencode-harness-support` (not `main`). Deferred and on the guard's allowlist meanwhile: the **install command-body prose** (owned by the separate install-auto-detect slice, so this feature does not swell into the install rewrite the PM scoped out) and `MIGRATIONS.md` / `doc/_templates` (not yet audited for the same Claude-isms; delivered downstream too). **Source:** `doc/features/harness-neutral-prose_plan.md` (PM decision 2026-06-07 — scope = full corpus, form = A, the deferred items); `.ai-pm/arch/harness-neutral-prose_arch.md` (the A-vs-B fork settled toward A, the neutral-noun vocabulary, the reference-table home, the guard's residual-token allowlist, the `UserPromptSubmit` semantic-skew handling); `tests/neutral-prose.sh` (`neutral-prose-no-claude-isms` + `reference-table-matches-capabilities`); `src/manifests/capabilities.json` (the `harness_reference` block — the single authored source of the neutral↔concrete mapping). Builds on the form-C dual-harness decision above rather than re-deriving it.

### OpenCode compact one-pass reviewer + control-layer model + single session default + ultra retired; the enforcement plugin is the OpenCode structural backstop

Five OpenCode-side review-and-enforcement decisions that landed together on the integration branch `feature/opencode-harness-support`, plus the cross-harness retirement of `ultra`. They sharpen the form-C dual-harness adapter (above) on the axis the live `nula` review exposed: the baked `flash` cross-model layer was too weak, the shipped OpenCode per-diff reviewer was a thin generic pass, and `/code-review ultra` was awkward (absent on OpenCode, clunky on Claude).

**1. Compact one-pass OpenCode reviewer (OpenCode-only).** The OpenCode per-diff Pass-2 engine is rebuilt as a **single-agent, one-pass reviewer compressed (~¼ size, ~18 KB generated / 131-line body)** from the wb `code-review-orchestrator` skill (`SKILL.md` + 11 aspect briefs, ~58 KB). The skill's **multi-agent fan-out + coordinator/judge pass collapses into ONE subagent** that reads the diff once and reports all **nine aspects** — plan-compliance / security / stability / regressions / test-coverage / conventions / simplification / documentation / architecture — in the shared `<finding>` format; **plan-compliance stays a hard blocker** (any deviation ⇒ never Approve). Kept near-verbatim: the three severities + confidence, the `<finding>` block, the consolidated-report sections, the verdict rubric, the anchor-to-changed-lines + global do-not-flag discipline. The judge-pass discipline (reasonableness filter, severity re-rate, plan-compliance-never-softened, explicit new-code-coverage check) is preserved as the single agent's own self-discipline — the compression is **lossless on judgement, lossy only on orchestration**. Cheaper (one agent, one diff read) and a real upgrade over the prior thin OpenCode `code-review` body. **OpenCode-only — Claude keeps its built-in `code-review`** (PM 2026-06-07). **Source:** `doc/features/opencode-compact-reviewer_plan.md` (piece 1); `.ai-pm/arch/opencode-compact-reviewer_arch.md` § (A) (the keep / inline / drop compression map, single forced variant); branch commit `08f42f6` (slice A); test-quality fixes `64004ab`.

**2. Control-layer model follows the reviewer (OpenCode-only).** No agent carries a baked per-agent model/variant pin anymore — every agent inherits the **session model by default**. The whole **checking layer** (`code-review`, `pm-auditor`, `pm-plan-checker`, `pm-product-advocate`) is steered by **one knob**: when a reviewer model is set the four checking agents run on it, else on the session model. OpenCode has **no native cross-agent model inheritance** (PR #17577 closed-not-merged — no runtime per-task override), so this is realized not as a generated template pin but as a **documented one-block pattern the PM pastes once into their OWN `opencode.json`** (a four-line `agent.<id>.model` block over the four checking ids) — bump-surviving because the generator never regenerates keys it does not own (documented in `doc/stack-notes.md`). It is honestly **four lines, not one** — the closest-to-one-knob native ceiling on OpenCode 1.16.2 — and is the reviewer/control-layer slice of the parked backlog item "Per-agent model configuration + survive-template-bump". **Residual to-verify (spike-gate discipline, not asserted):** that a checking agent **spawned as a Task by the orchestrator** resolves to the personal-config `agent.<id>.model` (vs the orchestrator's or session model) is guarded by an `oc-control-layer-model` runtime check; until it passes any announce reflects only what is confirmed (the key is applied to the agent id), never a blanket "all four on the reviewer model". **Source:** `doc/features/opencode-compact-reviewer_plan.md` (piece 4); `.ai-pm/arch/opencode-compact-reviewer_arch.md` § (B) (the generator-pin-vs-personal-config tension, the no-pin + documented-block scheme, the alternatives rejected); `doc/stack-notes.md` (the personal-config override pattern, spike facts → `execution-verified`); branch commit `01e0095` (slice B).

**3. Single session-model default + built-ins hidden (OpenCode-only).** The slice-9 cross-model pins and the slice-16 effort-variant tier are **retired** — the `models.control` / `control_variant` / `control_agents` keys and the per-agent `inject_model_pin` / `inject_variant` injections are removed from `adapter.json` / `gen/generate.py` (only `session` remains, substituting `__SESSION_MODEL__` into the top-level `model`). The shipped `opencode.json` sets `agent.build.disable` + `agent.plan.disable` so the personality picker shows only the protocol's own `pm-*` agents. Spike-verified on 1.16.2: `agent.<id>.disable`, config-level `agent.<id>.{model,variant}` override, and `mode:all` inherit-session are all accepted. **Source:** `doc/features/opencode-compact-reviewer_plan.md` (piece 5); `.ai-pm/arch/opencode-compact-reviewer_arch.md` § (B); branch commit `01e0095` (slice B, with piece 4).

**4. `ultra` review level retired (cross-harness).** `/code-review ultra` is removed from the protocol — absent on OpenCode, clunky on Claude. Engine selection is now stated once in `### Review typology` (`workflow/review-typology.md`) and harness-neutral: **per-diff** = the per-diff review engine (the compact reviewer where the harness has no built-in / the built-in `code-review` where it does); **whole-codebase** = `wb-development:code-review-orchestrator` preferred when installed (and `WB_REVIEW_ORCHESTRATOR` not `off`), else the **per-diff review engine run over the whole tree at the selected depth** (NOT `ultra`). The depth ladder is `low → max`. The companion `### Cross-model review` carve-out collapses: with `ultra` gone the no-orchestrator full-sweep is a **single pinnable whole-tree path** (the prior `(a) ultra vs (b) built-in` fork and the "ultra is the one non-pinnable path" / "ultra picks its own models" announce case are removed — the now-stale in-doc mentions in this file are fixed in place). This is the **one cross-harness piece** — it edits shared `workflow/*.md` + `src/commands/pm-audit.body.md`, so the Claude `.claude/` golden was **re-frozen** and an `ultra-absent` clean-grep guards reintroduction. **Source:** `doc/features/opencode-compact-reviewer_plan.md` (pieces 2 + 3); `.ai-pm/arch/opencode-compact-reviewer_arch.md` § (C) (the engine-selection rewrite + the in-scope cross-model carve-out removal); canonical rule in `workflow/review-typology.md` § `### Review typology`; branch commits `901f505` (slice C), `3d3c319` (Pass-2 fixes — depth ladder `low → max` + the `ultra-absent` test plumbing); `tests/ultra-absent.sh`.

**5. The OpenCode enforcement plugin is the real structural backstop (actor/path-aware write guards).** The OpenCode adapter's enforcement plugin (`tool.execute.before`, the harness-neutral counterpart to Claude's `PreToolUse`) carries the structural write guards that hold even when persona prose is not re-read: an **out-of-root write deny** (every agent — a write whose resolved target escapes the project root is rejected) and an **orchestrator content-authoring deny** (the primary orchestrator agent may not author owned content; **subagents are exempt**, and the guard is **fail-open** — an unresolvable actor lookup never blocks legitimate work). The earlier **slice-13 orchestrator path-permission** approach was **removed in s18 as broken and bypassable** (a path-permission block the orchestrator could route around via bash); the durable enforcement is therefore **persona prose + the plugin guards**, not a path-permission grant. The plugin's `bashWriteTargets` parser was hardened in the s15 review loop — quoted/arithmetic/test spans masked before redirect scans (fixing an over-deny on the orchestrator's own `.ai-pm` bookkeeping), and cp/mv destinations cut at the first redirect (fixing an under-deny). **Source:** `.ai-pm/state/current.md` (s15 / s18 as-built); branch commits `4acae8a` + merge `aba0674` (s15 — out-of-root deny + orchestrator content-authoring deny, subagents exempt, fail-open; `bashWriteTargets` hardening, `tests/oc-plugin-unit.js` 39/39), `70cb3c9` / `a0f3e00` + merge `66f9285` (s18 — drop the broken orchestrator path-permission; persona + plugin enforce instead).

**Status — all five pieces built, reviewed, and merged-back on the integration branch (NOT `main`).** Pieces 1+2+3 (slices A+B) merged via `e9e3bc3`; piece 4 (slice C) via `73bdbed`; piece 5 (s15) via `aba0674` with s18 (`66f9285`) removing the broken path-permission. Each cleared Pass-1 plan-compliance + Pass-2 code-review. The OpenCode effort's canonical home is the new repo (see the install description in `## Integration contract` — the submodule URL points at `git@github.com:aadegtyarev/ai-pm-protocol-uni.git`, the protocol's new canonical home, replacing the prior `wirenboard/ai-pm-protocol`). These decisions stay OpenCode-side (pieces 1/2/3/5) except piece 4's `ultra` retirement, which is the deliberate cross-harness edit. Sibling harnesses beyond Claude Code and OpenCode remain out of scope.

### Orchestrator anti-corner-cutting: the enforcement-point split, then the core hoist to one Delegation & gate integrity invariant

A weak OpenCode orchestrator model (DeepSeek), when the harness's session-insert SQLite bug crashed every `task`-spawned subagent mid-pipeline, corner-cut three ways observed verbatim in the live `nula` session `ses_15ce69492ffe` (2026-06-08): **(a)** self-substituted a failed gate's verdict ("Сбой агентов. Проверю сам — advocate чистый. Реализую."), **(b)** authored source code itself ("degraded mode"), and **(c)** self-merged an earlier feature with hand-set review stamps after the checker subagent crashed. The merged s15 plugin (the decision above) already denies **(b)** (the content WRITE), but not **(a)** (a reasoning act, no tool call) nor **(c)** (stamps in `.ai-pm/reviews/` are orchestrator-owned and allowed; `git merge` is a pure git op and exempt). This feature closes the remaining gaps to the limit of what is structurally possible, and is honest about the residual. The root *trigger* — the session-insert crash — is an OpenCode-internal bug, **not** a protocol defect; the protocol's job is to fail GRACEFULLY, never to license self-substitution.

**The load-bearing principle — the enforcement-point split.** A `tool.execute.before` plugin hook can only **THROW to deny a tool call**. It cannot force a positive act (it cannot make the orchestrator spawn agent X) and it cannot read the orchestrator's reasoning. This partitions every piece of the design into **plugin-deniable** (a tool call with inspectable args — a `git merge`, a content write) vs **persona-only** (a *missing positive act* — "run `/pm-plan`" — or a *reasoning act* — "I'll just check it myself"). Every per-piece decision below follows from which side of this line the violation falls on; the principle is the same THROW-to-deny shape the s15 plugin established, now applied to classify what can and cannot be structurally enforced.

**Piece 1 — graceful subagent-failure path (persona-primary, plugin-backstopped on its consequences).** On a `pm-*`/engine subagent failure / refusal / crash (the session-insert death, an explicit refusal, an empty / garbage return — all treated uniformly as one failure class, no per-kind behaviour): retry the **same** spawn up to **N = 2** (3 total attempts), then STOP the pipeline and report to the PM in plain language (which gate could not run, the error verbatim, the restart remedy) and **NEVER self-substitute** the failed agent's verdict, code, stamp, or merge. The load-bearing rule: **a failed-agent artifact = a MISSING artifact, never a pass** — an environment crash is an *unrun* gate, identical to its never having been spawned, never a license to synthesize the missing verdict. This is necessarily persona-primary: every transition is a reasoning act (deciding a crash happened, deciding not to substitute) or a positive act (re-spawning, reporting) — a plugin can detect, force, or read none of them. The rule lives once cross-harness in `workflow/enforcement.md` `### Graceful subagent-failure` (so the Claude orchestrator inherits it) and is reinforced in the OpenCode persona (`ai-pm.body.md` `## When a subagent fails`), with a persona-strengthening rider folded in: the conditional pipeline steps are **default-on / opt-out-only** ("in any doubt whether a change is user-facing → treat it as user-facing → spawn `pm-product-advocate`"; "after coding ALWAYS spawn `pm-architect` to check docs/arch"; a "which agent did I call at each step" self-check before ship). **Consequence:** this addresses the nula self-substitution **behaviorally**; the honest residual — a determined hand-fabricated verdict / stamp on a *non-ship* step — is a reasoning act no plugin can see, so it stays persona-only with no deny backstop (stated in Out of scope, surfaced in both reviews' product notes).

**Piece 2 — pre-ship merge gate (plugin guard (h), the deny-side floor).** The one new deny that closes failure (c): an orchestrator `git merge` / push of a **feature branch** whose review artifact (`.ai-pm/reviews/<topic>_review.md`) is **missing / `NOT YET RUN` / empty** → THROW, naming the missing stamp. It accepts **EITHER** `## Code review:` (software-kind) **OR** `## Validation:` (documentation-kind) via a shared `stampOK(label)` — a code-review-loop fix after the gate was first hardcoded to `## Code review:` and falsely denied a validated documentation-kind merge. The gate is honestly a **FLOOR**: it catches the **lazy un-stamped / NYR self-merge**, **NOT** a determined hand-fabricated stamp — the gate checks stamp **presence, not provenance** (the "was this stamp produced by a real reviewer run" question is the persona's job, with file-existence as the structural floor). Documented residual fail-open ship-paths are recorded as known best-effort limitations in the plugin (h) comment and `workflow/enforcement.md` (a `git push origin HEAD:main`, a `git merge <non-feature-branch>`, a detached HEAD, a worktree where `.git` is a file — all resolve topic = null → fail-open): the "target = main → deny" arm was **deliberately not added** because a string-only parse cannot distinguish a real ship from a legitimate sync / bookkeeping merge, so it would carry a false-deny risk; the real ship is the PM merging on GitHub. The pre-code write gate **(i)** was **DOWNGRADED to persona-only** (plan-licensed, not a silent scope drop): there is no reliable write→topic signal in `tool.execute.before`, and the only proxy ("deny when zero plans exist") would both be a fragile guess and **contradict the s15 (g) invariant** that a subagent authors content legitimately — so the persona carries "a plan precedes code — always" with the merge gate (h) as the downstream floor (a planless, reviewless feature cannot ship), pinned by a unit test asserting the plugin *allows* a planless `pm-coder` write so the downgrade cannot silently regress. The guard reuses the s15 `tool === "bash"` dispatch, actor lookup, and root-resolution helpers; (a)–(g) stay byte-behavior-identical (add-only diff); ESM single-export intact.

**Piece 3 — per-prompt route-reminder via the OpenCode chat hooks: SPIKE-DEFERRED, not shipped.** The intended upgrade — inject the protocol route reminder into the `ai-pm` orchestrator's context on **every** prompt (the OpenCode analogue of Claude's `UserPromptSubmit` reminder, single-sourced from the always-on reminder text), the two-hook shape from `doc/stack-notes.md` (10e): `chat.message` to learn the active agent + `experimental.chat.messages.transform` to inject — rests on `experimental.chat.*` hooks whose model-reach is **contested** (`#17100` runtime-discard, `#885` not-rendered). The mandatory marker-echo spike (run **before** building) was **ATTEMPTED 2026-06-08 on OpenCode 1.16.2 and BLOCKED**: `opencode run` would not complete a single turn in the test environment — a deterministic silent **HANG** at the startup session-insert step, the **same SQLite issue** that is this feature's own corner-cutting trigger (recorded as a HANG-mode of the (11) limitation; plugin / model / DB / sandbox / network were ruled out). The spike therefore could not reach the model-reach verification step, so chat-hook model-reach is **still unproven, NOT refuted**. **Decision:** do **NOT** ship the chat-hook injection — shipping an unconfirmed hook would be a possibly-dead no-op (the exact "silent dead hook" the plan's scenario 10 forbids). ### Role-delegation integrity: a generic harness built-in is never a stand-in for a protocol role

Every job in the pipeline has exactly one agent that is allowed to do it — the reviewer is the review engine, the auditor is `pm-auditor`, the planner is `/pm-plan`, and so on. The risk this guards against: when the easy path is to ask a generic, catch-all helper (`general`) to "just do the review" or "just run the audit," the output *looks* like the real deliverable but carries none of the role's discipline — no severity scale, no review aspects, no auditor dimensions. So a generic standing in for a protocol role is now denied the same way a look-alike agent from an installed toolset already was.

**What we decided.** A generic harness built-in (`general`, `build`, `plan`) spawned to fill a protocol role is structurally blocked — added to the **same named deny-list, on the same code path**, that already blocks the `wb-*` role duplicators (keyed on the spawned-subagent id). Both sources of substitution — an installed-toolset look-alike and the harness's own generic — are one violation family ("a non-protocol agent in a protocol seat") and are gated identically. The deny-list stays **named**, never an "everything but `pm-*`" block: the built-in **engines the protocol delegates to on purpose** — `code-review` (the designated reviewer) and `deep-research` (the `/pm-research` engine) — are explicitly **not** gated, because they *are* the designated filler for their role, not a stand-in. The deny is the structural floor under a persona rule (fill each seat with its designated agent; never reach for a generic because the change "looks small / is only infrastructure"), since a deny can see the spawn but not the intent behind it.

**Why.** A live incident (2026-06-08): the orchestrator spawned a `general` subagent to "update the review file and run code-review," recognized the substitution was wrong, then rationalized keeping the generic's output as "very little actual code." Blocking the spawn removes the easy wrong path; pairing it with the persona rule names the rationalization the model used. The over-block risk is nil — the roster has no "generic exploration" seat (the orchestrator reads/greps inline and routes deep reads to `pm-codebase-reader`), so a `general` task-spawn is by construction a role substitution with no legitimate use to false-deny.

- rejected an "everything but `pm-*`" block: would false-deny the engines (`code-review`, `deep-research`) the protocol uses on purpose, and is unmaintainable as a blanket.
- rejected a persona-only rule with no deny: the live incident shows the model can recognize the rule and still rationalize past it — the structural floor is what makes it hold without re-reading.
