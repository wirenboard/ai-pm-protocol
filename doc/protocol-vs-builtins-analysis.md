# Протокол vs встроенные инструменты — разбор клубка

> Рабочая аналитика, не канон. Цель: решить, что в протоколе **упразднить**, **переделать** (рамка поверх встроенного движка) и **оставить + обязать** пользоваться именно нашим. Плюс правило маршрутизации скилов (что блокировать, что защищать, что не трогать).

## Тезис

Когда протокол писался, у харнесса не было хорошего ресёрча и ревью — приходилось делать свои. Мир изменился: встроенные `code-review` (вплоть до `ultra` — мультиагентное облачное ревью) и `deep-research` (параллельный поиск + адверсариальная верификация) стали кратно лучше всего, что протокол может захардкодить. Параллельно появились платформозависимые `wb-*` скилы, которые частично пересекаются с нашими ролями, но не заменяют их.

**Правильная будущая форма протокола** — тонкий слой, который владеет только тем, что действительно протоколо-специфично:
- разговор с PM (продуктовый язык, AskUserQuestion);
- артефакты (plan / contract / state / audit / review-trail);
- комплаенс-ворота (`pm-plan-checker`, `pm-auditor`);
- дисциплина (git-flow, remote-boundary, edit-ownership).

Всё, что является **движком** (поиск, техническое ревью, запуск приложения), — делегировать встроенным. Всё, что является **платформенным знанием** (codestyle, packaging, jenkins, serial/mqtt), — делегировать `wb-*` скилам. Протокол уже делает это с `code-review` (Pass 2) — паттерн «adopt the engine, keep the frame» доказан внутри самого протокола.

## Таблица 1. Маршрутизация скилов/агентов/инструментов

Три класса + протокольные. Хук-блоклист строится **только** по классу 1, и только по именованному списку — никогда по паттерну «всё, что не `pm-*`».

| Класс | Что значит | Действие в проекте-с-протоколом |
|---|---|---|
| **1. Role-duplicator** | Занимает кресло, которое у протокола уже есть, и делает ту же работу параллельно/хуже | **Block** (именованный deny-list) |
| **2a. Несущий capability** | Протокол сам его вызывает; без него ломается | **Защитить** allow-list'ом, никогда не блокировать |
| **2b. Capability / knowledge** | Даёт силу/знание, которого у протокола нет; кресло не занимает | **Оставить, поощрять** загрузку |
| **P. Протокольный** | Наш агент/команда | — |

| Скил / агент / инструмент | Класс | Решение | Почему |
|---|---|---|---|
| `wb-development:coder` | 1 | block | дублирует `pm-coder` (во WB-флейворе, не коммитит); `_private`-правило у нас уже есть |
| `wb-development:plan-feature` | 1 | block | дублирует `/pm-plan` + `pm-architect` |
| `wb-development:design-review` | 1 | block | дублирует `pm-architect` (Section B) |
| `wb-development:code-reviewer` | 1 | block | дублирует `pm-plan-checker` + нативный `code-review` |
| `wb-git:pr-author` | 1 | block **после** закрытия PR-review-гэпа | дублирует `pm-pr-prep`; но несёт способность, которой у нас НЕТ (ответ на ревью-замечания) — сначала закрыть, потом блокировать |
| `wb-git:workflow` | 1 | block | git-правила покрыты в WORKFLOW.md |
| `wb-git:pr-review` | — | не трогать | ревью ЧУЖОГО PR — режима нет в протоколе, не дубликатор |
| `code-review` (нативный) | **2a** | защитить | **Pass 2 ревью-петли держится на нём**; блок = протокол падает |
| `deep-research` | 2b → движок | оставить + сделать движком pm-research/stack-researcher | мощнее нашего ручного WebSearch-цикла; адверсариальная верификация фактов |
| `verify` / `run` | 2b → adopt | оставить + добавить опц. шаг «реально запустить фичу» | протокол сейчас НЕ запускает приложение, только просит PM проверить вручную |
| `security-review` | 2b | оставить, предлагать в `/pm-audit full` | дополняет `code-review ultra` |
| `simplify` | 2b | оставить, но не встраивать | пересекается с cleanup-находками `code-review` |
| `Workflow` (tool) | 2b | оставить, рассмотреть для headless-петли | детерминированная оркестрация неинтерактивного fix-loop |
| `Task*` (tools) | 2b | оставить, не заменять `state/current.md` | харнесс-таски эфемерны; наш state — durable, git, PM-readable |
| `schedule` / `loop` | 2b | оставить, ортогонально | можно использовать ДЛЯ протокола (плановые аудиты) |
| `init` (нативный) | 2b | оставить; `pm-bootstrap` его надкласс | `init` генерит только CLAUDE.md; bootstrap делает кратно больше |
| `wiren-board`, `wb-serial`, `wb-mqtt-broker`, `wb-network`, `wb-rules`, `wb-zigbee`, `wb-troubleshooting`, `wb-controller-backup`, `wb-auth`, `wb-jenkins`, `wb-dev`, `wb-development:codestyle`, `wb-development:package-bootstrap` | 2b knowledge | оставить, **обязать pm-агентов консультироваться** | платформенное знание; на WB-проекте `pm-coder`/`pm-stack-researcher` должны их грузить |
| все `pm-*` | P | — | наши |

**Ловушка имён:** `code-review` (2a, защитить) vs `code-reviewer` (1, block) — один символ, противоположные классы. Хук обязан матчить по точному имени, и множество 2a физически не может попасть в deny-путь.

## Таблица 2. Протокол vs встроенные — что делать с каждым нашим куском

Колонки-вердикты: **RETIRE** (удалить, движок встроенного лучше) · **REWORK** (оставить рамку, движок делегировать) · **KEEP+MANDATE** (уникален, обязать вместо дубликатов) · **ADOPT** (новая способность из встроенного, которой у нас нет) · **NEW** (способность, которой нет ни у нас, ни у одного встроенного целиком).

| Наш кусок | Лучший встроенный | Перекрытие | Вердикт | Что сделать |
|---|---|---|---|---|
| технический ревьювер (был) | `code-review` | полное | **RETIRE** ✓ уже сделано | — (это и есть твой эталон: движок встроенного кратно лучше) |
| `/pm-research` | `deep-research` | движок поиска | **REWORK** | выбросить ручной WebSearch-цикл (шаги 2–3), вызвать `deep-research` как движок; оставить рамку: фильтр по architecture.md, PM-facing build/adapt/build-custom вывод, артефакт `.ai-pm/research/` |
| `pm-stack-researcher` | `deep-research` | движок fetch + верификация | **REWORK** (ниже приоритет) | делегировать `deep-research` сбор канонических доков и адверсариальную проверку цитат; рамка (структура stack-notes, таблица валидаторов, integration contracts, agent-facing) — наша |
| `pm-pr-prep` | — (нативного аналога нет) | — | **KEEP** + **NEW** рядом | оставить bump/CHANGELOG/push/PR; **добавить отдельно** ответ на ревью-замечания открытого PR (read треды → `pm-coder` → push → reply/resolve). Платформонезависимо (gh/git) |
| `pm-plan-checker` | `code-review` / `review` | НЕТ (комплементарны) | **KEEP+MANDATE** | проверка соответствия плану/контракту/DoD — этого не делает ни один встроенный |
| `pm-auditor` | `code-review ultra` | НЕТ | **KEEP+MANDATE** | протокольная бухгалтерия (есть ли у каждой фичи plan+review+contract) — уникально; уже корректно предлагает `code-review ultra` в конце |
| `pm-coder` | — | НЕТ | **KEEP** + **REWORK** промта | оставить уникальные ограничения; срезать handholding под Opus, добавить `tools:`-lock, обязать консультироваться wb-knowledge — детали в Таблице 3 |
| `pm-architect` | — | НЕТ | **KEEP+MANDATE** | уникален |
| `/pm-plan` | — | НЕТ | **KEEP+MANDATE** (над `wb-development:plan-feature`) | продуктовая беседа + категориальный/stack/interaction чек |
| `/pm-bootstrap` | `init` | `init` ⊂ bootstrap | **KEEP** | надкласс init; опц. вызвать init как под-шаг для CLAUDE.md-скелета (мелочь) |
| ручная review-петля (WORKFLOW «Step 5») | `Workflow` (tool) | неинтерактивный сегмент | **REWORK (explore)** | PM-невидимый fix-loop (coder→plan-checker→code-review) можно сделать `Workflow`; интерактивные точки (AskUserQuestion) остаются в главной сессии |

| «how to try it yourself» (Step 4/6) | `verify` / `run` | способность запуска | **ADOPT** | протокол сейчас доверяет тестам и просит PM проверить руками; добавить опц. шаг `verify` — реально запустить и убедиться, ловит «тесты зелёные, а фича не работает» |

Сквозная линза для всех семи: отдельный контекст · нужен ли `tools:`-lock · показан ли trim промта под Opus 4.8. Сейчас `tools:` не указан **ни у кого** — все наследуют полный набор.

| Агент | Модель | Отдельный контекст | Tool-lock (предлагаемый) | Trim под Opus |

| `pm-auditor` | наследует | **сильно** (свип проекта) | read-only + `Write`→audit-файл | **нет** (закалён) |

> ⚠ **Reversal — 2026-06-03 (supersedes the «оставить `haiku` у `pm-pr-prep`» decision recorded in this doc at Table 3, conclusion #4, and Шаг 0).**
> Решение «оставить `haiku` у `pm-pr-prep`» **отменено**. Falsifying evidence: на практике пиннутая Haiku-модель давала повторяющиеся **фактические ошибки в PM-facing записях CHANGELOG** — несуществующую команду `/pm-architect`, опечатки, выдуманные номера секций `§`, — которые приходилось править руками перед каждым недавним мержем. Исходный довод «чистая механика (bump/CHANGELOG/git), терять нечего» **ложен**: авторинг CHANGELOG — это PM-facing текст, а не чистая механика. Поэтому `model: haiku` снят, и `pm-pr-prep` теперь **наследует** модель сессии, как и все остальные `pm-*` агенты. **Источник реверса:** `doc/features/route-reminder-coverage-and-prprep-model_plan.md`.

## Сквозные выводы по агентам

1. **Главная системная дыра — tool-lock нет ни у кого.** Границы объявлены прозой («read-only», «пиши только в X»), но `tools:` во фронтматтере отсутствует → все наследуют всё. Для read-only агентов (`pm-plan-checker`, `pm-auditor`) это значит, что они физически могут править код и пушить — при том что вся их ценность в обратном. Прозовый запрет — ровно то, что модель нарушает «по доброте». **Один проход по семи фронтматтерам с правильными `tools:` — самый дешёвый и самый высокоэффективный фикс во всём разборе: механизирует то, что держится на тексте, по всему протоколу разом.**

2. **Trim неравномерен — не блок-трим.** Высокий выигрыш только у `pm-coder` (много «never do X», которые сильная модель и так не сделает). У `pm-auditor` trim **противопоказан** — многословие это процедурные гарантии против срезания углов, недавно закалённые (коммиты «prevent shortcutting», «contract check table mandatory»). У остальных низкий.

3. **Web нужен ровно одному** (`pm-stack-researcher`, и тот → `deep-research`). После tool-lock Web физически только у него — риск «coder/architect полез гуглить вместо stack-notes» закрыт механически, а не текстом.

4. **Model-tier — частично пересмотрено.** Исходно решено: убрать `model:` у всех судья-агентов, оставить `haiku` только у `pm-pr-prep`. ~~`pm-pr-prep` остаётся `haiku`: чистая механика (bump/CHANGELOG/git), терять нечего.~~ **Эта часть отменена 2026-06-03 (см. ⚠ Reversal выше): `pm-pr-prep` тоже наследует модель сессии** — пиннутая Haiku давала фактические ошибки в PM-facing CHANGELOG. Остаётся в силе: снять `model:` у судья-агентов (Opus вместо пиннутого `sonnet`). Итог — `model:` не пиннут **ни у кого**; все наследуют модель оркестратора. Это фронтматтер-правка — идёт тем же проходом, что tool-lock.

5. **«Встроенный с правильным промтом» = то, что уже есть.** Каждый агент — это дженерик-агент + пиннутый промт + (нужный) tool-lock. Магии нет ни в одном. Ценность не в «особом кодере», а в: отдельном контексте (не засорять разговор с PM объёмом реализации) + версионируемых правилах (не зависеть от памяти оркестратора) + tool-lock (механический запрет вместо прозового). Заменять не на что — надо дотюнить.

## План действий (единый, по приоритету)

**Шаг 0 — фронтматтер-проход (tool-lock + model-tier).** Самый дешёвый, ни от чего не зависит, закрывает реальный риск. По Таблице 3: (а) добавить `tools:` всем семи агентам — механизирует read-only и write-ownership по всему протоколу разом; (б) снять `model:` у судья-агентов (наследуют Opus). ~~оставить `haiku` у `pm-pr-prep`.~~ **Отменено 2026-06-03 (см. ⚠ Reversal выше): `model:` снят и у `pm-pr-prep` — он наследует модель сессии, как и все остальные.**

**Шаг 1 — NEW: ответ на ревью-замечания PR** (рядом с `pm-pr-prep`, платформонезависимо gh/git). Разблокирует выпил `wb-git:pr-author`.

**Шаг 2 — REWORK движков на встроенные:**
- `/pm-research` → рамка поверх `deep-research` (высокий приоритет).
- `pm-stack-researcher` → движок `deep-research` (средний).
- headless fix-loop → `Workflow` (explore).

**Шаг 3 — REWORK промтов точечно:** `pm-coder` под Opus 4.8 (срезать handholding, оставить политику). Остальных не трогать, особенно `pm-auditor`.

**Шаг 4 — ADOPT `verify`/`run`:** опц. шаг «реально запустить фичу» перед сдачей PM (сейчас протокол только просит PM проверить руками).

**Шаг 5 — Маршрутизация хуками:**
- `Skill`-deny = именованный класс-1 список; allow-list защищает `code-review` + все `pm-*`.
- `UserPromptSubmit` — мягкое переутверждение маршрута на каждом ходу (не блок способностей).
- Развести в `pm-coder`: запрет WebSearch ≠ запрет грузить wb-knowledge скил.

**RETIRE:** нечего, кроме уже удалённого тех-ревьювера. Дальнейшая работа — REWORK + ADOPT + tool-lock, не удаление.
