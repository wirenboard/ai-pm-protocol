# Project audit — 2026-05-30

## Summary

This template repository ships seven well-shaped agent personas, four commands, seven document templates, and a hooks layer — the protocol is internally coherent and the recent work (PRs #142–#145) closed real structural gaps. The biggest gap is that the project does not eat its own dog food: it has no `docs/architecture.md`, no `docs/stack-notes.md`, and the bulk of its own evolution since v1.0.0 happened without `doc/features/<topic>_plan.md` files — exactly the pattern this template tells downstream projects is a failure. The hooks layer has two regex bugs (under-blocks `ssh host "command"` quoted form and `find / -name x`) that mean guarantee #10 ("no silent prod edits") rests on agent goodwill rather than the gate the WORKFLOW.md text claims.

## Blocking

1. `doc/architecture.md` — missing entirely. **Why it matters:** every downstream project is told `docs/architecture.md` is mandatory and that agents read it before any feature. This template has its own architecture (Markdown personas + hooks + symlink convention + GitHub Actions release flow) and exposes its own constraints (e.g. "agents are plain Markdown", "settings.json is delivered via symlink", "WORKFLOW.md is imported via `@.ai-pm/tooling/WORKFLOW.md`") — none of this is documented in the place the template insists on. The auditor cannot evaluate the project against its own architecture rule. **Fix:** `/plan-feature audit-fixup-self-docs-architecture` — write `doc/architecture.md` describing the agents-as-Markdown design, the symlink/submodule integration contract, the hooks layer, and the release-via-CHANGELOG-tag-detection flow.

2. `doc/stack-notes.md` — missing entirely. **Why it matters:** this template's own #10 guarantee in README and the new dimension 10 in `reviewer.md`/`auditor.md` say a missing stack-notes for an actively-used component is blocking. The template relies on at least: `jq` (used in hook commands), `gh` (used by `pr-prep` and the GitHub Actions release flow), `git` (entire flow), GitHub Actions YAML schema, Claude Code's `hookSpecificOutput`/`PreToolUse` contract, and Markdown frontmatter conventions for agent files. None of these have a documented canonical-rule entry. Without stack-notes, dimension 10 of the audit on this very project is unverifiable — the template fails its own rule. **Fix:** `/plan-feature audit-fixup-self-stack-notes` — spawn `stack-researcher` for jq, gh, git, GitHub Actions, Claude Code hook API, Markdown frontmatter; populate `doc/stack-notes.md` with cited rules and validators.

3. `.claude/settings.json:32` — the ssh-mutating-action regex under-blocks the common quoted form. **Why it matters:** the regex requires a leading space before `systemctl`/`rm`/`kubectl`/`tee` etc. When the orchestrator issues `ssh host "systemctl restart x"` or `ssh host "rm /etc/foo"` (a syntactically standard ssh invocation), the keyword is preceded by `"` not by space, so the regex misses and the prompt is not raised. Verified empirically: `ssh wb6 "systemctl restart wb-mqtt-serial"` and `ssh wb6 "rm /etc/foo"` both pass through silently. Result: the guarantee "no silent prod edits" depends on the model choosing the un-quoted form, not on the gate the WORKFLOW.md text promises. **Fix:** `/plan-feature audit-fixup-hooks-quoted-form` — adjust the regex so the lookbehind also accepts `"` and `'` (not just space) before the protected keyword; add unit tests as comments showing the cases caught.

4. `.claude/settings.json:9-12, 20` — the `find` boundary hook under-blocks `find / -name x`. **Why it matters:** the extractor `find +\K/[^ |;&"\x27]+` requires the path to be terminated by space/quote/pipe etc. — `/` alone followed by a space-and-dash terminates inside the character class on space, so the matched path becomes empty and the hook falls through to `exit 0`. Verified: `find / -name foo` extracts nothing, hook skips, the full-filesystem search is permitted. **Fix:** part of the same `audit-fixup-hooks-quoted-form` plan — handle `find /<space>` and bare-root case explicitly; deny when no in-root match found at all.

5. `doc/features/` — no plan files for the bulk of post-template-v2 work. **Why it matters:** between commits `eba8ab5` (template-v2 rewrite, 2026-05-22) and `6e1bf14` (protocol-integrity, 2026-05-30), approximately 37 commits modified core agent/command/template files (v1.0.0 through v1.5.1) without any matching `doc/features/<topic>_plan.md`, `_arch.md`, or `_review.md` artefact. Furthermore, the three most recent PRs (#143 audit subagent, #144 post-cycle lessons, #145 hooks layer) each shipped substantive protocol changes without a corresponding plan. This is the exact failure pattern that dimension 1 of the new auditor catches on downstream projects, mirrored in the wb-mqtt-matter validation file (finding 10: "Eight commits, including `feat:` additions of Dockerfile/compose/schema — none of which have a corresponding plan/review/arch artifact"). The template fails the rule it just shipped. **Fix:** `/plan-feature audit-fixup-self-retroactive-plans` — retroactively document the largest landed clusters as `audit-retroactive-<area>` plan entries in `doc/features/`, citing commit SHAs. No code changes; paper-trail recovery so future audits can compare against intent.

6. `.claude/agents/architect.md:17` — references a non-existent `design-review` agent. **Why it matters:** the line "Don't force design-review on simple additions" reads as if there were a `design-review` agent or step, but no such agent exists in `.claude/agents/`. This is a stale reference from `doc/features/template-v2_plan.md` (which listed `design-review.md` as an agent that was never actually shipped). A downstream PM reading the architect persona could be confused or send the orchestrator on a wild-goose chase. **Fix:** part of `/plan-feature audit-fixup-self-retroactive-plans` or its own cosmetic pass — rewrite the line to "Don't force architecture review on simple additions" or similar.

7. `.gitignore` — references stale concepts. **Why it matters:** the file references `.bootstrap-state.local.md` (a v0.x artefact removed in template-v2), `AP-16 local-trace mode` (an AP number from the deleted anti-patterns system), and the `.ai-pm/.reviews/release-*.json` exception which only makes sense for *downstream* projects that have this template installed as a submodule at `.ai-pm/tooling/`, not for the template repo itself. It also cites "§ 9 finding in audit 2026-05-27" — but `.reviews/doc-audit-2026-05-27.md` is itself ignored, so a reader cannot find the cited finding without reaching outside git. This is documentation drift baked into a config file. **Fix:** `/plan-feature audit-fixup-self-gitignore-cleanup` — drop the `.bootstrap-state.local.md` line, drop the `AP-16` comment, drop the `.ai-pm/.reviews/release-*.json` exception (irrelevant in template repo), trim the explanatory comment.

## Notes

1. `.reviews/doc-audit-2026-05-27.md` — references many files (AP-27 through AP-33, `discipline-advisor.md`, `bootstrap-state.md.tmpl`, `check-spec-discipline.sh.tmpl`) that no longer exist in this repo after the v1.0.0 rewrite. **Why it matters:** the audit document is a snapshot of a pre-rewrite state and is now obsolete. It is gitignored locally (so does not pollute the published repo) but it does sit on disk inside the project root and could mislead anyone running a fresh `/audit`. Drop the file or mark it explicitly archived.

2. `CHANGELOG.md` — `release-helper` referenced in entries v1.0.0 through v1.0.7 even though the agent file was deleted in `15e89d6` (PR #139, "Enforce project root boundary in all agents"). **Why it matters:** acceptable as historical record — CHANGELOG entries describing past versions correctly mention the agent that existed at that time. No action needed unless PM wants a one-line note at the top of v1.5.x saying "release-helper merged into pr-prep". Pure cosmetic.

3. `WORKFLOW.md` vs README.md — both cover the same orchestration material in two places, mostly consistent but with subtle divergences. WORKFLOW.md is in English; README.md is in Russian. Examples: README says "вместе с PM фиксирует план" while WORKFLOW says "we plan together"; README guarantees list is 11 items, the protocol-integrity plan introduced guarantees #10 #11 #12 but README shows #10 #11 (the third — "Scope без тихой деформации" — was added without renumbering the rest). **Why it matters:** mild drift, not contradictory. PM can read either and reach the same conclusions, but the two will need synchronized updates on every protocol change. If they get further out of sync, the template's own guarantee #7 ("Документация не расходится с кодом") becomes meta-violated. Consider a one-page divisor: README is the marketing/quickstart face, WORKFLOW is the canonical orchestration spec — link from one to the other rather than duplicating.

4. `doc/_templates/CLAUDE.md.tmpl:60-66` — the `Validators` section in Pipeline is left as `<...>` placeholders with no validator entries pre-populated, despite the template promising `stack-researcher` will populate it. **Why it matters:** correct at bootstrap, but it means a downstream project that bootstraps and never touches an external system will carry visibly-empty placeholders in CLAUDE.md indefinitely. Worth an explicit `# (none yet — extended on first feature that needs a validator)` comment so it doesn't look broken.

5. `.claude/agents/auditor.md` description / persona shape — frontmatter `description:` says the auditor writes to `docs/audit-<YYYY-MM-DD>.md`, but for *this* (meta) audit the user explicitly directed the output to `doc/features/audit-2026-05-30.md` because the template repo itself uses `doc/` not `docs/`. **Why it matters:** the auditor's own persona does not document how to handle the meta-case (the template auditing itself). Not a defect for downstream use, but for self-audits the operator must override the path manually each time. Worth a one-line note in `auditor.md`: "If the project under audit uses `doc/` (not `docs/`) — write to `doc/audit-<YYYY-MM-DD>.md`. Detect by listing both."

6. `.github/workflows/auto-tag.yml` — the only CI file; covers releases only. No CI lint pass, no link checker, no schema-validate-of-settings.json. **Why it matters:** the template's own infrastructure is minimal, which is consistent with "no automated tests by design — validation by use". But the hooks regex bugs found above (findings 3, 4) would have been caught by even a simple shell-based unit test on the regexes. A small `tests/hooks.sh` and a single `lint` workflow that runs it would convert "hopefully correct" into "verified correct". Not blocking — this is a deliberate design choice in `template-v2_plan.md` — worth a backlog entry only if regex-regression becomes a real risk.

## What looks healthy

- The persona files are individually well-shaped: each has a clear "When you are invoked", "What to do", "Output", and "Hard rules" section. The structure repeats across `architect`, `coder`, `reviewer`, `pr-prep`, `docs-extractor`, `stack-researcher`, `auditor` — internal consistency is high.
- The protocol-integrity plan and its validation file (`doc/features/protocol-integrity-and-stack-literacy_*.md`) are exemplary: the plan explains the 3 structural gaps, lists exactly which files need to change, and the validation file traces the same audit against a real downstream project (wb-mqtt-matter) and shows every known failure surfacing as a finding. That is the kind of paper-trail this template wants downstream projects to produce.
- The git history shows discipline on PR-per-change (every commit since v1.0.0 has a PR number; no direct-to-main pushes visible in the listing). The gap is in `doc/features/`, not in git hygiene.
- The hooks layer concept is sound — `PreToolUse` deny/ask decisions backed by the bash-pipeline + jq pattern is the right shape for cross-session enforcement. The bugs found are in the regex tuning, not in the architecture.
- The README and WORKFLOW.md, despite the duplication note above, each cover their material with PM-appropriate framing — no jargon thrown unexplained, no agent names leaked into the user-facing surface.
- The auto-tag release workflow is clean and minimal: read CHANGELOG, check if release exists, create tag + release. No magic, easy to audit.
