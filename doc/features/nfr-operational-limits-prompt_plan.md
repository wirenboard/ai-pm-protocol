# nfr-operational-limits-prompt — plan

*Third slice of the EPIC whole-system-property auditor (`.ai-pm/backlog.md` → "From downstream artifact review — 2026-06-04 (wb-mqtt-matter docs)", the **NFR / operational-limits** sub-class — the expert called it "the most generalizable gap of the three and the strongest case for a protocol addition … a true missing-prompt, not a discipline lapse"). PM picked this slice 2026-06-04 to run autonomously. Slice 1 (`invariants-index`, v2.21.0) made `## Behavioral contract (taxonomies & invariants)` the single invariants home; slice 2 (`taxonomy-drift-sweep`, v2.22.0) gave `pm-auditor` the journeys-surface drift sweep. This slice closes the **NFR** gap.*

Close the gap the downstream review found: a resource-constrained / scale-bearing project writes its resource and scale budget **nowhere** — not because the discipline lapsed, but because **the protocol has no prompt** that forces "what is this feature's / the system's resource and scale budget?". This slice adds a **conditional NFR / operational-limits prompt to `/pm-plan`** (fires only on a resource-constrained platform or a scale-bearing feature) and gives NFRs a **home split by audience**: user-facing limits → the Product Contract; resource budgets → a new conditional `docs/architecture.md` section.

Meta-feature on the template repo: **software-kind**, the no-user-facing-contract exception; dev-docs in `doc/` (singular). Every scenario subject is `/pm-plan` / the planner / `pm-architect` / the architecture doc / the contract template (non-human) → not user-facing → no Product Contract, no product-readiness advocate gate, no `## Validation` gate (Pass-2 is `code-review`). Verification = editorial + clean-grep.

## Scenarios

1. **A scale-bearing feature → the NFR prompt fires (the missing prompt).** When the orchestrator judges, while planning, that the feature is **scale-bearing** — its core scales with a count the system must bound (devices / endpoints / connections / sessions / subscriptions / messages-per-second / queue depth) — `/pm-plan` runs a **conditional NFR / operational-limits check**: it surfaces *"what is this feature's / the system's resource and scale budget?"* and records the answer in the audience-split homes (Scenarios 4–5). The check sits beside the existing conditional pre-draft checks (Stack-component, Interaction-scenario, Security-surface) — same judgment-triggered, proportional shape.

2. **A resource-constrained platform → the NFR prompt fires (system-level, even for a non-scale feature).** Independently of the feature: when the **platform is resource-constrained** — the signal read from `docs/stack-notes.md` (an embedded / limited-RAM / Docker-on-controller platform) or `docs/architecture.md` `## Architectural constraints` (an "embedded / offline / must-fit-X" boundary) — the same prompt fires for the **system's** budget (RAM, boot time, CPU headroom), so a controller-class project gets its resource budget written even when an individual feature is not itself scale-bearing.

3. **Neither holds → silent (proportional).** A feature that is **not** scale-bearing on a platform that is **not** resource-constrained gets **no** NFR prompt — the check is silent, exactly like the Security-surface check on a non-security project. The prompt is **conditional / proportional, never blanket-mandatory on every tiny project** (the backlog's explicit form-remark). No new mandatory plan section, no hook (the trigger is a semantic judgement a regex cannot make — the same no-hook reality as the product-readiness gate).

4. **User-facing limits → the Product Contract (`## Must not break`).** The audience-split, half one: a **user-facing limit** the PM cares about — max devices / endpoints the product supports, throughput the user perceives — is a **promise to the user**, so it is routed to the feature's Product Contract `## Must not break` as a quantified invariant (its natural home; the contract template gains a one-line note that quantified user-facing limits belong there). This half is a **product question** for the PM (what the product promises to support), surfaced per the existing product-vs-technical rule.

5. **Resource budgets → a conditional `docs/architecture.md` `## Operational limits & budgets` section.** The audience-split, half two: a **resource budget** — RAM ceiling, boot-time budget, CPU headroom, system-level max-N — is an **engineering bound**, so it is routed to a **new conditional `## Operational limits & budgets` section** in `docs/architecture.md` (owned by `pm-architect`, authored/refreshed on the `/pm-plan` "Docs to update" post-coding handoff — the same trigger as a decision record). The section is **`N/A — <reason>`** for projects with no quantified budget (proportional, like every other conditional architecture section). A budget the orchestrator cannot quantify without real platform knowledge is recorded as `[?]` for the PM to confirm — **never invented** (the same "never invent to fill" discipline `pm-architect` already applies to the Behavioral contract).

6. **Structural / proportional, additive, no migration.** The prompt matches a **judgement** (scale-bearing? resource-constrained?), never a regex; it records into **existing-owner homes** (the contract + a new conditional architecture section), not a new parallel mandatory document — the backlog's form-remark verbatim. Existing projects are unaffected until their next `/pm-plan`, where a scale-bearing feature / constrained platform earns the prompt; the architecture template gains one **conditional** section (born `N/A`), the contract template gains one note line. No existing artifact gains a required field; no template structural migration; no `MIGRATIONS.md` change.

## Out of scope

- **The other whole-system-property sub-classes** from the same downstream review — later EPIC slices, each its own plan:
  - **State / event model lifted into a conditional artefact** — a conditional `docs/architecture.md` state-model section + a `/pm-plan` trigger when the feature's core is a state machine / protocol (the third gap of the review; deferred).
- **The other cross-document-consistency sub-classes** (temporal-status conflation, ADR ↔ stack-notes backing, state-machine ↔ journeys single diagram, journeys ↔ threat-model UX) — separate later slices, untouched here.
- **Other NFR classes** (the categorical-scope siblings of "resource & scale budget"): **reliability / availability** (uptime, failover, recovery SLOs), **latency SLOs** (p50/p99 response bounds as distinct from throughput), and **security-as-NFR** (already owned by `docs/threat-model.md` + `## Security constraints` — not re-homed here). Each is a separate prompt/home decision; this slice is scoped to **resource footprint + scale/capacity budgets** (RAM, boot time, CPU, max devices/endpoints, throughput) exactly as the backlog sub-class names it.
- **A hard gate / auditor backstop for the NFR prompt** — deliberately **not** added this slice. Making NFRs mandatory (a `pm-plan-checker` DoD block or a `pm-auditor` presence-note) would violate the backlog's explicit "conditional / proportional, never blanket-mandatory" form-remark, because the fire-condition is a semantic judgement no gate can evaluate without re-deriving it. The accepted trade-off (recorded in the decision record): the prompt **degrades silently** if the orchestrator misjudges the trigger — acceptable here because the homes make a *recorded* budget durable and auditable, and a future slice may add a *narrow* "if NFRs were declared, are they in the right home?" check (a did-it-right, not a did-it-at-all gate). Auto-fixing or auto-quantifying a budget is likewise out — the orchestrator records or marks `[?]`, never invents a number.
- **Re-homing existing NFRs** in downstream projects — no migration; existing projects gain the prompt at their next `/pm-plan` and the `N/A` section at their next architecture refresh, never a retroactive sweep.
