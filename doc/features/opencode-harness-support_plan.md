# OpenCode harness support — plan

Source: PM request 2026-06-07 ("использовать протокол с OpenCode … можно ли универсально, чтобы не держать два репозитория"). Prior research: this conversation's harness-portability analysis + the `doc/stack-notes.md` § "OpenCode (sst/opencode → anomalyco/opencode)" entry (added 2026-06-07 by `pm-stack-researcher`). **Pass 4 (PM, 2026-06-07): the sync-mechanism FORM is now DECIDED — variant C (build-time single-source, two-repo source/dist split) — superseding the pass-3 "FORM REOPENED, A/B/C undecided" status and the orchestrator's earlier lean toward A.** Variants A (runtime symlink+pointer) and B (build-time, one repo) are rejected in favor of C. With C, the protocol deliberately **changes** its recorded "Agents are plain Markdown files, **no build step**" architecture decision (`doc/architecture.md` § "Agents are plain Markdown files, not compiled artefacts"). Topology, staging, and the newly-surfaced repo-split problems are folded in below (see Key design decisions, pass 4).

> **Scope of THIS plan = design + groundwork slice, staged** (PM decision 2026-06-07). It locks the dual-harness architecture under form C, carves the harness-neutral source, builds the generator + both adapters **on a dedicated feature branch inside the current repo** (where the context lives), keeps the generated Claude Code adapter byte-equivalent to today's, and scaffolds the OpenCode adapter as a clearly-labeled **preview**. The actual **repo split** (extracting the source/generator into a new repo the PM will name) is a **defined-but-later staged step**, gated on the PM providing the target. The slice does **not** certify OpenCode as fully-supported (later slice, gated on two upstream blockers — see Out of scope), and does **not** itself perform the extraction.

## Scenarios

1. **The protocol declares a symmetric dual-harness architecture realized by build-time single-source (two-repo source/dist split).** A reader of `doc/architecture.md` sees a recorded decision: the protocol is developed as **source** (a harness-neutral metalanguage core + a generator) in one repository, and a CI build assembles the two full **per-harness adapters** (Claude Code `.claude/` + OpenCode `.opencode/`) and publishes them to a **distribution repository** that downstream projects consume via submodule. The two adapters are **symmetric** — neither harness is primary, neither adapter reaches into the other's directory; each harness reads **only its own** adapter, and `.opencode/` is fully self-contained (it never depends on `.claude/` existing). OpenCode ships as a preview until two named upstream gaps close.

2. **The harness-neutral source is the single source of truth; both adapters are generated, never hand-duplicated.** The protocol's harness-agnostic surface — `WORKFLOW.md`, `workflow/*.md`, the **bodies** of every `pm-*` agent and command, `doc/_templates/*` — lives **once** in the neutral source. The generator emits, for each harness, a **full self-contained adapter file** (harness-specific frontmatter + the neutral body inlined) — **no symlinks, no runtime read-and-follow pointers, no second hand-maintained copy.** A harness-neutral vocabulary layer lets the neutral source name capabilities ("the structured-question tool", "the change-intent route reminder", "the deny-list enforcement") without binding to one harness's primitive names; the generator resolves each neutral capability name to the target harness's primitive. The single-source guarantee is enforced by a **diff-clean integrity check**: regenerating the adapters from the source must reproduce exactly what is published — any divergence (a hand-edit to a generated file, a stale build) fails the check. (Body **duplication by hand** is rejected — see Key design decisions; under form C there is genuinely only one authored copy, the source, and everything else is a build product.)

3. **The Claude Code adapter's behaviour is byte-equivalent to today's.** A maintainer running the existing dogfood pipeline on Claude Code sees the **generated** `.claude/` adapter behave byte-identically to today's committed one: same agents, same commands, same `settings.json` hooks, `tests/hooks.sh` green at its current count, `@`-import of `WORKFLOW.md` intact. The build step is new; the **output the Claude session loads is unchanged**. Claude Code remains the self-host harness (PM decision 2026-06-07).

4. **A downstream project on OpenCode can install the protocol as a preview, build-free, with supported harnesses declared explicitly.** The distribution repository carries the committed, pre-built `.opencode/` adapter — agents, commands, an enforcement plugin, an `opencode.json` with an `instructions` array, and an `AGENTS.md` entry. Downstream gets it by the **same submodule mechanism as today** (see scenario 5) — no downstream build. Which harnesses the protocol supports is stated **declaratively** in the `README` and `AGENTS.md` headers (and folded into the preview label) — a reader learns the supported set from an explicit statement, **not** by inferring it from which adapter folder is present. A prominent **preview label** states OpenCode is not yet enforcement-certified and names the two upstream gaps it waits on.

5. **Submodule delivery to downstream is unchanged; install stays one flow, harness auto-detected.** A downstream project adds the **distribution repository** as the `.ai-pm/tooling` submodule and runs `git submodule update --remote` exactly as today — **same URL** (the current repo stays the distribution repo), same build-free plain files, same mental model. The source/generator split is invisible to the consumer: it lives in a separate repo the downstream never sees. Install then **detects** which harness the project uses (presence of `.claude` vs `.opencode`, or a flag) and wires the matching adapter; the OpenCode path wires its own adapter the analogous way (submodule + `instructions` config entry + the `.opencode/` files made discoverable — exact link/copy step spike-gated, see scenario 8).

6. **Each of the four OpenCode gaps has a recorded adapter strategy.** The design names, for each gap, exactly how the OpenCode adapter behaves and how it differs from Claude Code: the missing per-prompt route-reminder hook, the missing built-in review/research engines, the missing runtime model override (cross-model — handled via static model pins in OpenCode agent frontmatter), and the unsettled subagent enforcement containment.

7. **The repo split is a defined, later, gated step — not done in this slice.** The plan sequences the work in two stages: **(a) groundwork on a feature branch in the current repo** — build the neutral source, the generator, and both generated adapters, with the diff-clean integrity check, all committed to the branch (merging into the branch, never into `main`); **(b) the extraction** — once the PM provides the target, the source/generator is "pinched off" into the new source repository, the current repo is reduced to the distribution surface, and CI is wired source→dist. Stage (b) is **out of this slice's coding** (gated on the PM's repo target) but its shape is designed now by the arch note so stage (a) does not paint stage (b) into a corner.

8. **The two unsettled dependencies are de-risked by gated spikes.** **Spike A** verifies submodule-path sourcing for the OpenCode install design (whether OpenCode's agent/command/plugin loaders can source from a submodule subdir by config alone, or whether a symlink/copy step into `.opencode/` is required) — **note this may shift under the dist-repo topology** (the dist repo ships pre-built files, which can change what "sourcing" means). **Spike B** verifies whether plugin `tool.execute.before` actually constrains a `task`-spawned subagent (the disputed `#5894`). Their outcomes flip the affected design points from `doc-cited (unverified)` to `execution-verified` or trigger the documented fallback. **The `@`-inline question is not a spike** — it was resolved against canonical docs (see Key design decisions); under form C it is moot anyway, because the generator inlines the neutral body at build time rather than relying on any runtime `@`-import.

## Existing behaviors this feature touches

(What must not break — the dogfood self-host runs on this repo under Claude Code. Note: one recorded decision is **deliberately changed**, not preserved — flagged below.)

- **The Claude Code adapter (`.claude/agents`, `.claude/commands`, `.claude/settings.json`).** The **generated** adapter must be byte-equivalent in behaviour to today's: no agent, command, or hook changes shape or behaviour. `tests/hooks.sh` stays green at its current count.
- **The `@`-import chain** `CLAUDE.md` → `@WORKFLOW.md` → on-demand `workflow/*.md`. The neutralization in scenario 2 must not change what a Claude Code session loads (the generated `WORKFLOW.md` and `@`-line are byte-equivalent).
- **Submodule delivery to downstream — unchanged and must stay so.** Downstream adds the distribution repo (the current repo) as the `.ai-pm/tooling` submodule, **same URL**, `git submodule update --remote`, build-free plain files, the `@`-import + symlink install for the Claude path. The source/dist split is internal to development and **does not reach the consumer**; downstream install/mechanics are not altered by this feature (the OpenCode *path* is added alongside, never replacing the Claude path; the symmetry decision forbids the new path from depending on the old one's files).
- **DELIBERATELY CHANGED — "Agents are plain Markdown files, not compiled artefacts / no build step" (`doc/architecture.md`).** Form C introduces a build step on the **development** side. The downstream-facing guarantee the original decision protected ("downstream gets plain files, no build") is **preserved** (the dist repo ships pre-built plain files; downstream still does not build). What changes is that the *authored* form is now source-that-is-built, not the runnable files directly. This is recorded as an **amendment** to that decision, not a silent break (see Docs to update).
- **The tooling-submodule / dev-artifact inertness invariant** (`template-dev-artifacts-inert`, #251): adapter files for one harness must be inert under the other harness's session, the same way the tooling submodule is inert downstream.

## Stack expectations touched

(From `doc/stack-notes.md` § "OpenCode" + § "git" — the load-bearing rules the adapter + topology design must respect. Confidence is per-rule; the two genuinely-unsettled rules stay spike-gated.)

- **Instruction entry — config-listed file array, not in-file import** (`doc-confirmed` 2026-06-07). OpenCode's `opencode.json` `instructions` array accepts relative/glob paths, so a submodule-relative `.ai-pm/tooling/WORKFLOW.md` entry is expressible; but there is **no `@`-import inside an instruction file**. The generated OpenCode adapter re-expresses the Claude `@`-import as a config-array entry + the existing harness-agnostic Read-on-demand pattern for topic files. Source: <https://opencode.ai/docs/rules/>, <https://opencode.ai/docs/config/>
- **Subagent frontmatter shape differs; no `@`-inline in an agent body** (`doc-confirmed` 2026-06-07). `.opencode/agent/*.md` uses `mode`/`model`/`permission` and a `tools` **object map**, not Claude's comma-list; invocation is by name/description via the `task` tool (no `subagent_type` string). Under form C the neutral body is **inlined by the generator** into each adapter file, so the absence of an agent-body `@`-inline is moot. Source: <https://opencode.ai/docs/agents/>
- **Command body supports `@file` inline; agent body does not** (`doc-confirmed` 2026-06-07). Recorded so the generator does not over-apply the command-only capability to agent bodies. Source: <https://opencode.ai/docs/commands/>
- **Enforcement is JS-throw-to-block in a TS plugin** (`doc-cited (unverified)`), not a JSON `permissionDecision` contract: `tool.execute.before` throws to deny / mutates `output.args` to rewrite. Source: <https://opencode.ai/docs/plugins/>
- **No UserPromptSubmit-equivalent hook** (`doc-cited (unverified)`) — no hook injects context per prompt; message hooks are reactive/post-fact. Source: <https://opencode.ai/docs/plugins/>
- **`question` tool = AskUserQuestion equivalent (PRESENT)** and **`skill` tool = Skill equivalent (PRESENT)** (`doc-cited (unverified)`). The "surface forks via the structured-question tool" and skill-invocation rules port. The generated OpenCode adapter ships its own skill files under its own path (no `.claude/skills/` cross-read — symmetry decision). Source: <https://opencode.ai/docs/tools/>, <https://opencode.ai/docs/skills/>
- **No runtime per-`task` model override** (`doc-cited (unverified)`; PR `#17577` closed-not-merged); subagent model is pinned in agent frontmatter or inherited. The Cross-model-review mechanism must pin model in **agent frontmatter** on OpenCode. Source: <https://opencode.ai/docs/agents/>, <https://github.com/anomalyco/opencode/pull/17577>
- **No built-in `code-review` / `deep-research`** (`doc-cited (unverified)`) — must be re-bound to a protocol-shipped OpenCode agent/skill. Source: <https://opencode.ai/docs/agents/>
- **Subagent tool-hook containment unsettled** (`doc-cited (unverified)`; `#5894` closed-disputed, fix PR `#7473` closed-not-merged). **Spike B.** Source: <https://github.com/anomalyco/opencode/issues/5894>
- **Install — submodule-path sourcing for loaders unconfirmed** (`doc-cited (unverified)`). Config can point `instructions` at a submodule path, but the docs do **not** confirm the agent/command/plugin loaders can source from an arbitrary submodule subdir by config alone — a symlink/copy step into `.opencode/` may be required. **May shift under the dist-repo topology** (the dist repo ships built files). **Spike A.** Source: <https://opencode.ai/docs/config/>
- **git — submodule data model + two-repo coordination** (`doc-confirmed`; the topology's load-bearing rule). Downstream `git submodule update --remote` runs **no build hooks**, so the dist repo must ship pre-built files (the build happens in the source repo's CI before publish). The source→dist publish needs a cross-repo write credential and matched version tags (a half-failed publish leaves source and dist at different versions — split-brain). Source: `doc/stack-notes.md` § "git", <https://opencode.ai/docs/config/>

## Interaction scenarios

(Not isolated — two adapters share one generated tree + one submodule-delivery channel + one CI/validation surface; and after the split, two repos share a publish pipeline.)

- **When a Claude Code dogfood session runs in this repo while the generated `.opencode/` adapter is present:** the OpenCode files are inert — they do not register as Claude agents, do not alter `@`-import loading, and `tests/hooks.sh` is unaffected (mirrors the `template-dev-artifacts-inert` invariant).
- **When an OpenCode session runs against the same tree:** the `.claude/settings.json` hooks and `@`-import are inert for it; OpenCode reads its own `AGENTS.md` + `instructions` + `.opencode/` adapter, and **only** its own adapter (the symmetry/no-cross-read invariant).
- **When a downstream project runs `git submodule update --remote`:** both adapters arrive together from the dist repo in one update, build-free; neither harness's install is left stale by a bump.
- **When the neutral source is edited (a `pm-*` agent/command body, `WORKFLOW.md`, a `workflow/*.md` rule):** the edit lands once in the source; a regenerate reproduces both adapters; the diff-clean integrity check fails only if a generated file was hand-edited or the build was skipped. The build-time guarantee that "one source of truth" stays literally true.
- **When the source repo's CI publishes to the dist repo and the publish half-fails** (source pushed, dist push failed, or vice-versa): the version/release-coordination guard must catch the source↔dist version mismatch (split-brain) rather than leaving downstream pulling an inconsistent set. (Design point for the arch note + a CI check; full live behaviour is post-extraction.)
- **When the install helper runs in a project that already has a `.opencode/` or `.claude/` dir / where a symlink target exists:** install detects the harness and does not clobber an existing adapter (failure-path, see Test plan).

## Test plan

- **Existing tests that must pass:** all of `tests/hooks.sh` (current count, unchanged — the generated Claude adapter is byte-equivalent).
- **New tests:**
  - `generated-claude-adapter-byte-equivalent`: given the generator and the neutral source, when the `.claude/` adapter is regenerated, then it is byte-identical to the committed `.claude/` adapter (the build introduces no behavioural change to the self-host harness). This is the guard that the new build step does not silently alter what Claude Code loads.
  - `single-source-diff-clean`: given the neutral source + the generator, when the adapters are regenerated and compared to what is published, then the tree is **diff-clean** — no generated adapter file diverges from a fresh build. This is the build-time integrity guard: it fails loudly on a hand-edit to a generated file or a stale build (the build-time analogue of the rejected runtime `single-source-integrity`).
  - `opencode-adapter-inert-under-claude`: given a simulated Claude Code session over this tree, when `tests/hooks.sh`'s boundary/route checks run, then the `.opencode/` files neither register as Claude agents nor change any hook outcome (mirrors the dev-artifact-inert invariant).
  - `opencode-adapter-self-contained`: given the generated `.opencode/` adapter, when its agent/command/skill files are scanned, then **none** references a `.claude/` path — the symmetry/no-cross-read invariant is mechanically enforced.
  - `preview-label-and-supported-harnesses-present`: given the OpenCode adapter, when its entry surface (`AGENTS.md` / `README` / adapter README) is read, then the preview label, the two named upstream gaps, **and the explicit supported-harness declaration** are present — so "preview" can never silently read as "certified" and the supported set is never left to folder-inference.
- **Interaction scenario tests (one per Interaction scenario above):**
  - `hand-edit-trips-diff-clean`: hand-edit one generated adapter file; assert `single-source-diff-clean` fails (the integrity guard is live, not vacuous — it catches exactly the drift form C forbids).
  - `submodule-bump-delivers-both`: simulate a `submodule update` from the dist surface; assert both adapter trees are present and self-consistent after one bump, build-free.
  - `source-dist-version-match`: given a source build and its published dist artifact, assert their version tags match (a positive check against the split-brain failure mode; the static-form check now, the live publish behaviour post-extraction).
  - `install-no-clobber`: run the install helper where an adapter dir / symlink target already exists; assert it detects and refuses to overwrite, with a clear message.
- **Stack-spec tests (one per stack expectation above, verifying against the cited OpenCode contract, not the adapter's own mapping):**
  - `oc-agent-frontmatter-shape`: each generated `.opencode/agent/*.md` carries a `description` + a valid `mode` + a `permission`/`tools`-object (not a Claude comma-list). Comment cites <https://opencode.ai/docs/agents/>.
  - `oc-instructions-points-at-submodule-core`: the generated `opencode.json` `instructions` includes the submodule-relative always-on core path(s). Comment cites <https://opencode.ai/docs/config/>.
  - `oc-cross-model-pinned-in-frontmatter`: any generated OpenCode review/audit agent that must run cross-model carries an explicit frontmatter `model:` (no reliance on a runtime override). Comment cites <https://github.com/anomalyco/opencode/pull/17577>.
  - `oc-enforcement-plugin-throws`: the generated OpenCode enforcement plugin's `tool.execute.before` throws (or mutates args) for each deny-list case that `tests/hooks.sh` covers on the Claude side. Comment cites <https://opencode.ai/docs/plugins/>.

  > Note: these are **form** checks against the documented contract. Behavioural truth for the two unsettled rules (submodule-path sourcing; subagent hook containment) comes from the spikes, not a static test — the static tests freeze the documented shape, the spikes verify it runs.

## Out of scope

Its *shape* is designed now (arch note) so stage (a) does not foreclose it.
- **Full OpenCode certification / "fully-supported" status** — this slice ships a labeled preview only. Certification is a later plan, gated on the two upstream blockers below.
- **Closing the two upstream blockers ourselves** — runtime per-`task` model override (PR `#17577`, closed-not-merged) and subagent hook containment (`#5894`, disputed). We design around them and wait/track; we do not fork OpenCode to fix them.
- **Dogfooding the protocol on OpenCode** — PM decision 2026-06-07: Claude Code stays the self-host harness; OpenCode parity is validated by a smoke/preview path, not by self-hosting.
- **Form A (runtime symlink + read-and-follow pointer) and Form B (build-time, one repo)** — both **rejected** by PM decision 2026-06-07 (pass 4) in favor of form C. A's pointer-softness + symlink fragility and B's mixing of generated files into the dev tree are the recorded reasons C was chosen over them (see Key design decisions).
- **Hand-duplicated shared bodies** — never permitted. Under form C there is exactly one authored copy (the source); generated adapters are build products guarded by `single-source-diff-clean`. There is no duplication fallback.
- **Aligning the command/skill primitive across both harnesses** — keeping the protocol's `/pm-*` as Claude *commands* is unchanged here; convergence on a single primitive is a separate future step.
- **Live execution of the two spikes if no OpenCode runtime is available in-session** — they are scheduled as gated tasks; if no runtime, the first task records `spike-deferred` and the affected design points stay `doc-cited (unverified)` with the documented fallback.
- **Sibling harnesses of the `harness` category** — Cursor, Codex CLI, Aider, Continue, and any other AI-coding harness. This plan focuses on **OpenCode** (PM-named); each other harness is a separate plan with its own adapter. The architecture is designed to **generalize** to N adapters, but only OpenCode is built here.

## Key design decisions

- **Four PM forks resolved 2026-06-07, pass 1 (AskUserQuestion):** (1) scope = design + groundwork (OpenCode adapter as preview); (2) dogfood = Claude Code stays self-host, OpenCode = downstream-supported; (3) sync = single-source (no build step **at the time** — superseded by pass 4); (4) install = auto-detect harness.
- **Symmetry / no cross-read (PM decision 2026-06-07, pass 2).** The two adapters are symmetric — neither is primary, neither reads the other's directory. `.opencode/` is fully self-contained. Enforced by `opencode-adapter-self-contained`.
- **Single source of truth, no hand-duplicated bodies (PM decision 2026-06-07, pass 2; mechanism settled pass 4).** Every shared body lives in **exactly one** authored place — the neutral source. Pass 2 rejected hand-duplication outright (a self-maintained protocol cannot safely keep two copies in sync; "else simpler to make two repos"). Pass 4 settles the *form*: single-source is realized by **build-time generation** (form C), not the pass-2 runtime symlink/pointer.
- **Sync-mechanism FORM = C, DECIDED (PM decision 2026-06-07, pass 4) — supersedes the pass-3 "REOPENED/undecided".** Of the three single-source candidates, the PM chose **C — build-time, two repos: split source / distribution.** The protocol is developed as plain "source code" (a neutral metalanguage + a generator); CI builds the full adapters and publishes them to a **separate distribution repo** that downstream adds as the submodule.
  - **A (runtime symlink + read-and-follow pointer) — rejected.** Its pointer is "soft" (a system-prompt instruction the agent must obey, not a guaranteed load) and symlinks are fragile on Windows checkouts.
  - **B (build-time, one repo) — rejected.** It mixes generated files into the dev tree (diff-noise, "is this generated?" confusion) and still imposes regenerate-before-test on the dev loop, without C's clean source/dist separation.
  - **C — chosen.** The dev tree is clean source only; the downstream submodule is lean built adapters; downstream stays build-free; the generated adapters are hard, self-contained files (no runtime pointer, no symlink). Accepted costs are recorded as work items below.
- **Topology (PM decision 2026-06-07, pass 4).** The **current repo (`ai-pm-protocol`) becomes the DISTRIBUTION repo** — it holds only the compiled/built adapters downstream consumes via submodule; its **submodule URL to downstream is unchanged**, so the consumer sees no change. A **NEW separate repo** (PM will provide the push target later) becomes the **SOURCE repo** — neutral metalanguage + generator. CI in the source repo builds and publishes artifacts to the dist repo.
- **Staging (PM decision 2026-06-07, pass 4).** All migration work is done **temporarily on a dedicated feature branch inside the current repo** (merging into that branch, **not** into `main`), where the context lives; **later** the source/generator is "pinched off" into the new source repo once the PM provides the target. The plan sequences (a) groundwork-on-branch, then (b) extraction-later.
- **Newly-surfaced problems (PM dialog 2026-06-07, pass 4) — first-class, designed by the arch note, not afterthoughts:**
  - **Identity inversion of the current repo.** Today this repo holds the entire **development** surface (plans, backlog, `.ai-pm/`, the dogfood `.claude/` adapter, all dev artifacts). Under C with this topology, that development surface must **move out** to the new source repo, leaving only built adapters here. "Pinching off" is a **center-of-gravity move**, not a small extraction — "what moves to source vs what stays as dist" is an explicit work item for the arch note.
  - **Self-host / dogfood loop change.** The protocol develops itself under its own protocol. After the split, the thing you edit (source) and the thing that runs (compiled `.claude/`) live in **different repos**. To dogfood a source change, the source repo must either **build a runnable `.claude/` locally** (build-then-run loop) or **pull the compiled artifacts back**. The arch note must **name the concrete mechanism** — not defer it. (Note: while work is still on the branch *before* the split, dogfood works the old way — the generated `.claude/` sits right here; the loop change bites only post-extraction, which is exactly why stage (b) is designed now.)
  - **Operational cost of C (accepted by PM, recorded as work items).** (i) **Cross-repo CI credentials** — the source repo's CI needs write access to the dist repo. (ii) **Version/release coordination** — matching tags so a downstream `submodule update` gets a consistent set; **split-brain** risk if a publish half-fails. (iii) A **CI source↔dist diff-clean / in-sync check** — the build-time analogue of single-source integrity.
