# Architecture

This document describes the architecture of the **ai-pm-protocol template repository itself** ‚Äî not a downstream project that consumes the template. It is a meta-case: the template documents itself in the same shape (`doc/_templates/architecture.md.tmpl`) it expects downstream projects to use. Sections that do not apply to a documentation-and-config repo are kept as headers and marked `N/A ‚Äî <one-line reason>` so a downstream reader can see the template walked its own structure honestly.

**Last full review:** 2026-06-04

---

## Project

A template for product managers building their product solo or with AI assistance. The PM describes what they want; a fixed set of Markdown-defined agents (`pm-architect`, `pm-coder`, `pm-plan-checker`, `pm-pr-prep`, `pm-codebase-reader`, `pm-stack-researcher`, `pm-auditor`, `pm-product-advocate`) plan, implement, and verify changes against the PM's plan, the downstream project's `docs/architecture.md` constraints, and `docs/stack-notes.md` stack literacy. The PM makes product decisions and does not read code. Marketing-level claims and the natural-language workflow walkthrough live in `README.md` ‚Äî this document does not duplicate them; it describes the underlying structure.

## Tech stack

The template's stack is the small set of CLI tools, file formats, and APIs that the agents, commands, hooks, and release workflow rely on. Each component has a full entry in `doc/stack-notes.md` (role, canonical docs, idioms, gotchas, last-reviewed date); this table is a directory, not a duplicate.

| Component | Role in the template | Reference |
|---|---|---|
| Markdown frontmatter (YAML 1.2.2) | Persona declaration on every `.claude/agents/*.md` (`name`, `description`, `model`). Read by Claude Code at session start to register each subagent. | `doc/stack-notes.md` ¬ß "Markdown frontmatter (YAML in Claude Code agent files)" |
| Claude Code hooks API | `.claude/settings.json` registers `PreToolUse` hooks that gate Read / Bash boundary violations, ssh-driven content edits, ssh-driven mutating actions, `git push --force`, `git commit --no-verify`. | `doc/stack-notes.md` ¬ß "Claude Code hooks API" |
| jq | JSON processor used inside hook command strings to parse the `PreToolUse` stdin contract and emit `hookSpecificOutput` JSON on stdout. | `doc/stack-notes.md` ¬ß "jq" |
| git | Branch / commit conventions, `git rev-parse --show-toplevel` (project-root boundary in every hook), submodule semantics (the template ships as `.ai-pm/tooling/` submodule in downstream). | `doc/stack-notes.md` ¬ß "git" |
| gh (GitHub CLI) | Used by `.claude/agents/pm-pr-prep.md` (PR open / edit) and by `.github/workflows/auto-tag.yml` (`gh release view`, `gh release create`). | `doc/stack-notes.md` ¬ß "gh (GitHub CLI)" |
| GitHub Actions | `.github/workflows/auto-tag.yml` runs on every push to `main`, reads the top version from `CHANGELOG.md`, creates the tag and GitHub Release if missing. | `doc/stack-notes.md` ¬ß "GitHub Actions" |

Components not in the list (Conventional Commits, SemVer, Keep a Changelog) are conventions used by `CHANGELOG.md` and `pm-pr-prep`, not external systems whose idioms or validators the template integrates with ‚Äî they are intentionally out of scope of stack-notes (see `doc/features/audit-fixup-self-stack-notes_plan.md` scope decision). They are not listed here either, to keep this directory consistent with stack-notes.

## Architectural decisions

Each subsection records a decision the template made, with one-line rationale and the document / PR / commit where the decision was actually taken. No "we just did it" claims.

### Agents are plain Markdown files, not compiled artefacts

Persona files at `.claude/agents/*.md` are Markdown documents with YAML frontmatter. There is no compiled runtime, no plugin manifest, no build step ‚Äî Claude Code loads them at session start by reading the directory. **Source:** `doc/features/template-v2_plan.md` ("Template repo structure" + "What's removed vs current protocol", v2 full rewrite, commit `eba8ab5 feat: template v2 ‚Äî full rewrite`).

### Commands are Markdown procedures

Slash commands at `.claude/commands/*.md` (`pm-bootstrap`, `pm-plan`, `pm-audit`, `pm-research`, `pm-fixup`) are Markdown procedure documents ‚Äî the same loading mechanism as agents. They are developer-operator shortcuts, not the PM interface. **Source:** `doc/features/template-v2_plan.md` ("Key design decisions" section: "Slash commands are developer shortcuts, not PM interface").

### WORKFLOW.md imported into downstream `CLAUDE.md` via `@.ai-pm/tooling/WORKFLOW.md`

The orchestration rules (agent roster, edit-ownership rule, remote-system boundary, git workflow, PM communication style) live in `WORKFLOW.md` at the template root. Downstream projects import it from their own `CLAUDE.md` via the `@.ai-pm/tooling/WORKFLOW.md` import directive, so an update to the template's WORKFLOW.md reaches every downstream project at the next `git submodule update --remote`. **Source:** `README.md` ¬ß "–°—Ç—Ä—É–∫—Ç—É—Ä–∞ downstream-–ø—Ä–æ–µ–∫—Ç–∞" ‚Äî "`CLAUDE.md` ‚Ä¶ (–∞–≤—Ç–æ–º–∞—Ç–∏—á–µ—Å–∫–∏ –∏–º–ø–æ—Ä—Ç–∏—Ä—É–µ—Ç `.ai-pm/tooling/WORKFLOW.md`)"; original split decision in commit `dec55b4 feat: split CLAUDE.md into static project part + dynamic WORKFLOW.md`.

### Settings.json delivered via symlink, not copy

Downstream installation creates `.claude/settings.json` as a symlink to `../.ai-pm/tooling/.claude/settings.json`, not a copy. A `git submodule update --remote` therefore picks up new hooks immediately without a per-project re-install. **Source:** `README.md` ¬ß "–£—Å—Ç–∞–Ω–æ–≤–∫–∞" ‚Äî the `ln -s ../.ai-pm/tooling/.claude/settings.json .claude/settings.json` line, added in commit `2626106 docs: add settings.json symlink to install instructions (#141)`.

### Hooks layer for cross-session enforcement of WORKFLOW rules

`WORKFLOW.md` documents the remote-edit boundary and the force-push / no-verify gates as prose rules. The same rules are also wired as Claude Code `PreToolUse` hooks in `.claude/settings.json`, so they hold even when a future session does not re-read WORKFLOW.md. The hooks gate Read/Bash path boundaries, `find` outside root, ssh + content edit, ssh + mutating action, `git push --force / -f / --force-with-lease`, `git commit --no-verify / --no-gpg-sign`. **Source:** PR #145, commit `ac5827a feat(hooks): enforce remote-edit boundary + force-push + no-verify gates`. Cross-referenced in `WORKFLOW.md` ¬ß "Hook-level enforcement".

### pm-auditor / pm-stack-researcher / pm-codebase-reader as read-only subagents

Read-only project-wide work (audit sweep, stack research, existing-codebase extraction at bootstrap) runs in subagents with explicit read-only constraints, not in the main session. This keeps the main orchestrator focused on PM conversation while large reads happen in isolation. **Source for pm-auditor:** PR #143, commit `cf889c6 feat(audit): make /audit spawn auditor subagent instead of reading in main`. **Source for pm-codebase-reader (under its former name, see rename below):** commit `f767531 Extract docs-extractor subagent for modular codebase reading`. **Source for pm-stack-researcher:** PR #142, commit `6e1bf14 Protocol integrity + stack literacy: close 5 structural gaps`.

**Rename + journeys-ownership consolidation (2026-06-04).** The reader agent (formerly carrying a "legacy" name) was renamed to **`pm-codebase-reader`**: it only ever *reads* an existing codebase at bootstrap and *writes* raw doc drafts, and "legacy" mis-narrows (a pre-existing codebase may be modern), so "codebase-reader" names what it does ‚Äî read-only on code, drafting docs. The rename also **completed the drafts-vs-owns pattern** already in force for `docs/architecture.md` and `docs/threat-model.md` (the reader drafts raw, `pm-architect` finalizes and owns ‚Äî established in commit `ee99c10`): `docs/user-journeys.md` was the one doc where the pattern was missing (the reader was treated as standing owner). Ownership of `docs/user-journeys.md` is now consolidated under **`pm-architect`** ‚Äî which already owns `architecture.md`, `product.md`, `threat-model.md`, and the `## Behavioral contract (taxonomies & invariants)` that journeys reference move-not-copy ‚Äî covering per-feature updates, gap-fill, standalone-when-gaps, and stale-journeys remediation. `pm-codebase-reader` is reduced to a pure bootstrap raw-drafter: it still drafts `user-journeys.md` from code at bootstrap legacy-full, and `pm-architect` finalizes that draft in the same finalize spawn that already finalizes the architecture and threat-model drafts. No new agent was added (the rejected alternative was a dedicated `pm-journeys` agent). **Source:** `doc/features/legacy-reader-role-split_plan.md`; branch commits `7e63ab1` (rename: agent file + frontmatter `name:`) and `462d61a` (journeys ownership ‚Üí `pm-architect`, reader narrowed to raw-drafter); pattern precedent commit `ee99c10`.

### Edit-ownership split: content artefacts vs orchestration artefacts

The rule is not "the orchestrator writes no content" ‚Äî the orchestrator already writes the **outputs of the processes it drives**: PM conversation ‚Üí backlog / contracts; `/pm-plan` ‚Üí the plan; running `code-review` ‚Üí the Pass-2 trail; finding a protocol gap ‚Üí a protocol-feedback report. What it never does is **freehand-edit canon owned by an autonomous agent** ‚Äî source code, schemas, manifests, `docs/architecture.md`, `docs/user-journeys.md`, `docs/stack-notes.md`, plan / arch files under `docs/features/`, and (inside a review artefact under `.ai-pm/reviews/`) everything through `## Verdict`, owned by `pm-plan-checker`. When one of those needs to change, even by a line, it respawns the responsible agent.

**The one carve-out** is the Pass-2 code-review trail inside `.ai-pm/reviews/<topic>_review.md`: `pm-plan-checker` owns everything through `## Verdict`; the orchestrator owns only the `## Code review` trail below it. This is the single review section the orchestrator writes ‚Äî it runs the `code-review` skill and already holds the findings, so routing them through an agent would pay a hop for data in hand. It is made safe by a **gate, not by discipline**: `pm-pr-prep` refuses to release a feature whose `## Code review` section is unstamped, and `pm-auditor` blocks an unstamped trail.

Orchestration artefacts (`docs/backlog.md` entries, PM decision logs, the Pass-2 trail above, protocol-gap reports under `.ai-pm/protocol-feedback/`, git operations, deployment script invocations) are normal orchestrator work. On discovering a structural gap in the protocol spec itself (a contradiction, an ungated manual step, a deceptive-empty template, an ownerless artefact ‚Äî in `.ai-pm/tooling`, not project code), the orchestrator writes a structured report to `.ai-pm/protocol-feedback/<topic>.md` and surfaces it for upstreaming; it never silently works around the gap and never edits `.ai-pm/tooling` in place. **Source:** PR #144, commit `9f81f64 Post-cycle lessons: notes split, edit ownership, pr-prep flexibility`; carve-out, reframed model, and protocol-gap mechanism in `WORKFLOW.md` ¬ß "Edit-ownership rule" + ¬ß "When the protocol itself has a gap" (branch `feature/review-stamp-gate`); `doc/features/review-stamp-gate_plan.md` (scenarios 2 and 7).

### Categorical scope check at plan-feature stage

Every feature plan that mentions a categorical concept (type, mode, role, state, operation, category) must explicitly choose: take the whole set or one element. If one element ‚Äî siblings are listed in "Out of scope" with one-line reason each. pm-coder cannot extend semantics of the chosen element to cover a sibling case; pm-plan-checker blocks. This catches scope drift before code. **Source:** PR #142, commit `6e1bf14 Protocol integrity + stack literacy: close 5 structural gaps`; rule text in `README.md` ¬ß "–ß—Ç–æ –≥–∞—Ä–∞–Ω—Ç–∏—Ä—É–µ—Ç —à–∞–±–ª–æ–Ω" ‚Äî "Scope –±–µ–∑ —Ç–∏—Ö–æ–π –¥–µ—Ñ–æ—Ä–º–∞—Ü–∏–∏"; full plan in `doc/features/protocol-integrity-and-stack-literacy_plan.md`.

### Release flow: CHANGELOG entry ‚Üí auto-tag ‚Üí tag + GitHub Release

`.github/workflows/auto-tag.yml` runs on every push to `main`, reads the top `## [<VERSION>]` entry from `CHANGELOG.md`, creates the matching `v<VERSION>` tag if missing, extracts the section as release notes, and calls `gh release create`. No release branch, no manual tagging, no confirmation gates. **Source:** `.github/workflows/auto-tag.yml` (current shape); decision sequence: `3f1ef5b Streamline release workflow: auto-tag + no confirmation gates`, `e14b720 fix: auto-tag reads CHANGELOG instead of commit message`, `ad3d30f fix: merge auto-tag and create-github-release into one workflow`, `c240181 fix: release workflow checks GitHub Release, not tag`.

### Execution State as the single source of progress

One file ‚Äî `.ai-pm/state/current.md` (per downstream project) ‚Äî holds the active task's Status / Done / Remaining / Touched files / Next step / Validation. Every pm-coder run reads it first and writes it last. /pm-plan initializes it. Completed tasks are archived to `.ai-pm/state/archive/`. The intent is that any future session ‚Äî same model, different model, same agent, different agent ‚Äî can resume from this file alone, without scrolling chat history. **Source:** `doc/features/integrate-consultancy_plan.md` ¬ß "Scenarios"; `doc/_templates/state.md.tmpl`.

### Product Contracts as the product-side complement to stack-notes

Stack-notes describes technical idioms cited from upstream docs. Product Contracts describe user-visible behavior in the project's own words: User value / Who uses it / Must work / Must not break / Acceptance checks / Out of scope. One contract per user-facing feature, in `.ai-pm/contracts/<feature>.md`. pm-coder reads before implementing; pm-plan-checker verifies the diff against the contract (dimension 1, Plan & Contract compliance); pm-auditor flags missing or stale contracts. Backend-only changes (refactor, infra) skip contract checks explicitly. **Source:** `doc/features/integrate-consultancy_plan.md` ¬ß "Key design decisions"; `doc/_templates/contract.md.tmpl`.

### Definition of Done as an explicit pm-plan-checker subsection

pm-plan-checker verdict now contains a "Definition of Done" subsection with 7 hard checks: scope respected; stack expectations honored; Product Contract honored; pipeline green; state updated; Product Impact Report present (when contract touched); plan's docs updates landed. The subsection ends with `DoD: pass | fail`. A pass with any unchecked box is a contradiction; pm-plan-checker must re-read its own findings. A fail forces `request-changes` even when Blocking is empty. **Source:** `doc/features/integrate-consultancy_plan.md` ¬ß "Key design decisions" point 4; `.claude/agents/pm-plan-checker.md` ¬ß "Verdict format".

### Eight review dimensions, decision matrix, trivial fast path, audit scope

Four orthogonal optimizations applied together:
1. **pm-plan-checker / pm-auditor dimensions reduced from 11 to 8** by merging three overlapping pairs (Plan + Product Contract ‚Üí Plan & Contract compliance; Security + Stability ‚Üí Correctness; Docs drift + Stack expectations ‚Üí Documentation and canon compliance). No defect class lost; only the heading collapses.
2. **Decision matrix in WORKFLOW.md** (`## What is mandatory when`) ‚Äî single table covering Execution State, Product Contract, DoD scope, Stack expectations by change type (user-facing / backend / docs-only / trivial). Replaces scattered inline conditions in pm-coder.md and pm-plan-checker.md.
3. **`/pm-fixup` command** for changes meeting four conditions (‚â§ 50 LOC, no user-visible behavior change, no stack-notes touch, no new source file). Skips plan-feature; pm-plan-checker runs in trivial mode (re-validates the four conditions; only escape hatch).
4. **pm-auditor `--scope=diff`** mode reads only files changed since the most recent `docs/audit-*.md` + their cross-references. `full` remains default and is explicitly recommended quarterly.

**Source:** `doc/features/optimize-without-losing-rigor_plan.md`; commits d09ac14, e441949, d563b02, 310695d.

### Realignment around best-in-class built-in skills/tools

The protocol predates the maturation of Claude Code's built-in skills. It now delegates *engines* to them and keeps only its protocol-specific frame: technical review ‚Üí built-in `code-review` (already Pass 2); research ‚Üí `deep-research` (driven by `/pm-research` and `pm-stack-researcher`); real-run verification ‚Üí `verify` / `run` (WORKFLOW Step 5.5). Discipline is made mechanical rather than prose-only: every `pm-*` agent has an explicit `tools:` frontmatter (read-only reviewers cannot edit code; Web is confined to `pm-stack-researcher`); judgment agents drop pinned `model:` to inherit the orchestrator model (only `pm-pr-prep` stays `haiku`); a `PreToolUse` guard denies the named `wb-*` role duplicators while letting `code-review` / `deep-research` / `wb-*` knowledge skills through; a `UserPromptSubmit` hook reasserts the route on change-intent prompts. The protocol also grew its own PR-review-response path (WORKFLOW Step 7), closing the gap that made the model reach for `wb-git:pr-author`. **Source:** `doc/protocol-vs-builtins-analysis.md`, `doc/features/protocol-builtins-realignment_plan.md`.

**Currency note (2026-06-03):** Two points in the paragraph above have since moved. (1) `pm-pr-prep` no longer stays `haiku` ‚Äî *no agent pins a `model:` now*; `pm-pr-prep` inherits the session model like every other `pm-*` agent (pinned Haiku produced repeated factual errors in PM-facing CHANGELOG entries ‚Äî CHANGELOG authoring is PM-facing, not pure mechanics). (2) The `UserPromptSubmit` route reminder's trigger vocabulary was broadened beyond the original change-intent words to also cover removal/edit verbs (e.g. —É–±–µ—Ä–∏/—É–¥–∞–ª–∏/–ø–µ—Ä–µ–∏–º–µ–Ω—É–π/–æ–±–Ω–æ–≤–∏, remove/delete/rename/update), keeping the keyword gate. **Source:** `doc/features/route-reminder-coverage-and-prprep-model_plan.md`.

**Currency note (2026-06-03):** The `UserPromptSubmit` route reminder's `additionalContext` text now also carries a short clause pointing at the "surface substantive PM decision-forks via the `AskUserQuestion` tool, not plain prose" convention ‚Äî the convention itself lives in `WORKFLOW.md` (near the PM-communication guidance), and simple proceed/confirm gates (merge-auth, "ready?", plain yes/no) stay exempt to avoid tool-spam. Text-only addition to the reminder: the matcher, the trigger regex, and the `hookSpecificOutput.additionalContext` output shape are unchanged. **Source:** `doc/features/pm-decision-via-askuserquestion_plan.md`.

### Contract-centric product map replaces the feature index

`docs/features/_index.md` (component ‚Üí feature) was a write-only output, useful to no reader. It is replaced by `docs/product-map.md` (component ‚Üí contract ‚Üí the features + reviews that built it, plus an `Infrastructure` bucket for contract-less work) ‚Äî contracts are the source of truth for behaviour, so the PM-facing view is inverted to lead with them. This map is **generated** from contracts/plans/reviews/git, never hand-filled, rebuilt wholesale by `pm-auditor` from source (Product map generation procedure). Each per-contract block is **value-first**: it leads with `User value` (projected from the contract's `## User value`) + `Out of scope` (from `## Out of scope`), with the build-history table demoted below under a `Built by:` label ‚Äî a render-only projection of fields that already exist (no new contract field or generator input), pure markdown with no HTML (the generated map ‚Äî like every PM-facing rendered doc ‚Äî keeps lists/tables/headings blank-line-separated for cross-renderer + markdownlint MD022/MD032 correctness, per the markdown-authoring rule that now lives in `WORKFLOW.md`), and old-format maps (literal `Guarantees:` label) are detected and offered a regenerate at `/pm-plan` and `/pm-audit`. Alongside it, an **authored** `docs/product.md` front door now exists ‚Äî owned by `pm-architect`, hand-written (not generated), funnelling the PM from Why this exists ‚Üí What it does today ‚Üí Documents ‚Üí Features. The per-feature "did all artefacts appear" checklist the index implicitly served moves to an explicit `pm-plan-checker` DoD item, with `pm-auditor` dimension 1 as the project-wide backstop. The scaffolded downstream project README now owns **no** capability list ‚Äî it carries a one-paragraph intro plus a single pointer link to `docs/product.md`, which is the single owner of "what it does / for whom / current limits"; existing downstream projects are migrated **move-not-copy** (their `## What it does` capabilities are first reconciled into `docs/product.md` `## What it does today` by `pm-architect`, then the README list is dropped ‚Äî never a blind rewrite). **This template repo is a deliberate exception** ‚Äî it has no user-facing Product Contracts (its "product" is the protocol itself), so neither the generated `docs/product-map.md` nor an authored `docs/product.md` is produced here; the protocol's capabilities are documented in this `architecture.md` and in `doc/features/` instead. **Source:** `doc/features/contract-centric-product-map_plan.md`; `doc/features/product-md-front-door_plan.md`; `doc/features/product-map-value-first_plan.md`; `doc/features/readme-front-gate_plan.md`; `doc/features/markdown-blank-line-sweep_plan.md`; self-audit 2026-06-02.

### Behavioral contract owns technical taxonomies; journeys stay human-language

Technical taxonomies and invariants (status enums, topic/ID grammars, QoS levels, reachability rules) now have a single owner ‚Äî a `## Behavioral contract (taxonomies & invariants)` section in the project `architecture.md` template ‚Äî distinct from `## Integration contract`, which stays "how external consumers integrate" (install / entry points / API). User-journey steps are rewritten to human language ("what the user does / expects / can go wrong") and **reference** the Behavioral contract move-not-copy: the identifier lives in one place and the journey points to it, never restating it beside a link. This is slice 3 of the two-layer-docs sequence (slices 1‚Äì2 already shipped: product-map value-first, README front-gate). **Source:** `doc/features/behavioral-contract-and-human-journeys_plan.md`.

### Contracts are two-layer; wire-tokens are single-owned in the Behavioral contract and referenced

Contracts now follow the same two-layer split as `architecture.md` and `user-journeys.md`: `## User value` / `## Out of scope` are the **token-free PM layer** (plain product language), while machine grammars and taxonomies ‚Äî topic conventions, `<x>_<y>` id/name formats, status enums, dotted config keys, `retain` / `QoS` flags, raw wire ranges ‚Äî are **single-owned** in `docs/architecture.md ## Behavioral contract (taxonomies & invariants)` and **referenced** from `## Must work` / `## Must not break`, never restated inline. This extends slice 3 (Behavioral contract) into `contract.md.tmpl`, making it slice 4 of the two-layer-docs sequence. A **structural wire-token lint** backs the split ‚Äî `pm-plan-checker` on a plan/contract change, `pm-auditor` on the project sweep ‚Äî flagging wire-token shapes in the contract's PM-facing sections and in the generated map's value / Out-of-scope lines as a **non-blocking structural note**; it is a pattern match over wire-token shapes only, explicitly **not** prose-policing (it never judges the prose's meaning or quality, consistent with `pm-auditor`'s no-prose-policing rule). Its boundary is **wire-tokens only** (topic paths, `<x>_<y>` grammars, dotted config keys, `retain` / `QoS`, raw wire ranges); **domain vocabulary is allowed** (`DimmableLight`, `Matter`, `fabric` are product language, never flagged) ‚Äî targeted, not strip-everything. Existing token-laden contracts are migrated **move-not-copy** by `pm-architect`: grammars relocate into the Behavioral contract (one shared grammar converges on one entry), PM sections are rephrased token-free, and **every Must-work / Must-not-break guarantee is preserved** ‚Äî `pm-plan-checker` compares migrated-vs-original and blocks if any guarantee is dropped or weakened. Motivation: caught live on wb-mqtt-matter 2026-06-03 ‚Äî wire-tokens leaked through a contract's `## Out of scope` into the PM-facing product-map. **Source:** `doc/features/contract-two-layer-token-lint_plan.md`.

### On-disk artifacts are English-canonical; the conversation stays in the PM's language

Everything written to disk ‚Äî files, source code, commits, and agent-authored docs/reports/reviews ‚Äî is authored in **English**. The **conversation** with the PM stays in the PM's own language: agents translate artifacts on read (relay findings to the PM in the PM's language) and write only English to disk. This is a two-axis rule ‚Äî *artifact language = English* is orthogonal to *conversation language = the PM's* ‚Äî and it is recorded once (in `WORKFLOW.md`, which every agent reads, and `CLAUDE.md.tmpl` for the downstream record) so it is stable across sessions and for subagents. It supersedes the earlier **rejected** "language-as-a-parameter" design (a single configurable artifact language threaded through templates and generators); the canon is a written instruction agents follow, with no enforcement hook (consistent with how other conversation-language guidance lives in `WORKFLOW.md`). Concretely, the `product.md` funnel headers (`Why this exists` / `What it does today` / `Documents` / `Features`), the product-map value-first labels (`User value` / `Out of scope` / `Built by:`), and the `‚Üë same work` repeat marker are all English; an un-migrated downstream `product.md` carrying the prior Russian headers is treated as a **migration trigger** (header-only rewrite, authored prose preserved verbatim), never machine-translated. **Source:** `doc/features/english-canonical-artifacts_plan.md` (PM decision 2026-06-03).

### Migration catalogue is a single protocol-root reference `MIGRATIONS.md`, sibling to `WORKFLOW.md`

The pending-migration detection conditions and every per-version migration procedure (the catalogue for moving a downstream project between protocol template versions) live in **one** protocol-root reference, `MIGRATIONS.md`, sibling to `WORKFLOW.md` (`WORKFLOW.md` = how the protocol behaves day-to-day; `MIGRATIONS.md` = how a project moves between protocol versions). It is referenced by **bare filename** so it resolves in both contexts: this repo (dogfood, read at the repo root) and downstream (read at `.ai-pm/tooling/MIGRATIONS.md`) ‚Äî the already-proven `WORKFLOW.md` mechanism. The catalogue was **extracted** from `.claude/commands/pm-bootstrap.md`, which had grown an inline catalogue that expanded every time a migration was added and mixed "how to bootstrap" with "how to migrate N template versions"; `pm-bootstrap.md` keeps only a short pointer where the catalogue was. The **single-source-of-conditions** invariant is preserved across the move: `pm-plan`, `pm-audit`, `pm-auditor`, and `pm-plan-checker` reference the one home (`### Pending-migration detection` in `MIGRATIONS.md`) **by name** and never re-encode the conditions. The `## Product map generation procedure` deliberately **stays in `pm-bootstrap.md`** (it is a generation, not a migration); migration procedures that call it cross-reference it there by filename. Home choice: a protocol-root `.md` rather than `.claude/commands/<name>.md` ‚Äî a command file would register a phantom `/<name>` slash command inviting a PM to "run migrations" directly and bypass the `/pm-bootstrap` detect-and-run flow ‚Äî and rather than `doc/<name>.md` ‚Äî a `doc/`-vs-`docs/` path resolves in only one of the two contexts, fatal for a hot-path reference four agents follow on every plan/audit. **Source:** `doc/features/extract-migrations-reference_plan.md` (pure structural relocation, no behavior change; home + reference mechanism chosen in `.ai-pm/arch/extract-migrations-reference_arch.md`).

### Blast-radius preflight is soft prose, single-sourced, referenced from Step 5.5 and Step A.5

Before any on-hardware "run it for real" (Step 5.5) or a diagnostic probe that restarts or structurally mutates a live target (Step A.5), the orchestrator must first check whether the target is coupled to a live external stateful peer whose state a local revert won't undo. This **Blast-radius preflight** is defined once in `WORKFLOW.md` as a single named concept and *referenced* from both steps (the protocol's "one fact, one owner" principle ‚Äî not a rule copied into two places that would drift). It is enforced by **prose plus orchestrator discipline, not a `PreToolUse` hook**: detecting whether a live target is coupled to an external peer requires runtime state a regex guard cannot read, so a hard gate is impossible ‚Äî consistent with the 2026-06-02 rejection of a hard edit-ownership guard, where a soft `UserPromptSubmit` reminder stayed the enforcement surface. The rule is **domain-agnostic**: it is phrased around "a live system coupled to an external stateful peer", with the originating downstream incident (`wb-mqtt-matter` ‚Äî a structural hardware test on a live paired bridge corrupted the ecosystem's device record, which a local revert / service restart did not undo) named only as the worked example, never encoded as protocol vocabulary. **Source:** `doc/features/on-hardware-blast-radius-preflight_plan.md`; `WORKFLOW.md` ¬ß "Blast-radius preflight" (commit `bf75c63`).

### Threat-model gains a full lifecycle owned by pm-architect, gated on security-bearing projects

`docs/threat-model.md` was scaffold-and-forget ‚Äî conditionally copied at bootstrap, then never owned, filled, or audited. It now has a full lifecycle (bootstrap-draft, per-feature update, audit, plan-checker gate), owned by **`pm-architect`** with **no new agent**: the agent already owns the adjacent `architecture.md ¬ß Security constraints`, and the two are complementary (threat-model = *what we protect / from whom / likelihood-impact*; Security constraints = *the enforceable rules*), so a single owner keeps them from drifting. The lifecycle is gated on **security-bearing projects only**, signalled by the **presence of `docs/threat-model.md`** (a non-security project never gets one scaffolded, so it is silently unaffected). The **"Security-relevant surfaces" list** (the per-feature trigger ‚Äî its items enumerated only at their single home) lives **once** as a named subsection in `WORKFLOW.md` and is referenced **by name** from `pm-plan` / `pm-plan-checker` / `pm-auditor`, never re-encoded ‚Äî the same single-source-of-conditions invariant as the migration conditions in `MIGRATIONS.md`, and chosen over a template-header home to avoid the `doc/`-vs-`docs/` resolution trap that home would re-introduce. The **threat ‚Üí constraint wiring is one-way and ID-keyed**: a Threats-row `Mitigation` column points at an `SCn` ID in `architecture.md ¬ß Security constraints`; the constraint keeps no back-link, and the rule text is never duplicated (move-not-copy, the contract-two-layer precedent), so each document stays independently readable. Bootstrap **drafts a populated** threat-model rather than an empty skeleton (the root cause of the downstream pain); **staleness** is detected via a dated `Last reviewed` field compared against the merge dates of security-touching features; enforcement is **soft** (a `pm-plan-checker` block plus a `pm-auditor` finding), **not a hook**, because "security-touch" is a semantic judgement a regex guard cannot make ‚Äî consistent with the Blast-radius preflight and the 2026-06-02 rejection of a hard edit-ownership guard. **Source:** `doc/features/threat-model-ownership-and-lifecycle_plan.md`; `.ai-pm/arch/threat-model-ownership-and-lifecycle_arch.md` (home + wiring chosen in the arch note); commits `6b15b05..1e93bd7`.

### pm-product-advocate: an independent product-axis referee with a soft, single-sourced, owner-split-with-gate mechanism

The protocol front-loaded and independently refereed the **technical** axis (architecture, stack-notes, threat-model, `SCn` invariants) ‚Äî `pm-plan-checker`, `code-review`, `pm-auditor`, `pm-stack-researcher` ‚Äî but the **product** axis (value, the zero-to-working story, usability, scope boundary) had **zero** independent referee: the orchestrator both *elicits* product detail from the PM and *drives* toward coding (player + referee), so an under-specified product sailed through. `pm-product-advocate` (`.claude/agents/pm-product-advocate.md`) is that missing referee ‚Äî the product-axis twin of `code-review` on the technical axis.

**Why a new agent, not a fold-in.** Independence beats self-check: research verified that LLMs cannot reliably self-correct reasoning without external feedback, and that an LLM judging its own output scores it higher than humans do (self-preference bias, causally linked to self-recognition) ‚Äî so the orchestrator judging its own elicitation is the weak configuration (`.ai-pm/research/pm-product-advocate_research.md`, citing Huang et al. ICLR 2024 / Panickssery et al. NeurIPS 2024). The advocate is player-referee-**incompatible** with `pm-architect` (which *writes* `product.md` / journeys) and with `pm-plan-checker` (which runs *post*-coding with a compliance, not advocacy, stance), so neither could host it without re-introducing the player-as-referee flaw. It inherits the established read-only-referee shape verbatim (`tools: Read, Grep, Glob, Bash, Write`, no pinned `model:` ‚Äî inherits the session model like every `pm-*` agent).

**Two run points.** (1) A **per-feature pre-coding gate** ‚Äî `WORKFLOW.md` **Step 3.5**, between the architecture decision (Step 3) and the coder (Step 4), spawned from `/pm-plan` after the plan is approved and the contract drafted, **user-facing only** via the same human-role-subject extraction `pm-auditor` already performs (backend / infrastructure / docs-only / trivial-fixup / diagnostic-probe are exempt, with no feature-category special-casing ‚Äî a blanket gate would be the Definition-of-Ready anti-pattern that strangles legitimate substrate work). (2) A **`/pm-bootstrap` foundational-question pass** ‚Äî the once-per-project, zero-to-working tier, run after the product Q&A and before the project is ready for its first feature. (This pass forces the *questions* only; auto-drafting a populated `user-journeys.md` from the answers is a separate deferred feature.)

**Block-but-sovereign.** "Block" = the coder handoff is held until each foundational gap is **answered** or **consciously descoped with a recorded rationale** ‚Äî never a permanent veto (the Shape Up bet / dismissible-blocking-review / risk-acceptance precedent). Self-preference bias is structurally sidestepped rather than merely mitigated: the advocate's output goes to the **human PM** (relayed by the orchestrator in one `AskUserQuestion` pass ‚Äî preserving the "only the orchestrator talks to the PM" invariant), **not** back to the orchestrator to be scored. The final arbiter is the human, so the advocate inheriting the session model is acceptable; independence comes from a separate role / context / prompt / fixed rubric. The advocate judges **presence of a recorded answer, never answer quality** (the PM owns meaning) ‚Äî the same "shape, not meaning" discipline as the structural wire-token lint and `pm-auditor`'s no-prose-policing rule.

**Single-sourced checklist.** The foundational questions live **once** as the named subsection `### Foundational product questions` in `WORKFLOW.md`, in two tiers (`per-feature`, `bootstrap`), in a fixed order, referenced **by name** from `pm-product-advocate`, `/pm-plan`, and `/pm-bootstrap` ‚Äî never re-encoded. This is the identical single-source-of-conditions pattern already used by `### Security-relevant surfaces` (threat-model) and `### Pending-migration detection` (`MIGRATIONS.md`), and `WORKFLOW.md` is the home for the same `doc/`-vs-`docs/` resolution reason those two decisions recorded (it resolves in both the dogfood and downstream contexts).

**Artifact + owner-split ‚Äî the second edit-ownership carve-out.** The advocate writes `.ai-pm/reviews/<topic>_advocate.md` (per-feature) / `.ai-pm/reviews/bootstrap_advocate.md` (bootstrap), owning everything **through `## Verdict`**; the orchestrator owns **only** the `## Resolutions` trail below it (one numbered entry per gap ‚Äî the PM's answer or descope-with-rationale, the output of the one `AskUserQuestion` pass it drives). This is the **second** carve-out to the Edit-ownership rule, worded by direct analogy to the Pass-2 code-review trail in `<topic>_review.md`, and made safe the same way ‚Äî by a **gate, not by discipline**. The verdict is a fixed, **greppable** token ‚Äî `gaps: N` (N ‚â• 1, blocking until resolved) or `clean` (zero gaps, silent pass) ‚Äî a positive-presence signal, never an absence that reads as "passed" (the review-stamp lesson); each gap and each resolution is a stably-numbered list item so the N-gaps-‚Üî-N-resolutions count-match is mechanical. Three mechanically-distinguishable states: *absent* (no file), *unresolved* (`gaps: N` with fewer than N resolutions), *resolved* (`gaps: N` with N resolutions **or** `clean`).

**Soft enforcement, no `PreToolUse` hook, two load-bearing backstops.** There is deliberately **no hook** ‚Äî "is this feature user-facing / is the product under-specified" is a semantic judgement a regex cannot decide, the same reasoning the threat-model and blast-radius decisions used to reject a hook (a hook here would be the fourth member of a consistent family of *rejected* hooks). Primary enforcement is the `/pm-plan` Step 3.5 step; the backstops, both keyed on the same greppable verdict token, are `pm-plan-checker`'s DoD (a user-facing feature whose advocate gate is unresolved ‚Üí blocking) and `pm-auditor` dimension 1 (a merged user-facing feature with no resolved advocate artifact ‚Üí blocking; a non-user-facing feature with none ‚Üí clean). The pre-gate-/post-backstop temporal gap (the gate is pre-coding; both backstops are post-coding) is **acceptable for the identical reason the review-stamp gate's is**: the backstops are not the primary enforcement ‚Äî they exist so a skip *cannot land silently* ‚Äî and the cost of catching a skip post-coding is strictly better than a hook that cannot decide the trigger and false-blocks substrate work.

**Family framing (the load-bearing rationale).** `pm-product-advocate` is the **product-axis member of an established family** of mechanisms in this protocol ‚Äî **soft-enforced + single-sourced + owner-split-with-gate** ‚Äî alongside the threat-model lifecycle, the blast-radius preflight, the review-stamp gate, and the migrations single-source. It is *not* a standalone with three incidental backstops: each backstop is the same load-bearing element these siblings already carry (a hook is rejected for a semantic trigger; conditions live once and are referenced by name; an orchestrator-written trail inside an agent-owned file is made safe by a gate, never by discipline). Reading any one backstop as removable ‚Äî "the DoD item is redundant with the pre-coding step", "the auditor check duplicates the plan-checker" ‚Äî is the failure this framing exists to prevent: drop the post-coding backstops and a skipped pre-coding gate lands silently, exactly the degradation the review-stamp gate was built to close. **Source:** `doc/features/pm-product-advocate_plan.md`; `.ai-pm/research/pm-product-advocate_research.md`; `.ai-pm/arch/pm-product-advocate_arch.md` (pipeline placement, checklist home, owner-split, and enforcement shape chosen in the arch note); commits `71e18fb` (agent), `40ad3ee` (WORKFLOW Step 3.5 + `### Foundational product questions` + second carve-out), `f7388ed` (`/pm-plan` + `/pm-bootstrap` gate steps), `8578791` (`pm-plan-checker` DoD + `pm-auditor` dimension-1 backstops), `431ced3` (README mention + state).

### Semantic doc-migration on template bump: an expected-discipline registry + an enhanced remediation, no new detection

When a downstream project bumps the template version (`git submodule update --remote`), the **mechanical** half of doc migration was already covered (`### Pending-migration detection` in `MIGRATIONS.md` lists greppable stale-artifact conditions, and `/pm-audit` detects them and offers the matching structural migration). The missing **semantic** half is this decision: a newer template version can introduce a *content discipline* that needs **PM-authored** content ‚Äî a populated threat-model, a foundational user-journey, a value-first product story ‚Äî which is not "a stale artifact is present" (mechanical) but "an expected discipline is absent / under-filled, and filling it needs product knowledge." The first build of this feature added a parallel detector for these gaps; **code review found that detector double-flagged what `pm-auditor` dimension 5 already detects** (the threat-model skeleton check is verbatim the existing one; the journeys / `product.md` checks overlap the existing dimension-5 notes), so it was reworked to **remediation-only**: the existing dimension-5 findings keep doing the detection, and the feature's net-new value is purely the **PM-collaborative remediation**. It is realized by **extending `/pm-audit`** (PM decision 2026-06-04, reworked post-code-review 2026-06-04): **no new command, no new agent, no new hook, and no new pm-auditor dimension / sub-check**. Three small parts: (1) a version-keyed **`### Expected-discipline manifest`** in `MIGRATIONS.md`, reframed as a **registry** (single-source, sibling to `### Pending-migration detection`, referenced **by name** by `pm-audit.md`'s remediation step and never re-encoded) ‚Äî a `####`-per-discipline subsection that maps each discipline to `Introduced in:` (the version that added it) + **Detected by** (the *existing* dimension-5 doc finding that already detects the gap ‚Äî skeleton `threat-model`, missing journey, missing/empty `product.md` funnel) + **Question source** (the by-name foundational-question list for the fill); it carries **no satisfied-checks** of its own, because detection lives in dimension 5; seeded with three real disciplines (populated threat-model lifecycle, foundational user-journeys, value-first product story); (2) an **enhancement of `/pm-audit`'s existing "Stale docs ‚Üí `pm-architect`" remediation** ‚Äî when a dimension-5 doc finding corresponds to a discipline registered in the manifest and the PM chose fix-now, the **orchestrator relays** that discipline's foundational questions in **one `AskUserQuestion` pass** (the manifest entry's by-name question source, no `pm-product-advocate` spawn ‚Äî the gap is already identified and the question list is fixed) then spawns **`pm-architect`** (the owner of `architecture.md` / `user-journeys.md` / `threat-model.md` / `product.md`) to author the content from the answers; this adds **no** new finding type ‚Äî it enriches the fix-now path of a finding dimension 5 already emits; and (3) a **"after the bump, run `/pm-audit`"** step in `WORKFLOW.md` ¬ß Maintenance so the disciplines a new version introduced surface on the same sweep that already runs the mechanical detection.

**Why this shape.** Detection of these gaps **already exists** in dimension 5 (missing journey / missing `product.md` funnel / skeleton threat-model); adding a parallel detector double-flagged every applicable discipline (e.g. a security project with a skeleton threat-model reported twice ‚Äî blocking + note), so the feature's net-new value is **purely the PM-collaborative remediation** and the manifest is the **registry that maps an existing finding to its remediation question-source**, not a second detector. With that correction it remains the **semantic complement** to the mechanical `### Pending-migration detection` (structural detect ‚Üí procedure), and it **composes existing machinery** rather than adding surface ‚Äî the `/pm-audit` finding loop, `accept-with-context` as the conscious-defer escape hatch (the load-bearing-marker analogue: a deferred discipline gap is recorded in `.ai-pm/backlog.md`, not lost, and not re-raised), `pm-architect` as the established doc owner, and the advocate's `### Foundational product questions` as a named question source ‚Äî consistent with the protocol's resistance to new commands / agents / hooks / dimensions. The cross-ecosystem research (Copier template-update, dpkg three-way proportionality, the Next.js codemod load-bearing-marker-plus-escape-hatch, terraform plan‚Üíapply) maps cleanly onto the protocol's own loop. It **generalizes** the epic's "product story fell behind" idea: "N substrate features shipped while `user-journeys.md` is still skeletal" is simply the existing dimension-5 missing-journey finding firing (the foundational-journeys discipline), not a separate one-off check ‚Äî that backlog item is subsumed here.

**Key properties.** **Presence-not-quality + proportional + idempotent are inherited from the existing dimension-5 findings, not re-implemented** ‚Äî the detection these properties describe lives entirely in dimension 5: each dimension-5 doc finding is already a positive-presence-of-a-gap test (structure absent / `<placeholder>` present / skeletal table or funnel), never a prose-quality judgment (a populated-but-"weak" artifact is silent ‚Äî the PM owns meaning); each already fires only when it **applies** (file-presence and the dimension-1 human-role-subject extraction, never re-derived) AND is **unsatisfied** (the dpkg three-way "prompt only when both changed" property); and a re-run on an already-satisfying project yields **zero** findings (the terraform "no changes" property). The feature adds none of this ‚Äî it adds **no new state file** (a committed "last-migrated-to" version stamp, the `.copier-answers.yml` analogue, is a deferred optimization, not needed for correctness) and **no new command / agent / dimension / hook**. The manifest is **disjoint** from `### Pending-migration detection` ‚Äî it carries only content disciplines (remediated by PM authoring), never a greppable stale-artifact condition (remediated mechanically), so the two registries never overlap.

**Family placement.** This is the newest member of the **soft-enforced + single-sourced** family alongside the threat-model lifecycle, the blast-radius preflight, the review-stamp gate, the migrations single-source, and `pm-product-advocate` ‚Äî the same "single-source-of-conditions + reuse-the-existing-loop" discipline: the conditions live once (the manifest registry) and are referenced by name; enforcement rides the existing `/pm-audit` loop and the existing dimension-5 detection rather than a new gate or a new detector; and no hook is added (a hook here would mis-decide the semantic "is this discipline present / does it apply" trigger a regex cannot read ‚Äî the same reasoning the threat-model, blast-radius, and advocate decisions used to reject a hook). **Source:** `doc/features/doc-migration-on-template-bump_plan.md` (the DESIGN CORRECTION banner at the top of Key design decisions records the reworked shape); `.ai-pm/research/doc-migration-on-template-bump_research.md`; `.ai-pm/arch/doc-migration-on-template-bump_arch.md` (manifest encoding, home, dimension placement, remediation wiring, and idempotence-without-state-file chosen in the arch note); `.ai-pm/reviews/doc-migration-on-template-bump_review.md` ¬ß Code review findings (finding 1: the parallel-detector overlap and the remediation-only fix); branch commits `ca412a2` (manifest in `MIGRATIONS.md`), `50c092a` + `6748af3` (initial detector + remediation, since reworked), `d14f451` (`WORKFLOW.md` ¬ß Maintenance post-bump step + README mention + state), `a940508` (this decision record), and `e8bf2ab` (the remediation-only rework: drop the duplicate detector, reframe the manifest as a question-source registry, enhance the existing dimension-5 remediation).

### Project kind: a `software | documentation` conditional split lets the protocol develop documentation projects, not only code (v1 slice; generalizes the v2.18 `process` flavor)

The protocol assumed every project's deliverable was source code. It now develops **documentation** projects too ‚Äî an SOP, a runbook, a guide, a spec, diagrams (integrating a device into a company ecosystem, diagnosing a crashed server, soldering a board, a user guide) ‚Äî via a declared **project kind** (`software | documentation`) that **conditions** the existing pipeline rather than forking it. This **generalizes** the v2.18 `process` flavor, which shipped **too narrow**: v2.18 got the frame right (kind conditional-split, shared dev-docs, a born-loud + greppable + downstream-gated validation stamp, the pm-coder deliverable remit, the advocate firing on a human-role subject, no new agent / command / hook) but rigidly narrowed the deliverable to a single SOP. This feature keeps every v2.18 mechanism and changes three things: the kind **name** (`process` ‚Üí `documentation`), the deliverable **shape** (one rigid SOP ‚Üí open document(s)), and the validation / tier / template wording (SOP-specific ‚Üí documentation-general). v1 is the slice: whole-project kind (one kind per project), a bootstrap kind question, an open deliverable, and the no-code validation wiring. This repo is itself a docs-as-code project (the "no automated tests by design ‚Äî validation by use" constraint), so the no-code mode was partly dogfooded.

**`kind` is declared once and the rule is single-sourced.** The per-project value lives as a `## Project kind: software | documentation` line in the downstream **`CLAUDE.md`** (every agent already reads `CLAUDE.md`; a root file dodges the `doc/`-vs-`docs/` trap), carried defaulting to `software` by `CLAUDE.md.tmpl`. The **rule** ‚Äî the enum plus the **load-bearing `absent OR unrecognized ‚áí software` back-compat default** ‚Äî is single-sourced in the **`### Project kind` subsection of `WORKFLOW.md`**; every kind conditional (the mandatory-table rider, Pass-2 routing, the no-code validation discipline, the `## Validation` stamp, the advocate `documentation` tier) **references it by name** and never re-encodes the enum or the default. This is the identical single-source-of-conditions pattern already used by `### Security-relevant surfaces` (threat-model) and `### Pending-migration detection` (`MIGRATIONS.md`), and `WORKFLOW.md` is the home for the same `doc/`-vs-`docs/` resolution reason those decisions recorded. The default is the airtight back-compat guarantee: a project with no `## Project kind` line **or** an unrecognized value is treated as `software`, so the documentation machinery is fully dormant unless `documentation` is explicitly declared, and an unknown value never hard-errors or silently picks a random branch ‚Äî it falls to the safe, byte-unchanged software default. The default was **extended** from the v2.18 `absent ‚áí software` to `absent OR unrecognized ‚áí software` precisely so the `process` ‚Üí `documentation` rename is back-compat-safe: a stale `## Project kind: process` line (an early v2.18 adopter) falls safely to software *and* is non-silently flagged for the one-line rename (see the migration entry below).

**The mandatory-table gains a kind dimension as a rider, not a column.** The "What is mandatory when" table gets a **Project-kind rider** below it (the proven Product-readiness / Threat-model rider precedent) ‚Äî **not** a new column and **not** kind-prefixed row duplication, so the existing software rows are **byte-unchanged**. On a `documentation`-kind project the code-only obligations (automated tests, the Pass-2 `code-review`, the build pipeline) are **inert**; what still applies (shared): the plan, `docs/user-journeys.md`, Product Contracts (as the good-outcome definition), threat-model where security is in play, `pm-auditor`, and `.ai-pm/state/current.md`. The dev-docs (architecture / stack-notes / threats / journeys) all apply, minus code-specifics ‚Äî `stack-notes` reframes from libraries/runtime to "what / who / tools we work with" (people, instruments, company systems & standards), a wording/guidance change, not a new section. Proportionality still scales through the existing change-type matrix and the per-section `N/A` discipline ‚Äî no documentation-specific stakes mechanism.

**The deliverable is open document(s) in a fixed `deliverable/` directory.** A documentation project's produced artefact ‚Äî the analogue of source code ‚Äî lives under a fixed **`deliverable/`** directory at the downstream project root (the `src/`-analogue), deliberately distinct from `docs/` (dev-docs canon) and `.ai-pm/` (protocol working area), so there is no `doc/`-vs-`docs/` homophone collision. One file (`deliverable/runbook.md`) or several (`deliverable/onboarding.md`, `deliverable/diagrams/topology.mmd`, `deliverable/img/rack.png`); the **plan names the exact leaf file(s)** while `deliverable/` is the standing convention, single-sourced once in `### Project kind`. Gates find it **presence-keyed**, exactly as they find `docs/` ‚Äî a software project has no `deliverable/` and is never flagged. The deliverable is **open in form** (markdown, diagrams, images) and **open in shape** (no mandated section set): **diagrams are first-class** ‚Äî `pm-coder` authors text-based diagrams (mermaid `.mmd`, ascii fenced in markdown) as deliverable content, while **raster images are human-supplied** and `pm-coder` only references them (`deliverable/img/...`), never fabricates binaries. There is **no mandated deliverable scaffold** (openness = absence of a forced template); the v2.18 `process.md.tmpl` is **demoted to an optional starter** ‚Äî renamed `doc/_templates/starters/sop.md.tmpl` (content preserved, RACI/SIPOC tables intact), shelved under `starters/` alongside a new `doc/_templates/starters/guide.md.tmpl` (a light reference-doc skeleton). The plan may name a starter as a seed, or none; a starter is a starting point, never a contract. `pm-coder` authors the deliverable via the v2.18 remit generalization ‚Äî its hard rule is "author the plan's deliverable artefact" (source code in software mode, the `deliverable/` file(s) in documentation mode); the **`docs/`-canon ban is preserved verbatim** (`architecture.md` / `user-journeys.md` / `product.md` / `stack-notes.md` / `threat-model.md` stay `pm-architect`-owned).

**No-code validation is one generalized, stamped gate cloned from the review-stamp triad.** A documentation feature has no `code-review`/test analogue, so its load-bearing gate is the **document-type's validation**, mapped onto the existing Step 5.5 "run it for real" ‚Äî and which validation fits depends on the doc type, which **the plan declares**: an **actionable** doc (SOP / runbook) is validated by a **dry-run / tabletop**; a **reference** doc (guide / spec / diagram) is validated by **editorial review + expert sign-off** (a dry-run may be N/A). Both record the **same single stamp**: the v2.18 `## Dry-run` token is **absorbed into one `## Validation` stamp**. `pm-plan-checker` writes it born-loud as **`## Validation: NOT YET RUN`** (never an empty heading that greps as "passed"); after the validation clears, the orchestrator **replaces the whole line** with **`## Validation: <date> ‚Äî <method> ‚Äî passed`** where **`<method>` ‚àà `dry-run` | `sign-off`** (the method the plan declared), recording observations as `## Validation findings` above it. Enforcement reuses the **same two presence-keyed gates** that already enforce the code-review stamp ‚Äî `pm-pr-prep` step 0 and `pm-auditor` dimension 1 block a `## Validation` section not stamped `‚Äî passed`; a software review file (no `## Validation` section) is exempt by section-absence, so **no new gate and no filename special-casing**. One stamp with a `<method>` field ‚Äî rather than separate `## Dry-run` + `## Sign-off` stamps ‚Äî keeps the gate logic byte-identical to today (one section, presence-keyed) while the `<method>` token and findings body carry the actionable-vs-reference distinction. A markdownlint structural-lint **pre-gate** (MD022/MD032, already in the repo's authoring discipline; Vale noted optional/future) and a **kind-conditioned DoD line** (a checklist item, never a lone rubber-stamp) complete the layering. **Step 5 Pass 2 routes on kind**: `software` (and kind-absent) ‚áí `code-review` (the existing path, **byte-for-byte untouched** ‚Äî `## Code review` is the software-kind stamp only, `## Validation` the documentation-kind stamp only); `documentation` ‚áí editorial review + the structural-lint pre-gate + the `## Validation` gate.

**The advocate finds holes in the document via a general `documentation` tier.** A single general **`documentation` tier** in the single-source `### Foundational product questions` (audience ¬∑ scope ¬∑ coverage ¬∑ decision-points ¬∑ exceptions+recovery ¬∑ zero-to-done) lets `pm-product-advocate` find holes before the document ships ‚Äî one superset list covering both actionable and reference docs, presence-only, never grading the prose; the branch / recovery questions are **N/A-able** for a pure reference doc (answered `N/A ‚Äî reference doc`, a recorded answer, so it does not flag). This generalizes the v2.18 SOP-specific `process` tier (roles/RACI ¬∑ prerequisites/SIPOC ¬∑ ‚Ä¶): RACI/SIPOC are no longer *named* in the tier (they were SOP jargon) ‚Äî a project that *is* an SOP recovers them from the `starters/sop.md.tmpl` starter. A documentation project is human-facing ‚Äî the **reader / operator is a human role** ‚Äî so the human-role-subject extraction that gates the per-feature advocate **fires** on a documentation-kind feature. The advocate, its relay (the one `AskUserQuestion` pass), and its backstops (`pm-plan-checker` DoD, `pm-auditor` dimension 1) are **reused verbatim** ‚Äî only the question source gains the renamed tier, the proven extension shape.

**Back-compat is airtight, and the rename leaves no dangling kind reference.** `process` shipped only in v2.18 (~zero downstream adoption), so the rename to `documentation` is clean across every consumer (`WORKFLOW.md` rule + rider + Pass-2 routing + the foundational-questions tier; `pm-bootstrap`; `pm-coder`; `pm-plan-checker`; `pm-pr-prep`; `pm-auditor`; `CLAUDE.md.tmpl`; `README`) ‚Äî the load-bearing check is a clean grep: no live `process` *kind* reference remains. The one intentional survivor is a **`MIGRATIONS.md ### Pending-migration detection` entry** that detects a stale `## Project kind: process` line (the durable greppable anchor) and remediates it with a trivial single-line rename to `documentation`, owned by the orchestrator (`CLAUDE.md`'s runtime owner). This anchor is what makes the extended `absent OR unrecognized ‚áí software` default **non-silent**: until migrated, a stale `process` line falls safely to software (no break); the flag nudges the rename so the adopter regains the documentation flavor.

**Why this shape.** Reuse-not-new-surface throughout: the rider, the `## Validation` stamp triad, the advocate tier, the open-deliverable convention, and the single-source `### Project kind` rule are each cloned from or generalized out of an existing pattern ‚Äî **no new agent, no new command, no new hook, no new gate, no new pm-auditor dimension**. The conditional-not-fork choice is validated by parameterized-kind build tooling (Bazel `select()` / Gradle variants / CI-matrix ‚Äî shared defaults + per-kind overrides). Back-compat is airtight via the one named `absent OR unrecognized ‚áí software` rule every consumer defaults to. **Boundaries / deferred:** v1 is **whole-project kind only** ‚Äî the full per-feature artifact-kind axis (mixed projects) is the deferred generalization; **composition** (a documentation project's deliverable consumed as another project's cited `stack-notes` standard, riding the existing doc-migration staleness machinery) is **optional and documented, not required** ‚Äî a documentation deliverable may be a terminal human artefact; and the automation-opportunity scanner (process‚Üícode bridge) plus a diagram-rendering/validation toolchain are separate backlog items. The deferrals are recorded in `.ai-pm/backlog.md`.

**Family placement.** This joins the **soft-enforced + single-sourced + reuse-the-existing-loop** family alongside the threat-model lifecycle (the rider precedent), the review-stamp gate (the `## Validation` stamp's source), the migrations single-source, `pm-product-advocate` (whose checklist the documentation tier extends), and the doc-migration registry ‚Äî the same "single-source-of-conditions + reuse-the-existing-loop" discipline: conditions live once and are referenced by name; enforcement rides the existing review-loop / pr-prep / auditor gates rather than a new mechanism; and no hook is added (a hook would mis-decide the semantic "what kind is this / is the document under-specified" trigger a regex cannot read ‚Äî the same reasoning the threat-model, blast-radius, and advocate decisions used to reject a hook).

**This template repo stays `software`-kind.** It carries no `## Project kind: documentation` line, so by the `absent OR unrecognized ‚áí software` default the machinery is **dormant here except as the template it ships** ‚Äî there is no roster or count change (the agent roster and the eight-section walk are unchanged; only the `doc/_templates/starters/` starters are added under `doc/_templates/`). **Source:** the original v2.18 record ‚Äî `doc/features/protocol-process-flavor_plan.md`, `.ai-pm/research/protocol-process-flavor_research.md`, `.ai-pm/arch/protocol-process-flavor_arch.md` ‚Äî generalized by `doc/features/documentation-flavor_plan.md` (¬ß Architecture outcomes ‚Äî Variant A on all six forks) and `.ai-pm/arch/documentation-flavor_arch.md` (deliverable location, openness vs the SOP starter, the `## Validation` stamp generalization, the `documentation` tier, the rename mechanics + migration, and the back-compat default chosen across forks 1‚Äì6); branch commits `bc4717b` (`process.md.tmpl` ‚Üí `starters/sop.md.tmpl`), `1d282b2` (open deliverable ‚Äî optional SOP + `guide.md.tmpl` starters), `935a045` (`WORKFLOW.md` ‚Äî `process` ‚Üí `documentation` rule + rider + Pass-2 routing + `## Validation` discipline + `documentation` tier + extended default), `bd05bb6` (`/pm-bootstrap` documentation-kind scaffolding + `CLAUDE.md.tmpl` kind line + `deliverable/` note), `7af0185` (agents ‚Äî documentation kind + generalized `## Validation` stamp), `31cedcd` (`MIGRATIONS.md` `process` ‚Üí `documentation` entry + `pm-pr-prep` `## Validation` stamp), `4492919` (README capability note).

### Automode: a decision-authority mode (graded, capped autonomy) ‚Äî one engine, two scopes, riding the pm-product-advocate gate

The protocol was **interactive by default** ‚Äî the PM answers every product fork, every time. **Automode** adds the opposite end of that same dial as a declared **decision-authority mode** (`autonomous | interactive`, default `interactive`): in `autonomous` mode the pipeline resolves a feature's product-forks itself from the PM's bootstrap mandate + the project's canon, **announces** each decision with a cited rationale, escalates only what the canon does not cover, and **always stops before merge**. It is a **wiring/redirect**, not a new engine: the existing `pm-product-advocate` gap output (Step 3.5) is routed through an *announce + derive-from-cited-canon-or-escalate* gate instead of being blanket-relayed to the PM, and the advocate's `## Resolutions` trail doubles as the auto-decision log. The advocate agent, its prompt, and its `clean` / `gaps: N` verdict are **byte-unchanged**; only the orchestrator's handling branches on authority. The full semantics ‚Äî the enum, the load-bearing default, the resolution order, the derivability test, the escalate cap, the announce rule, the markers ‚Äî are single-sourced in **`### Decision authority` in `WORKFLOW.md`**; this record captures *what* the feature is and *why* the shape was chosen, and references that subsection by name rather than re-encoding its rules.

**Two scopes, one engine.** Autonomy has two entry points ‚Äî **project-wide** (`.ai-pm/decision-authority.md` `mode: autonomous` ‚Äî every feature) and **per-feature** (the plan's `Decision authority: autonomous` line on an otherwise-interactive project). These are two **reads of one effective-authority value** (per-feature plan line ‚Üí file `mode:` ‚Üí `interactive`), not two engines: the only scope-dependent step is computing that value; everything downstream (announce ‚Üí derivability test ‚Üí auto-resolve-or-escalate) exists once and is scope-agnostic, so a duplicated engine cannot drift. Autonomy is an **upper bound**, never required.

**ODD-as-baseline + the no-number derivability proxy (and why not a confidence threshold).** Automode treats the PM's bootstrap mandate + project canon (`docs/product.md` / `docs/architecture.md` / `.ai-pm/contracts/` / prior `## Resolutions` / declared standards) as an **operational design domain**: it acts autonomously only *inside* what the baseline implies. The in-domain test is a **no-number derivability proxy** ‚Äî *is the gap's answer derivable from a citable canon passage?* ‚Äî auto-resolve with the citation if yes, **escalate** if no (so automode never confabulates a direction the baseline never implied: no canon passage ‚áí no auto-decision). It is deliberately **not** a numeric self-confidence threshold: the research is explicit that LLM self-confidence is **overconfident / mis-calibrated**, so a "decide when confidence > X" gate would license exactly the unsupported decisions the citation gate refuses (`.ai-pm/research/automode_research.md`). The cap also escalates regardless of derivability when a gap touches a `### Security-relevant surfaces` item or a PM-marked irreversible/high-stakes fork.

**Value home = a dedicated `.ai-pm/decision-authority.md`, not a `CLAUDE.md` line.** The *semantics* (enum + default) stay single-sourced in `### Decision authority` exactly as `### Project kind` does ‚Äî but the per-project *value* (`mode` + `veto-window-seconds`) lives in a dedicated `.ai-pm/decision-authority.md`, **not** a `CLAUDE.md` line. The split is principled by **lifecycle**: project-kind is **set-once** (a line among many in `CLAUDE.md` is fine ‚Äî nothing rewrites it), whereas authority is a **PM-flippable mid-flight toggle**; a flip-often value buried in a frequently-edited multi-purpose file is a clobber surface, and because `absent ‚áí interactive`, a dropped line would be a **silent revert** to interactive with no signal. A single-purpose `.ai-pm/` file isolates the toggle (read natively downstream, same shape as `.ai-pm/state/current.md`) so a clobber is far less likely and, if it happens, surfaces as a missing file rather than a missing line. The `.ai-pm/` home does **not** re-open the `doc/`-vs-`docs/` resolution trap (that is specific to `doc/<name>.md` spec files; `.ai-pm/` paths read natively in both this repo and downstream). This is a *refinement* of the single-source precedent, not a break ‚Äî different lifecycle, different home for the value, semantics unchanged.

**Announce-before-act + an advisory veto window that is recorded, not enforced (no v1 countdown).** Every autonomous decision is printed to console (fork ¬∑ chosen option ¬∑ brief rationale citing the canon ref ¬∑ invariants kept) **before** the orchestrator acts; it then proceeds, remaining interruptible at any moment. `veto-window-seconds` (default 15) is **recorded as the intended pause but NOT enforced as a literal countdown in v1** ‚Äî there is no Claude Code primitive that waits N seconds for *optional* input, so the actual behavior is announce-then-proceed-interruptible and a hard countdown is a deferred future hook. This is stated honestly and at the single source (`### Decision authority`) so a future reader does not mistake the field for an enforced timer; the console line never renders a live countdown. The honest framing is **graded autonomy with a human-revocable license**, not a timed pause.

**Merge/ship stays manual in both scopes.** Even in `autonomous` mode, Step 6 (the A/B/C ship gate) is unchanged ‚Äî automode carries a feature to a finished, reviewed result and **stops before merge**; it never opens or merges a PR. Autonomy relaxes only the *product-fork* questions, never merge/release.

**Dual backstop ‚Äî not redundant.** The anti-confabulation guard (every `## Resolutions` entry marked `auto` must carry a cited canon reference ‚Äî a presence/shape check, never a judgement of whether the citation truly supports the decision) is enforced at **two** points on purpose: `pm-plan-checker`'s product-readiness DoD (**per-feature**, pre-merge) **and** `pm-auditor` dimension 1 (**project-wide**, the sweep). These are **not** duplicates and must not be "simplified" into one: the plan-checker catches an uncited auto-decision *in the feature under review*; the auditor catches one that already landed *anywhere in the project* (and across features the plan-checker never re-examines). Drop either and a skipped/uncited auto-decision lands silently in its scope ‚Äî the same load-bearing-backstop reasoning as the review-stamp gate and the advocate's own two backstops (which these ride on, adding only the `auto`-entry citation sub-check; the `gaps: N` ‚Üî N-resolutions count-match is unchanged).

**Back-compat: absent / unrecognized ‚áí interactive, no migration.** An absent `.ai-pm/decision-authority.md` **or** an unrecognized `mode:` value resolves to `interactive`, so every existing downstream behaves exactly as before ‚Äî the autonomous machinery is dormant unless `autonomous` is explicitly declared, no consumer may require the file to exist, and **no migration** is introduced (mirroring the `absent OR unrecognized ‚áí software` rule of `### Project kind`).

**Family placement.** Automode is a **rider on the already-shipped `pm-product-advocate` family member**, inheriting the protocol's **soft-enforced + single-sourced + owner-split-with-gate** discipline wholesale: the semantics live once in `### Decision authority` and every consumer references it by name; enforcement is prose at the gated Step 3.5 plus the two presence-keyed backstops, **no `PreToolUse` hook** (whether a gap is derivable from canon is a semantic judgement a regex cannot make ‚Äî the same reasoning the threat-model, blast-radius, and advocate decisions used to reject a hook); and the `auto` | `escalated` markers ride the orchestrator-owned `## Resolutions` trail (the Edit-ownership second carve-out), made safe by the same gate, not by discipline. **Source:** `doc/features/automode_plan.md`; `.ai-pm/research/automode_research.md`; `.ai-pm/arch/automode_arch.md` (value-vs-semantics split + lifecycle rationale, one-engine-two-scopes, the timer-honesty caveat, and the dual-backstop framing chosen in the arch note); canonical spec in `WORKFLOW.md` ¬ß `### Decision authority`; branch commits `0017be4` (research), `5c7b8de` + `82c8a8a` (plan + PM revision), `cd1b2a9` + `4f2926a` (arch note), `e23090d` (`### Decision authority` single source + Step 3.5 autonomous branch + PM-talk rider + Edit-ownership marker note), `18608a5` (`/pm-bootstrap` authority question + `pm-plan` per-feature override line), `7758c4a` (`pm-plan-checker` + `pm-auditor` `auto`-entry citation backstops), `f71001b` (README mode line).

### System invariants get a single index in the Behavioral contract, by-reference like threat ‚Üí constraint

The `## Behavioral contract (taxonomies & invariants)` section gains a `### System invariants` index ‚Äî the single entry point for a project's cross-cutting system invariants, so a top-down reader finds one list instead of three partial ones keyed differently (format / taxonomy invariants in the Behavioral contract, rule-shaped invariants as `SCn` in `## Security constraints`, journey-level preconditions in the per-journey `**Invariants:**` blocks). Each invariant is listed **by reference to its single home**: **inline** when the Behavioral contract is itself the home (a format / taxonomy / transport invariant it already states); **by `SCn` ID** when it is enforced as a security constraint (the rule text stays in `## Security constraints`, referenced with a human hint only); **by journey name** when it is a journey-level precondition (the rule stays in that journey's `**Invariants:**` block). The index never restates an identifier and the section keeps its existing taxonomy content unchanged. The full index role + examples live once in the template's `### System invariants` subsection (`doc/_templates/architecture.md.tmpl`); this record does not duplicate them.

**Why / key decisions.** The index applies the protocol's own *one-fact-one-owner / reference-don't-duplicate* and *threat ‚Üí constraint-by-ID* patterns (see the Threat-model lifecycle decision above) to invariants: it points *out* at each home by `SCn` ID or journey name, the home keeps **no back-link**, and **no rule text is duplicated** (the only cross-doc datum is the bare ID or name). `SCn`-enforced invariants are **indexed by ID, never relocated** into the Behavioral contract ‚Äî moving them would break the one-way threat ‚Üí constraint-by-ID wiring, so the index references them where they live. The Behavioral contract already declares itself the single home for invariants (its title carries `& invariants`), so an additive subsection there satisfies one-fact-one-owner and **inherits the A4 exclusion for free** ‚Äî the index is authored domain content with no external artifact to diverge from, so it is **not** added to the A4 cross-check set (the set stays exactly File layout ‚Üî tree, Release flow ‚Üî CI, Integration contract ‚Üî README) and never produces an A4 finding. `pm-architect` (already owner of `architecture.md`, `user-journeys.md`, and the `SCn` IDs) authors and maintains the index by reference; `pm-auditor` dimension 5 gains a **structural, presence-conditional, note-not-blocking** check ‚Äî sibling to the existing threat ‚Üî constraint wiring check ‚Äî that fires only when the project has invariant-bearing homes (**‚â•1 `SCn`** in `## Security constraints` **and/or ‚â•1 journey `**Invariants:**` block**, **not** gated on security-bearing) but no index, or when an index entry names a dangling `SCn` no constraint defines; it matches shapes and IDs only, never prose meaning, and remediates by spawning `pm-architect`. The change is additive with no migration and full back-compat ‚Äî a project with no invariant homes sees nothing change; existing downstreams get at most the new note (never a block) at their next `/pm-audit`.

**Scope ‚Äî first slice of an EPIC.** This is the first slice of the EPIC **cross-document-consistency auditor**; the sibling sub-classes are deferred to later slices: the single-source enum / taxonomy drift sweep (a taxonomy restated in journeys/contracts and drifted from its Behavioral-contract home), temporal-status conflation ("known limitation / planned" in one doc vs "done / current" in another), ADR ‚Üî stack-notes backing (every architectural decision cites stack-notes), state-machine ‚Üî journeys single diagram (one canonical diagram referenced, not redrawn per journey), and journeys ‚Üî threat-model UX (does a journey surface the mitigation a threat implies). Explicitly rejected: relocating `SCn` invariants into the Behavioral contract (breaks the threat ‚Üí constraint-by-ID wiring) and promoting the auditor note to blocking (kept a note ‚Äî a wake-up, not a wall). **Source:** `doc/features/invariants-index_plan.md`; `.ai-pm/arch/invariants-index_arch.md` (index-as-additive-subsection vs new top-level section, the by-ID/by-name reference mechanics, the A4 exclusion, and the auditor-check placement chosen in the arch note); branch commits `e9d847d` (template `### System invariants` subsection), `f4d3bb6` (`pm-architect` index-authoring role), `3b63cc0` (`pm-auditor` dimension-5 note).

### Journey taxonomy-drift sweep: pm-auditor dimension 5 backstops the journeys move-not-copy discipline

`pm-auditor` dimension 5 gains a **journey identifier-restatement** check ‚Äî a structural, presence-conditional, **note-not-blocking** finding that catches a taxonomy value / enum / identifier-grammar **restated in `docs/user-journeys.md`** (step bodies or `**Invariants:**` blocks) instead of referencing the `## Behavioral contract (taxonomies & invariants)` single home. Until now the journeys move-not-copy discipline (identifiers live once in the Behavioral contract; journeys reference it, never restate it ‚Äî owned by `pm-architect`, stated in `doc/_templates/user-journeys.md.tmpl`) had **no auditor backstop**: the rule was advisory only, so a stale restated copy (the "6 statuses in the docs, 7 in the code" drift) went uncaught. This slice makes the discipline auditor-backed.

**Why / key decisions.** The new check is the **journeys-surface complement** of the existing structural-token note (the contract PM-sections + product-map value-line wire-token note already in dimension 5) ‚Äî and it is **structurally disjoint** from it: the two read different files (`docs/user-journeys.md` vs contracts / the product map), so no token can fire on both and there is no double-flag. The wire-token vocabulary is **reused by reference, not re-encoded** ‚Äî the new check points at the structural-token note's shape list rather than carrying its own copy (re-encoding the list would itself be the second-list drift this EPIC exists to remove, and self-parody in a move-not-copy slice). The declared-taxonomy exact-token match is **gated to backticked / fenced code-span tokens**: a backtick is the author asserting "this is a machine token", which keeps the check provably shape-not-meaning and avoids over-firing on common-word enums (`active` / `done` / `error` ‚Äî the template's own example values are ordinary English words, so a bare-prose match would over-fire and, via the dimension-wide two-consecutive-audits ‚Üí blocking rule, could escalate a false positive to blocking); a bare-prose restatement is therefore a **deliberate, documented miss**, the boundary of structural detection (semantic value-diffing is explicitly out of scope). The intended `## Behavioral contract` reference is **exempt** ‚Äî it inherits verbatim the same relative-doc-reference carve-out the structural-token note already applies, so the correct move-not-copy form is never mistaken for a restatement. A **dangling Behavioral-contract reference** (a journey references the section where it is absent or `N/A`) is flagged too ‚Äî the journeys-side sibling of the dangling-`SCn` check. The check sits in dimension 5 beside the threat ‚Üî constraint wiring and the System-invariants-index checks (same structural family, same shape-not-meaning + note-not-blocking discipline, same two-consecutive-audits escalation ‚Äî not a new one); it is silent on a journeys-absent / identifier-free project (proportional) and remediates by spawning `pm-architect` to move the identifier into the Behavioral contract and reference it.

**Scope ‚Äî slice 2 of an EPIC.** This is the second slice of the EPIC **cross-document-consistency auditor** (slice 1 = the System-invariants index, v2.21.0, above). The remaining sibling sub-classes are deferred to later slices, each its own plan: temporal-status conflation ("known limitation / planned" in one doc vs "done / current" in another), ADR ‚Üî stack-notes backing (every architectural decision cites stack-notes), state-machine ‚Üî journeys single diagram (one canonical diagram referenced, not redrawn per journey), journeys ‚Üî threat-model UX (does a journey surface the mitigation a threat implies), the NFR / operational-limits prompt, and the conditional state-model section. Explicitly out of scope: semantic value-diffing (which enum list is "right"), promoting the note to blocking (kept a note ‚Äî subject only to the existing dimension-wide escalation), re-doing the contract / product-map structural-token note (untouched; referenced, not duplicated), and auto-fixing the restatement (the auditor only flags; remediation is a `pm-architect` move-not-copy). **Source:** `doc/features/taxonomy-drift-sweep_plan.md`; `.ai-pm/arch/taxonomy-drift-sweep_arch.md` (the code-span gate on the sc1 declared-taxonomy match, the by-reference vocabulary reuse, and the surface-split non-overlap chosen in the arch note); branch commits `6c62ea2` (plan), `d6f5dea` (sc1 backtick gate + Behavioral-contract-reference exemption in plan + arch), `3e5c894` (`pm-auditor` dimension-5 journey identifier-restatement check), `efdbf7c` (`pm-architect` move-not-copy rule noted as backstopped by `pm-auditor`).

## Architectural constraints

Hard constraints ‚Äî agents and the orchestrator treat these as non-negotiable without an explicit PM decision.

- **No automated tests in the protocol repo by design ‚Äî validation by use, with a narrow meta-infrastructure exception.** The template ships as documentation + config; it has no runtime to assert against. Validation of feature behaviour is whether the next feature on the template (or on a downstream project) flows through the workflow correctly. **Exception:** tests that cover the template's own meta-infrastructure ‚Äî specifically, the regexes inside `.claude/settings.json` `PreToolUse` hooks ‚Äî *are* allowed and live in `tests/hooks.sh`, gated by `.github/workflows/lint-hooks.yml`. The reasoning: hook regexes are the gate behind several `WORKFLOW.md` guarantees (no silent remote edits, no unsupervised force-push); a silent regression there is not caught by "validation by use" because the user only notices when the gate fails to fire. Runtime / feature-behaviour tests for downstream projects are still out ‚Äî the exception is for protocol gates inside this repo, not for product behaviour anywhere. **Source:** `doc/features/template-v2_plan.md` ¬ß "Test plan" (the original constraint) and `doc/features/audit-fixup-hooks-quoted-form_plan.md` (the decision point that introduced the exception, after audit-2026-05-30 findings #3 / #4 / #6).
- **Template stays application-agnostic.** No domain-specific content in agents, commands, or templates. Examples must be cross-domain (checkout flow, ajv / kubectl / terraform as validators) so the template fits any project type. **Source:** commit `b10a9fa chore(template): degeneralize examples ‚Äî remove WB/Matter project bias` ‚Äî first pass leaked wb-mqtt-matter specifics; the cleanup pass replaced them with neutral examples.
- **Settings.json delivered as symlink, not copy.** Downstream `.claude/settings.json` must point at the submodule's copy so a `git submodule update --remote` immediately propagates new hooks without per-project re-install. Documented install path is `ln -s ../.ai-pm/tooling/.claude/settings.json .claude/settings.json`. **Source:** `README.md` ¬ß "–£—Å—Ç–∞–Ω–æ–≤–∫–∞".
- **Hooks are `PreToolUse` guards plus one `UserPromptSubmit` route reminder.** No `PostToolUse`, `PostToolUseFailure`, `Stop`, `SessionStart`, or other hook events are used ‚Äî the surface is intentionally minimal: gate at decision time with `PreToolUse`, and add a single soft `UserPromptSubmit` reminder that re-asserts the protocol route on change-intent prompts (the every-turn counterpart to the `PreToolUse` guards), but never react after the fact. **Source:** `.claude/settings.json` shape ‚Äî the `hooks` object contains a `PreToolUse` key and one `UserPromptSubmit` key.

## Integration contract (template ‚Üî downstream)

How a downstream project consumes the template, and what it commits to in return.

**Consumption:**

1. **Submodule at `.ai-pm/tooling/`.** Downstream runs `git submodule add git@github.com:aadegtyarev/ai-pm-protocol.git .ai-pm/tooling`. The template's `git` semantics (the submodule lives at `$GIT_DIR/modules/ai-pm/tooling/` in the superproject's git directory; see `doc/stack-notes.md` ¬ß "git" submodule data model) apply. Default `git clone` of the downstream does not check out the submodule ‚Äî `--recurse-submodules` or `git submodule update --init` is required (gotcha documented in `doc/stack-notes.md` ¬ß "git").
2. **Symlinks for agents, commands, settings.** Downstream creates three symlinks pointing into the submodule:
    - `.claude/agents` ‚Üí `../.ai-pm/tooling/.claude/agents`
    - `.claude/commands` ‚Üí `../.ai-pm/tooling/.claude/commands`
    - `.claude/settings.json` ‚Üí `../.ai-pm/tooling/.claude/settings.json`

    Submodule update therefore propagates new agent personas, command procedures, and hooks without any downstream merge.
3. **WORKFLOW.md imported via `@`-directive.** Downstream `CLAUDE.md` contains the line `@.ai-pm/tooling/WORKFLOW.md` ‚Äî Claude Code expands it inline at session start. Orchestration rules update with the submodule pointer.
4. **Templates consumed by `/pm-bootstrap`.** On first `/pm-bootstrap` in a downstream, the command reads `.ai-pm/tooling/doc/_templates/*.tmpl` and writes filled-in counterparts into the downstream's `docs/` (e.g., `docs/architecture.md`, `docs/user-journeys.md`, `docs/stack-notes.md`). After bootstrap the downstream owns those files ‚Äî they are not symlinked, they live in the downstream's own history.

**Version bumping:** When downstream wants a newer template version, it runs `git submodule update --remote .ai-pm/tooling`, stages the new submodule pointer, and commits with a `chore: bump ai-pm-protocol to <sha>` message. The submodule pointer is the only thing in the downstream's diff ‚Äî agent / command / WORKFLOW changes are transparent. **Source:** `README.md` ¬ß "–£—Å—Ç–∞–Ω–æ–≤–∫–∞" (the inline `git submodule update --remote .ai-pm/tooling` line at the end of the install block) + `WORKFLOW.md` ¬ß "Maintenance".

## Release flow

How a feature commit in this template becomes a published GitHub Release. Cross-checked against `.github/workflows/auto-tag.yml`.

1. **Feature branch + conventional commits.** Orchestrator branches from current `main` (`feature/<topic>`), commits atomically; `WORKFLOW.md` ¬ß "Git workflow ‚Äî orchestrator owns this" documents the shape.
2. **PR opened by `pm-pr-prep`.** Agent bumps the version, edits `CHANGELOG.md` (adds a `## [<VERSION>] ‚Äî <DATE>` section under `## [Unreleased]`), pushes the branch, opens (or updates) the PR via `gh pr create` / `gh pr edit`.
3. **PM merges via squash on GitHub.** The squashed commit lands on `main`. Branch / commit message at this step do not matter for the release workflow ‚Äî `auto-tag.yml` reads `CHANGELOG.md`, not the commit log.
4. **`auto-tag.yml` runs on push to `main`.**
    - `actions/checkout@v4` with `fetch-depth: 0` (full history ‚Äî required so `git tag` can see all existing tags; default `fetch-depth: 1` would make tag-existence checks unreliable, gotcha documented in `doc/stack-notes.md` ¬ß "GitHub Actions").
    - "Detect new version" step: `grep -m1 '^## \[[0-9]' CHANGELOG.md` extracts the top entry; if a `v<VERSION>` GitHub Release already exists (`gh release view "$TAG"` succeeds) the step sets `is_release=false` and the workflow no-ops. This is the idempotency guarantee ‚Äî pushing the same `main` twice does not double-tag.
    - "Create tag if missing" step (gated by `if: steps.detect.outputs.is_release == 'true'`): annotated tag `v<VERSION>` created and pushed to `origin`. Workflow-level `permissions: contents: write` grants the necessary token scope.
    - "Extract CHANGELOG notes" step: `awk "/^## \[${VERSION}\]/,/^## \[[0-9]/"` slices the section between the new version's header and the next version header; trailing line removed, separator + blank lines stripped, capped at 200 lines.
    - "Create GitHub Release" step: `gh release create "$TAG" --title "Release $TAG" --notes-file /tmp/release_notes.md` publishes the release. `GH_TOKEN` is `secrets.GITHUB_TOKEN`, which is the per-job installation access token (lifetime ‚â§ 6h on GitHub-hosted runners; documented in `doc/stack-notes.md` ¬ß "GitHub Actions").

**Why no tag-on-merge or release-branch model:** the decision was to read the CHANGELOG (single source of truth for what changed) instead of inferring versions from commit messages or branch names. Commits `e14b720 fix: auto-tag reads CHANGELOG instead of commit message`, `ad3d30f fix: merge auto-tag and create-github-release into one workflow`, `c240181 fix: release workflow checks GitHub Release, not tag`, `ba5f613 refactor: release model = tag main, not release branch` record the iterations that landed at the current shape.
