# Architecture

This document describes the architecture of the **ai-pm-protocol template repository itself** — not a downstream project that consumes the template. It is a meta-case: the template documents itself in the same shape (`doc/_templates/architecture.md.tmpl`) it expects downstream projects to use. Sections that do not apply to a documentation-and-config repo are kept as headers and marked `N/A — <one-line reason>` so a downstream reader can see the template walked its own structure honestly.

**Last full review:** 2026-06-04

---

## Project

A template for product managers building their product solo or with AI assistance. The PM describes what they want; a fixed set of Markdown-defined agents (`pm-architect`, `pm-coder`, `pm-plan-checker`, `pm-pr-prep`, `pm-codebase-reader`, `pm-stack-researcher`, `pm-auditor`, `pm-product-advocate`) plan, implement, and verify changes against the PM's plan, the downstream project's `docs/architecture.md` constraints, and `docs/stack-notes.md` stack literacy. The PM makes product decisions and does not read code. Marketing-level claims and the natural-language workflow walkthrough live in `README.md` — this document does not duplicate them; it describes the underlying structure.

## Tech stack

The template's stack is the small set of CLI tools, file formats, and APIs that the agents, commands, hooks, and release workflow rely on. Each component has a full entry in `doc/stack-notes.md` (role, canonical docs, idioms, gotchas, last-reviewed date); this table is a directory, not a duplicate.

| Component | Role in the template | Reference |
|---|---|---|
| Markdown frontmatter (YAML 1.2.2) | Persona declaration on every `.claude/agents/*.md` (`name`, `description`, `model`). Read by Claude Code at session start to register each subagent. | `doc/stack-notes.md` § "Markdown frontmatter (YAML in Claude Code agent files)" |
| Claude Code hooks API | `.claude/settings.json` registers `PreToolUse` hooks that gate Read / Bash boundary violations, ssh-driven content edits, ssh-driven mutating actions, `git push --force`, `git commit --no-verify`. | `doc/stack-notes.md` § "Claude Code hooks API" |
| jq | JSON processor used inside hook command strings to parse the `PreToolUse` stdin contract and emit `hookSpecificOutput` JSON on stdout. | `doc/stack-notes.md` § "jq" |
| git | Branch / commit conventions, `git rev-parse --show-toplevel` (project-root boundary in every hook), submodule semantics (the template ships as `.ai-pm/tooling/` submodule in downstream). | `doc/stack-notes.md` § "git" |
| gh (GitHub CLI) | Used by `.claude/agents/pm-pr-prep.md` (PR open / edit) and by `.github/workflows/auto-tag.yml` (`gh release view`, `gh release create`). | `doc/stack-notes.md` § "gh (GitHub CLI)" |
| GitHub Actions | `.github/workflows/auto-tag.yml` runs on every push to `main`, reads the top version from `CHANGELOG.md`, creates the tag and GitHub Release if missing. | `doc/stack-notes.md` § "GitHub Actions" |

Components not in the list (Conventional Commits, SemVer, Keep a Changelog) are conventions used by `CHANGELOG.md` and `pm-pr-prep`, not external systems whose idioms or validators the template integrates with — they are intentionally out of scope of stack-notes (see `doc/features/audit-fixup-self-stack-notes_plan.md` scope decision). They are not listed here either, to keep this directory consistent with stack-notes.

## Architectural decisions

Each subsection records a decision the template made, with one-line rationale and the document / PR / commit where the decision was actually taken. No "we just did it" claims.

### Agents are plain Markdown files, not compiled artefacts

Persona files at `.claude/agents/*.md` are Markdown documents with YAML frontmatter. There is no compiled runtime, no plugin manifest, no build step — Claude Code loads them at session start by reading the directory. **Source:** `doc/features/template-v2_plan.md` ("Template repo structure" + "What's removed vs current protocol", v2 full rewrite, commit `eba8ab5 feat: template v2 — full rewrite`).

### Commands are Markdown procedures

Slash commands at `.claude/commands/*.md` (`pm-bootstrap`, `pm-plan`, `pm-audit`, `pm-research`, `pm-fixup`) are Markdown procedure documents — the same loading mechanism as agents. They are developer-operator shortcuts, not the PM interface. **Source:** `doc/features/template-v2_plan.md` ("Key design decisions" section: "Slash commands are developer shortcuts, not PM interface").

### WORKFLOW.md imported into downstream `CLAUDE.md` via `@.ai-pm/tooling/WORKFLOW.md`

The orchestration rules (agent roster, edit-ownership rule, remote-system boundary, git workflow, PM communication style) live in `WORKFLOW.md` at the template root. Downstream projects import it from their own `CLAUDE.md` via the `@.ai-pm/tooling/WORKFLOW.md` import directive, so an update to the template's WORKFLOW.md reaches every downstream project at the next `git submodule update --remote`. **Source:** `README.md` § "Структура downstream-проекта" — "`CLAUDE.md` … (автоматически импортирует `.ai-pm/tooling/WORKFLOW.md`)"; original split decision in commit `dec55b4 feat: split CLAUDE.md into static project part + dynamic WORKFLOW.md`.

### Settings.json delivered via symlink, not copy

Downstream installation creates `.claude/settings.json` as a symlink to `../.ai-pm/tooling/.claude/settings.json`, not a copy. A `git submodule update --remote` therefore picks up new hooks immediately without a per-project re-install. **Source:** `README.md` § "Установка" — the `ln -s ../.ai-pm/tooling/.claude/settings.json .claude/settings.json` line, added in commit `2626106 docs: add settings.json symlink to install instructions (#141)`.

### Hooks layer for cross-session enforcement of WORKFLOW rules

`WORKFLOW.md` documents the remote-edit boundary and the force-push / no-verify gates as prose rules. The same rules are also wired as Claude Code `PreToolUse` hooks in `.claude/settings.json`, so they hold even when a future session does not re-read WORKFLOW.md. The hooks gate Read/Bash path boundaries, `find` outside root, ssh + content edit, ssh + mutating action, `git push --force / -f / --force-with-lease`, `git commit --no-verify / --no-gpg-sign`. **Source:** PR #145, commit `ac5827a feat(hooks): enforce remote-edit boundary + force-push + no-verify gates`. Cross-referenced in `WORKFLOW.md` § "Hook-level enforcement".

### pm-auditor / pm-stack-researcher / pm-codebase-reader as read-only subagents

Read-only project-wide work (audit sweep, stack research, existing-codebase extraction at bootstrap) runs in subagents with explicit read-only constraints, not in the main session. This keeps the main orchestrator focused on PM conversation while large reads happen in isolation. **Source for pm-auditor:** PR #143, commit `cf889c6 feat(audit): make /audit spawn auditor subagent instead of reading in main`. **Source for pm-codebase-reader (under its former name, see rename below):** commit `f767531 Extract docs-extractor subagent for modular codebase reading`. **Source for pm-stack-researcher:** PR #142, commit `6e1bf14 Protocol integrity + stack literacy: close 5 structural gaps`.

**Rename + journeys-ownership consolidation (2026-06-04).** The reader agent (formerly carrying a "legacy" name) was renamed to **`pm-codebase-reader`**: it only ever *reads* an existing codebase at bootstrap and *writes* raw doc drafts, and "legacy" mis-narrows (a pre-existing codebase may be modern), so "codebase-reader" names what it does — read-only on code, drafting docs. The rename also **completed the drafts-vs-owns pattern** already in force for `docs/architecture.md` and `docs/threat-model.md` (the reader drafts raw, `pm-architect` finalizes and owns — established in commit `ee99c10`): `docs/user-journeys.md` was the one doc where the pattern was missing (the reader was treated as standing owner). Ownership of `docs/user-journeys.md` is now consolidated under **`pm-architect`** — which already owns `architecture.md`, `product.md`, `threat-model.md`, and the `## Behavioral contract (taxonomies & invariants)` that journeys reference move-not-copy — covering per-feature updates, gap-fill, standalone-when-gaps, and stale-journeys remediation. `pm-codebase-reader` is reduced to a pure bootstrap raw-drafter: it still drafts `user-journeys.md` from code at bootstrap legacy-full, and `pm-architect` finalizes that draft in the same finalize spawn that already finalizes the architecture and threat-model drafts. No new agent was added (the rejected alternative was a dedicated `pm-journeys` agent). **Source:** `doc/features/legacy-reader-role-split_plan.md`; branch commits `7e63ab1` (rename: agent file + frontmatter `name:`) and `462d61a` (journeys ownership → `pm-architect`, reader narrowed to raw-drafter); pattern precedent commit `ee99c10`.

### Edit-ownership split: content artefacts vs orchestration artefacts

The rule is not "the orchestrator writes no content" — the orchestrator already writes the **outputs of the processes it drives**: PM conversation → backlog / contracts; `/pm-plan` → the plan; running `code-review` → the Pass-2 trail; finding a protocol gap → a protocol-feedback report. What it never does is **freehand-edit canon owned by an autonomous agent** — source code, schemas, manifests, `docs/architecture.md`, `docs/user-journeys.md`, `docs/stack-notes.md`, plan / arch files under `docs/features/`, and (inside a review artefact under `.ai-pm/reviews/`) everything through `## Verdict`, owned by `pm-plan-checker`. When one of those needs to change, even by a line, it respawns the responsible agent.

**The one carve-out** is the Pass-2 code-review trail inside `.ai-pm/reviews/<topic>_review.md`: `pm-plan-checker` owns everything through `## Verdict`; the orchestrator owns only the `## Code review` trail below it. This is the single review section the orchestrator writes — it runs the `code-review` skill and already holds the findings, so routing them through an agent would pay a hop for data in hand. It is made safe by a **gate, not by discipline**: `pm-pr-prep` refuses to release a feature whose `## Code review` section is unstamped, and `pm-auditor` blocks an unstamped trail.

Orchestration artefacts (`docs/backlog.md` entries, PM decision logs, the Pass-2 trail above, protocol-gap reports under `.ai-pm/protocol-feedback/`, git operations, deployment script invocations) are normal orchestrator work. On discovering a structural gap in the protocol spec itself (a contradiction, an ungated manual step, a deceptive-empty template, an ownerless artefact — in `.ai-pm/tooling`, not project code), the orchestrator writes a structured report to `.ai-pm/protocol-feedback/<topic>.md` and surfaces it for upstreaming; it never silently works around the gap and never edits `.ai-pm/tooling` in place. **Source:** PR #144, commit `9f81f64 Post-cycle lessons: notes split, edit ownership, pr-prep flexibility`; carve-out, reframed model, and protocol-gap mechanism in `WORKFLOW.md` § "Edit-ownership rule" + § "When the protocol itself has a gap" (branch `feature/review-stamp-gate`); `doc/features/review-stamp-gate_plan.md` (scenarios 2 and 7).

### Categorical scope check at plan-feature stage

Every feature plan that mentions a categorical concept (type, mode, role, state, operation, category) must explicitly choose: take the whole set or one element. If one element — siblings are listed in "Out of scope" with one-line reason each. pm-coder cannot extend semantics of the chosen element to cover a sibling case; pm-plan-checker blocks. This catches scope drift before code. **Source:** PR #142, commit `6e1bf14 Protocol integrity + stack literacy: close 5 structural gaps`; rule text in `README.md` § "Что гарантирует шаблон" — "Scope без тихой деформации"; full plan in `doc/features/protocol-integrity-and-stack-literacy_plan.md`.

### Release flow: CHANGELOG entry → auto-tag → tag + GitHub Release

`.github/workflows/auto-tag.yml` runs on every push to `main`, reads the top `## [<VERSION>]` entry from `CHANGELOG.md`, creates the matching `v<VERSION>` tag if missing, extracts the section as release notes, and calls `gh release create`. No release branch, no manual tagging, no confirmation gates. **Source:** `.github/workflows/auto-tag.yml` (current shape); decision sequence: `3f1ef5b Streamline release workflow: auto-tag + no confirmation gates`, `e14b720 fix: auto-tag reads CHANGELOG instead of commit message`, `ad3d30f fix: merge auto-tag and create-github-release into one workflow`, `c240181 fix: release workflow checks GitHub Release, not tag`.

### Execution State as the single source of progress

One file — `.ai-pm/state/current.md` (per downstream project) — holds the active task's Status / Done / Remaining / Touched files / Next step / Validation. Every pm-coder run reads it first and writes it last. /pm-plan initializes it. Completed tasks are archived to `.ai-pm/state/archive/`. The intent is that any future session — same model, different model, same agent, different agent — can resume from this file alone, without scrolling chat history. **Source:** `doc/features/integrate-consultancy_plan.md` § "Scenarios"; `doc/_templates/state.md.tmpl`.

### Product Contracts as the product-side complement to stack-notes

Stack-notes describes technical idioms cited from upstream docs. Product Contracts describe user-visible behavior in the project's own words: User value / Who uses it / Must work / Must not break / Acceptance checks / Out of scope. One contract per user-facing feature, in `.ai-pm/contracts/<feature>.md`. pm-coder reads before implementing; pm-plan-checker verifies the diff against the contract (dimension 1, Plan & Contract compliance); pm-auditor flags missing or stale contracts. Backend-only changes (refactor, infra) skip contract checks explicitly. **Source:** `doc/features/integrate-consultancy_plan.md` § "Key design decisions"; `doc/_templates/contract.md.tmpl`.

### Definition of Done as an explicit pm-plan-checker subsection

pm-plan-checker verdict now contains a "Definition of Done" subsection with 7 hard checks: scope respected; stack expectations honored; Product Contract honored; pipeline green; state updated; Product Impact Report present (when contract touched); plan's docs updates landed. The subsection ends with `DoD: pass | fail`. A pass with any unchecked box is a contradiction; pm-plan-checker must re-read its own findings. A fail forces `request-changes` even when Blocking is empty. **Source:** `doc/features/integrate-consultancy_plan.md` § "Key design decisions" point 4; `.claude/agents/pm-plan-checker.md` § "Verdict format".

### Eight review dimensions, decision matrix, trivial fast path, audit scope

Four orthogonal optimizations applied together:
1. **pm-plan-checker / pm-auditor dimensions reduced from 11 to 8** by merging three overlapping pairs (Plan + Product Contract → Plan & Contract compliance; Security + Stability → Correctness; Docs drift + Stack expectations → Documentation and canon compliance). No defect class lost; only the heading collapses.
2. **Decision matrix in WORKFLOW.md** (`## What is mandatory when`) — single table covering Execution State, Product Contract, DoD scope, Stack expectations by change type (user-facing / backend / docs-only / trivial). Replaces scattered inline conditions in pm-coder.md and pm-plan-checker.md.
3. **`/pm-fixup` command** for changes meeting four conditions (≤ 50 LOC, no user-visible behavior change, no stack-notes touch, no new source file). Skips plan-feature; pm-plan-checker runs in trivial mode (re-validates the four conditions; only escape hatch).
4. **pm-auditor `--scope=diff`** mode reads only files changed since the most recent `docs/audit-*.md` + their cross-references. `full` remains default and is explicitly recommended quarterly.

**Source:** `doc/features/optimize-without-losing-rigor_plan.md`; commits d09ac14, e441949, d563b02, 310695d.

**Currency note (2026-06-05):** Point 4's audit-scope decision was a silent auto-decide-and-announce. It now splits on **who initiated** the audit. A **PM-initiated** project-analysis request ("анализ проекта" / "review the project" by intent, no scope named) surfaces an **upfront `AskUserQuestion` scope menu** — Quick (`diff`, protocol-compliance only) vs Full (whole project + the code-quality review sweep) — with the threshold logic (first audit / >60 days / >15 feature commits) as the menu's **recommended default** the PM sees and can override; for Full the review-sweep **depth** choice is folded in. An explicit scope in the PM's words skips the menu (Full still surfaces depth). A **system-initiated** audit (the retrospective nudge / orchestrator-decided) keeps its **announce-and-proceed** behavior in both authority modes. A "no double-prompt" guard ensures the post-`diff` "run a full audit now?" offer fires only when a Quick/`diff` audit actually ran. PM-facing flow refinement, not a structural change. **Source:** `doc/features/audit-scope-menu_plan.md`.

### Realignment around best-in-class built-in skills/tools

The protocol predates the maturation of Claude Code's built-in skills. It now delegates *engines* to them and keeps only its protocol-specific frame: technical review → built-in `code-review` (already Pass 2); research → `deep-research` (driven by `/pm-research` and `pm-stack-researcher`); real-run verification → `verify` / `run` (WORKFLOW Step 5.5). Discipline is made mechanical rather than prose-only: every `pm-*` agent has an explicit `tools:` frontmatter (read-only reviewers cannot edit code; Web is confined to `pm-stack-researcher`); judgment agents drop pinned `model:` to inherit the orchestrator model (only `pm-pr-prep` stays `haiku`); a `PreToolUse` guard denies the named `wb-*` role duplicators while letting `code-review` / `deep-research` / `wb-*` knowledge skills through; a `UserPromptSubmit` hook reasserts the route on change-intent prompts. The protocol also grew its own PR-review-response path (WORKFLOW Step 7), closing the gap that made the model reach for `wb-git:pr-author`. **Source:** `doc/protocol-vs-builtins-analysis.md`, `doc/features/protocol-builtins-realignment_plan.md`.

**Currency note (2026-06-03):** Two points in the paragraph above have since moved. (1) `pm-pr-prep` no longer stays `haiku` — *no agent pins a `model:` now*; `pm-pr-prep` inherits the session model like every other `pm-*` agent (pinned Haiku produced repeated factual errors in PM-facing CHANGELOG entries — CHANGELOG authoring is PM-facing, not pure mechanics). (2) The `UserPromptSubmit` route reminder's trigger vocabulary was broadened beyond the original change-intent words to also cover removal/edit verbs (e.g. убери/удали/переименуй/обнови, remove/delete/rename/update), keeping the keyword gate. **Source:** `doc/features/route-reminder-coverage-and-prprep-model_plan.md`.

**Currency note (2026-06-03):** The `UserPromptSubmit` route reminder's `additionalContext` text now also carries a short clause pointing at the "surface substantive PM decision-forks via the `AskUserQuestion` tool, not plain prose" convention — the convention itself lives in `WORKFLOW.md` (near the PM-communication guidance), and simple proceed/confirm gates (merge-auth, "ready?", plain yes/no) stay exempt to avoid tool-spam. Text-only addition to the reminder: the matcher, the trigger regex, and the `hookSpecificOutput.additionalContext` output shape are unchanged. **Source:** `doc/features/pm-decision-via-askuserquestion_plan.md`.

### Contract-centric product map replaces the feature index

`docs/features/_index.md` (component → feature) was a write-only output, useful to no reader. It is replaced by `docs/product-map.md` (component → contract → the features + reviews that built it, plus an `Infrastructure` bucket for contract-less work) — contracts are the source of truth for behaviour, so the PM-facing view is inverted to lead with them. This map is **generated** from contracts/plans/reviews/git, never hand-filled, rebuilt wholesale by `pm-auditor` from source (Product map generation procedure). Each per-contract block is **value-first**: it leads with `User value` (projected from the contract's `## User value`) + `Out of scope` (from `## Out of scope`), with the build-history table demoted below under a `Built by:` label — a render-only projection of fields that already exist (no new contract field or generator input), pure markdown with no HTML (the generated map — like every PM-facing rendered doc — keeps lists/tables/headings blank-line-separated for cross-renderer + markdownlint MD022/MD032 correctness, per the markdown-authoring rule that now lives in `WORKFLOW.md`), and old-format maps (literal `Guarantees:` label) are detected and offered a regenerate at `/pm-plan` and `/pm-audit`. Alongside it, an **authored** `docs/product.md` front door now exists — owned by `pm-architect`, hand-written (not generated), funnelling the PM from Why this exists → What it does today → Documents → Features. The per-feature "did all artefacts appear" checklist the index implicitly served moves to an explicit `pm-plan-checker` DoD item, with `pm-auditor` dimension 1 as the project-wide backstop. The scaffolded downstream project README now owns **no** capability list — it carries a one-paragraph intro plus a single pointer link to `docs/product.md`, which is the single owner of "what it does / for whom / current limits"; existing downstream projects are migrated **move-not-copy** (their `## What it does` capabilities are first reconciled into `docs/product.md` `## What it does today` by `pm-architect`, then the README list is dropped — never a blind rewrite). **This template repo is a deliberate exception** — it has no user-facing Product Contracts (its "product" is the protocol itself), so neither the generated `docs/product-map.md` nor an authored `docs/product.md` is produced here; the protocol's capabilities are documented in this `architecture.md` and in `doc/features/` instead. **Source:** `doc/features/contract-centric-product-map_plan.md`; `doc/features/product-md-front-door_plan.md`; `doc/features/product-map-value-first_plan.md`; `doc/features/readme-front-gate_plan.md`; `doc/features/markdown-blank-line-sweep_plan.md`; self-audit 2026-06-02.

### Behavioral contract owns technical taxonomies; journeys stay human-language

Technical taxonomies and invariants (status enums, topic/ID grammars, QoS levels, reachability rules) now have a single owner — a `## Behavioral contract (taxonomies & invariants)` section in the project `architecture.md` template — distinct from `## Integration contract`, which stays "how external consumers integrate" (install / entry points / API). User-journey steps are rewritten to human language ("what the user does / expects / can go wrong") and **reference** the Behavioral contract move-not-copy: the identifier lives in one place and the journey points to it, never restating it beside a link. This is slice 3 of the two-layer-docs sequence (slices 1–2 already shipped: product-map value-first, README front-gate). **Source:** `doc/features/behavioral-contract-and-human-journeys_plan.md`.

### Contracts are two-layer; wire-tokens are single-owned in the Behavioral contract and referenced

Contracts now follow the same two-layer split as `architecture.md` and `user-journeys.md`: `## User value` / `## Out of scope` are the **token-free PM layer** (plain product language), while machine grammars and taxonomies — topic conventions, `<x>_<y>` id/name formats, status enums, dotted config keys, `retain` / `QoS` flags, raw wire ranges — are **single-owned** in `docs/architecture.md ## Behavioral contract (taxonomies & invariants)` and **referenced** from `## Must work` / `## Must not break`, never restated inline. This extends slice 3 (Behavioral contract) into `contract.md.tmpl`, making it slice 4 of the two-layer-docs sequence. A **structural wire-token lint** backs the split — `pm-plan-checker` on a plan/contract change, `pm-auditor` on the project sweep — flagging wire-token shapes in the contract's PM-facing sections and in the generated map's value / Out-of-scope lines as a **non-blocking structural note**; it is a pattern match over wire-token shapes only, explicitly **not** prose-policing (it never judges the prose's meaning or quality, consistent with `pm-auditor`'s no-prose-policing rule). Its boundary is **wire-tokens only** (topic paths, `<x>_<y>` grammars, dotted config keys, `retain` / `QoS`, raw wire ranges); **domain vocabulary is allowed** (`DimmableLight`, `Matter`, `fabric` are product language, never flagged) — targeted, not strip-everything. Existing token-laden contracts are migrated **move-not-copy** by `pm-architect`: grammars relocate into the Behavioral contract (one shared grammar converges on one entry), PM sections are rephrased token-free, and **every Must-work / Must-not-break guarantee is preserved** — `pm-plan-checker` compares migrated-vs-original and blocks if any guarantee is dropped or weakened. Motivation: caught live on wb-mqtt-matter 2026-06-03 — wire-tokens leaked through a contract's `## Out of scope` into the PM-facing product-map. **Source:** `doc/features/contract-two-layer-token-lint_plan.md`.

### On-disk artifacts are English-canonical; the conversation stays in the PM's language

Everything written to disk — files, source code, commits, and agent-authored docs/reports/reviews — is authored in **English**. The **conversation** with the PM stays in the PM's own language: agents translate artifacts on read (relay findings to the PM in the PM's language) and write only English to disk. This is a two-axis rule — *artifact language = English* is orthogonal to *conversation language = the PM's* — and it is recorded once (in `WORKFLOW.md`, which every agent reads, and `CLAUDE.md.tmpl` for the downstream record) so it is stable across sessions and for subagents. It supersedes the earlier **rejected** "language-as-a-parameter" design (a single configurable artifact language threaded through templates and generators); the canon is a written instruction agents follow, with no enforcement hook (consistent with how other conversation-language guidance lives in `WORKFLOW.md`). Concretely, the `product.md` funnel headers (`Why this exists` / `What it does today` / `Documents` / `Features`), the product-map value-first labels (`User value` / `Out of scope` / `Built by:`), and the `↑ same work` repeat marker are all English; an un-migrated downstream `product.md` carrying the prior Russian headers is treated as a **migration trigger** (header-only rewrite, authored prose preserved verbatim), never machine-translated. **Source:** `doc/features/english-canonical-artifacts_plan.md` (PM decision 2026-06-03).

### Migration catalogue is a single protocol-root reference `MIGRATIONS.md`, sibling to `WORKFLOW.md`

The pending-migration detection conditions and every per-version migration procedure (the catalogue for moving a downstream project between protocol template versions) live in **one** protocol-root reference, `MIGRATIONS.md`, sibling to `WORKFLOW.md` (`WORKFLOW.md` = how the protocol behaves day-to-day; `MIGRATIONS.md` = how a project moves between protocol versions). It is referenced by **bare filename** so it resolves in both contexts: this repo (dogfood, read at the repo root) and downstream (read at `.ai-pm/tooling/MIGRATIONS.md`) — the already-proven `WORKFLOW.md` mechanism. The catalogue was **extracted** from `.claude/commands/pm-bootstrap.md`, which had grown an inline catalogue that expanded every time a migration was added and mixed "how to bootstrap" with "how to migrate N template versions"; `pm-bootstrap.md` keeps only a short pointer where the catalogue was. The **single-source-of-conditions** invariant is preserved across the move: `pm-plan`, `pm-audit`, `pm-auditor`, and `pm-plan-checker` reference the one home (`### Pending-migration detection` in `MIGRATIONS.md`) **by name** and never re-encode the conditions. The `## Product map generation procedure` deliberately **stays in `pm-bootstrap.md`** (it is a generation, not a migration); migration procedures that call it cross-reference it there by filename. Home choice: a protocol-root `.md` rather than `.claude/commands/<name>.md` — a command file would register a phantom `/<name>` slash command inviting a PM to "run migrations" directly and bypass the `/pm-bootstrap` detect-and-run flow — and rather than `doc/<name>.md` — a `doc/`-vs-`docs/` path resolves in only one of the two contexts, fatal for a hot-path reference four agents follow on every plan/audit. **Source:** `doc/features/extract-migrations-reference_plan.md` (pure structural relocation, no behavior change; home + reference mechanism chosen in `.ai-pm/arch/extract-migrations-reference_arch.md`).

### Blast-radius preflight is soft prose, single-sourced, referenced from Step 5.5 and Step A.5

Before any on-hardware "run it for real" (Step 5.5) or a diagnostic probe that restarts or structurally mutates a live target (Step A.5), the orchestrator must first check whether the target is coupled to a live external stateful peer whose state a local revert won't undo. This **Blast-radius preflight** is defined once in `WORKFLOW.md` as a single named concept and *referenced* from both steps (the protocol's "one fact, one owner" principle — not a rule copied into two places that would drift). It is enforced by **prose plus orchestrator discipline, not a `PreToolUse` hook**: detecting whether a live target is coupled to an external peer requires runtime state a regex guard cannot read, so a hard gate is impossible — consistent with the 2026-06-02 rejection of a hard edit-ownership guard, where a soft `UserPromptSubmit` reminder stayed the enforcement surface. The rule is **domain-agnostic**: it is phrased around "a live system coupled to an external stateful peer", with the originating downstream incident (`wb-mqtt-matter` — a structural hardware test on a live paired bridge corrupted the ecosystem's device record, which a local revert / service restart did not undo) named only as the worked example, never encoded as protocol vocabulary. **Source:** `doc/features/on-hardware-blast-radius-preflight_plan.md`; `WORKFLOW.md` § "Blast-radius preflight" (commit `bf75c63`).

### Threat-model gains a full lifecycle owned by pm-architect, gated on security-bearing projects

`docs/threat-model.md` was scaffold-and-forget — conditionally copied at bootstrap, then never owned, filled, or audited. It now has a full lifecycle (bootstrap-draft, per-feature update, audit, plan-checker gate), owned by **`pm-architect`** with **no new agent**: the agent already owns the adjacent `architecture.md § Security constraints`, and the two are complementary (threat-model = *what we protect / from whom / likelihood-impact*; Security constraints = *the enforceable rules*), so a single owner keeps them from drifting. The lifecycle is gated on **security-bearing projects only**, signalled by the **presence of `docs/threat-model.md`** (a non-security project never gets one scaffolded, so it is silently unaffected). The **"Security-relevant surfaces" list** (the per-feature trigger — its items enumerated only at their single home) lives **once** as a named subsection in `WORKFLOW.md` and is referenced **by name** from `pm-plan` / `pm-plan-checker` / `pm-auditor`, never re-encoded — the same single-source-of-conditions invariant as the migration conditions in `MIGRATIONS.md`, and chosen over a template-header home to avoid the `doc/`-vs-`docs/` resolution trap that home would re-introduce. The **threat → constraint wiring is one-way and ID-keyed**: a Threats-row `Mitigation` column points at an `SCn` ID in `architecture.md § Security constraints`; the constraint keeps no back-link, and the rule text is never duplicated (move-not-copy, the contract-two-layer precedent), so each document stays independently readable. Bootstrap **drafts a populated** threat-model rather than an empty skeleton (the root cause of the downstream pain); **staleness** is detected via a dated `Last reviewed` field compared against the merge dates of security-touching features; enforcement is **soft** (a `pm-plan-checker` block plus a `pm-auditor` finding), **not a hook**, because "security-touch" is a semantic judgement a regex guard cannot make — consistent with the Blast-radius preflight and the 2026-06-02 rejection of a hard edit-ownership guard. **Source:** `doc/features/threat-model-ownership-and-lifecycle_plan.md`; `.ai-pm/arch/threat-model-ownership-and-lifecycle_arch.md` (home + wiring chosen in the arch note); commits `6b15b05..1e93bd7`.

### pm-product-advocate: an independent product-axis referee with a soft, single-sourced, owner-split-with-gate mechanism

The protocol front-loaded and independently refereed the **technical** axis (architecture, stack-notes, threat-model, `SCn` invariants) — `pm-plan-checker`, `code-review`, `pm-auditor`, `pm-stack-researcher` — but the **product** axis (value, the zero-to-working story, usability, scope boundary) had **zero** independent referee: the orchestrator both *elicits* product detail from the PM and *drives* toward coding (player + referee), so an under-specified product sailed through. `pm-product-advocate` (`.claude/agents/pm-product-advocate.md`) is that missing referee — the product-axis twin of `code-review` on the technical axis.

**Why a new agent, not a fold-in.** Independence beats self-check: research verified that LLMs cannot reliably self-correct reasoning without external feedback, and that an LLM judging its own output scores it higher than humans do (self-preference bias, causally linked to self-recognition) — so the orchestrator judging its own elicitation is the weak configuration (`.ai-pm/research/pm-product-advocate_research.md`, citing Huang et al. ICLR 2024 / Panickssery et al. NeurIPS 2024). The advocate is player-referee-**incompatible** with `pm-architect` (which *writes* `product.md` / journeys) and with `pm-plan-checker` (which runs *post*-coding with a compliance, not advocacy, stance), so neither could host it without re-introducing the player-as-referee flaw. It inherits the established read-only-referee shape verbatim (`tools: Read, Grep, Glob, Bash, Write`, no pinned `model:` — inherits the session model like every `pm-*` agent).

**Two run points.** (1) A **per-feature pre-coding gate** — `WORKFLOW.md` **Step 3.5**, between the architecture decision (Step 3) and the coder (Step 4), spawned from `/pm-plan` after the plan is approved and the contract drafted, **user-facing only** via the same human-role-subject extraction `pm-auditor` already performs (backend / infrastructure / docs-only / trivial-fixup / diagnostic-probe are exempt, with no feature-category special-casing — a blanket gate would be the Definition-of-Ready anti-pattern that strangles legitimate substrate work). (2) A **`/pm-bootstrap` foundational-question pass** — the once-per-project, zero-to-working tier, run after the product Q&A and before the project is ready for its first feature. (This pass forces the *questions* only; auto-drafting a populated `user-journeys.md` from the answers is a separate deferred feature.)

**Block-but-sovereign.** "Block" = the coder handoff is held until each foundational gap is **answered** or **consciously descoped with a recorded rationale** — never a permanent veto (the Shape Up bet / dismissible-blocking-review / risk-acceptance precedent). Self-preference bias is structurally sidestepped rather than merely mitigated: the advocate's output goes to the **human PM** (relayed by the orchestrator in one `AskUserQuestion` pass — preserving the "only the orchestrator talks to the PM" invariant), **not** back to the orchestrator to be scored. The final arbiter is the human, so the advocate inheriting the session model is acceptable; independence comes from a separate role / context / prompt / fixed rubric. The advocate judges **presence of a recorded answer, never answer quality** (the PM owns meaning) — the same "shape, not meaning" discipline as the structural wire-token lint and `pm-auditor`'s no-prose-policing rule.

**Single-sourced checklist.** The foundational questions live **once** as the named subsection `### Foundational product questions` in `WORKFLOW.md`, in two tiers (`per-feature`, `bootstrap`), in a fixed order, referenced **by name** from `pm-product-advocate`, `/pm-plan`, and `/pm-bootstrap` — never re-encoded. This is the identical single-source-of-conditions pattern already used by `### Security-relevant surfaces` (threat-model) and `### Pending-migration detection` (`MIGRATIONS.md`), and `WORKFLOW.md` is the home for the same `doc/`-vs-`docs/` resolution reason those two decisions recorded (it resolves in both the dogfood and downstream contexts).

**Artifact + owner-split — the second edit-ownership carve-out.** The advocate writes `.ai-pm/reviews/<topic>_advocate.md` (per-feature) / `.ai-pm/reviews/bootstrap_advocate.md` (bootstrap), owning everything **through `## Verdict`**; the orchestrator owns **only** the `## Resolutions` trail below it (one numbered entry per gap — the PM's answer or descope-with-rationale, the output of the one `AskUserQuestion` pass it drives). This is the **second** carve-out to the Edit-ownership rule, worded by direct analogy to the Pass-2 code-review trail in `<topic>_review.md`, and made safe the same way — by a **gate, not by discipline**. The verdict is a fixed, **greppable** token — `gaps: N` (N ≥ 1, blocking until resolved) or `clean` (zero gaps, silent pass) — a positive-presence signal, never an absence that reads as "passed" (the review-stamp lesson); each gap and each resolution is a stably-numbered list item so the N-gaps-↔-N-resolutions count-match is mechanical. Three mechanically-distinguishable states: *absent* (no file), *unresolved* (`gaps: N` with fewer than N resolutions), *resolved* (`gaps: N` with N resolutions **or** `clean`).

**Soft enforcement, no `PreToolUse` hook, two load-bearing backstops.** There is deliberately **no hook** — "is this feature user-facing / is the product under-specified" is a semantic judgement a regex cannot decide, the same reasoning the threat-model and blast-radius decisions used to reject a hook (a hook here would be the fourth member of a consistent family of *rejected* hooks). Primary enforcement is the `/pm-plan` Step 3.5 step; the backstops, both keyed on the same greppable verdict token, are `pm-plan-checker`'s DoD (a user-facing feature whose advocate gate is unresolved → blocking) and `pm-auditor` dimension 1 (a merged user-facing feature with no resolved advocate artifact → blocking; a non-user-facing feature with none → clean). The pre-gate-/post-backstop temporal gap (the gate is pre-coding; both backstops are post-coding) is **acceptable for the identical reason the review-stamp gate's is**: the backstops are not the primary enforcement — they exist so a skip *cannot land silently* — and the cost of catching a skip post-coding is strictly better than a hook that cannot decide the trigger and false-blocks substrate work.

**Family framing (the load-bearing rationale).** `pm-product-advocate` is the **product-axis member of an established family** of mechanisms in this protocol — **soft-enforced + single-sourced + owner-split-with-gate** — alongside the threat-model lifecycle, the blast-radius preflight, the review-stamp gate, and the migrations single-source. It is *not* a standalone with three incidental backstops: each backstop is the same load-bearing element these siblings already carry (a hook is rejected for a semantic trigger; conditions live once and are referenced by name; an orchestrator-written trail inside an agent-owned file is made safe by a gate, never by discipline). Reading any one backstop as removable — "the DoD item is redundant with the pre-coding step", "the auditor check duplicates the plan-checker" — is the failure this framing exists to prevent: drop the post-coding backstops and a skipped pre-coding gate lands silently, exactly the degradation the review-stamp gate was built to close. **Source:** `doc/features/pm-product-advocate_plan.md`; `.ai-pm/research/pm-product-advocate_research.md`; `.ai-pm/arch/pm-product-advocate_arch.md` (pipeline placement, checklist home, owner-split, and enforcement shape chosen in the arch note); commits `71e18fb` (agent), `40ad3ee` (WORKFLOW Step 3.5 + `### Foundational product questions` + second carve-out), `f7388ed` (`/pm-plan` + `/pm-bootstrap` gate steps), `8578791` (`pm-plan-checker` DoD + `pm-auditor` dimension-1 backstops), `431ced3` (README mention + state).

### Semantic doc-migration on template bump: an expected-discipline registry + an enhanced remediation, no new detection

When a downstream project bumps the template version (`git submodule update --remote`), the **mechanical** half of doc migration was already covered (`### Pending-migration detection` in `MIGRATIONS.md` lists greppable stale-artifact conditions, and `/pm-audit` detects them and offers the matching structural migration). The missing **semantic** half is this decision: a newer template version can introduce a *content discipline* that needs **PM-authored** content — a populated threat-model, a foundational user-journey, a value-first product story — which is not "a stale artifact is present" (mechanical) but "an expected discipline is absent / under-filled, and filling it needs product knowledge." The first build of this feature added a parallel detector for these gaps; **code review found that detector double-flagged what `pm-auditor` dimension 5 already detects** (the threat-model skeleton check is verbatim the existing one; the journeys / `product.md` checks overlap the existing dimension-5 notes), so it was reworked to **remediation-only**: the existing dimension-5 findings keep doing the detection, and the feature's net-new value is purely the **PM-collaborative remediation**. It is realized by **extending `/pm-audit`** (PM decision 2026-06-04, reworked post-code-review 2026-06-04): **no new command, no new agent, no new hook, and no new pm-auditor dimension / sub-check**. Three small parts: (1) a version-keyed **`### Expected-discipline manifest`** in `MIGRATIONS.md`, reframed as a **registry** (single-source, sibling to `### Pending-migration detection`, referenced **by name** by `pm-audit.md`'s remediation step and never re-encoded) — a `####`-per-discipline subsection that maps each discipline to `Introduced in:` (the version that added it) + **Detected by** (the *existing* dimension-5 doc finding that already detects the gap — skeleton `threat-model`, missing journey, missing/empty `product.md` funnel) + **Question source** (the by-name foundational-question list for the fill); it carries **no satisfied-checks** of its own, because detection lives in dimension 5; seeded with three real disciplines (populated threat-model lifecycle, foundational user-journeys, value-first product story); (2) an **enhancement of `/pm-audit`'s existing "Stale docs → `pm-architect`" remediation** — when a dimension-5 doc finding corresponds to a discipline registered in the manifest and the PM chose fix-now, the **orchestrator relays** that discipline's foundational questions in **one `AskUserQuestion` pass** (the manifest entry's by-name question source, no `pm-product-advocate` spawn — the gap is already identified and the question list is fixed) then spawns **`pm-architect`** (the owner of `architecture.md` / `user-journeys.md` / `threat-model.md` / `product.md`) to author the content from the answers; this adds **no** new finding type — it enriches the fix-now path of a finding dimension 5 already emits; and (3) a **"after the bump, run `/pm-audit`"** step in `WORKFLOW.md` § Maintenance so the disciplines a new version introduced surface on the same sweep that already runs the mechanical detection.

**Why this shape.** Detection of these gaps **already exists** in dimension 5 (missing journey / missing `product.md` funnel / skeleton threat-model); adding a parallel detector double-flagged every applicable discipline (e.g. a security project with a skeleton threat-model reported twice — blocking + note), so the feature's net-new value is **purely the PM-collaborative remediation** and the manifest is the **registry that maps an existing finding to its remediation question-source**, not a second detector. With that correction it remains the **semantic complement** to the mechanical `### Pending-migration detection` (structural detect → procedure), and it **composes existing machinery** rather than adding surface — the `/pm-audit` finding loop, `accept-with-context` as the conscious-defer escape hatch (the load-bearing-marker analogue: a deferred discipline gap is recorded in `.ai-pm/backlog.md`, not lost, and not re-raised), `pm-architect` as the established doc owner, and the advocate's `### Foundational product questions` as a named question source — consistent with the protocol's resistance to new commands / agents / hooks / dimensions. The cross-ecosystem research (Copier template-update, dpkg three-way proportionality, the Next.js codemod load-bearing-marker-plus-escape-hatch, terraform plan→apply) maps cleanly onto the protocol's own loop. It **generalizes** the epic's "product story fell behind" idea: "N substrate features shipped while `user-journeys.md` is still skeletal" is simply the existing dimension-5 missing-journey finding firing (the foundational-journeys discipline), not a separate one-off check — that backlog item is subsumed here.

**Key properties.** **Presence-not-quality + proportional + idempotent are inherited from the existing dimension-5 findings, not re-implemented** — the detection these properties describe lives entirely in dimension 5: each dimension-5 doc finding is already a positive-presence-of-a-gap test (structure absent / `<placeholder>` present / skeletal table or funnel), never a prose-quality judgment (a populated-but-"weak" artifact is silent — the PM owns meaning); each already fires only when it **applies** (file-presence and the dimension-1 human-role-subject extraction, never re-derived) AND is **unsatisfied** (the dpkg three-way "prompt only when both changed" property); and a re-run on an already-satisfying project yields **zero** findings (the terraform "no changes" property). The feature adds none of this — it adds **no new state file** (a committed "last-migrated-to" version stamp, the `.copier-answers.yml` analogue, is a deferred optimization, not needed for correctness) and **no new command / agent / dimension / hook**. The manifest is **disjoint** from `### Pending-migration detection` — it carries only content disciplines (remediated by PM authoring), never a greppable stale-artifact condition (remediated mechanically), so the two registries never overlap.

**Family placement.** This is the newest member of the **soft-enforced + single-sourced** family alongside the threat-model lifecycle, the blast-radius preflight, the review-stamp gate, the migrations single-source, and `pm-product-advocate` — the same "single-source-of-conditions + reuse-the-existing-loop" discipline: the conditions live once (the manifest registry) and are referenced by name; enforcement rides the existing `/pm-audit` loop and the existing dimension-5 detection rather than a new gate or a new detector; and no hook is added (a hook here would mis-decide the semantic "is this discipline present / does it apply" trigger a regex cannot read — the same reasoning the threat-model, blast-radius, and advocate decisions used to reject a hook). **Source:** `doc/features/doc-migration-on-template-bump_plan.md` (the DESIGN CORRECTION banner at the top of Key design decisions records the reworked shape); `.ai-pm/research/doc-migration-on-template-bump_research.md`; `.ai-pm/arch/doc-migration-on-template-bump_arch.md` (manifest encoding, home, dimension placement, remediation wiring, and idempotence-without-state-file chosen in the arch note); `.ai-pm/reviews/doc-migration-on-template-bump_review.md` § Code review findings (finding 1: the parallel-detector overlap and the remediation-only fix); branch commits `ca412a2` (manifest in `MIGRATIONS.md`), `50c092a` + `6748af3` (initial detector + remediation, since reworked), `d14f451` (`WORKFLOW.md` § Maintenance post-bump step + README mention + state), `a940508` (this decision record), and `e8bf2ab` (the remediation-only rework: drop the duplicate detector, reframe the manifest as a question-source registry, enhance the existing dimension-5 remediation).

### Project kind: a `software | documentation` conditional split lets the protocol develop documentation projects, not only code (v1 slice; generalizes the v2.18 `process` flavor)

The protocol assumed every project's deliverable was source code. It now develops **documentation** projects too — an SOP, a runbook, a guide, a spec, diagrams (integrating a device into a company ecosystem, diagnosing a crashed server, soldering a board, a user guide) — via a declared **project kind** (`software | documentation`) that **conditions** the existing pipeline rather than forking it. This **generalizes** the v2.18 `process` flavor, which shipped **too narrow**: v2.18 got the frame right (kind conditional-split, shared dev-docs, a born-loud + greppable + downstream-gated validation stamp, the pm-coder deliverable remit, the advocate firing on a human-role subject, no new agent / command / hook) but rigidly narrowed the deliverable to a single SOP. This feature keeps every v2.18 mechanism and changes three things: the kind **name** (`process` → `documentation`), the deliverable **shape** (one rigid SOP → open document(s)), and the validation / tier / template wording (SOP-specific → documentation-general). v1 is the slice: whole-project kind (one kind per project), a bootstrap kind question, an open deliverable, and the no-code validation wiring. This repo is itself a docs-as-code project (the "no automated tests by design — validation by use" constraint), so the no-code mode was partly dogfooded.

**`kind` is declared once and the rule is single-sourced.** The per-project value lives as a `## Project kind: software | documentation` line in the downstream **`CLAUDE.md`** (every agent already reads `CLAUDE.md`; a root file dodges the `doc/`-vs-`docs/` trap), carried defaulting to `software` by `CLAUDE.md.tmpl`. The **rule** — the enum plus the **load-bearing `absent OR unrecognized ⇒ software` back-compat default** — is single-sourced in the **`### Project kind` subsection of `WORKFLOW.md`**; every kind conditional (the mandatory-table rider, Pass-2 routing, the no-code validation discipline, the `## Validation` stamp, the advocate `documentation` tier) **references it by name** and never re-encodes the enum or the default. This is the identical single-source-of-conditions pattern already used by `### Security-relevant surfaces` (threat-model) and `### Pending-migration detection` (`MIGRATIONS.md`), and `WORKFLOW.md` is the home for the same `doc/`-vs-`docs/` resolution reason those decisions recorded. The default is the airtight back-compat guarantee: a project with no `## Project kind` line **or** an unrecognized value is treated as `software`, so the documentation machinery is fully dormant unless `documentation` is explicitly declared, and an unknown value never hard-errors or silently picks a random branch — it falls to the safe, byte-unchanged software default. The default was **extended** from the v2.18 `absent ⇒ software` to `absent OR unrecognized ⇒ software` precisely so the `process` → `documentation` rename is back-compat-safe: a stale `## Project kind: process` line (an early v2.18 adopter) falls safely to software *and* is non-silently flagged for the one-line rename (see the migration entry below).

**The mandatory-table gains a kind dimension as a rider, not a column.** The "What is mandatory when" table gets a **Project-kind rider** below it (the proven Product-readiness / Threat-model rider precedent) — **not** a new column and **not** kind-prefixed row duplication, so the existing software rows are **byte-unchanged**. On a `documentation`-kind project the code-only obligations (automated tests, the Pass-2 `code-review`, the build pipeline) are **inert**; what still applies (shared): the plan, `docs/user-journeys.md`, Product Contracts (as the good-outcome definition), threat-model where security is in play, `pm-auditor`, and `.ai-pm/state/current.md`. The dev-docs (architecture / stack-notes / threats / journeys) all apply, minus code-specifics — `stack-notes` reframes from libraries/runtime to "what / who / tools we work with" (people, instruments, company systems & standards), a wording/guidance change, not a new section. Proportionality still scales through the existing change-type matrix and the per-section `N/A` discipline — no documentation-specific stakes mechanism.

**The deliverable is open document(s) in a fixed `deliverable/` directory.** A documentation project's produced artefact — the analogue of source code — lives under a fixed **`deliverable/`** directory at the downstream project root (the `src/`-analogue), deliberately distinct from `docs/` (dev-docs canon) and `.ai-pm/` (protocol working area), so there is no `doc/`-vs-`docs/` homophone collision. One file (`deliverable/runbook.md`) or several (`deliverable/onboarding.md`, `deliverable/diagrams/topology.mmd`, `deliverable/img/rack.png`); the **plan names the exact leaf file(s)** while `deliverable/` is the standing convention, single-sourced once in `### Project kind`. Gates find it **presence-keyed**, exactly as they find `docs/` — a software project has no `deliverable/` and is never flagged. The deliverable is **open in form** (markdown, diagrams, images) and **open in shape** (no mandated section set): **diagrams are first-class** — `pm-coder` authors text-based diagrams (mermaid `.mmd`, ascii fenced in markdown) as deliverable content, while **raster images are human-supplied** and `pm-coder` only references them (`deliverable/img/...`), never fabricates binaries. There is **no mandated deliverable scaffold** (openness = absence of a forced template); the v2.18 `process.md.tmpl` is **demoted to an optional starter** — renamed `doc/_templates/starters/sop.md.tmpl` (content preserved, RACI/SIPOC tables intact), shelved under `starters/` alongside a new `doc/_templates/starters/guide.md.tmpl` (a light reference-doc skeleton). The plan may name a starter as a seed, or none; a starter is a starting point, never a contract. `pm-coder` authors the deliverable via the v2.18 remit generalization — its hard rule is "author the plan's deliverable artefact" (source code in software mode, the `deliverable/` file(s) in documentation mode); the **`docs/`-canon ban is preserved verbatim** (`architecture.md` / `user-journeys.md` / `product.md` / `stack-notes.md` / `threat-model.md` stay `pm-architect`-owned).

**No-code validation is one generalized, stamped gate cloned from the review-stamp triad.** A documentation feature has no `code-review`/test analogue, so its load-bearing gate is the **document-type's validation**, mapped onto the existing Step 5.5 "run it for real" — and which validation fits depends on the doc type, which **the plan declares**: an **actionable** doc (SOP / runbook) is validated by a **dry-run / tabletop**; a **reference** doc (guide / spec / diagram) is validated by **editorial review + expert sign-off** (a dry-run may be N/A). Both record the **same single stamp**: the v2.18 `## Dry-run` token is **absorbed into one `## Validation` stamp**. `pm-plan-checker` writes it born-loud as **`## Validation: NOT YET RUN`** (never an empty heading that greps as "passed"); after the validation clears, the orchestrator **replaces the whole line** with **`## Validation: <date> — <method> — passed`** where **`<method>` ∈ `dry-run` | `sign-off`** (the method the plan declared), recording observations as `## Validation findings` above it. Enforcement reuses the **same two presence-keyed gates** that already enforce the code-review stamp — `pm-pr-prep` step 0 and `pm-auditor` dimension 1 block a `## Validation` section not stamped `— passed`; a software review file (no `## Validation` section) is exempt by section-absence, so **no new gate and no filename special-casing**. One stamp with a `<method>` field — rather than separate `## Dry-run` + `## Sign-off` stamps — keeps the gate logic byte-identical to today (one section, presence-keyed) while the `<method>` token and findings body carry the actionable-vs-reference distinction. A markdownlint structural-lint **pre-gate** (MD022/MD032, already in the repo's authoring discipline; Vale noted optional/future) and a **kind-conditioned DoD line** (a checklist item, never a lone rubber-stamp) complete the layering. **Step 5 Pass 2 routes on kind**: `software` (and kind-absent) ⇒ `code-review` (the existing path, **byte-for-byte untouched** — `## Code review` is the software-kind stamp only, `## Validation` the documentation-kind stamp only); `documentation` ⇒ editorial review + the structural-lint pre-gate + the `## Validation` gate.

**The advocate finds holes in the document via a general `documentation` tier.** A single general **`documentation` tier** in the single-source `### Foundational product questions` (audience · scope · coverage · decision-points · exceptions+recovery · zero-to-done) lets `pm-product-advocate` find holes before the document ships — one superset list covering both actionable and reference docs, presence-only, never grading the prose; the branch / recovery questions are **N/A-able** for a pure reference doc (answered `N/A — reference doc`, a recorded answer, so it does not flag). This generalizes the v2.18 SOP-specific `process` tier (roles/RACI · prerequisites/SIPOC · …): RACI/SIPOC are no longer *named* in the tier (they were SOP jargon) — a project that *is* an SOP recovers them from the `starters/sop.md.tmpl` starter. A documentation project is human-facing — the **reader / operator is a human role** — so the human-role-subject extraction that gates the per-feature advocate **fires** on a documentation-kind feature. The advocate, its relay (the one `AskUserQuestion` pass), and its backstops (`pm-plan-checker` DoD, `pm-auditor` dimension 1) are **reused verbatim** — only the question source gains the renamed tier, the proven extension shape.

**Back-compat is airtight, and the rename leaves no dangling kind reference.** `process` shipped only in v2.18 (~zero downstream adoption), so the rename to `documentation` is clean across every consumer (`WORKFLOW.md` rule + rider + Pass-2 routing + the foundational-questions tier; `pm-bootstrap`; `pm-coder`; `pm-plan-checker`; `pm-pr-prep`; `pm-auditor`; `CLAUDE.md.tmpl`; `README`) — the load-bearing check is a clean grep: no live `process` *kind* reference remains. The one intentional survivor is a **`MIGRATIONS.md ### Pending-migration detection` entry** that detects a stale `## Project kind: process` line (the durable greppable anchor) and remediates it with a trivial single-line rename to `documentation`, owned by the orchestrator (`CLAUDE.md`'s runtime owner). This anchor is what makes the extended `absent OR unrecognized ⇒ software` default **non-silent**: until migrated, a stale `process` line falls safely to software (no break); the flag nudges the rename so the adopter regains the documentation flavor.

**Why this shape.** Reuse-not-new-surface throughout: the rider, the `## Validation` stamp triad, the advocate tier, the open-deliverable convention, and the single-source `### Project kind` rule are each cloned from or generalized out of an existing pattern — **no new agent, no new command, no new hook, no new gate, no new pm-auditor dimension**. The conditional-not-fork choice is validated by parameterized-kind build tooling (Bazel `select()` / Gradle variants / CI-matrix — shared defaults + per-kind overrides). Back-compat is airtight via the one named `absent OR unrecognized ⇒ software` rule every consumer defaults to. **Boundaries / deferred:** v1 is **whole-project kind only** — the full per-feature artifact-kind axis (mixed projects) is the deferred generalization; **composition** (a documentation project's deliverable consumed as another project's cited `stack-notes` standard, riding the existing doc-migration staleness machinery) is **optional and documented, not required** — a documentation deliverable may be a terminal human artefact; and the automation-opportunity scanner (process→code bridge) plus a diagram-rendering/validation toolchain are separate backlog items. The deferrals are recorded in `.ai-pm/backlog.md`.

**Family placement.** This joins the **soft-enforced + single-sourced + reuse-the-existing-loop** family alongside the threat-model lifecycle (the rider precedent), the review-stamp gate (the `## Validation` stamp's source), the migrations single-source, `pm-product-advocate` (whose checklist the documentation tier extends), and the doc-migration registry — the same "single-source-of-conditions + reuse-the-existing-loop" discipline: conditions live once and are referenced by name; enforcement rides the existing review-loop / pr-prep / auditor gates rather than a new mechanism; and no hook is added (a hook would mis-decide the semantic "what kind is this / is the document under-specified" trigger a regex cannot read — the same reasoning the threat-model, blast-radius, and advocate decisions used to reject a hook).

### Automode: a decision-authority mode (graded, capped autonomy) — one engine, two scopes, riding the pm-product-advocate gate

The protocol was **interactive by default** — the PM answers every product fork, every time. **Automode** adds the opposite end of that same dial as a declared **decision-authority mode** (`autonomous | interactive`, default `interactive`): in `autonomous` mode the pipeline resolves a feature's product-forks itself from the PM's bootstrap mandate + the project's canon, **announces** each decision with a cited rationale, escalates only what the canon does not cover, and **always stops before merge**. It is a **wiring/redirect**, not a new engine: the existing `pm-product-advocate` gap output (Step 3.5) is routed through an *announce + derive-from-cited-canon-or-escalate* gate instead of being blanket-relayed to the PM, and the advocate's `## Resolutions` trail doubles as the auto-decision log. The advocate agent, its prompt, and its `clean` / `gaps: N` verdict are **byte-unchanged**; only the orchestrator's handling branches on authority. The full semantics — the enum, the load-bearing default, the resolution order, the derivability test, the escalate cap, the announce rule, the markers — are single-sourced in **`### Decision authority` in `WORKFLOW.md`**; this record captures *what* the feature is and *why* the shape was chosen, and references that subsection by name rather than re-encoding its rules.

**Two scopes, one engine.** Autonomy has two entry points — **project-wide** (`.ai-pm/decision-authority.md` `mode: autonomous` — every feature) and **per-feature** (the plan's `Decision authority: autonomous` line on an otherwise-interactive project). These are two **reads of one effective-authority value** (per-feature plan line → file `mode:` → `interactive`), not two engines: the only scope-dependent step is computing that value; everything downstream (announce → derivability test → auto-resolve-or-escalate) exists once and is scope-agnostic, so a duplicated engine cannot drift. Autonomy is an **upper bound**, never required.

**ODD-as-baseline + the no-number derivability proxy (and why not a confidence threshold).** Automode treats the PM's bootstrap mandate + project canon (`docs/product.md` / `docs/architecture.md` / `.ai-pm/contracts/` / prior `## Resolutions` / declared standards) as an **operational design domain**: it acts autonomously only *inside* what the baseline implies. The in-domain test is a **no-number derivability proxy** — *is the gap's answer derivable from a citable canon passage?* — auto-resolve with the citation if yes, **escalate** if no (so automode never confabulates a direction the baseline never implied: no canon passage ⇒ no auto-decision). It is deliberately **not** a numeric self-confidence threshold: the research is explicit that LLM self-confidence is **overconfident / mis-calibrated**, so a "decide when confidence > X" gate would license exactly the unsupported decisions the citation gate refuses (`.ai-pm/research/automode_research.md`). The cap also escalates regardless of derivability when a gap touches a `### Security-relevant surfaces` item or a PM-marked irreversible/high-stakes fork.

**Value home = a dedicated `.ai-pm/decision-authority.md`, not a `CLAUDE.md` line.** The *semantics* (enum + default) stay single-sourced in `### Decision authority` exactly as `### Project kind` does — but the per-project *value* (`mode` + `veto-window-seconds`) lives in a dedicated `.ai-pm/decision-authority.md`, **not** a `CLAUDE.md` line. The split is principled by **lifecycle**: project-kind is **set-once** (a line among many in `CLAUDE.md` is fine — nothing rewrites it), whereas authority is a **PM-flippable mid-flight toggle**; a flip-often value buried in a frequently-edited multi-purpose file is a clobber surface, and because `absent ⇒ interactive`, a dropped line would be a **silent revert** to interactive with no signal. A single-purpose `.ai-pm/` file isolates the toggle (read natively downstream, same shape as `.ai-pm/state/current.md`) so a clobber is far less likely and, if it happens, surfaces as a missing file rather than a missing line. The `.ai-pm/` home does **not** re-open the `doc/`-vs-`docs/` resolution trap (that is specific to `doc/<name>.md` spec files; `.ai-pm/` paths read natively in both this repo and downstream). This is a *refinement* of the single-source precedent, not a break — different lifecycle, different home for the value, semantics unchanged.

**Announce-before-act + an advisory veto window that is recorded, not enforced (no v1 countdown).** Every autonomous decision is printed to console (fork · chosen option · brief rationale citing the canon ref · invariants kept) **before** the orchestrator acts; it then proceeds, remaining interruptible at any moment. `veto-window-seconds` (default 15) is **recorded as the intended pause but NOT enforced as a literal countdown in v1** — there is no Claude Code primitive that waits N seconds for *optional* input, so the actual behavior is announce-then-proceed-interruptible and a hard countdown is a deferred future hook. This is stated honestly and at the single source (`### Decision authority`) so a future reader does not mistake the field for an enforced timer; the console line never renders a live countdown. The honest framing is **graded autonomy with a human-revocable license**, not a timed pause.

**Merge/ship stays manual in both scopes.** Even in `autonomous` mode, Step 6 (the A/B/C ship gate) is unchanged — automode carries a feature to a finished, reviewed result and **stops before merge**; it never opens or merges a PR. Autonomy relaxes only the *product-fork* questions, never merge/release.

**Dual backstop — not redundant.** The anti-confabulation guard (every `## Resolutions` entry marked `auto` must carry a cited canon reference — a presence/shape check, never a judgement of whether the citation truly supports the decision) is enforced at **two** points on purpose: `pm-plan-checker`'s product-readiness DoD (**per-feature**, pre-merge) **and** `pm-auditor` dimension 1 (**project-wide**, the sweep). These are **not** duplicates and must not be "simplified" into one: the plan-checker catches an uncited auto-decision *in the feature under review*; the auditor catches one that already landed *anywhere in the project* (and across features the plan-checker never re-examines). Drop either and a skipped/uncited auto-decision lands silently in its scope — the same load-bearing-backstop reasoning as the review-stamp gate and the advocate's own two backstops (which these ride on, adding only the `auto`-entry citation sub-check; the `gaps: N` ↔ N-resolutions count-match is unchanged).

**Back-compat: absent / unrecognized ⇒ interactive, no migration.** An absent `.ai-pm/decision-authority.md` **or** an unrecognized `mode:` value resolves to `interactive`, so every existing downstream behaves exactly as before — the autonomous machinery is dormant unless `autonomous` is explicitly declared, no consumer may require the file to exist, and **no migration** is introduced (mirroring the `absent OR unrecognized ⇒ software` rule of `### Project kind`).

**Family placement.** Automode is a **rider on the already-shipped `pm-product-advocate` family member**, inheriting the protocol's **soft-enforced + single-sourced + owner-split-with-gate** discipline wholesale: the semantics live once in `### Decision authority` and every consumer references it by name; enforcement is prose at the gated Step 3.5 plus the two presence-keyed backstops, **no `PreToolUse` hook** (whether a gap is derivable from canon is a semantic judgement a regex cannot make — the same reasoning the threat-model, blast-radius, and advocate decisions used to reject a hook); and the `auto` | `escalated` markers ride the orchestrator-owned `## Resolutions` trail (the Edit-ownership second carve-out), made safe by the same gate, not by discipline. **Source:** `doc/features/automode_plan.md`; `.ai-pm/research/automode_research.md`; `.ai-pm/arch/automode_arch.md` (value-vs-semantics split + lifecycle rationale, one-engine-two-scopes, the timer-honesty caveat, and the dual-backstop framing chosen in the arch note); canonical spec in `WORKFLOW.md` § `### Decision authority`; branch commits `0017be4` (research), `5c7b8de` + `82c8a8a` (plan + PM revision), `cd1b2a9` + `4f2926a` (arch note), `e23090d` (`### Decision authority` single source + Step 3.5 autonomous branch + PM-talk rider + Edit-ownership marker note), `18608a5` (`/pm-bootstrap` authority question + `pm-plan` per-feature override line), `7758c4a` (`pm-plan-checker` + `pm-auditor` `auto`-entry citation backstops), `f71001b` (README mode line).

### System invariants get a single index in the Behavioral contract, by-reference like threat → constraint

The `## Behavioral contract (taxonomies & invariants)` section gains a `### System invariants` index — the single entry point for a project's cross-cutting system invariants, so a top-down reader finds one list instead of three partial ones keyed differently (format / taxonomy invariants in the Behavioral contract, rule-shaped invariants as `SCn` in `## Security constraints`, journey-level preconditions in the per-journey `**Invariants:**` blocks). Each invariant is listed **by reference to its single home**: **inline** when the Behavioral contract is itself the home (a format / taxonomy / transport invariant it already states); **by `SCn` ID** when it is enforced as a security constraint (the rule text stays in `## Security constraints`, referenced with a human hint only); **by journey name** when it is a journey-level precondition (the rule stays in that journey's `**Invariants:**` block). The index never restates an identifier and the section keeps its existing taxonomy content unchanged. The full index role + examples live once in the template's `### System invariants` subsection (`doc/_templates/architecture.md.tmpl`); this record does not duplicate them.

**Why / key decisions.** The index applies the protocol's own *one-fact-one-owner / reference-don't-duplicate* and *threat → constraint-by-ID* patterns (see the Threat-model lifecycle decision above) to invariants: it points *out* at each home by `SCn` ID or journey name, the home keeps **no back-link**, and **no rule text is duplicated** (the only cross-doc datum is the bare ID or name). `SCn`-enforced invariants are **indexed by ID, never relocated** into the Behavioral contract — moving them would break the one-way threat → constraint-by-ID wiring, so the index references them where they live. The Behavioral contract already declares itself the single home for invariants (its title carries `& invariants`), so an additive subsection there satisfies one-fact-one-owner and **inherits the A4 exclusion for free** — the index is authored domain content with no external artifact to diverge from, so it is **not** added to the A4 cross-check set (the set stays exactly File layout ↔ tree, Release flow ↔ CI, Integration contract ↔ README) and never produces an A4 finding. `pm-architect` (already owner of `architecture.md`, `user-journeys.md`, and the `SCn` IDs) authors and maintains the index by reference; `pm-auditor` dimension 5 gains a **structural, presence-conditional, note-not-blocking** check — sibling to the existing threat ↔ constraint wiring check — that fires only when the project has invariant-bearing homes (**≥1 `SCn`** in `## Security constraints` **and/or ≥1 journey `**Invariants:**` block**, **not** gated on security-bearing) but no index, or when an index entry names a dangling `SCn` no constraint defines; it matches shapes and IDs only, never prose meaning, and remediates by spawning `pm-architect`. The change is additive with no migration and full back-compat — a project with no invariant homes sees nothing change; existing downstreams get at most the new note (never a block) at their next `/pm-audit`.

**Scope — first slice of an EPIC.** This is the first slice of the EPIC **cross-document-consistency auditor**; the sibling sub-classes are deferred to later slices: the single-source enum / taxonomy drift sweep (a taxonomy restated in journeys/contracts and drifted from its Behavioral-contract home), temporal-status conflation ("known limitation / planned" in one doc vs "done / current" in another), ADR ↔ stack-notes backing (every architectural decision cites stack-notes), state-machine ↔ journeys single diagram (one canonical diagram referenced, not redrawn per journey), and journeys ↔ threat-model UX (does a journey surface the mitigation a threat implies). ### Journey taxonomy-drift sweep: pm-auditor dimension 5 backstops the journeys move-not-copy discipline

`pm-auditor` dimension 5 gains a **journey identifier-restatement** check — a structural, presence-conditional, **note-not-blocking** finding that catches a taxonomy value / enum / identifier-grammar **restated in `docs/user-journeys.md`** (step bodies or `**Invariants:**` blocks) instead of referencing the `## Behavioral contract (taxonomies & invariants)` single home. Until now the journeys move-not-copy discipline (identifiers live once in the Behavioral contract; journeys reference it, never restate it — owned by `pm-architect`, stated in `doc/_templates/user-journeys.md.tmpl`) had **no auditor backstop**: the rule was advisory only, so a stale restated copy (the "6 statuses in the docs, 7 in the code" drift) went uncaught. This slice makes the discipline auditor-backed.

**Why / key decisions.** The new check is the **journeys-surface complement** of the existing structural-token note (the contract PM-sections + product-map value-line wire-token note already in dimension 5) — and it is **structurally disjoint** from it: the two read different files (`docs/user-journeys.md` vs contracts / the product map), so no token can fire on both and there is no double-flag. The wire-token vocabulary is **reused by reference, not re-encoded** — the new check points at the structural-token note's shape list rather than carrying its own copy (re-encoding the list would itself be the second-list drift this EPIC exists to remove, and self-parody in a move-not-copy slice). The declared-taxonomy exact-token match is **gated to backticked / fenced code-span tokens**: a backtick is the author asserting "this is a machine token", which keeps the check provably shape-not-meaning and avoids over-firing on common-word enums (`active` / `done` / `error` — the template's own example values are ordinary English words, so a bare-prose match would over-fire and, via the dimension-wide two-consecutive-audits → blocking rule, could escalate a false positive to blocking); a bare-prose restatement is therefore a **deliberate, documented miss**, the boundary of structural detection (semantic value-diffing is explicitly out of scope). The intended `## Behavioral contract` reference is **exempt** — it inherits verbatim the same relative-doc-reference carve-out the structural-token note already applies, so the correct move-not-copy form is never mistaken for a restatement. A **dangling Behavioral-contract reference** (a journey references the section where it is absent or `N/A`) is flagged too — the journeys-side sibling of the dangling-`SCn` check. The check sits in dimension 5 beside the threat ↔ constraint wiring and the System-invariants-index checks (same structural family, same shape-not-meaning + note-not-blocking discipline, same two-consecutive-audits escalation — not a new one); it is silent on a journeys-absent / identifier-free project (proportional) and remediates by spawning `pm-architect` to move the identifier into the Behavioral contract and reference it.

**Scope — slice 2 of an EPIC.** This is the second slice of the EPIC **cross-document-consistency auditor** (slice 1 = the System-invariants index, v2.21.0, above). The remaining sibling sub-classes are deferred to later slices, each its own plan: temporal-status conflation ("known limitation / planned" in one doc vs "done / current" in another), ADR ↔ stack-notes backing (every architectural decision cites stack-notes), state-machine ↔ journeys single diagram (one canonical diagram referenced, not redrawn per journey), journeys ↔ threat-model UX (does a journey surface the mitigation a threat implies), the NFR / operational-limits prompt, and the conditional state-model section. Explicitly out of scope: semantic value-diffing (which enum list is "right"), promoting the note to blocking (kept a note — subject only to the existing dimension-wide escalation), re-doing the contract / product-map structural-token note (untouched; referenced, not duplicated), and auto-fixing the restatement (the auditor only flags; remediation is a `pm-architect` move-not-copy). **Source:** `doc/features/taxonomy-drift-sweep_plan.md`; `.ai-pm/arch/taxonomy-drift-sweep_arch.md` (the code-span gate on the sc1 declared-taxonomy match, the by-reference vocabulary reuse, and the surface-split non-overlap chosen in the arch note); branch commits `6c62ea2` (plan), `d6f5dea` (sc1 backtick gate + Behavioral-contract-reference exemption in plan + arch), `3e5c894` (`pm-auditor` dimension-5 journey identifier-restatement check), `efdbf7c` (`pm-architect` move-not-copy rule noted as backstopped by `pm-auditor`).

### Conditional NFR / operational-limits prompt: scale/resource budgets get a home, split by audience

`/pm-plan` gains a **conditional NFR / operational-limits check** that closes a true missing-prompt: a resource-constrained / scale-bearing project had no prompt forcing *"what is this feature's / the system's resource and scale budget?"*, so a real budget went written **nowhere** — not a discipline lapse but an absent question. The check fires when the orchestrator judges (while planning) the feature **scale-bearing** — its core scales with a bounded count the system must hold (devices / endpoints / connections / sessions / subscriptions / messages-per-second / queue depth) — **OR** the platform **resource-constrained** (the signal *read*, not present, from `docs/stack-notes.md` or an "embedded / offline / must-fit-X" boundary in `docs/architecture.md` `## Architectural constraints`). It sits beside the existing conditional pre-draft checks (Stack-component / Interaction-scenario / Security-surface), same judgment-triggered proportional shape; it is **silent** on a non-scale feature on a non-constrained platform.

**Audience-split homes (one fact, one owner).** A declared NFR is routed by *who the number is for*, the protocol's standing two-layer / one-owner split: a **user-facing limit** (max devices / endpoints the product supports, perceived throughput) is a promise to the user → the feature's Product Contract `## Must not break` as a quantified invariant (a **product question** for the PM, per the existing product-vs-technical rule; the contract template gains one note line); a **resource budget** (RAM ceiling, boot-time, CPU headroom, system-level max-N) is an engineering bound → a **new conditional `## Operational limits & budgets` section** in `docs/architecture.md`, owned by `pm-architect`, authored / refreshed on the `/pm-plan` "Docs to update" post-coding handoff (the same trigger as this decision record). The `/pm-plan` prompt **references the new section by name**, never re-encoding its content.

**Why / key decisions (from the arch note).** The real fork was **a new section vs folding budgets into `## Architectural constraints`**: a quantified budget is a *distinct content shape with a distinct audience* (a sized number + unit + scope, read by an engineer sizing the platform — not a qualitative don't-violate rule), so it earns its own labelled, conditional, born-`N/A` home the way taxonomies earned theirs in the Behavioral contract. Folding into `## Architectural constraints` was rejected: it gives the prompt's "reference by name" no clean pointer target, mixes two content shapes a future check would have to prose-police apart, and bolts conditional-section semantics onto a non-conditional section. The new section is **A2-walked but A4-EXCLUDED** — authored engineering content with no source-tree / CI / README artifact to diverge from, so it inherits the same A4 cross-check exclusion the `## Behavioral contract` and the `### System invariants` index already carry (the A4 set stays exactly File layout ↔ tree, Release flow ↔ CI, Integration contract ↔ README); a budget the orchestrator cannot quantify without real platform knowledge is recorded as `[?]` for the PM, **never invented** (the verbatim Behavioral-contract "never invent to fill" discipline — a confabulated "RAM ≤ 64 MB" reads as authoritative and is worse than `[?]`). The trigger is **irreducibly a semantic judgement** — there is no durable "file present ⇒ resource-constrained" signal analogous to threat-model.md, so the resource-constrained read is gated to the two named sources and **no hook** is added (the same no-hook reality as the product-readiness / threat-model / blast-radius family).

**No hard gate this slice — and a did-it-*at-all* gate would be incoherent, not merely over-built.** A `pm-plan-checker` DoD block or `pm-auditor` presence-note "did the NFR check fire when it should have?" would have to **re-derive the semantic trigger** the orchestrator already judged — the exact judgement no checker can evaluate (this is why the Security-surface DoD block *can* exist: it gates on a *presence* signal, both greppable). The accepted trade-off: the prompt **degrades silently** if the orchestrator misjudges the trigger — acceptable because the homes make a *recorded* budget durable and auditable at the next refresh. A narrow **did-it-*right*** check ("if NFRs were declared, are they in the right home?") *is* coherent (presence-keyed, shape-not-meaning — the direct sibling of the dangling-`SCn` and slice-2 journeys-drift checks) and is the natural **slice-4 candidate**, but is deferred: there is no non-`N/A` corpus to tune it against until the homes ship and downstream projects record into them (the premature-tuning trap), and the EPIC's home-first / check-next cadence puts it in a later slice.

**Scope — slice 3 of an EPIC.** This is the third slice of the EPIC **cross-document-consistency auditor** (slice 1 = the System-invariants index, v2.21.0; slice 2 = the journey taxonomy-drift sweep, v2.22.0). This slice is the **resource/scale-budget sub-class** — one of the three whole-system-property gaps the downstream review surfaced (invariants index, NFR / operational-limits, conditional state-model) — *within* that EPIC, not a competing EPIC name. It is scoped to the **resource footprint + scale/capacity budget** sub-class exactly as the backlog names it (RAM, boot time, CPU, max devices / endpoints, throughput). Deferred siblings, each its own plan: the conditional **state / event-model section** (a `docs/architecture.md` state-model section + a `/pm-plan` trigger when the feature's core is a state machine / protocol); the other **NFR classes** — **reliability / availability** (uptime, failover, recovery SLOs) and **latency SLOs** (p50/p99 bounds, distinct from throughput); **security-as-NFR** is *not* re-homed here (already owned by `docs/threat-model.md` + `## Security constraints`). Additive with no migration — existing downstreams gain the prompt at their next `/pm-plan` and the `N/A` section at their next architecture refresh, never a retroactive re-home sweep. **Source:** `doc/features/nfr-operational-limits-prompt_plan.md`; `.ai-pm/arch/nfr-operational-limits-prompt_arch.md` (the new-section-vs-fold fork, the A2-walk / A4-exclude treatment, the `[?]`-not-invented discipline, and the no-hard-gate altitude with the did-it-right slice-4 deferral chosen in the arch note); branch commits `cb77385` (plan + arch note), `100955c` (`/pm-plan` conditional NFR check + `architecture.md.tmpl` `## Operational limits & budgets` section + `contract.md.tmpl` note line + `pm-architect` A2-walk / A4-exclude wiring).

### Autonomous procedural-gate progression: announce-and-proceed through routine pipeline gates, derive-or-escalate only at genuine product forks

The shipped automode (v2.20.0, above) relaxed **only** the per-feature product-fork question — the `pm-product-advocate` gaps inside an already-chosen feature. It never relaxed the pipeline's **routine procedural gates**, so an `autonomous` project still blocked on every one of them: which feature to build next, *do you approve this plan?*, the architect-review offer, the retrospective/audit + pending-migration nudges, the contract-existence question. The PM observed exactly this — an already-`autonomous` project hitting *both* "which feature?" (on idle-after-merge) and "approve the plan?" (after a draft) — which contradicts the automode vision the backlog records (`.ai-pm/backlog.md:80`: front-load at bootstrap, then carry to the end with **no further questions**). **Autonomous procedural-gate progression** closes that general gap — not one instance of it — by giving the `### Decision authority` engine a second, scope-extended rule beside the product-fork resolution it already encodes: in `autonomous` mode a **routine procedural checkpoint** becomes **announce-and-proceed**; a **genuine product fork** still uses the unchanged derivability test (auto-resolve from cited canon or escalate); **merge/ship stays manual**. This is a **graded extension of the one engine** — it reuses the derivability test, announce-before-act, the escalate-regardless cap, and the merge-manual invariant by name; **no new engine, no new enum value, no migration**, and interactive mode is byte-unchanged. The full rule is single-sourced in **`### Decision authority` in `WORKFLOW.md`**; this record captures *what* the extension is and *why*, and references that subsection by name rather than re-encoding it.

**The dividing line — "does it decide what the user gets?"** This is the load-bearing classification, and it is stated once at the single source. A gate that asks *"ok to proceed? / which order? / want this optional step?"* and does **not** decide what the user gets is **procedural** → announce-and-proceed (state the decision + a brief cited rationale on the existing announce-before-act console line — `… (proceeding — interrupt to override)` — then proceed; no `AskUserQuestion`). A choice that **shapes what the user gets** is a **product fork** → derive-or-escalate, exactly the shipped automode, untouched. The named procedural instances are: feature-selection, plan-approval, the architect-review offer, the retrospective/audit + pending-migration nudges, and the contract-existence question. Announce-and-proceed **sequences/advances PM-authored work**, it never invents product direction — every announced decision carries a cited rationale (the backlog item / mandate passage / the rule that fired), and a procedural decision **escalates** (one `AskUserQuestion`, reusing the escalate-regardless cap) only when it is **not derivable**, or collides with a high-stakes / irreversible / security-surface concern.

**Feature-selection — one named procedural instance (idle + bootstrap, two entry points).** When `.ai-pm/state/current.md` is `idle`/`done` (no feature in flight) on an `autonomous` project with open backlog items, the orchestrator **selects** the next feature (the open candidate most aligned with the bootstrap mandate + recorded priorities), **announces** it, and proceeds into `/pm-plan`; the same path serves `/pm-bootstrap`'s "before the first feature" transition (one fork, two entry points — only the candidate-set computation is scope-dependent, the select→announce→proceed path is identical). Because the trigger is the **idle-after-merge** transition, the PM's manual merge of each feature naturally **paces** successive selections — **no numeric "max N features" cap** is needed. The escalation trigger is **absence of a derivable tiebreak from cited canon — not absence of a formal rank** (a backlog the mandate disambiguates is not ambiguous; a formal priority-ranking model is explicitly out of scope): ≥2 candidates with no citable tiebreak, or an empty backlog, ⇒ escalate. Selection **precedes** the feature, so it has no advocate `## Resolutions` trail to ride; it records on the selected plan's `Source:` provenance line (the plan-level provenance line **defined by this feature** in the plan format, not a parallel field — `selected autonomously per ### Decision authority; source: <backlog item / mandate passage>`) plus the announce line + a note in `.ai-pm/state/current.md`. **Backstop (mirrors automode's `auto`-entry citation guard):** `pm-plan-checker` carries one greppable presence assertion — a plan whose provenance says `selected autonomously` **must** carry a `source:` token (shape-not-meaning; the PM owns whether the source justifies the pick, at PR review). This makes a mis-recorded selection **non-silent** at the same altitude automode chose, the symmetry the arch note found load-bearing rather than cosmetic.

**Plan-approval auto-proceed does NOT skip the product-fork checks.** In `autonomous` mode the `/pm-plan` Handoff "Show the draft to PM. It is safe because the plan is committed, independently reviewed, and the **PR is reviewed before the manual merge**. The other optional/offer gates (architect-review, retrospective/audit, pending-migration, contract-existence) likewise gain a one-line auto-decide-and-announce branch — each references the rule, none decides product direction, so none escalates by itself. Throughout, **merge/ship stays manual** in both scopes (Step 6 A/B/C unchanged; the autonomous selection slots at the idle-after-merge point, never inside the ship gate) — this is the load-bearing safety that makes announce-and-proceed sound, and the **advocate product-fork flow is untouched**.

**Scope.** A direct extension of the shipped automode (v2.20.0) — second rule on the one `### Decision authority` engine, additive, no migration. ### Conditional State-model section: a state machine / protocol gets a single referenceable home, by transitions-not-states reference

`docs/architecture.md` gains a **conditional `## State model` section** and `/pm-plan` gains a **conditional State-model check** that closes the last of three whole-system-property gaps the downstream review named: a state / event model that is implemented and justified but **embedded in prose** — status values, transitions, recompute / transition triggers, debounce written as paragraph-form "why" — forces a reader to reverse-engineer "the state machine" from narrative. The new section is a **states×transitions table + a triggers / debounce list** that gives that machine a single labelled home. The check fires when the orchestrator judges (while planning) that the feature's **core is a state machine / protocol** — a status ladder with defined transitions, a named-state lifecycle plus the events that move between states, or a wire / connection protocol with message / connection states — an **irreducible semantic judgement**, not a hook and **not** keyed on a status enum alone (many projects carry a status field without their feature core being a state machine; that would over-fire). It sits beside the existing conditional pre-draft checks (Stack-component / Interaction-scenario / Security-surface / NFR-operational-limits), same judgment-triggered proportional shape; it is **silent** on a non-state-bearing feature.

**Column-level drift-bound (one fact, one owner — the load-bearing discipline).** A states×transitions table names the states, so it touches the same value set as the `## Behavioral contract` status enum. The bound that keeps it from being a partial restatement is a **column-level split of ownership**, the same one-fact-one-owner discipline the threat → constraint and journey → Behavioral-contract splits already use: the **`## Behavioral contract` status enum owns the value set** (which states exist, what each means) — the table's state column is a *reference into* that enum, never an independent declaration; the **State model owns only the transitions + triggers** (which state moves to which, on what trigger, with what debounce) — its own new datum, existing nowhere else. This is the section's one-line guidance and `pm-architect`'s section guidance; the existing prose's "why" justification stays where it is and is referenced, never re-derived (Variant A, by reference).

### Review typology: a single-sourced, layered review-type registry with proportional new-code gating (EPIC slice 1 — smell/hygiene built, three types registered)

The protocol reviewed every change **by diff** and audited **compliance** periodically — but it had **no whole-codebase, multi-type review discipline**: "is the codebase clean overall?" was unguaranteed, because per-diff review never re-examines already-merged code and the compliance audit checks process adherence, not code quality. **Review typology** closes that gap by establishing review as a **layered system** rather than one undifferentiated pass — a single-sourced **`### Review typology` registry in `WORKFLOW.md`** naming the review **TYPES**, each described by **cadence · depth · scope · deterministic-half · AI-half**: **per-diff** (built, every change — lint + low/med AI semantic); **smell / hygiene** (built this slice, frequent — detection[det, future hook] + prioritization/root-cause[AI]); and **architectural**, **functional / integration**, **criticality-prioritization** (each **registered here as a named later slice, visibly marked "not built"**, with its cadence/depth/scope/det-vs-AI sketched so a future slice slots into the agreed shape). The registry is **research-backed** — a deep-research pass validated the five-type taxonomy as a coherent layering and the dominant cadence rule across mature tooling (SonarQube *Clean-as-You-Code*: gate on new/changed code, do not re-review already-clean code). Consumers **reference `### Review typology` by name** and re-encode neither the type list nor the cadence rule — the same single-source discipline as `### Decision authority` / `### Project kind`.

**Key decisions (from the arch note).** (1) **Registry home = `WORKFLOW.md`, not `/pm-audit`.** The typology is a *discipline* spanning multiple consumers (the per-diff type is a `/pm-plan` review-loop concern; the architectural / functional types are future cross-command concerns), so it lives **above** any one consumer — exactly as `### Decision authority` does not live inside any one command. `/pm-audit` is only the **runner of one type** (the smell sweep); the `/pm-audit`-section-as-registry alternative was **rejected**. (2) **Proportional new-code gating** (research-validated Clean-as-You-Code): the smell sweep scopes to **(a) changed since the last sweep** (`git diff <last-sweep-sha>..HEAD`) **+ (b) never-swept areas** (legacy onboarded via full-mode bootstrap, documented but never diff-reviewed) **+ (c) a periodic full re-sweep**, and is **silent** when nothing un-swept remains. The **last-sweep marker = a `## Quality sweep: <date> — swept <sha>..HEAD at depth <d>` line in the `audit-*.md` report** the sweep already produces — **NO new file**: it extends the auditor's existing derive-cutoff-from-the-latest-`audit-*.md` pattern (`pm-auditor.md` step 2) from date-only to date+SHA for the sweep line only, and **reuses the auditor's "first run = full" fallback verbatim** (no `## Quality sweep` line anywhere ⇒ full re-sweep — no parallel fallback invented). **Marker↔audit-report coupling (the explicit constraint):** the marker exists only where an `audit-*.md` does, which is safe **today** because the sweep's sole trigger is the `/pm-audit` full-scope hook (always co-produces a report); a **future standalone sweep trigger** (outside an audit) would have no report to carry the line and would need a **dedicated marker home** (the rejected Variant-B `.ai-pm/state/` file) — flagged here so the constraint is visible, not discovered later. (3) **Selectable depth** (low → ultra) for economy — the sweep **drops the hard-wired `ultra`**: interactive presents the cost/depth trade-off and the PM picks; autonomous picks a proportionate depth and announces (never silently the costliest), PM can override. (4) **Per-type deterministic-vs-AI split** as the concrete landing of **backlog #211**: each type names its deterministic half (an enumerable catalog a hook/linter runs every time) and its AI half (judgement at the selected depth). For the **smell type** the deterministic detection half is **named as a downstream/future hook, not built** — this markdown-prose protocol repo has **no linter to host smell-*detection*** (so building it now is impossible, not deferred-by-choice; the row reads "det half = a future smell-detection hook, downstream code projects, #211"), and slice 1 ships only the **AI** sweep (prioritization / root-cause via the `code-review` skill).

**Reuse, not re-encode (what this slice does not duplicate).** The smell sweep strengthens `/pm-audit`'s existing `## Technical quality (full scope only)` hook **in place** (no new standalone command — rejected): it references `### Review typology` by name, runs the unchanged **`code-review` skill** as the engine over the proportional scope, routes findings through `/pm-audit`'s existing **fix-now / next-sprint / accept-with-context triage** (the `accepted (quality-sweep-<date>)` backlog marker mirrors `accepted (auditor-<date>)` verbatim), and rides the **`### Decision authority` procedural-gate progression** (v2.24.0) for the autonomous offer — **bounded by the proportionality gate** (auto-runs only when there is un-swept/changed surface, never a full-tree ultra sweep every audit; **merge/ship stays manual**). The per-diff `code-review` and the compliance audit are **untouched**; the registry states the boundary keeping the types distinct (smell ≠ bug; smell-sweep ≠ compliance audit ≠ the cross-model-review backlog item). No `.claude/settings.json` / hook / template / `MIGRATIONS.md` change — **additive, no migration**: existing projects gain the proportional smell sweep at their next full `/pm-audit`, whose first run records the baseline marker.

### AI-specific minimums are deterministically linter-enforced, not AI-self-policed (the #211 deterministic half, made real)

The protocol **declared** the **AI-specific minimums** (the ones declared in `### AI-specific minimums`) as conventions in `docs/architecture.md` + `CLAUDE.md`, and *expected* the project's `<lint command>` to enforce them — but it never **guaranteed** the linter actually encoded them. So they degraded to **AI-self-policed**: a diff that crossed a minimum still passed lint + per-diff review. This decision makes the minimums **deterministically enforced by the downstream project's real linter, run every diff**, not self-policed. The split of responsibility is three-layer: the **numbers stay single-sourced** in `### AI-specific minimums`; `pm-stack-researcher` produces the per-stack **AI-minimums→linter-rule mapping** into `docs/stack-notes.md` (each minimum → the concrete rule that encodes it, doc-URL cited, **referencing** the numbers, never re-stating them); the **linter config is the enforcement**. A minimum the stack's linter cannot express is recorded **convention-only**, explicitly — backstopped by the AI per-diff/smell review (the `### Review typology` smell type, referenced by name, not paraphrased), so nothing is silently dropped.

**Key decisions (from the arch note — Variant A).** (1) **The reviewer check extends the existing dim-9 "validator present + run" mechanism across both cadences — no new gate, no new dimension.** The AI-minimums linter rules *are* validators once the mapping wires them, so they fall under the existing present-+-run discipline: `pm-plan-checker`'s **per-diff Pipeline-green DoD** is clarified to require that the `<lint command>` **encodes** the minimums per the stack-notes mapping (not just a default lint — the per-diff guarantee is free, the linter already runs every diff), and `pm-auditor` **dimension 5** gains a periodic **non-blocking note** for the un-wired-project case (a project declares the minimums but its lint config does not encode them). This mirrors the existing two-cadence validator discipline (rule-text single-homed in `CLAUDE.md` Pipeline; run by `pm-coder`; checked per-diff by `pm-plan-checker`; re-checked periodically by `pm-auditor`) rather than forking a parallel gate. `code-review` is **not** a third owner — it is the per-diff AI-semantic half backstopping *unexpressible* minimums. (2) **Single-source holds — a config parameter is an enforcement encoding, not a forbidden re-declaration.** A linter-config value that *carries* the number (`max-module-lines=300`) is an enforcement parameter, by the same principle as threat→`SCn` and journey→Behavioral-contract; the distinction the prose must hold is that the config may **contain** the number to enforce it, but must **not** become a second place a reader is *told what the minimum is* (no competing prose authority outside `### AI-specific minimums`). (3) **The protocol stays stack-agnostic.** The per-stack linter rules (pylint `max-module-lines` / mccabe / `R0801`, eslint `max-lines` / `complexity`, golangci-lint `funlen` / `gocyclo`, …) are `pm-stack-researcher`'s output, **examples only**, derived per project from the linter's docs — **never hardcoded** in the core (`/pm-bootstrap` carries them as illustration, not a fixed list).

**Why.** The protocol *declared* the AI-minimums but never *enforced* them, so they degraded to AI-self-policed. The **DriveBox uber-review (#224)** showed `cli.py` growing to 331 lines past the **max-source-file** minimum over many diffs — because nothing **ran** the rule; the per-diff coder/review never failed on it. This decision is the **deterministic half of the #211** deterministic-enforceable-vs-AI-evaluated boundary made real, and the bootstrap-wiring of **#218** (AI-minimums risk being AI-self-policed).

**Ties + scope.** The deterministically-linted minimums are the **per-diff deterministic half** the `### Review typology` registry already names; the **smell type** (review-typology slice 1) owns the accumulated / cross-module cases a single diff's lint cannot catch (cross-module duplication) — this decision fills in the per-diff deterministic half the registry pointed at. Backlog: **#211** (the det-vs-AI boundary), **#218** (the self-policing risk), **#224** (the DriveBox `cli.py` evidence). Additive, no migration: new projects get the AI-minimums-encoded linter at bootstrap; existing downstreams are flagged as a **note** at their next `/pm-audit` (PM opts in — never a retroactive forced rewrite). **NB — this protocol repo itself is markdown-prose with no linter**, so it cannot dogfood the enforcement, only the *discipline* (the AI-minimums numbers stay stated here; no linter config is forced on this repo). **Source:** `doc/features/ai-minimums-linter-wiring_plan.md`; `.ai-pm/arch/ai-minimums-linter-wiring_arch.md` (the dim-9-extension-across-both-cadences fork → Variant A, the three-layer single-source treatment with the config-parameter-is-encoding distinction, the honest-partial #211 boundary tied by name to the `### Review typology` smell type, and the stack-agnostic researcher-derives-not-hardcoded split); branch commits `7412a28` (feat — `pm-stack-researcher` AI-minimums→linter-rule mapping + `/pm-bootstrap` stack-setup wiring), `517ae54` (feat — `CLAUDE.md.tmpl` + `architecture.md.tmpl` notes: enforced-by-linter, not self-policed), `0080a6a` (feat — extend dim-9 validator discipline to AI-minimums encoding across both cadences: `pm-plan-checker` per-diff DoD + `pm-auditor` dim-5 periodic note).

### WORKFLOW.md progressive-disclosure boundary: a freeform-reasoning rule keeps its kernel in the always-on core

### Review-engine selection: the orchestrator becomes a legitimate engine; per-diff stays built-in by routing, not by a hook deny (supersedes deny-review-orchestrator)

`wb-development:code-review-orchestrator` was **hard-denied** by a `PreToolUse` arm in `.claude/settings.json` (the v2.25.1 `deny-review-orchestrator` decision), with a `WB_ALLOW_REVIEW_ORCHESTRATOR=1` per-session launch-escape, because its broad auto-trigger could intercept the protocol's own per-diff `/code-review` (Pass-2). This decision **reframes** the orchestrator from a denied skill into a **legitimate engine** and moves the per-diff guarantee from a hard hook to protocol routing. Concretely: (1) the orchestrator `case` arm **and** its `WB_ALLOW_REVIEW_ORCHESTRATOR=1` escape guard are **removed** from the `PreToolUse` command — the skill now falls through to the trailing `exit 0` (no decision; normal permission flow), so anyone (PM or operator) may invoke it anytime; the **7 `wb-*` role-duplicator denies stay byte-identical** (only the orchestrator arm leaves the deny). (2) The **per-diff Pass-2 stays on the built-in `/code-review`** by **protocol routing discipline** — the automated per-diff loop uses the lighter built-in engine that suffices for a single diff; it never auto-escalates a routine per-diff review to the orchestrator. This is no longer a hook block but the routing rule plus the existing soft `UserPromptSubmit` route-reminder. (3) The **`/pm-audit` whole-codebase smell/hygiene sweep** now **prefers the orchestrator when available**, falling back to the built-in `/code-review` (at the selected depth, `ultra` for a deep/first/legacy sweep) when it is **unavailable**; the environment switch **`WB_REVIEW_ORCHESTRATOR=off`** forces the built-in fallback even when the orchestrator is installed (cost control). The engine-selection rule itself is **single-sourced** in `### Review typology` in `workflow/review-typology.md` and referenced by name from `.claude/commands/pm-audit.md` and `workflow/enforcement.md`, never re-encoded.

**Supersedes `deny-review-orchestrator` (v2.25.1).** The hard per-diff deny is **lifted** — the orchestrator is removed from the deny-list and is no longer a denied skill — and the per-diff → built-in guarantee now rests on **protocol routing discipline + the soft route-reminder**, not a `PreToolUse` deny. The prior decision's record in `doc/features/deny-review-orchestrator_plan.md` is left intact (history stays honest); this record supersedes it rather than rewriting it.

**Security framing.** `.claude/settings.json` is a **security-relevant hook** (it gates agent/skill routing and the boundary/force-push/no-verify guards), so **removing a deny arm was a deliberate PM decision** (2026-06-05: "remove the block on `code-review-orchestrator` so it is legally usable; within the protocol's own movement it just must not be auto-invoked on diffs"), reviewed in the Pass-2 loop with the settings-file's security relevance called out. The change is provably isolated: the `PreToolUse` hook is stateless (each invocation reads its own stdin + env and exits), so removing one `case` arm + the escape guard cannot loosen the other arms — verified by the `tests/hooks.sh` cases (`code-review-orchestrator is NOT denied (let-through)` and `wb-development:coder still denied`, 73/73 green). This repo's `## Security constraints` stays `N/A` (the template runs no service and the hooks are gated by Claude Code's own permission flow); the security relevance here is the *review discipline* applied to editing the hook, not a new runtime constraint. **Source:** `doc/features/review-engine-selection_plan.md` (backlog #372/#378 engine-selection residue of #269; PM directive 2026-06-05); single-source rule in `workflow/review-typology.md` § `### Review typology`; consumers `.claude/commands/pm-audit.md` `## Technical quality` + `workflow/enforcement.md`; deterministic hook half tested in `tests/hooks.sh`.

### Cross-model review: review and audit run on a different model than the session by default, for independent blind spots (the model axis, orthogonal to engine-selection)

The protocol planned, coded, **and reviewed/audited** every change on **one** model — the running session model — so a blind spot the maker-model has is a blind spot its own review shares. **Cross-model review** adds a project-level config that runs review and audit on a **different** model from the session *by default*: *plan and code on the session model; review and audit run on a different model, because a different model has different blind spots and catches what the maker-model misses*. This is the **model axis** — *which brain runs the review* — **orthogonal** to the already-shipped **engine axis** (review-engine-selection, above: *which reviewer* — built-in `/code-review` vs the orchestrator skill) and to the **idioms/rules axis** (the deferred canonical-idioms library, feature C of the reviewability track). The cross-model default is opinionated, and the value was **proven live**: an Opus authoring pass plus an Opus Pass-1 missed **five real consistency / single-source findings on this very feature** that a cross-model **Sonnet** Pass-2 review then caught — and a cross-model Sonnet review had earlier caught a real gap in `agent-reporting-discipline` the same-model passes missed. Same-brain review shares the maker's blind spots; a different brain does not.

**Four settings, value home `.ai-pm/review-config.md` (mirroring `.ai-pm/decision-authority.md`).** Three model settings — `review-diff-model` (per-diff Pass-2, every feature), `review-full-model` (the whole-codebase `## Technical quality` sweep), `audit-model` (the `pm-auditor` compliance audit) — each taking `session | auto | opus | sonnet`, plus one scope setting `review-scope` (`auto` = all per-diff changes get the non-session review model | `high-risk` = only a security-surface / irreversible / PM-flagged change does, others stay on the session; default `auto`). `review-scope` gates the **per-diff path (`review-diff-model`) ONLY** — the whole-codebase sweep (`review-full-model`) runs regardless of `review-scope` (it is not a per-diff path, so scope never applies to it). The model settings **default `auto`** — an **opinionated default**: review and audit run on a model **different from the session** out of the box, and **absent file OR unrecognized value ⇒ `auto`** (a new project, or one bumping the template, gets cross-model review by default — the back-compat shape mirrors `### Decision authority`'s `absent ⇒ interactive`, but the *value* is the opinionated cross-model, not the no-op). `auto` resolves to **a review-capable model different from the session** (real diversity, never the same brain twice). **Haiku is blacklisted for review** — the enum omits it, `auto` never selects it (even a Haiku session resolves `auto` to a stronger reviewer), and an explicit `haiku` is **refused with a warning and treated as `session`**.

**Why `auto` is the default (the opinionated-default justification).** The cross-model default is an intentional, argued choice, not a coin-flip: **(a) the value is proven live** — a cross-model Sonnet Pass-2 caught **five real consistency / single-source findings on this very feature** that the Opus authoring pass and Opus Pass-1 both missed (and earlier caught a real gap in `agent-reporting-discipline`); a same-model review structurally shares the maker's blind spots, a different model does not. **(b) The mechanism is spike-verified** (below), not doc-claimed. **(c) `auto` → Sonnet from an Opus session is *cheaper*** as well as more independent — the diversity is not bought with a cost penalty. **(d) The fallback makes it safe** — if the resolved model equals the session or is unavailable, review/audit silently runs on the session model and the orchestrator announces the fallback, so the default **never breaks** a project. The default is **non-destructive** (only *which model* reviews changes — never *whether* review runs, never the `## Code review` stamp gate), **announced** at every review, and **reversible** with one line (`review-*-model: session`).

**Mechanism — a model-pinned subagent, spike-verified for the built-in path (not doc-claimed).** Review/audit runs in a subagent pinned to the resolved model via the Claude Code Agent/subagent model override; the findings land in the existing trail / stamp unchanged (the `## Code review` stamp gate untouched). Per setting:

- **`review-diff-model`** runs the **built-in `/code-review`** inside a pinned subagent — **execution-verified by a throwaway spike** (a subagent reported `claude-sonnet-4-6`, `/code-review` available, produced a real finding), the load-bearing idiom proven by running it, not asserted from docs (dogfooding the integration-risk-spike-gate principle).
- **`review-full-model`** pins the existing `## Technical quality` engine chain **where the engine is ours**, branching on whether the orchestrator skill is installed:
  - **`code-review-orchestrator` installed** → run it in a subagent pinned to `review-full-model`; the skill has **no model of its own**, so it runs on the pinned model — **pinnable**.
  - **No orchestrator** → the orchestrator **offers** the PM (`AskUserQuestion`, at `/pm-audit`): **(a) `/code-review ultra`** — cloud multi-agent, broad, **model not ours** (announce "ultra picks its own models"; the `review-full-model` pin does **not** apply); **or (b) the built-in `/code-review` over the whole codebase** — **pinnable** to `review-full-model`, so cross-model works here without the orchestrator. The PM picks breadth (`ultra`) vs cross-model-pinned whole-base (built-in). `ultra` is the **one non-pinnable path** (the deliberate cloud exception); every other full-sweep path is pinnable.
- **`audit-model`** pins `pm-auditor` directly, since it is the protocol's own Agent spawn.

**Fan-out verify-before-rely caveat.** That the `code-review-orchestrator` skill's **fan-out** subagents inherit the pinned model is recorded as a **to-verify, not asserted as fact** (spike-gate discipline — the spike covered the built-in path, the orchestrator fan-out is unverified). Until verified, the per-review announce must reflect **only what is actually pinned** (the top-level orchestrator subagent) — never a blanket "all on Sonnet" while the fan-out may still be running on the session model.

**No restart, command-scoped change dialog, MANDATORY per-review model-announce.** The orchestrator reads `.ai-pm/review-config.md` **fresh right before each review/audit** (never cached at session start), so a config change applies **from the next review with no session restart** — the review runs in a spawned worker pinned to the configured model, independent of the session (the only thing a restart governs is the *session* model itself, a Claude Code relaunch, unrelated to this config). The PM does not hand-edit the file in the normal flow: a command opens an `AskUserQuestion` dialog that **asks only what the command did not already name** (both named → no dialog, confirm + write; which named, value missing → one question on that one setting; unspecified → multi-select which setting(s), then value(s)).

**The review/audit model is ALWAYS highlighted at the moment of invocation — a hard, every-path transparency requirement.** A review or audit is **never** run silently with respect to its model, so the PM is never left guessing which brain is reviewing. This holds on **every** path — the per-diff review, the whole-codebase full sweep, AND the compliance audit — and across **every** model case: the **cross-model** case names the exact model (e.g. "Code review on Sonnet (independent of your Opus session)"); the **fallback** case (resolved model equals the session or is unavailable) runs on the session model and announces "no cross-model this run (review model = session / unavailable)"; and the **ultra** cloud path announces "ultra — picks its own models". Review/audit always still runs — a config value never blocks it (never an error, never false diversity); the announce is the transparency, not a gate.

**Single-sourced, no hook, sharpens #101.** The rule — the enum, `auto` semantics, the default, the mechanism, the fallback, the Haiku-blacklist, the change UX — is single-sourced **once** as `### Cross-model review` in `workflow/review-typology.md`; every consumer (`workflow/pipeline.md` Step 5 Pass 2, `.claude/commands/pm-audit.md` `## Technical quality` + the `pm-auditor` spawn, `.claude/commands/pm-bootstrap.md`, `MIGRATIONS.md`, the change-dialog + announce in `workflow/pm-comms.md`) **references it by name** and re-encodes none of it — the same single-source discipline as `### Decision authority` / `### Review typology`. There is **no `PreToolUse` hook**: model selection is config + routing, the **soft-config family** alongside the `decision-authority` `mode:` (a model choice is not a regex-decidable safety boundary).

**Downstream migration — a documented default-change, NOT an interactive setup-offer.** On bump, an existing downstream project with **no `.ai-pm/review-config.md`** picks up **absent ⇒ `auto`**: review and audit begin running cross-model (a model different from the session). This **is** a behavior change, stated honestly in `MIGRATIONS.md` (`### Pending-migration detection` detects the absent file; the `### Expected-discipline`-style note documents it) — but it is **non-destructive** (only the review *model* changes; review still runs, the `## Code review` stamp gate is untouched), **announced** at the first review ("Code review on Sonnet…"), and **reversible** with one line (`review-*-model: session`). Deliberately there is **no interactive offer, no re-nag, and no opt-out-state to record** — the default simply applies and is documented (here, the CHANGELOG, and the migration note). A project that wants the old session-model behavior writes `review-*-model: session`. This is a change from the earlier design's interactive `/pm-audit` / `/pm-plan` setup-offer: the opinionated `auto` default removes the need for an offer, and with it the dangling-`/pm-plan` and re-nag problems the cross-model Pass-2 review surfaced. This **sharpens backlog #101** (cross-model review) from an idea into project-level config. **Pass-1 (`pm-plan-checker`) stays on the session model** — out of scope v1; the narrative is review + audit, and a custom reviewer / forcing the session model itself / a hard gate were all rejected. **Source:** `doc/features/cross-model-review_plan.md` (PM directive 2026-06-05; sharpens #101); single-source rule `workflow/review-typology.md` § `### Cross-model review`; value home `.ai-pm/review-config.md`; consumers `workflow/pipeline.md` Step 5, `.claude/commands/pm-audit.md`, `.claude/commands/pm-bootstrap.md`, `MIGRATIONS.md`, `workflow/pm-comms.md` (`/pm-plan` is **not** a consumer — no per-feature setup-offer exists); mechanism execution-verified by the recorded spike (no `docs/stack-notes.md` in this markdown-prose repo — the spike is the verification of record).

### Changeset hygiene: the diff carries only feature-serving hunks, the reviewer notes diff-noise, and authored human-facing text is read-before-ship (feature A of the reviewability track)

Motivated by the PM constraint *the PM does not read code, but their colleagues do* — discipline the protocol so the changeset is not painful for a human to review. Three disciplines land together: (1) **Diff-scope** — `pm-coder` keeps the changeset to **only hunks that serve the plan**, folding in **no** cosmetic-only churn (whitespace, reformatting of untouched lines, reordering, opportunistic micro-optimizations) **even when harmless**; a worthwhile unrelated change goes to the report (→ backlog), never the diff. This sharpens the prior "note unrelated, don't fix" rule from *functional* fixes to *cosmetic* too. (2) **Reviewer backstop** — `pm-plan-checker` surfaces hunk-level cosmetic noise as a **non-blocking structural note** (the wire-token-note shape), so the orchestrator/PM can prune or consciously keep the hunk. (3) **Legibility** — durable human-facing text the protocol authors (CHANGELOG, PR body, decision records, code comments) is read-before-ship and rewritten-if-unclear, never pasted verbatim; this rule is **single-sourced** as `## Human-facing text legibility` in `workflow/pm-comms.md` and referenced **by name** from `pm-coder` (comments) and `pm-pr-prep` (CHANGELOG/PR), never re-encoded.

**Soft enforcement, no hook.** "Is this hunk feature-related" is a semantic judgement a regex guard cannot make, so the diff-noise enforcement is a **structural note, not a hook or a hard block**, and **necessary incidental edits are explicitly NOT flagged** (a rename's call-site update, an import the new code needs, the line a real edit sits on) — the same "categorical, not strip-everything" boundary the wire-token note uses. This places the feature in the established **soft-enforced + single-sourced** family alongside the wire-token lint, the threat-model lifecycle, the blast-radius preflight, and the review-stamp gate.

**Sequencing.** This is feature **A** of a PM-sequenced reviewability track. **B** (deterministic linters: file≤300 / function≤50 / complexity / dead-code / duplication wired into bootstrap — backlog #218/#211) and **C** (canonical-idioms / Semgrep-rules library — backlog #227, needs `/pm-research` first) are the next, separate features and are out of scope here. The deeper comment-restraint rubric (backlog #386) is also separate — A's legibility rule is the lighter read-before-ship / rewrite-if-unclear discipline only. **Source:** `doc/features/changeset-hygiene_plan.md`.

### Test-wiring-parity: a wiring-dependent feature must test the production registration path, enforced by pm-plan-checker

A feature whose correctness depends on **initialization / registration / wiring order** — a provider registered on a shared singleton, a side-effect import, a plugin/factory hook, env/DI setup — must carry **≥1 test that drives the same registration path the production entry point uses** and asserts the observable post-condition that path produces (e.g. `registerX()` → `container.has(X)` is true), **not a hand-rolled equivalent setup**. A test that wires the dependency differently from how `main` does measures a path the app never takes, so it can stay green while production is broken. The rule is added to `/pm-plan`'s Test-plan rules and `pm-plan-checker` **blocks** (Implementation compliance) a wiring-dependent feature whose tests bypass the production registration path.

**Judgement-triggered, single-sourced, no hook.** The trigger — "does this feature's correctness depend on init/registration/wiring order?" — is a semantic judgement, the same shape as the NFR / state-model / security-surface triggers, so there is **no hook**: the rule text lives **once** in `/pm-plan`'s Test-plan rules and `pm-plan-checker` references it **by name**, enforcing presence without re-encoding it (the standing single-source-of-conditions discipline). It is **proportional** — a feature whose correctness does not depend on wiring (a pure function, a self-contained transform) needs no wiring-parity test and is not flagged.

**Sibling of the Stack-spec test rule in the "test the real thing, not a self-consistent stand-in" family.** It sits beside the existing Stack-spec test rule (don't test against a self-consistent stand-in) and the Interaction-scenario test rule (missing test → blocking); all three guard against a green test that measures something other than the real production behavior. The Stack-spec rule and its enforcement are untouched — this is its sibling, not a replacement. The defense **moves a slice of Step 5.5's run-it-for-real protection earlier**, into the test suite, so the modal wiring bug fails in CI rather than on hardware; it **does not replace** the on-hardware run, which stays the ultimate catch.

**Honest limitation — the Pass-2 half is out of scope.** The PM's proposed text also named a `code-review` "test initializes differently than main → finding" half; that half is **out of scope** because `code-review` is a built-in Claude Code engine the protocol delegates to, not ours to reprogram. The durable, ownable enforcement is therefore the `/pm-plan` rule + the `pm-plan-checker` clause (both protocol-owned) — the wiring-parity defense lands at plan-time + Pass-1, not inside the Pass-2 built-in engine. Recorded as a known limitation, not silently dropped. Motivation: caught live on hardware — a BLE-provider feature had green tests and passed both review passes but did not work, because the test registered the provider differently than `main` (production used a soft reactive flag); only the hardware caught it. **Source:** `doc/features/test-wiring-parity_plan.md`.

### Diagnostic-flow discipline: passive observation is read-only, bisect-before-hypothesize, an anti-thrash tripwire, and mid-debug stack-research

The `workflow/incident.md` "doesn't work in production" flow gains four **additive** diagnostic disciplines: (1) **Passive observation is read-only-grade (Step A).** Sniffing / scanning / capturing what is already on the wire or in the air (`tcpdump`, a BLE / mDNS scan, a bus capture, reading advertisements) is the same class as reading a log — it changes nothing and reaches no external peer, so it is **not** a probe: no Step A.5 authorization, no Blast-radius preflight, reach for it early. The carve-out is bounded by an explicit **observe-vs-emit boundary** — anything that *transmits* toward a coupled external stateful peer (injecting traffic, an active scan that solicits scan-responses from a paired peer, pairing / commissioning, writing to a bus) stays under Step A.5 + the preflight. (2) **Bisect before you hypothesize** — on a failed multi-stage path with a silent / ambiguous symptom, the first move is the cheapest observation that splits the path in half ("does the signal physically reach us at all?"), establishing ground truth at the lowest cheaply-observable layer before reasoning top-down. (3) **Stop-and-research tripwire (anti-thrash)** — two failed fixes in a row on the same symptom is a hard signal the model is wrong, not the line; the next move is then **not another patch** but consulting the canonical source (reference implementation / official example / spec). (4) **Stack-research as a mid-debug escalation** — `pm-stack-researcher` is spawned *whenever reality contradicts an already-cited stack rule* mid-diagnosis, not only up-front from `/pm-plan`.

**Additive — relaxes NONE of the safety rules.** Passive observation is placed *outside* the Blast-radius preflight / Step A.5 precisely because it is read-only and reaches no peer; the observe-vs-emit boundary keeps everything that emits toward a coupled peer *inside* the preflight. The tripwire adds a **brake** — it gates the *next* move after a failed fix and still requires Step A.5 authorization exactly as before; it does not loosen authorization. The Step A read-only default and the preflight rule itself are unchanged.

**No new mechanism.** The bisect and the tripwire are **orchestrator-discipline prose, not hooks** — "have two fixes failed?" / "does reality contradict the cited rule?" are semantic judgements a regex guard cannot make, placing them in the established **no-hook-for-semantic-judgement** family (alongside the Blast-radius preflight, the changeset-hygiene note, and the test-wiring-parity gate). The mid-debug stack-research **reuses** the existing `pm-stack-researcher` (one added invocation-context line), **no new agent**. Motivation: a live BLE-import incident where reactive patch-thrash on a live system continued until the PM said "давай ресёрч, ты делаешь костыли" — patch a guess vs. re-research the canonical flow. **Source:** `doc/features/diagnostic-flow-discipline_plan.md`.

### Agent reporting discipline: verify-or-stay-silent for repo/VCS facts + doc/review agents do not narrate git state

Every agent's report holds to two reporting rules, single-sourced as `### Reporting discipline` in `workflow/enforcement.md` (under the existing "git is the orchestrator's domain, not subagents'" boundary). (1) **Verify-or-stay-silent.** Any claim about repository / VCS state — tracked vs untracked, committed, current branch, file presence, "byte-identical to main", test / pass counts — must be backed by a command the agent **actually ran this turn**, or **not stated at all**; narrated-from-memory repo claims are forbidden. (2) **Lane discipline.** Doc and review agents author their artifact and report on **that** — they do **not** narrate git / tracking / branch / commit state, which is the orchestrator's domain; an irrelevant tracking observation is omitted, not volunteered.

**Single-sourced, referenced by name for reach.** The rule lives once in `workflow/enforcement.md`. The reporting agents that already read that file (`pm-coder`, `pm-plan-checker`, `pm-product-advocate`) inherit it; the five that do **not** read it today (`pm-architect`, `pm-auditor`, `pm-codebase-reader`, `pm-stack-researcher`, `pm-pr-prep`) gained a one-line by-name reference so the rule actually reaches them — never a re-encoding. The by-name reference exists precisely because the offender, `pm-architect`, does not read `enforcement.md`, so inheritance alone would have missed it.

**No hook — the semantic-judgement family.** "Did the agent verify this claim?" is not regex-decidable, so this is discipline, not a mechanical guarantee — the same no-hook-for-semantic-judgement family as the changeset-hygiene and test-wiring-parity disciplines. The rule narrows the surface (cite-or-omit) and the lane boundary removes the most common occasion for a fabricated claim; an LLM can still confabulate, so this is honestly a soft fix. Motivation: `pm-architect` reported a tracked file as "untracked" — a confabulated git-state claim it never verified, made outside its remit; a fabricated repo-state claim is a trust tax on every report the orchestrator relays. **Source:** `doc/features/agent-reporting-discipline_plan.md`.

## Architectural constraints

Hard constraints — agents and the orchestrator treat these as non-negotiable without an explicit PM decision.

- **No automated tests in the protocol repo by design — validation by use, with a narrow meta-infrastructure exception.** The template ships as documentation + config; it has no runtime to assert against. Validation of feature behaviour is whether the next feature on the template (or on a downstream project) flows through the workflow correctly. **Exception:** tests that cover the template's own meta-infrastructure — specifically, the regexes inside `.claude/settings.json` `PreToolUse` hooks — *are* allowed and live in `tests/hooks.sh`, gated by `.github/workflows/lint-hooks.yml`. The reasoning: hook regexes are the gate behind several `WORKFLOW.md` guarantees (no silent remote edits, no unsupervised force-push); a silent regression there is not caught by "validation by use" because the user only notices when the gate fails to fire. Runtime / feature-behaviour tests for downstream projects are still out — the exception is for protocol gates inside this repo, not for product behaviour anywhere. **Source:** `doc/features/template-v2_plan.md` § "Test plan" (the original constraint) and `doc/features/audit-fixup-hooks-quoted-form_plan.md` (the decision point that introduced the exception, after audit-2026-05-30 findings #3 / #4 / #6).
- **Template stays application-agnostic.** No domain-specific content in agents, commands, or templates. Examples must be cross-domain (checkout flow, ajv / kubectl / terraform as validators) so the template fits any project type. **Source:** commit `b10a9fa chore(template): degeneralize examples — remove WB/Matter project bias` — first pass leaked wb-mqtt-matter specifics; the cleanup pass replaced them with neutral examples.
- **Settings.json delivered as symlink, not copy.** Downstream `.claude/settings.json` must point at the submodule's copy so a `git submodule update --remote` immediately propagates new hooks without per-project re-install. Documented install path is `ln -s ../.ai-pm/tooling/.claude/settings.json .claude/settings.json`. **Source:** `README.md` § "Установка".
- **Hooks are `PreToolUse` guards plus one `UserPromptSubmit` route reminder.** No `PostToolUse`, `PostToolUseFailure`, `Stop`, `SessionStart`, or other hook events are used — the surface is intentionally minimal: gate at decision time with `PreToolUse`, and add a single soft `UserPromptSubmit` reminder that re-asserts the protocol route on change-intent prompts (the every-turn counterpart to the `PreToolUse` guards), but never react after the fact. **Source:** `.claude/settings.json` shape — the `hooks` object contains a `PreToolUse` key and one `UserPromptSubmit` key.

## Integration contract (template ↔ downstream)

How a downstream project consumes the template, and what it commits to in return.

**Consumption:**

1. **Submodule at `.ai-pm/tooling/`.** Downstream runs `git submodule add git@github.com:aadegtyarev/ai-pm-protocol.git .ai-pm/tooling`. The template's `git` semantics (the submodule lives at `$GIT_DIR/modules/ai-pm/tooling/` in the superproject's git directory; see `doc/stack-notes.md` § "git" submodule data model) apply. Default `git clone` of the downstream does not check out the submodule — `--recurse-submodules` or `git submodule update --init` is required (gotcha documented in `doc/stack-notes.md` § "git").
2. **Symlinks for agents, commands, settings.** Downstream creates three symlinks pointing into the submodule:
    - `.claude/agents` → `../.ai-pm/tooling/.claude/agents`
    - `.claude/commands` → `../.ai-pm/tooling/.claude/commands`
    - `.claude/settings.json` → `../.ai-pm/tooling/.claude/settings.json`

    Submodule update therefore propagates new agent personas, command procedures, and hooks without any downstream merge.
3. **WORKFLOW.md imported via `@`-directive.** Downstream `CLAUDE.md` contains the line `@.ai-pm/tooling/WORKFLOW.md` — Claude Code expands it inline at session start. Orchestration rules update with the submodule pointer. The `@`-line itself is byte-unchanged and now imports only the thin constitution + router core; its `workflow/*.md` topic siblings ride the same submodule and are pulled on demand by the consuming agents / commands (read just-in-time, not `@`-imported), so they update transparently with the submodule pointer exactly as the symlinked agents / commands do. **Source:** `WORKFLOW.md` (the `Read workflow/<topic>.md` router pointers) + `doc/features/workflow-progressive-disclosure_plan.md`.
4. **Templates consumed by `/pm-bootstrap`.** On first `/pm-bootstrap` in a downstream, the command reads `.ai-pm/tooling/doc/_templates/*.tmpl` and writes filled-in counterparts into the downstream's `docs/` (e.g., `docs/architecture.md`, `docs/user-journeys.md`, `docs/stack-notes.md`). After bootstrap the downstream owns those files — they are not symlinked, they live in the downstream's own history.

**Version bumping:** When downstream wants a newer template version, it runs `git submodule update --remote .ai-pm/tooling`, stages the new submodule pointer, and commits with a `chore: bump ai-pm-protocol to <sha>` message. The submodule pointer is the only thing in the downstream's diff — agent / command / WORKFLOW changes are transparent. **Source:** `README.md` § "Установка" (the inline `git submodule update --remote .ai-pm/tooling` line at the end of the install block) + `WORKFLOW.md` § "Maintenance".

## Release flow

How a feature commit in this template becomes a published GitHub Release. Cross-checked against `.github/workflows/auto-tag.yml`.

1. **Feature branch + conventional commits.** Orchestrator branches from current `main` (`feature/<topic>`), commits atomically; `WORKFLOW.md` § "Git workflow — orchestrator owns this" documents the shape.
2. **PR opened by `pm-pr-prep`.** Agent bumps the version, edits `CHANGELOG.md` (adds a `## [<VERSION>] — <DATE>` section under `## [Unreleased]`), pushes the branch, opens (or updates) the PR via `gh pr create` / `gh pr edit`.
3. **PM merges via squash on GitHub.** The squashed commit lands on `main`. Branch / commit message at this step do not matter for the release workflow — `auto-tag.yml` reads `CHANGELOG.md`, not the commit log.
4. **`auto-tag.yml` runs on push to `main`.**
    - `actions/checkout@v4` with `fetch-depth: 0` (full history — required so `git tag` can see all existing tags; default `fetch-depth: 1` would make tag-existence checks unreliable, gotcha documented in `doc/stack-notes.md` § "GitHub Actions").
    - "Detect new version" step: `grep -m1 '^## \[[0-9]' CHANGELOG.md` extracts the top entry; if a `v<VERSION>` GitHub Release already exists (`gh release view "$TAG"` succeeds) the step sets `is_release=false` and the workflow no-ops. This is the idempotency guarantee — pushing the same `main` twice does not double-tag.
    - "Create tag if missing" step (gated by `if: steps.detect.outputs.is_release == 'true'`): annotated tag `v<VERSION>` created and pushed to `origin`. Workflow-level `permissions: contents: write` grants the necessary token scope.
    - "Extract CHANGELOG notes" step: `awk "/^## \[${VERSION}\]/,/^## \[[0-9]/"` slices the section between the new version's header and the next version header; trailing line removed, separator + blank lines stripped, capped at 200 lines.
    - "Create GitHub Release" step: `gh release create "$TAG" --title "Release $TAG" --notes-file /tmp/release_notes.md` publishes the release. `GH_TOKEN` is `secrets.GITHUB_TOKEN`, which is the per-job installation access token (lifetime ≤ 6h on GitHub-hosted runners; documented in `doc/stack-notes.md` § "GitHub Actions").

**Why no tag-on-merge or release-branch model:** the decision was to read the CHANGELOG (single source of truth for what changed) instead of inferring versions from commit messages or branch names. Commits `e14b720 fix: auto-tag reads CHANGELOG instead of commit message`, `ad3d30f fix: merge auto-tag and create-github-release into one workflow`, `c240181 fix: release workflow checks GitHub Release, not tag`, `ba5f613 refactor: release model = tag main, not release branch` record the iterations that landed at the current shape.
