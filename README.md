# ai-pm-protocol

Шаблон для тех, кто пилит свой продукт один или в паре с AI.

PM описывает что хочет — агенты планируют, пишут код, проверяют. PM принимает решения по продукту и не читает код.

## Как это работает

```
PM: "хочу добавить групповые чаты"
        ↓
Claude читает docs/ проекта, задаёт уточняющие вопросы
Вместе с PM фиксирует план: сценарии, контракты, тест-план
(если в backlog есть подходящие заметки — предлагает взять в план)
PM: "ок"
        ↓
Architect-агент решает структурный вопрос (если есть): где живёт новый код?
Объясняет PM решение на понятном языке — PM может возразить
        ↓
Coder-агент реализует, запускает тесты + линтеры
Объясняет PM что сделано и как попробовать — без кода
        ↓
Reviewer-агент проверяет по 9 измерениям: план, тесты, безопасность,
стабильность, регрессии, конвенции, мёртвый код, документация, инфраструктура
        ↓
PM получает вердикт: "готово" или "нашёл проблему: X"
Если есть заметки (notes) — PM решает: починить сейчас, отложить в backlog, или игнорировать
```

PM не знает про агентов, команды и пайплайн — только разговор и продуктовые решения.

## Установка

```bash
git submodule add git@github.com:aadegtyarev/ai-pm-protocol.git .ai-pm/tooling
mkdir -p .claude
ln -s ../.ai-pm/tooling/.claude/agents .claude/agents
ln -s ../.ai-pm/tooling/.claude/commands .claude/commands
ln -s ../.ai-pm/tooling/.claude/settings.json .claude/settings.json
```

Первый запуск — скажи Claude: **«начни проект»**.

- **Новый проект** — Claude задаёт вопросы и создаёт документацию
- **Существующий проект (legacy)** — два режима:
  - *Быстрый старт* — читает минимум, создаёт черновую документацию с пробелами `[?]`, можно сразу работать
  - *Полное документирование* — читает весь код сам, восстанавливает архитектуру и пользовательские сценарии, PM только валидирует результат. После этого можно портировать на новый стек — Claude проанализирует систему и предложит 2–3 варианта.

Обновление шаблона:

```bash
git submodule update --remote .ai-pm/tooling
```

## Что гарантирует шаблон

**Код соответствует плану.** Reviewer проверяет каждый сценарий из плана — есть реализация и тест?

**Существующее поведение не ломается.** Тесты = поведенческий контракт. Кодер не может изменить существующий тест — только починить код чтобы тест прошёл. Reviewer ловит регрессии: изменённые сигнатуры, сломанные callers, скрытые поведенческие изменения.

**Архитектурные ограничения соблюдаются.** Агенты читают `docs/architecture.md` проекта — не добавят SQLite в Postgres-проект без явного решения.

**Безопасность.** Reviewer думает adversarially о каждом диффе: что может утечь, что хранится небезопасно, что можно подделать.

**Нет хардкода.** Reviewer блокирует любые credentials, пароли, device ID в коде — только из конфига или env.

**Код чистый.** Мёртвый код, дублирование, нарушение конвенций из `CLAUDE.md` — blocking, не пожелания.

**Документация не расходится с кодом.** Если изменилось поведение — reviewer блокирует PR пока `docs/` не обновлены.

**Инфраструктура соответствует архитектуре.** Если `docs/architecture.md` указывает Docker — reviewer блокирует PR, если `Dockerfile` отсутствует.

**Backlog не теряется.** Незначительные находки (notes) не блокируют релиз — PM явно решает что отложить. Отложенное попадает в `docs/backlog.md` и автоматически предлагается когда планируется подходящая фича.

## Что остаётся за PM

- Описать фичу или баг
- Ответить на уточняющие вопросы при планировании
- Принять архитектурное решение, если есть развилка
- Принять вердикт reviewer или попросить доработать
- Решить что делать с заметками (notes): починить сейчас, в backlog, или проигнорировать
- Сказать «выпускай» когда фич накопилось на релиз

Всё остальное — агенты.

## Структура шаблона

```
WORKFLOW.md           — правила работы агентов (импортируется в CLAUDE.md проекта)
.claude/agents/       — architect, coder, reviewer, pr-prep, release-helper
.claude/commands/     — bootstrap, plan-feature, research, audit
doc/_templates/       — шаблоны документов проекта
```

## Структура downstream-проекта

```
README.md             — что и зачем, для людей
CLAUDE.md             — статическая часть: проект, архитектура, pipeline
                         (автоматически импортирует .ai-pm/tooling/WORKFLOW.md)
docs/
  architecture.md     — стек, решения, ограничения
  user-journeys.md    — пользовательские сценарии
  ui-guide.md         — UI-конвенции (если есть UI)
  threat-model.md     — модель угроз (если нужна)
  backlog.md          — отложенные заметки (создаётся при первом использовании)
  features/
    <topic>_plan.md   — накапливается по ходу
    <topic>_review.md — вердикт reviewer
  audit-<date>.md     — результат /audit (по запросу PM)
.ai-pm/tooling/       — этот шаблон (submodule)
```

При обновлении submodule (`git submodule update --remote .ai-pm/tooling`) агенты и `WORKFLOW.md` обновляются автоматически. Статическая часть `CLAUDE.md` остаётся нетронутой.
