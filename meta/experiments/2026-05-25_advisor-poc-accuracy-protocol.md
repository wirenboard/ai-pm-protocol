# Discipline-advisor PoC accuracy protocol (2026-05-25)

**Scope:** validation protocol для discipline-advisor (v0.4.0+) перед тем как advisor станет mandatory в Stage F triggers. Closes plan-review concerning-2 (PoC accuracy gate ≥80% not measurable as written).

**Pre-condition:** этот документ существует **перед** PoC runs. Generated после PoC — `meta/experiments/<date>_advisor-poc-accuracy.md` с numbers.

## Test set design

**10-15 synthetic test projects** покрывающих разные axes complexity:

| Category | Project type | Expected complexity |
|---|---|---|
| 1 | Single-file CLI utility (e.g., color picker) | minimal preset, ~5 skip-safe artifacts |
| 2 | TUI tool без persistence | minimal preset, ~6 skip-safe |
| 3 | Pure backend API без public surface | standard preset, ~3 conditional |
| 4 | Public web app + auth + payments | full preset, all mandatory |
| 5 | Mobile app + custom backend | full preset, multi-stack |
| 6 | Embedded device + companion app | standard preset, conditional UI per kind |
| 7 | Multi-tenant SaaS + compliance | full preset, hard floors fire |
| 8 | Single-user personal utility | minimal preset, most skip-safe |
| 9 | Internal tool (no external users) | minimal preset, skip customer-interview |
| 10 | E-commerce checkout flow | full preset, payments hard floor |
| 11 | Static site generator | minimal preset, ~7 skip-safe |
| 12 | CI/CD pipeline tool | standard preset, no PII |
| 13 | Database migration tool | standard preset, no UI but schema-heavy |
| 14 | Browser extension | conditional UI = web, minimal otherwise |
| 15 | Cross-platform desktop + sync | full preset, multi-kind UI |

Каждый project — minimal `package.json` / `go.mod` / `Cargo.toml` + 5-10 representative files (entry points + sample modules + manifests). Files generated synthetic или взяты sanitized из real OSS projects.

## Labelling rubric — known-correct expected output

Для каждого test project — pre-labelled expected advisor output:

```yaml
test_project_N:
  name: <slug>
  capabilities:
    uses-crypto: <yes|no>
    uses-auth: <yes|no>
    processes-pii: <yes|no>
    uses-payments: <yes|no>
    public-web: <yes|no>
    ui_kind: [<list>]
    db_kind: [<list>]
  expected_preset: minimal | standard | full
  expected_skip_eligibility:
    vision: required
    mvp-scope: required
    personas: <recommended|skip-safe>  # based on use case
    user-journeys: <...>
    threat-model: <mandatory|skip-safe>
    legal: <mandatory|skip-safe>
    # ... per artifact
```

**Labels generated by operator** (Trust profile A in real PoC; для bootstrap PoC — Claude Code session с deliberate human override на ambiguous cases). NOT generated by advisor itself (это circular).

## Measurement protocol

### Procedure

1. **Setup test project** в isolated git repo (one per test case)
2. **Run advisor** в blind mode (no access to expected labels):
   ```
   claude --agent discipline-advisor "Run bootstrap entry analysis. Report mandatory/recommended/skip-safe per artifact."
   ```
3. **Capture output** — structured advisory YAML
4. **Compare** advisor output vs expected labels:
   - Per artifact: match = correct, mismatch = incorrect
   - Per axis: aggregate correctness across all 6 axes

### Metrics

Per axis (понятность / поддерживаемость / технические качество / UI / UX / learning):

```
axis_accuracy = (correct_decisions_for_axis / total_decisions_for_axis) × 100%
```

**Bar:** ≥80% per axis. Если ниже на любой axis:
- **Below 80%:** advisor НЕ становится mandatory для этой axis. Fallback — static rules (`skip_eligibility` metadata + manual operator quiz).
- **70-80%:** retry с improved advisor prompt (iterate on hard detection rules / scan heuristics)
- **<70%:** abandon advisor для этой axis в этой версии; reconsider в v0.5.0+

### Per-trigger breakdown

Также measure accuracy per trigger type:
- **Bootstrap entry** — preset + skip_eligibility recommendations
- **Stage F Step 1** (spec quality)
- **Stage F Step 2** (architecture proportionality)
- **Stage F Step 4** (gap checks)
- **Stage F Step 7** (multi-axis coverage)

Каждый trigger может иметь different accuracy → mandatory per trigger, не per advisor в целом.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| Test set too synthetic — doesn't represent real projects | Use 5+ sanitized OSS projects, не only synthetic |
| Labelling bias (single operator) | Document labelling rationale per project; future PoC re-run с different operator |
| Advisor overfits на test set | Hold-out validation: 10 train + 5 test; report holdout-only numbers |
| Accuracy threshold ≥80% arbitrary | Document trade-off: 70% = too many false positives; 90% = too restrictive given LLM uncertainty |
| PoC takes weeks | Mitigation: start с 5 test projects (smaller axes coverage), expand to 15 если initial results promising |
