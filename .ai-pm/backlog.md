## Agent-handoff durability when "continue the same agent" is unavailable — 2026-06-04 (this repo)

- **An agent that stops mid-task at a blocking question holds valuable intermediate analysis only in its own context; if it cannot be resumed (the continue-the-same-agent tool was unavailable this session), that analysis is re-derived from scratch by a fresh agent — wasted tokens, plus a thin-loss risk if the orchestrator's hand-off brief omits a conclusion the stopped agent never wrote to disk.** PM observation 2026-06-04, from a live incident: `pm-coder` correctly stopped at a blocking conflict (a plan reversal — "sensors have no virtual device" — collided with three old test assertions, and its hard "never touch existing tests → stop and report" rule fired) after ~130k tokens of code analysis. The orchestrator authorized the fix (PM-approved reversal, not a regression) and wanted to resume the **same** agent with its context — but the continue-same-agent path was unavailable, so a fresh agent was spawned and re-briefed. **Correctness did not suffer** — the protocol deliberately keeps the source of truth in files (`state/current.md`, plan, arch notes, contracts), so the new agent rebuilt context from disk and finished cleanly (350 tests green); that is the designed fault-tolerance. **Efficiency and thin-loss did** — the new agent re-read the code, and any conclusion the stopped agent held only "in head" had to be hand-carried by the orchestrator into the brief. *Recommendation (three parts):* (1) **Make explicit in `WORKFLOW.md` that durable hand-off is the file-state, not agent memory** — continue-the-same-agent is an optional optimization the protocol must never depend on (saves tokens where available, breaks nothing where not). (2) **Agent rule on stopping at a blocking question:** before yielding control, write the work-so-far (the conflict area, what was already checked, what remains) to a durable artifact — its own review file, `state/current.md`, or a scratch note — so the successor/orchestrator doesn't re-derive it; then "hand-off" = a file, not a lost context. This extends the stop-and-report rule (e.g. `pm-coder`'s "stop and report" + the doc-conflict stop) from *report-the-conflict* to *also-flush-the-analysis*. (3) **Orchestrator rule:** when the same agent can't be continued, spawning a fresh agent with a brief that explicitly quotes the stopped agent's conclusions is the **standard** path, not an emergency. *One line:* the problem isn't a missing continue-tool — it's that valuable intermediate analysis lives only in the agent's context; the fix is the agent flushing it to disk on stop, which makes the continue-tool a pure optimization. *Where it lands:* `WORKFLOW.md` § "How state is kept" (durable-handoff = files principle) + the agent stop-and-report rules (`pm-coder.md` and the others' "stop and surface" paths gain "flush your analysis first"). Likely `/pm-plan` (small, mostly WORKFLOW.md + agent-prompt wording). PM observation 2026-06-04.

### Design principle (why this stays readable AND doesn't drift — the frame for planning)

The three constraints — PM-readable / strict contract↔doc↔code chain intact / not smeared across many files — are not in tension. They are all served by **one invariant: one fact, one owner; the render *selects* a layer, it never *copies* it.** Two consequences worth carrying into the plans:

**Two parallel funnels, one entry each, linked one-directionally.** The PM never wades back into topics because the technical layer is a *separate* funnel reached only by an explicit "↓ Технически" link — not interleaved into the PM path.

```
PM funnel:   README (1 para) → product.md (why/for-whom) → product-map (features)
                                                              │ ↓ Технически
Tech funnel: architecture.md «## Behavioral contract» ────────┘ + contract Must-work + stack-notes
```

This adds **zero new files and exactly one new section** (`## Behavioral contract` inside `architecture.md`). The fact-ownership map is itself the anti-smear mechanism: smearing comes from the COPY pattern (a fact in two places), which the move-not-copy guard forbids at every level.

**Single owner applied at every level of the PM funnel — this is what keeps the PM description from drifting:**

| Level | Owns | Defers (links, never copies) |
|---|---|---|
| README | nothing about capabilities | → `product.md` |
| `product.md` `## Что умеет` | "why / for whom" + overall maturity boundary | → the map for the per-feature list |
| `product-map` | a *projection* of `User value` + `Out of scope` | (generated; re-derived & compared by auditor) |
| `contract.User value` | the single authored PM sentence per feature | rides the plan-gate + audit |

Drift analysis of the PM layer, level by level:
- **map ↔ contract: safe by construction** — the map is generated and the auditor re-derives & compares (`pm-auditor.md:111`). The cheap slice *removes* a drift surface: today the map carries a separately-authored guarantee string; making it a pure projection of `User value`/`Out of scope` deletes that extra copy.
- **`product.md` ↔ map: the real "two PM descriptions" risk** — resolved structurally by the README-fix applied one level down: `product.md` must NOT re-list per-feature capabilities; it owns only why/for-whom + the overall boundary and links to the map for the feature list. (If it re-lists, it will drift exactly as README↔product.md did downstream.)
- **`contract.User value` ↔ code: the one irreducible authored text** — but it drifts no worse than the technical contract does, because it rides the same enforcement: behaviour changed without a plan = orphaned-implementation **blocking** (auditor dim 3, `:90-95`); behaviour changed with a plan = the plan must touch the contract and its `Out of scope` "must not silently relax Must not break" (`contract.md.tmpl:61`); code changed after `Last reviewed` = stale-contract **note** (dim 4). The only gap no mechanism sees is a feature whose *value framing* changed while its *mechanics* (Must-work) did not — rare, low-harm. Two cheap backstops: a **structural** lint "PM layer contains no machine tokens (`<id>_<…>`, `retain`, `QoS`)" (the legitimate structural anti-drift, NOT prose-policing) and the PM re-reading the map at the validation point above.

Net: the PM description drifts only as much as the technical contract drifts — because it is a projection of the same single-owned source, not a parallel prose. The thing that would make it drift is COPY; the design forbids copy at every level.

## Deterministic-enforceable vs AI-evaluated check boundary — 2026-06-04 (this repo)

PM observation 2026-06-04: the protocol sometimes makes **AI-dependent** what is by nature **deterministically enforceable** — there is no explicit rule for *which* checks must be a hook/linter vs a semantic AI judgement. Two concrete instances:

- **AI-specific minimums risk being AI-self-policed, not linted.** `### AI-specific minimums` (`architecture.md.tmpl` + `CLAUDE.md.tmpl`: max file 300 lines, max function 50, cyclomatic ≤10, no file-level lint-suppressions, coverage ≥80%) are **motivated by AI failure modes** ("AI tends to write monolithic files and complex functions") and are **deterministically lintable** — but the protocol only *states* them as conventions and *expects* them in the project's real linter config; it does not **guarantee** they land there. If the project's `### Linter commands` don't encode them, only the agent enforces them by reading `CLAUDE.md` → they degrade to AI-self-policed. *Fix direction:* at `/pm-bootstrap`, wire the AI-specific minimums into the deterministic linter config (or record explicitly which are unenforceable on this stack and stay convention-only), so they are linted, not read.
- **Many "shape-not-meaning" auditor checks are deterministically hook-able but shipped as LLM checks.** Several `pm-auditor` / `pm-plan-checker` checks added recently are explicitly described as **structural / shape-not-meaning** (the structural-token note, the journey identifier-restatement check, the System-invariants index presence check, the `selected autonomously` `source:` citation backstop, etc.). By their own framing they match **shapes / token presence**, not meaning — i.e. a deterministic hook/linter *could* enforce them, cheaper and more reliably than an agent re-reading prose each audit. They are shipped as agent-prompt checks because there is no general rule pushing structural checks down to hooks.

*The missing rule:* draw an explicit boundary in the protocol — **deterministically-enforceable ⇒ hook/linter (run every time, free, reliable); semantic-judgement-only ⇒ AI check (no hook, by design)** — and apply it both ways: (a) guarantee the deterministic ones are actually wired to tooling (AI-specific minimums → linter), (b) audit the existing "shape-not-meaning" agent checks for hook-ability and migrate the genuinely-structural ones to hooks. ## Periodic whole-codebase code-quality review (not just per-diff) — 2026-06-05 (this repo)

PM question 2026-06-05: *"do we have a code-review of the WHOLE project periodically — to guarantee the codebase is clean overall — or do we only ever look by diff?"* The honest answer surfaced a real gap.

- **What exists:** every change is reviewed **by diff** — `pm-plan-checker` (Pass-1) + `code-review` (Pass-2) on every feature, plus `/code-review` / `/code-review ultra` on demand. And `/pm-audit` runs **periodically** — but it is **protocol-COMPLIANCE** (every feature has plan + review + contract; plans match implementations; contracts / docs current; cross-doc consistency), **not** a code-QUALITY sweep of the whole tree.
- **The gap:** there is **no periodic whole-codebase code-quality review**. The protocol rests on the assumption *"sum of reviewed diffs = clean whole"*, which is leaky: (a) **legacy code** onboarded via `/pm-bootstrap` full mode was *documented* but never diff-reviewed for quality; (b) **cross-cutting / emergent** issues (a pattern locally-fine per diff but globally problematic, accumulated drift, inter-feature inconsistency) are invisible to a per-diff window; (c) a diff clean in isolation can interact badly with the whole. So "the codebase is clean overall" is **not** periodically guaranteed — only "each change was reviewed at merge".
- **Fix direction:** add a **periodic whole-codebase code-quality sweep**, distinct from `/pm-audit` (compliance) and from per-diff `code-review`. Candidates: a `/pm-audit` mode / sibling that runs `code-review` over the whole tree (or over "hot" / recently-churned / never-diff-reviewed areas, for proportionality — a full-tree sweep every audit is too heavy); or a scheduled `/code-review ultra` over the whole base at a chosen cadence. Decisions for planning: (1) trigger/cadence (every Nth audit? on PM request? on a legacy-onboarded project once?); (2) scope (whole tree vs changed-since-last-sweep vs never-reviewed-legacy); (3) where findings land (a new audit-adjacent report? backlog items the PM triages?); (4) proportionality — must not re-review the same clean code every cycle. **Distinct from** the existing backlog item "cross-model review for high-risk / especially-painful changes" (that is about *individual high-risk diffs* reviewed by a different model, not a *whole-codebase* sweep). *Likely path:* `/pm-plan` (the machinery — `code-review` + a scoping/cadence rule — already exists; mostly a triggering + scoping + reporting design).

## Flag-controlled mode: don't commit project-generated docs into the project repo — 2026-06-05 (this repo)

PM idea 2026-06-05: a mode, toggled by a flag in `.env` (or equivalent config), where the protocol's **project-generated documentation is NOT committed into the project's own git repository**. Sometimes the PM wants those artifacts to live elsewhere — a separate repo, or out of the product repo entirely. Storage location still undecided ("может быть в отдельном репо").

- **What it covers (decide in planning):** the docs/meta the protocol produces — `docs/` (product.md, product-map.md, architecture.md, user-journeys.md, stack-notes.md, threat-model.md, features/*_plan.md) and/or `.ai-pm/` operational artifacts (contracts, reviews, arch notes, audits, research, backlog, state). Possibly all, possibly a subset (only `docs/`, or only `.ai-pm/`), possibly configurable.
- **Behavior:** the pipeline still **writes** these locally (agents must read them — `pm-coder` reads state, `pm-auditor` reads everything, contracts/plans are the source of truth); it just does not **commit** them into the project repo. The flag suppresses staging/commit of those paths (via `.gitignore` and/or a commit-scoping rule in `pm-pr-prep` + the orchestrator), so the project's git history carries only the product code, not the protocol paperwork.
- **The hard tension — durability.** The protocol **relies** on these artifacts being durable + readable across sessions: `.ai-pm/state/current.md` survives a pause, `pm-auditor` reads the git history of contracts/reviews, plans/contracts are the canonical source. If they are not in the project repo, durability needs another home. *Decisions for planning:* (1) **where they go** when not in the project repo — a **separate dedicated repo** for the `docs/` + `.ai-pm/` bundle (auto-init / commit there?), a git-ignored local dir, or an external store; (2) **which artifacts** the flag covers (`docs/` vs `.ai-pm/` vs both vs configurable); (3) how **session-durable state** (`current.md`) and **auditor history** survive if not committed to the project repo (the separate-repo answer must still give the auditor a history to read); (4) how **`pm-pr-prep`** and the orchestrator's commit steps learn to **skip those paths**; (5) interaction with the **submodule** (`.ai-pm/tooling/` stays committed — it is the read-only template, distinct from the project's own generated artifacts).
- **Why:** some projects should not carry their PM/protocol paperwork in the product repo — clean product history, separation of concerns, client repos where only code ships. *Likely path:* `/pm-research` (how others separate generated-docs / meta from the product repo — docs-as-separate-repo, git worktrees, git-ignored + external sync, `$GIT_DIR` tricks) then `/pm-plan`. PM idea 2026-06-05, recorded while the review-typology research ran.
