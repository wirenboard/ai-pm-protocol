## Automode ‚Äî autonomous product-fork resolution from the bootstrap baseline ‚Äî 2026-06-04 (this repo)

- **A run mode where the PM front-loads everything at bootstrap, then the orchestrator + agents carry the feature/product to the end with NO further questions ‚Äî autonomously resolving product forks from (a) the initial bootstrap, (b) project logic, standards, and accumulated experience inside the project.** Today the protocol is interactive-by-default: every substantive product fork is surfaced to the PM (`AskUserQuestion`), and the new `pm-product-advocate` gate even *blocks* the coder handoff until the PM answers or descopes. Automode is the **inverse**: the PM declares intent + boundaries once, up front, and grants the pipeline authority to decide the rest. *Mechanism sketch:* a project-level setting (e.g. `CLAUDE.md` `## Decision authority: autonomous | interactive`, default interactive) that flips how forks resolve ‚Äî in autonomous mode the orchestrator/advocate **resolve** each gap from the bootstrap baseline + project canon (`product.md`, `architecture.md`, contracts, prior decisions, standards) instead of relaying it, and **record the auto-decision with its rationale** (a decision log the PM can review after the fact, not a silent choice). *Key tension to resolve in planning:* this is in **direct tension with `pm-product-advocate`** ‚Äî that gate is built to *force* the conversation. Automode does not delete the advocate; it **redirects its output**: the advocate still generates the foundational-question gaps (same fixed checklist, same artifact), but in autonomous mode the gaps are **answered by the pipeline from the baseline + recorded as auto-decisions**, blocking only when a gap is genuinely unresolvable from the available context (a true ambiguity the baseline does not cover ‚Äî escalate even in automode). So the advocate becomes the *list of decisions that got auto-made*, giving the PM an audit trail. *Open design questions:* (1) what is the **escape valve** ‚Äî a confidence/coverage threshold below which automode still asks (so it does not confabulate a product direction the bootstrap never implied)? (2) does automode also relax the human-in-the-loop ship gate (Step 6 A/B/C), or only the *product-fork* questions, keeping merge authority with the PM? (3) granularity ‚Äî whole-project automode vs per-feature opt-in ("bootstrap this feature, then run it autonomously"). (4) how the recorded auto-decisions feed back into the project canon so later features inherit them (the "accumulated experience" loop). *Why:* for an experienced PM with a well-bootstrapped product, the interactive cadence is friction; automode trades PM attention for speed while keeping a reviewable trail ‚Äî the opposite end of the same dial the advocate sits on. **Sequence after `pm-product-advocate` lands** ‚Äî automode is defined *relative to* the advocate (it redirects the advocate's output), so the advocate must exist first. Likely `/pm-research` (autonomy/decision-authority patterns, confidence-gating for when an agent should ask vs decide ‚Äî note the research open-question on clarifying-question generation that the advocate plan also left open) then `/pm-plan`. PM idea 2026-06-04.


This adds **zero new files and exactly one new section** (`## Behavioral contract` inside `architecture.md`). The fact-ownership map is itself the anti-smear mechanism: smearing comes from the COPY pattern (a fact in two places), which the move-not-copy guard forbids at every level.

**Single owner applied at every level of the PM funnel ‚Äî this is what keeps the PM description from drifting:**

| Level | Owns | Defers (links, never copies) |
|---|---|---|
| README | nothing about capabilities | ‚Üí `product.md` |
| `product.md` `## –ß—Ç–æ —É–º–µ–µ—Ç` | "why / for whom" + overall maturity boundary | ‚Üí the map for the per-feature list |
| `product-map` | a *projection* of `User value` + `Out of scope` | (generated; re-derived & compared by auditor) |
| `contract.User value` | the single authored PM sentence per feature | rides the plan-gate + audit |

Drift analysis of the PM layer, level by level:
- **map ‚Üî contract: safe by construction** ‚Äî the map is generated and the auditor re-derives & compares (`pm-auditor.md:111`). The cheap slice *removes* a drift surface: today the map carries a separately-authored guarantee string; making it a pure projection of `User value`/`Out of scope` deletes that extra copy.
- **`product.md` ‚Üî map: the real "two PM descriptions" risk** ‚Äî resolved structurally by the README-fix applied one level down: `product.md` must NOT re-list per-feature capabilities; it owns only why/for-whom + the overall boundary and links to the map for the feature list. (If it re-lists, it will drift exactly as README‚Üîproduct.md did downstream.)
- **`contract.User value` ‚Üî code: the one irreducible authored text** ‚Äî but it drifts no worse than the technical contract does, because it rides the same enforcement: behaviour changed without a plan = orphaned-implementation **blocking** (auditor dim 3, `:90-95`); behaviour changed with a plan = the plan must touch the contract and its `Out of scope` "must not silently relax Must not break" (`contract.md.tmpl:61`); code changed after `Last reviewed` = stale-contract **note** (dim 4). The only gap no mechanism sees is a feature whose *value framing* changed while its *mechanics* (Must-work) did not ‚Äî rare, low-harm. Two cheap backstops: a **structural** lint "PM layer contains no machine tokens (`<id>_<‚Ä¶>`, `retain`, `QoS`)" (the legitimate structural anti-drift, NOT prose-policing) and the PM re-reading the map at the validation point above.

Net: the PM description drifts only as much as the technical contract drifts ‚Äî because it is a projection of the same single-owned source, not a parallel prose. The thing that would make it drift is COPY; the design forbids copy at every level.
