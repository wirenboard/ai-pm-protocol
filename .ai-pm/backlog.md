## State-archive (post-ship bookkeeping) has no rule-compliant committed home — 2026-06-04 (this repo)

- **When a feature ships, `.ai-pm/state/current.md` is archived to `.ai-pm/state/archive/<topic>-<date>.md` and reset — but the workflow defines no clean, rule-compliant place to COMMIT that archive, so it either gets committed to `main` (which the git rule forbids) or rides as an untracked file onto the next feature's branch (delayed into git, and `git clean`-losable until then).** Surfaced live 2026-06-04: right after `git checkout main && git pull` (post-merge sync) the orchestrator committed the state-archive straight to `main` "by habit", because the archive step logically happens *after* merge — at which point you are standing on `main` and the protocol says never commit there. Caught and reset (local only, not pushed), but it is a real gap: the archive-and-reset is a **manual step with no defined home**, the exact "degrades silently" class the review-stamp gate named. *Why it happens:* archiving conceptually belongs *after* the task is done = after the PR merges, but the only rule-compliant commit surface (a feature branch / PR) is gone by then. *Options to settle in planning (do NOT pre-decide):* (1) **archive on the shipping feature branch as its final pre-`pr-prep` step** — set `Status: done`, write the archive + reset on the branch so it merges *with* the feature PR (the archive records "shipped via PR #N"); eliminates the post-merge orphan entirely. This looks cleanest — the state-archive becomes part of the feature it describes. (2) **formalize the current de-facto carry-over** — `/pm-plan` explicitly stages and commits any pending prior-feature archive as part of the next feature's first commit (turns the slip into a defined step). (3) a narrow, explicit **bookkeeping-only carve-out** allowing state-archive commits to `main` (rejected-leaning: bypasses review, and although a no-CHANGELOG-bump push to `main` does not create a release, it still erodes the no-commit-to-main discipline). *Where it lands:* `WORKFLOW.md` § "How state is kept" + Step 6 (Ship) — define the archive's commit point in the lifecycle so it is always inside a PR, never stranded. *Why:* the protocol enforces "never commit to main" but gives one of its own routine artifacts no compliant way to honor it; the fix is to give state-archiving a defined home in the ship flow. PM decision 2026-06-04: protocol must cover this. Likely `/pm-plan` (small, mostly WORKFLOW.md wording).

## PM-collaborative documentation migration on template bump — 2026-06-04 (this repo)

- **When a downstream project bumps the template version, give it a flow that migrates the project's EXISTING documentation up to the new template's expectations IN PARTNERSHIP WITH THE PM — not just the mechanical structural migrations we have today.** PM idea 2026-06-04. The shape the PM described: on a template bump, the protocol should (1) **analyze** the project's current docs against what the *new* template version expects, (2) **find the gaps** (sections/disciplines the project predates — e.g. `## Behavioral contract`, foundational journeys, threat-model lifecycle, contract two-layer, value-first product-map), (3) **go to the PM with proposals + clarifying questions** (not silently rewrite), and (4) **apply** the migration with the PM's answers. This is the "ревизия документации на downstream" made into a **repeatable protocol feature** rather than a one-off manual pass. *Why it's new (not already covered):* today's `### Pending-migration detection` migrations in `MIGRATIONS.md` are **structural and mechanical** — a detect-condition fires a fixed procedure (feature-index→product-map, product.md split, README front-gate, header rename, contract two-layer), all agent-run with no PM gap-conversation. `/pm-audit` is close (doc-currency sweep + a PM finding-loop) but is framed as *protocol-compliance*, not *content-migration-to-the-new-template-on-upgrade*. `WORKFLOW.md` § "Maintenance" runs the mechanical migrations on bump but has **no** PM-collaborative content layer. *Design notes for planning:* (a) reuse the established **"agent generates the gap questions → orchestrator relays them in one `AskUserQuestion` pass → PM answers or descopes"** pattern from `pm-product-advocate` (do not let an agent talk to the PM directly); (b) the gap source is the **delta between the project's docs and the new template version's expected structure/disciplines** — needs a machine-readable notion of "what version expects what" (this is the hard part: today expectations are scattered across templates + WORKFLOW.md + MIGRATIONS.md); (c) decide whether this is a new command (e.g. `/pm-migrate-docs` run after a bump), an extension of `/pm-audit` with an `--upgrade` mode, or an extension of the WORKFLOW Maintenance bump flow; (d) keep it **proportional** — only surface real gaps the new version introduced, never re-interrogate already-adequate docs (the `clean`-is-silent discipline); (e) strong overlap with the EPIC "technical-over-product bias" items (cross-document consistency auditor, product-lag note) and with `bootstrap-populated-journeys` — sequence so they compose, not collide. *Likely path:* `/pm-research` (how others do doc-migration-on-dependency-upgrade; whether to model template expectations as a versioned manifest) then `/pm-plan`. PM idea 2026-06-04 — this is the protocolized form of the "downstream documentation revision" request.

## Automode — autonomous product-fork resolution from the bootstrap baseline — 2026-06-04 (this repo)

- **A run mode where the PM front-loads everything at bootstrap, then the orchestrator + agents carry the feature/product to the end with NO further questions — autonomously resolving product forks from (a) the initial bootstrap, (b) project logic, standards, and accumulated experience inside the project.** Today the protocol is interactive-by-default: every substantive product fork is surfaced to the PM (`AskUserQuestion`), and the new `pm-product-advocate` gate even *blocks* the coder handoff until the PM answers or descopes. Automode is the **inverse**: the PM declares intent + boundaries once, up front, and grants the pipeline authority to decide the rest. *Mechanism sketch:* a project-level setting (e.g. `CLAUDE.md` `## Decision authority: autonomous | interactive`, default interactive) that flips how forks resolve — in autonomous mode the orchestrator/advocate **resolve** each gap from the bootstrap baseline + project canon (`product.md`, `architecture.md`, contracts, prior decisions, standards) instead of relaying it, and **record the auto-decision with its rationale** (a decision log the PM can review after the fact, not a silent choice). *Key tension to resolve in planning:* this is in **direct tension with `pm-product-advocate`** — that gate is built to *force* the conversation. Automode does not delete the advocate; it **redirects its output**: the advocate still generates the foundational-question gaps (same fixed checklist, same artifact), but in autonomous mode the gaps are **answered by the pipeline from the baseline + recorded as auto-decisions**, blocking only when a gap is genuinely unresolvable from the available context (a true ambiguity the baseline does not cover — escalate even in automode). So the advocate becomes the *list of decisions that got auto-made*, giving the PM an audit trail. *Open design questions:* (1) what is the **escape valve** — a confidence/coverage threshold below which automode still asks (so it does not confabulate a product direction the bootstrap never implied)? (2) does automode also relax the human-in-the-loop ship gate (Step 6 A/B/C), or only the *product-fork* questions, keeping merge authority with the PM? (3) granularity — whole-project automode vs per-feature opt-in ("bootstrap this feature, then run it autonomously"). (4) how the recorded auto-decisions feed back into the project canon so later features inherit them (the "accumulated experience" loop). *Why:* for an experienced PM with a well-bootstrapped product, the interactive cadence is friction; automode trades PM attention for speed while keeping a reviewable trail — the opposite end of the same dial the advocate sits on. **Sequence after `pm-product-advocate` lands** — automode is defined *relative to* the advocate (it redirects the advocate's output), so the advocate must exist first. Likely `/pm-research` (autonomy/decision-authority patterns, confidence-gating for when an agent should ask vs decide — note the research open-question on clarifying-question generation that the advocate plan also left open) then `/pm-plan`. PM idea 2026-06-04.

### Design principle (why this stays readable AND doesn't drift — the frame for planning)

The three constraints — PM-readable / strict contract↔doc↔code chain intact / not smeared across many files — are not in tension. They are all served by **one invariant: one fact, one owner; the render *selects* a layer, it never *copies* it.** Two consequences worth carrying into the plans:

**Two parallel funnels, one entry each, linked one-directionally.** The PM never wades back into topics because the technical layer is a *separate* funnel reached only by an explicit "↓ Технически" link — not interleaved into the PM path.

```
PM funnel:   README (1 para) → product.md (why/for-whom) → product-map (features)
                                                              │ ↓ Технически
Tech funnel: architecture.md «## Behavioral contract» ────────┘ + contract Must-work + stack-notes
```

This adds **zero new files and exactly one new section** (`## Behavioral contract` inside `architecture.md`). The fact-ownership map is itself the anti-smear mechanism: smearing comes from the COPY pattern (a fact in two places), which the move-not-copy guard forbids at every level.

**Single owner applied at every level of the PM funnel — this is what keeps the PM description from drifting:**

| Level | Owns | Defers (links, never copies) |
|---|---|---|
| README | nothing about capabilities | → `product.md` |
| `product.md` `## Что умеет` | "why / for whom" + overall maturity boundary | → the map for the per-feature list |
| `product-map` | a *projection* of `User value` + `Out of scope` | (generated; re-derived & compared by auditor) |
| `contract.User value` | the single authored PM sentence per feature | rides the plan-gate + audit |

Drift analysis of the PM layer, level by level:
- **map ↔ contract: safe by construction** — the map is generated and the auditor re-derives & compares (`pm-auditor.md:111`). The cheap slice *removes* a drift surface: today the map carries a separately-authored guarantee string; making it a pure projection of `User value`/`Out of scope` deletes that extra copy.
- **`product.md` ↔ map: the real "two PM descriptions" risk** — resolved structurally by the README-fix applied one level down: `product.md` must NOT re-list per-feature capabilities; it owns only why/for-whom + the overall boundary and links to the map for the feature list. (If it re-lists, it will drift exactly as README↔product.md did downstream.)
- **`contract.User value` ↔ code: the one irreducible authored text** — but it drifts no worse than the technical contract does, because it rides the same enforcement: behaviour changed without a plan = orphaned-implementation **blocking** (auditor dim 3, `:90-95`); behaviour changed with a plan = the plan must touch the contract and its `Out of scope` "must not silently relax Must not break" (`contract.md.tmpl:61`); code changed after `Last reviewed` = stale-contract **note** (dim 4). The only gap no mechanism sees is a feature whose *value framing* changed while its *mechanics* (Must-work) did not — rare, low-harm. Two cheap backstops: a **structural** lint "PM layer contains no machine tokens (`<id>_<…>`, `retain`, `QoS`)" (the legitimate structural anti-drift, NOT prose-policing) and the PM re-reading the map at the validation point above.

Net: the PM description drifts only as much as the technical contract drifts — because it is a projection of the same single-owned source, not a parallel prose. The thing that would make it drift is COPY; the design forbids copy at every level.
