## Integration-risk spike gate: docs-research is necessary-not-sufficient — verify a load-bearing idiom by execution before design (the upstream root of the diagnostic items) — 2026-06-05 (this repo)

PM-relayed feedback from the live BLE-import session — the **highest-severity** of the four items (the root where the whole cost arose: three reactive research rounds + several rebuilds/redeploys on live hardware). The up-front `pm-stack-researcher` (pre-plan) **and** the arch note (design) both missed three silent integration bugs: (1) provider-registration timing via ServiceBundle/Environment, (2) the controller scanner inheriting a deliberately-off `network.ble` export, (3) the BLE-central/dialer gated separately from the scanner. All three surfaced only **reactively, on real hardware**, after code+design had already committed to a wrong README registration idiom. Crucially the research was **not** empty — it correctly cleared the big risks (right package, dedup trap, mandatory Wi-Fi handoff, capabilities, disabling bluetoothd, the device-role BLE sharp edge). It missed exactly the **emergent-silent** class.

**Two linked root gaps:**

**A. `pm-stack-researcher` is positioned as docs+source feasibility research, with no mandatory execution verification.** It answers "is it possible, which package, which idiom, what inputs" — but (1) README/docs describe the happy path of the typical case; a non-standard combination (a BLE controller on a node where BLE export is deliberately OFF) breaks their premises; (2) traps are **absence bugs** (a silent default/gate, no error, no log) — research finds what code *says/does*, not what it *silently does not do* under your config; (3) docs-research **inherits the docs' blind spots** — the cited idiom was literally wrong for our case, and a citation can't catch that, only a run can.

**B. No "walking skeleton / tracer-bullet before design" gate.** The flow is research → plan → arch → code → review → (opt.) verify. For a feature that combines SDK capabilities atypically or hinges the design on a load-bearing idiom, the cheapest insurance is a **minimal end-to-end throwaway run that proves the idiom by execution, before plan/arch/code commit to it**. We did the equivalent — but reactively, at the end, on prod, instead of proactively, at the start, on a throwaway. Architecture builds on the research premise and is not obliged to re-litigate the cited stack-idiom → a wrong premise leaks silently into the design.

**Proposed text (PM-supplied):**
- *Integration-risk spike gate (`/pm-plan`):* "Before finalizing a plan/arch whose design hinges on a load-bearing SDK idiom OR that combines stack capabilities in a way the docs do not directly cover (a novel interaction — e.g. enabling capability X on a component where the sibling capability is deliberately off), require an early **execution spike (walking skeleton)**: the minimal end-to-end path, run on a throwaway target, that PROVES the idiom works in this configuration — before code and architecture are committed to it. Doc-cited feasibility is necessary but not sufficient; integration traps live in the interaction under the specific config and are 'absence' failures (a silent default/gate, no error) that only an actual run surfaces."
- *Stack-research must mark idiom confidence:* "`pm-stack-researcher` tags each load-bearing rule as **doc-cited (unverified)** vs **execution-verified**. Any idiom the design will hinge on must be escalated from doc-cited to execution-verified (via the spike) before `/pm-plan` finalizes — research-from-docs inherits the docs' happy-path blind spots, and a wrong idiom propagates silently through architecture, which builds on the research premise and does not re-litigate it."

**Design notes for planning.** (1) Trigger is a **judgement** ("does the design hinge on a load-bearing idiom / a novel cross-capability interaction the docs don't directly cover?") — the no-hook-for-semantic-judgement family; a `/pm-plan` gate + a `pm-stack-researcher` confidence tag, not a regex. (2) The spike is a **throwaway execution**, distinct from Step 5.5 "run it for real" (that validates the *finished* feature; this proves the *idiom* before design) and from a diagnostic probe (Step A.5 is reactive, post-incident; this is proactive, pre-design) — and the **Blast-radius preflight applies** (a spike on a live coupled peer is exactly what the preflight guards). (3) It is the **upstream root of the diagnostic-flow items** (passive-observation / bisect / anti-thrash) — those are reactive recovery; this is the proactive prevention. (4) Idiom-confidence tagging pairs with the `### Reporting discipline` verify-or-stay-silent rule (doc-cited vs execution-verified IS a confidence mark on a claim). *Path:* `/pm-plan` (a spike gate in `/pm-plan` + a `doc-cited | execution-verified` tag in `pm-stack-researcher` + the `workflow/incident.md` or a new spike home), single-source, no new agent (reuses `pm-stack-researcher` + the existing throwaway-spike concept). Highest leverage of the four — one early throwaway run would have caught all three layers before code+design. PM-relayed 2026-06-05.

## Diagnostic-flow gap (2): no anti-thrash brake — stack-research is a mid-debug escalation, not only an up-front step — 2026-06-05 (this repo)

PM-relayed feedback from the same live BLE-import session (sibling to the "Diagnostic-flow gap (passive observation + bisect)" entry below — both edit the `workflow/incident.md` "doesn't work in production" flow, and should be built together as one coherent diagnostic-discipline feature). The import failed; the agent fixed reactively — logs → patch one thing → still broken → probe → patch another → still broken → another probe — several patch iterations on the live system, each chasing the immediate symptom, never stepping back to "how is this actually supposed to work?" The PM stopped it directly: *"давай ресёрч, ты делаешь костыли."* Only then did the agent do a proper stack-research against the canonical matter.js flow.

**Two linked root gaps:**
1. **No "thrash detector" in the diagnostic flow.** `workflow/incident.md` is diagnose → probe → plan → fix. There is **no rule**: if behavior still contradicts your model *after the first fix*, the next move is **not another patch** — it is a check against the canonical source (reference implementation / spec / official example). Without it, the natural dynamic is a series of reactive edits, each locally cheaper than "stop and understand," but slower and dirtier in aggregate — especially on live hardware.
2. **`pm-stack-researcher` is positioned only as an up-front step.** It is invoked "from `/pm-plan` when a component is not yet in stack-notes." There is **no explicit trigger** for "re-research when hardware/reality contradicts an already-cited stack rule." That is exactly what happened: stack-notes said one thing (a soft BLE-registration idiom), the hardware showed another; then the high-level `commission()` did not scan BLE though the docs implied it should. Stack-research as a **mid-debug escalation** is a missing-but-needed first-class move.

**The subtle point.** When integrating an external SDK and the observed behavior diverges from your model *after the very first fix*, that is a signal the **model** is wrong, not the line. The right reaction is to find the reference implementation (how the library's own examples do it), not to patch further. The protocol does not encode this.

**Proposed text (PM-supplied, for `workflow/incident.md`):**
- *Stop-and-research tripwire (anti-thrash):* "If a fix attempt does NOT resolve a 'doesn't work' symptom and the observed behavior still contradicts your model of how the component works, the next move is NOT another patch. Stop, and consult the canonical source — the library's reference implementation / official example / spec — before the next code change. Two failed fixes in a row on the same symptom is a hard signal that the model is wrong, not the line; reactive patching past that point burns live-system iterations and converges slowly."
- *Stack-research is a mid-diagnosis escalation, not only an up-front step:* "Spawn `pm-stack-researcher` whenever reality contradicts an already-cited stack rule (hardware behaves unlike the documented idiom; the high-level API does not do what stack-notes says). Re-researching the canonical flow mid-debug — and correcting the stale cited rule — is a first-class move, not an admission of failure. Prefer 'find how the library's own example does it' over 'iterate a guess.'"

Severity: **medium.** Did not break the result (research was eventually run — the PM initiated it), but without the brake the protocol permits a long series of live-system edits where one canonical check would have saved them all. About the discipline "model before patch" and treating research as a mid-debug tool, not only a start-of-work step. *Path:* fold into the same `/pm-plan` on `workflow/incident.md` as the passive-observation + bisect item (one diagnostic-discipline feature); single-source, no new agent/command/hook (it *re-uses* `pm-stack-researcher`, just adds a mid-debug trigger). PM-relayed 2026-06-05.

## Agents must not assert VCS / repo state from narrative — verify-or-stay-silent, and stay in their lane — 2026-06-05 (this repo)

Live incident, this session. `pm-architect` (on the `test-wiring-parity` run) reported in its summary that `.ai-pm/state/archive/orchestrator-read-discipline-2026-06-05.md` was *"untracked … left alone."* **Verified false by git forensics:** the file's exact path was first committed in `0cc7ecc` (the #238 squash, v2.32.0) and at the time the architect ran it was **tracked, unmodified, identical to HEAD** on the branch. No data was lost — but the claim was a **confabulation**: the agent had read `.ai-pm/state/current.md`, whose prose narrates the "state-archive home gap" and names that archive file (e.g. "carried onto this branch at Step 0"), and it **blended that documented *history* into a present-tense git observation it never ran** (`git status` / `git ls-files` would have shown it tracked).

**Two distinct faults, both trust-eroding:**
1. **Asserting a fact from narrative/memory instead of verifying it.** The agent stated a checkable VCS fact (tracked vs untracked) without running the one-line command that decides it. The whole protocol rests on agent reports being accurate; a fabricated repo-state claim is worse than a missing one, because the orchestrator/PM may *act* on it (re-commit a "lost" file → duplicate, or panic over phantom loss).
2. **Lane-bleed.** `pm-architect`'s remit is the decision record in `doc/architecture.md`. Git / tracking bookkeeping is the **orchestrator's** domain. The false claim lived entirely in a comment *outside the agent's job* — it had no reason to narrate tracking state at all.

**Why it matters beyond this one file.** "We rely on these agents and they assert things that aren't true" (PM, 2026-06-05). The cost is not the file (it was fine) — it is that an unverified, out-of-lane factual claim in an agent summary looks identical to a verified one. Every agent report the orchestrator relays to the PM inherits this risk. This is the same root as the session's other items: *the report measures/claims something other than the verified reality* (the test-wiring-parity "test measures a path the app never takes") and *text shipped without being read against the real thing* (the changeset-hygiene legibility discipline).

**Where a fix could land (settle in planning):**
- **Verify-or-stay-silent for repo/VCS facts.** Any agent claim about repository state — tracked/untracked, committed, branch, file presence, test/pass counts, "X is byte-identical to main" — must be backed by a command the agent ran **this turn**, or not stated. (Several agents already do run `git`/`tests` and cite real output — `pm-coder` reports `hooks.sh` counts, `pm-plan-checker` re-runs the suite — so this codifies the good existing habit and bans the narrated-from-memory variant.)
- **Lane discipline for doc/review agents.** `pm-architect` (and the other doc/review agents) author their artifact and report on *that*; they do **not** narrate git/tracking/branch state — that is the orchestrator's domain. If a tracking observation is irrelevant to the agent's task (it always is, for the doc owner), stay silent.
- **Home:** likely a short rule in the shared agent guidance (the base instruction every `pm-*` agent inherits) + a one-liner in `pm-architect`'s remit; possibly a generalized "do not assert unverified facts; cite the command or omit the claim" line near the existing reporting conventions. No new agent/command/hook — a discipline + a remit boundary.

*Honest scope note:* this is a **discipline / prompt** fix, not a mechanically enforceable one — an LLM can still confabulate; the rule narrows the surface (don't volunteer VCS claims; cite-or-omit) and the lane boundary removes the most common occasion for it. Pairs with the "AI is bad at self-checking → independent verification" principle and the legibility discipline. Severity: **trust-level** — no breakage this time, but a fabricated repo-state claim is a latent foot-gun and a credibility tax on every agent report. *Path:* small `/pm-plan` (shared agent reporting rule + `pm-architect` remit boundary), single-source, no new agent/command/hook. PM-flagged 2026-06-05 ("заведи, мы же надеемся на них, а они врут").

## Diagnostic-flow gap: passive observation is read-only (not a probe) + "bisect before you hypothesize" — 2026-06-05 (this repo)

PM-relayed feedback from a live on-hardware test (Wiren Board Matter bridge, BLE device import). The feature silently failed; the agent went top-down — logs → dependency sources → a long stretch of code-level hypotheses. The PM suggested the trivial move: *"just listen to the air."* One **passive BLE scan** instantly bisected the problem (hardware / transport OK → the bug is in the pairing logic) and reached the root in minutes. The agent did not arrive at that observation on its own.

**Two linked root gaps in `workflow/incident.md` ("When you say it doesn't work in production"):**

1. **The protocol does not distinguish passive observation from a mutating probe.** Step A is read-only (logs, statuses, config files, deployed artifacts) and Step A.5 is a probe (needs explicit PM authorization, gated by the Blast-radius preflight). But **passive observation of a live signal** — sniffing/scanning/capturing what is already on the wire or the air (tcpdump, a BLE/mDNS scan, a bus capture, reading advertisements) — is materially **read-only** (changes nothing, reaches no external peer), yet it *feels* like "touching live hardware," so it lands in the Step A.5 psychological bucket and gets deferred. The cheapest, most informative diagnostic tool is under-used. **Honest correction to the PM's framing:** the feedback assumed `tcpdump`/`strace` are already named as allowed in Step A — they are **not** in the current `incident.md` text (Step A names only log/status/config/artifact reads). So the gap is slightly larger than described: passive signal observation is not named at all, which reads as a probe.
2. **No "bisect before you hypothesize" principle.** The flow runs logs → hypothesis → probe-to-confirm. It lacks the step: when a multi-stage path fails (hardware → driver → transport → protocol → app logic) and the symptom is silent/ambiguous, **first make the cheapest observation that splits the path in half**, and only then build a hypothesis. Standard debugging discipline; its absence invites top-down reasoning from the "most likely" software cause.

**The subtle side-effect worth naming directly.** The protocol's strong (and correct) "everything live = be careful" bias has a cost: it dulls cheap, safe observations. State explicitly that **passive observation is a read-only class, not a risk** — so the caution bias does not suppress it.

**Proposed text (PM-supplied, for `workflow/incident.md`):**
- *Step A (extend the allowed list):* "Passive observation of a live signal — sniffing/scanning/capturing what is already on the wire or the air (tcpdump, a BLE/mDNS scan, a bus capture, reading advertisements) — is read-only-grade diagnostics, in the same class as reading a log. It changes nothing and reaches no external peer, so it is not a probe: it needs no Step A.5 authorization and the Blast-radius preflight does not apply. Reach for it early."
- *New principle — "Bisect before you hypothesize":* "When a multi-stage path fails (hardware → driver → transport → protocol → app logic) and the symptom is silent/ambiguous, the first move is the cheapest observation that splits the path in half (does the signal physically reach us at all?), before reasoning top-down from the most likely software cause. Establish ground truth at the lowest cheaply-observable layer first."

**Boundary to preserve when implementing:** the passive-observation carve-out must stay genuinely passive — it permits *listening* to what is already on the air/wire, NOT injecting traffic, pairing, writing to a bus, or any action that reaches an external peer (those stay Step A.5 + Blast-radius preflight). The line is "observe vs emit": word it so an active scan that *transmits* (some BLE active scans request scan-responses; an mDNS query emits a packet) is judged on whether it reaches/mutates an external stateful peer — keep the carve-out for purely passive capture, and let the existing preflight govern anything that emits toward a coupled peer. Severity: **limited but real** — it did not break the result (the live test found the bug — that is what it is for) but lengthened the path to root. About diagnostic *speed and discipline*, not safety. *Path:* small `/pm-plan` on `workflow/incident.md` (Step A extension + the bisect principle), single-source, no new agent/command/hook. PM-relayed 2026-06-05.

## Test-wiring-parity: tests that diverge from the production initialization/registration path — 2026-06-05 (this repo)

PM-relayed feedback from the same live session. A feature (BLE provider registration) had **green tests and passed both review passes** (`pm-plan-checker` + `code-review` high) but **did not work on hardware**. Root cause: production registered the dependency one way (a "soft" reactive flag) while the unit tests registered the provider another way (explicitly / bypassing). The test verified *its own* registration scheme, not the one actually executed in `main`. A textbook "green tests, broken feature" — uncaught precisely because **the path in the test ≠ the path in production**.

**Root gap.** The protocol's test rules (`/pm-plan` Test plan rules, `pm-plan-checker`) require: every scenario covered; stack-spec tests assert against the *cited rule*, not a self-consistent mock. That is good against one class of falseness (a mock that froze a wrong spec). It does **not** cover another: a test that **initializes / wires the dependency differently from the production entry point**. If the test setup does not exercise the same registration/wiring path `main` uses, the test is green while production is dead. The stack-spec rule says "don't mock against yourself" — it does **not** say "initialize like production does."

**Why neither review nor checker caught it.** `pm-plan-checker` checks plan-compliance + presence of tests; `code-review` hunts bugs in the diff. Neither is obligated to verify that the **test setup reproduces the production initialization path** — this is not "no test" and not "a bug on a line," it is "the test measures the wrong thing." This is the sharper failure: the whole two-pass review gate passed a non-working feature — only the hardware caught it. Step 5.5 ("run it for real") exists for exactly this, but it is **optional and PM-owned**; the rule below moves the defense **earlier, into the test itself**.

**Proposed text (PM-supplied):**
> **Test-wiring-parity rule.** A test that sets up the unit's dependencies, registration, or environment differently from how the production entry point does can pass while production is broken — it measures a path the app never takes. For any feature whose correctness depends on initialization/registration order or wiring (a provider registered on a shared singleton, a side-effect import, a plugin/factory hook, env/DI setup), at least one test must drive the **same registration path the production entry point uses** and assert the observable post-condition that path is supposed to produce (e.g. `registerX()` → `container.has(X)` is true), not a hand-rolled equivalent setup. `pm-plan-checker` flags a wiring-dependent feature whose tests bypass the production registration path; `code-review` treats "test initializes the dependency differently than `main`" as a finding, not a style nit.

**Honest design notes for planning.** (1) The trigger is **judgement, not a regex** — "does this feature's correctness depend on init/registration/wiring order?" is a semantic call (like the NFR / state-model / security-surface triggers), so this is a **plan-time discipline + a `pm-plan-checker` semantic check**, not a hook. (2) It pairs naturally with the existing **stack-spec test rule** (same family: "test the real thing, not a self-consistent stand-in") — likely lands as a sibling Test-plan rule in `/pm-plan` + a `pm-plan-checker` dimension clause, single-sourced. (3) It overlaps the **seam-completeness** theme (#226) and the "плохо проверяет то что сделал" independent-review principle, but is distinct: those are about *unstated failure modes* / *who reviews*; this is specifically **test setup ≠ production wiring**. (4) Relation to Step 5.5: this does not replace run-it-for-real (the ultimate catch) — it moves a cheap, deterministic slice of that protection into every diff's test suite, so the modal wiring bug fails in CI, not on hardware. Severity: **higher than the diagnostic-flow item** — this is about the review gate passing a dead feature, not about diagnosis speed. *Path:* `/pm-plan` (a Test-plan rule + `pm-plan-checker` clause + a `code-review` finding-class note), single-source, no new agent/command/hook; sits alongside the B/C reviewability track and the seam-completeness work. PM-relayed 2026-06-05.

## Pre-review self-review hygiene — read the DIFF, prune leftovers, and READ the human-facing text before it ships (human pain, from a non-protocol workflow) — 2026-06-05 (this repo)

PM relayed a colleague's pain (a human who does **not** use the protocol): "sometimes code sent to review hasn't passed a simple self-review." Their four-point self-review checklist: **(1) always look at the DIFF** (the diff immediately shows what was forgotten and whether unused/leftover code remains — must look at the *diff* specifically, not the files); **(2) understand what we're doing and whether all the steps the code performs are actually needed**; **(3) are the variable names etc. understandable**; **(4) read the text the AI writes for humans — don't copy it unread; read it twice, thoughtfully; rewrite if it's unclear or hard to read.**

- **The honest tension to settle first — the protocol deliberately rejects "ask the maker to self-review."** The recorded principle (this backlog, the `pm-product-advocate` research lineage): *an AI is bad at checking its own output*, so the protocol's answer is **structural independence** — `pm-plan-checker` (compliance) + `code-review` (quality) are separate passes by separate agents, and `pm-coder` never signs off on its own work. So the colleague's advice is a **human** self-review discipline; for the AI maker the protocol's equivalent is the **independent** review, not a maker self-review step. **Do not add a "pm-coder self-reviews its diff" gate** — that would re-introduce the weak configuration the protocol designed away.
- **Points 1–3 are largely ALREADY covered — but by the independent diff-review, not self-review.** "Look at the diff / leftover unused code / unclear variables / unnecessary steps" is exactly what the per-diff `code-review` Pass-2 (diff-scoped) and the smell/hygiene sweep (dead code, over-complexity, simplification) already hunt. The net-new question is only whether anything in 1–3 escapes that net: e.g. *the orchestrator itself* relaying a coder diff onward without ever looking at the diff (the orchestrator runs `code-review` but may not *read* the hunks) — worth checking against the just-reinforced "per-diff = built-in code-review" routing.
- **Point 4 is the sharpest, partly-uncovered one — a human-facing-text legibility discipline.** "Read what the AI wrote for humans, twice, and rewrite if unclear" targets the **prose the protocol emits**: CHANGELOG entries, PR bodies, doc records, contracts, and code comments — not the code logic. It connects to **comment-restraint + documentation-minimalism (#386)** (rationale-leaked-into-code, AI over-polish) but extends past comments to *all* human-facing artifacts. The risk it names: AI-authored text pasted into a durable artifact **without a human (or the orchestrator) actually reading it for legibility**, so machine-texture / unclear prose ships. *Where a fix could land (settle in planning):* a lightweight **"read-before-ship the human-facing text" discipline** — the orchestrator (which already owns relaying to the PM and authoring the CHANGELOG/PR trail) must read AI-authored human-facing prose for legibility and rewrite-if-unclear before it lands, rather than copy-pasting agent output verbatim; possibly a `### Code conventions` / pm-comms rule + a `code-review` / `pm-auditor` *legibility* note (presence/clarity of human-facing text), kept proportional and **not** prose-policing-by-rubric. Pairs with #386's comment-restraint and the AI-texture work.
- **Honest scoping:** this is mostly a **re-confirmation that the existing independent-review + smell-sweep cover 1–3**, plus a **genuinely new legibility discipline for 4**. Risk of over-building: adding ceremony that duplicates code-review. *Path:* fold point 4 into the #386 comment-restraint/AI-texture plan (extend "comments" → "all human-facing artifacts: CHANGELOG/PR/docs read-before-ship"); for 1–3, a quick audit that the orchestrator-reads-the-diff expectation is real, no new gate. PM-relayed human pain 2026-06-05.

## Agent-handoff durability when "continue the same agent" is unavailable — 2026-06-04 (this repo)

- **An agent that stops mid-task at a blocking question holds valuable intermediate analysis only in its own context; if it cannot be resumed (the continue-the-same-agent tool was unavailable this session), that analysis is re-derived from scratch by a fresh agent — wasted tokens, plus a thin-loss risk if the orchestrator's hand-off brief omits a conclusion the stopped agent never wrote to disk.** PM observation 2026-06-04, from a live incident: `pm-coder` correctly stopped at a blocking conflict (a plan reversal — "sensors have no virtual device" — collided with three old test assertions, and its hard "never touch existing tests → stop and report" rule fired) after ~130k tokens of code analysis. The orchestrator authorized the fix (PM-approved reversal, not a regression) and wanted to resume the **same** agent with its context — but the continue-same-agent path was unavailable, so a fresh agent was spawned and re-briefed. **Correctness did not suffer** — the protocol deliberately keeps the source of truth in files (`state/current.md`, plan, arch notes, contracts), so the new agent rebuilt context from disk and finished cleanly (350 tests green); that is the designed fault-tolerance. **Efficiency and thin-loss did** — the new agent re-read the code, and any conclusion the stopped agent held only "in head" had to be hand-carried by the orchestrator into the brief. *Recommendation (three parts):* (1) **Make explicit in `WORKFLOW.md` that durable hand-off is the file-state, not agent memory** — continue-the-same-agent is an optional optimization the protocol must never depend on (saves tokens where available, breaks nothing where not). (2) **Agent rule on stopping at a blocking question:** before yielding control, write the work-so-far (the conflict area, what was already checked, what remains) to a durable artifact — its own review file, `state/current.md`, or a scratch note — so the successor/orchestrator doesn't re-derive it; then "hand-off" = a file, not a lost context. This extends the stop-and-report rule (e.g. `pm-coder`'s "stop and report" + the doc-conflict stop) from *report-the-conflict* to *also-flush-the-analysis*. (3) **Orchestrator rule:** when the same agent can't be continued, spawning a fresh agent with a brief that explicitly quotes the stopped agent's conclusions is the **standard** path, not an emergency. *One line:* the problem isn't a missing continue-tool — it's that valuable intermediate analysis lives only in the agent's context; the fix is the agent flushing it to disk on stop, which makes the continue-tool a pure optimization. *Where it lands:* `WORKFLOW.md` § "How state is kept" (durable-handoff = files principle) + the agent stop-and-report rules (`pm-coder.md` and the others' "stop and surface" paths gain "flush your analysis first"). Original framing kept below for history.
- **A run mode where the PM front-loads everything at bootstrap, then the orchestrator + agents carry the feature/product to the end with NO further questions — autonomously resolving product forks from (a) the initial bootstrap, (b) project logic, standards, and accumulated experience inside the project.** Today the protocol is interactive-by-default: every substantive product fork is surfaced to the PM (`AskUserQuestion`), and the new `pm-product-advocate` gate even *blocks* the coder handoff until the PM answers or descopes. Automode is the **inverse**: the PM declares intent + boundaries once, up front, and grants the pipeline authority to decide the rest. *Mechanism sketch:* a project-level setting (e.g. `CLAUDE.md` `## Decision authority: autonomous | interactive`, default interactive) that flips how forks resolve — in autonomous mode the orchestrator/advocate **resolve** each gap from the bootstrap baseline + project canon (`product.md`, `architecture.md`, contracts, prior decisions, standards) instead of relaying it, and **record the auto-decision with its rationale** (a decision log the PM can review after the fact, not a silent choice). *Key tension to resolve in planning:* this is in **direct tension with `pm-product-advocate`** — that gate is built to *force* the conversation. Automode does not delete the advocate; it **redirects its output**: the advocate still generates the foundational-question gaps (same fixed checklist, same artifact), but in autonomous mode the gaps are **answered by the pipeline from the baseline + recorded as auto-decisions**, blocking only when a gap is genuinely unresolvable from the available context (a true ambiguity the baseline does not cover — escalate even in automode). So the advocate becomes the *list of decisions that got auto-made*, giving the PM an audit trail. *Open design questions:* (1) what is the **escape valve** — a confidence/coverage threshold below which automode still asks (so it does not confabulate a product direction the bootstrap never implied)? (2) does automode also relax the human-in-the-loop ship gate (Step 6 A/B/C), or only the *product-fork* questions, keeping merge authority with the PM? (3) granularity — whole-project automode vs per-feature opt-in ("bootstrap this feature, then run it autonomously"). (4) how the recorded auto-decisions feed back into the project canon so later features inherit them (the "accumulated experience" loop). *Why:* for an experienced PM with a well-bootstrapped product, the interactive cadence is friction; automode trades PM attention for speed while keeping a reviewable trail — the opposite end of the same dial the advocate sits on. **Sequence after `pm-product-advocate` lands** — automode is defined *relative to* the advocate (it redirects the advocate's output), so the advocate must exist first. Likely `/pm-research` (autonomy/decision-authority patterns, confidence-gating for when an agent should ask vs decide — note the research open-question on clarifying-question generation that the advocate plan also left open) then `/pm-plan`. ### Design principle (why this stays readable AND doesn't drift — the frame for planning)

The three constraints — PM-readable / strict contract↔doc↔code chain intact / not smeared across many files — are not in tension. They are all served by **one invariant: one fact, one owner; the render *selects* a layer, it never *copies* it.** Two consequences worth carrying into the plans:

**Two parallel funnels, one entry each, linked one-directionally.** The PM never wades back into topics because the technical layer is a *separate* funnel reached only by an explicit "↓ Технически" link — not interleaved into the PM path.

```
PM funnel:   README (1 para) → product.md (why/for-whom) → product-map (features)
                                                              │ ↓ Технически
Tech funnel: architecture.md «## Behavioral contract» ────────┘ + contract Must-work + stack-notes
```

This adds **zero new files and exactly one new section** (`## Behavioral contract` inside `architecture.md`). The fact-ownership map is itself the anti-smear mechanism: smearing comes from the COPY pattern (a fact in two places), which the move-not-copy guard forbids at every level.

**Single owner applied at every level of the PM funnel — this is what keeps the PM description from drifting:**

| Level | Owns | Defers (links, never copies) |
|---|---|---|
| README | nothing about capabilities | → `product.md` |
| `product.md` `## Что умеет` | "why / for whom" + overall maturity boundary | → the map for the per-feature list |
| `product-map` | a *projection* of `User value` + `Out of scope` | (generated; re-derived & compared by auditor) |
| `contract.User value` | the single authored PM sentence per feature | rides the plan-gate + audit |

Drift analysis of the PM layer, level by level:
- **map ↔ contract: safe by construction** — the map is generated and the auditor re-derives & compares (`pm-auditor.md:111`). The cheap slice *removes* a drift surface: today the map carries a separately-authored guarantee string; making it a pure projection of `User value`/`Out of scope` deletes that extra copy.
- **`product.md` ↔ map: the real "two PM descriptions" risk** — resolved structurally by the README-fix applied one level down: `product.md` must NOT re-list per-feature capabilities; it owns only why/for-whom + the overall boundary and links to the map for the feature list. (If it re-lists, it will drift exactly as README↔product.md did downstream.)
- **`contract.User value` ↔ code: the one irreducible authored text** — but it drifts no worse than the technical contract does, because it rides the same enforcement: behaviour changed without a plan = orphaned-implementation **blocking** (auditor dim 3, `:90-95`); behaviour changed with a plan = the plan must touch the contract and its `Out of scope` "must not silently relax Must not break" (`contract.md.tmpl:61`); code changed after `Last reviewed` = stale-contract **note** (dim 4). The only gap no mechanism sees is a feature whose *value framing* changed while its *mechanics* (Must-work) did not — rare, low-harm. Two cheap backstops: a **structural** lint "PM layer contains no machine tokens (`<id>_<…>`, `retain`, `QoS`)" (the legitimate structural anti-drift, NOT prose-policing) and the PM re-reading the map at the validation point above.

Net: the PM description drifts only as much as the technical contract drifts — because it is a projection of the same single-owned source, not a parallel prose. The thing that would make it drift is COPY; the design forbids copy at every level.

## Seam-completeness + failure-inventory: closing the "flawless-where-stated, gap-on-the-unstated" blind spot — 2026-06-05 (this repo)

Source: PM's DriveBox uber-review (#224) + an AI-texture analysis. The protocol's strength — explicit requirements (SC1–SC5) → flawless implementation — is also its blind spot: the coder reasons **locally**, so a failure mode that falls on a **cross-module seam** and was never written into the requirements becomes a hole (offline `TransportError` escapes as a traceback; `KeyError` on a store record missing `real_id`) **even while every stated invariant is perfectly covered**. **Rules for the coder cannot fix this** — the seam is by definition not inside a function; ten more rules get obeyed within each function and the stitch is missed again. The PM's own diagnosis: the reviewer (pm-plan-checker + code-review) is tuned to plan-compliance + quality, **not to seam-completeness** — so this is a protocol gap, recorded as protocol-feedback (this repo is the upstream).

**1. Bug-class plug rules → the DOWNSTREAM project's `docs/stack-notes.md` (Python/Google idioms, via pm-stack-researcher; NOT this repo):**
- **Module-boundary catch-completeness:** enumerate every exception a function doing external I/O (API/network/FS) can raise; any exception crossing a module boundary uncaught at the CLI top level = defect (→ `TransportError` at `refresh()`).
- **Contract-consistency on shared data:** if one module treats corrupt/partial input as empty/refused (not a crash), every module reading the same source must behave the same; never subscript `dict[key]` on store data — `.get()` + explicit refusal (→ `KeyError` on missing `real_id`).
- **Pin lower version bounds for behavior-bearing deps:** a `getattr(o, attr, None)` without a fallback that silently changes behavior = defect; pin the min version (→ `HttpError.status_code` silent degradation).

**2. Meta-rules against the blind spot → `### Code conventions` (cross-cutting; these are protocol-level features):**
- **Failure inventory before implementation** (strongest idea): for each public function, list explicitly what invalid inputs it takes, what external failures are possible, what it returns/raises in each case. Most found holes are items nobody wrote down — making the implicit **explicit** lets the coder cover it as perfectly as it covers SC1–SC5. **→ a `/pm-plan` discipline (new feature).**
- **Test the negative space, not the invariant:** ≥1 test per function on an **unstated** path (network failure, partial data, race), not just the happy path with a documented rule. **→ a test-plan discipline (pairs with the failure inventory).**
- **Global pass after local work** (a *revision* step over the whole diff: file/function lengths vs limits; a pattern repeated that a sibling module already extracted; dead `__all__` exports). **→ MOSTLY ALREADY IN FLIGHT: `ai-minimums-linter-wiring` (lengths/dead) + the review-typology smell type (cross-module duplication, shipped slice 1). The "sibling already extracted this" dup is the smell type's job.**
- **Symmetry of paranoia:** if you guarded against failure X in one file, check sibling entry points of the same kind. **→ part of the seam pass.**

**3. THE LEVER (headline protocol gap) — a seam-completeness review pass.** A narrow reviewer checklist whose sole job is seams: "walk every exception that can cross a module boundary and confirm the top level catches it; for every read from a shared store, confirm the contract is consistent (corrupt → empty/refused, not crash)." This catches what the coder structurally cannot see. **→ a new review lens/dimension in `code-review` (and/or a reviewer instruction), per-change for newly-introduced seams; + the EPIC review-typology functional/integration type (slice 3) for the whole-system seams.** Highest leverage of everything here — more than any coder rule.

*Decomposition / sequence:* the **seam-completeness review pass** + the **failure-inventory `/pm-plan` discipline** are the two near-term features (highest leverage); negative-space testing pairs with the inventory; the global-pass is the in-flight linter + smell type; the Python idioms go to the DriveBox project's stack-notes (downstream). ## Idiom-as-pre-solved-edge-case + local idiom-inconsistency (deterministic) + shared-LLM-blind-spot — 2026-06-05 (this repo)

PM framing that sharpens the seam (#226) and stack-idioms-library (#227) notes — a theory of *why* idiom checks catch bugs, plus a cheap new deterministic signal and a confirmed reason to prefer tooling over the AI reviewer.

- **Theory: an idiom is not "prettier" — it is "a way that already accounted for the edge case."** `dict.get(k, default)` accounted for a missing key; `with` accounted for closing the resource; `frozenset` accounted for immutability. **Where code deviates from an idiom is, most often, exactly where a bug breeds** — the deviation is an edge case left uncovered. This reframes idiom enforcement from style to **edge-case coverage**, and explains why Semgrep-style idiom rules catch real bugs. *Design consequence (sharpens #227's economical standard):* each library entry names **the edge case the idiom pre-solves** — schema `idiom → edge case it covers → deviation = which bug` (e.g. `dict.get(k, d)` → "missing key" → `d[k]` subscript = `KeyError` on a partial record). The "why" is built into the one-line entry; no justifying prose needed.
- **A cheap new DETERMINISTIC signal: local idiom-inconsistency.** Real example (DriveBox `content.py:61-62`): `mime_type = meta.get("mimeType", "")` (defensive) immediately followed by `object_id = meta["id"]` (subscript → `KeyError`) — two different accesses to the *same source in the same scope*, one line apart. **Per-diff-detectable, Semgrep-able** ("mixed dict-access on the same object in one scope"), cheaper than full seam analysis — the *local shadow* of the seam blind spot (asymmetric paranoia in miniature). **This belongs as a deterministic rule, NOT a reviewer-lens instruction** (see next point).
- **Shared-LLM-blind-spot, confirmed on a live example.** This two-adjacent-line inconsistency was **missed by the protocol's own per-diff `code-review`** — the reviewer is an LLM and shares the coder-LLM's defect (locally-exhaustive, inconsistency-blind): it reads `.get()` ✓ and `[id]` ✓ and **never spontaneously asks "are these two consistent?"** because nobody asked. The same brain does not catch its own handwriting. *Consequence:* don't fix this by telling the AI reviewer to "look harder" — it structurally glosses idiom-deviation. **Encode the detectable cases as deterministic rules (Semgrep)**; reserve **cross-model AI review** (the separate "cross-model review for high-risk changes" backlog item — different model = different blind spots) for the *genuinely semantic* residue a rule can't express. This is the live justification for both the deterministic-vs-AI boundary (#211) and the cross-model item.
- **Unified frame (three tiers, one root — "idiom deviation = bug"):** deterministic (idiom-deviation, local-inconsistency, seam-exception-completeness) → **Semgrep/rules, every diff, free**; genuinely-semantic → AI, and for high-risk → **cross-model** AI. Folds into the stack-idioms library (#227, entries as Semgrep rules) + the seam-completeness pass (#226, the cross-file semantic tier). PM-supplied 2026-06-05.

- **Backlog in/out discipline (PM's "what can go in the backlog and what can't"):** keep the backlog the canonical ledger of *consciously deferred* work.

  - **Out / dedup:** a finding already accepted (do not add a second copy); a re-discovery of an accepted item (the review pass should suppress it at raise-time, per above); a fresh higher severity slapped on an item the team already triaged lower.
- **Reuses existing machinery:** the protocol already records `accepted (auditor-<date>)` / `accept-with-context → backlog`. The gap is only that a *fresh review pass does not read it back*. Fix = make review passes read-and-dedup against the accepted ledger; state the in/out discipline once (where the triage loop lives — `/pm-audit` + the reviewer instruction). *Likely path:* fold into the seam-completeness-pass plan (#226) as a required property + a short in/out rule near the accept-with-context discipline. ## Severity triage must be deployment-context-aware; context-enrichment before cross-model — 2026-06-05 (this repo)

Source: PM's DriveBox review of a second deferred item — an unbounded `download_text_blob` (`get_media().execute()` loads the whole blob into memory, then decodes to `str` → peak memory doubles; no streaming, and unlike `export_text` no Drive-API 10 MB safeguard). The Pass-2 triage assessed it **Low** ("curated workspace, usually small; v1 didn't promise streaming") — but that was **incomplete**, and the contrast with the (accurately-Low) `real_id` item is the lesson.

- **Deployment context changes severity, and static code analysis is blind to it.** Two factors the triage missed: (1) the **target is embedded WB controllers** (low RAM, OOM both more likely and more harmful — it can take down neighbouring device services), not a 16 GB dev laptop; (2) **folder-relative allow-list entries weaken "curated"** — the user allow-lists a *folder*, not a file, so `read <folder>/huge.log` resolves an un-vetted child at call time, and big text files (logs / CSV / NDJSON, all in `_TEXT_BLOB_MIMETYPES`) are exactly that. So "curated and small" does not hold for folder branches. Verdict: **Low–Medium**, not Low (still not critical — read-only, local, single-user; OOM kills the process, no token leak / no data loss / trivial recovery; availability, not security), and **higher priority than the `real_id` item** (that needs manual config corruption — rare; this triggers on a normal path on normal hardware), worth pulling out of backlog before deploying to real controllers.
- **The protocol ALREADY has the missing context.** `docs/architecture.md` `## Operational limits & budgets` (the RAM budget — shipped in **nfr-operational-limits-prompt v2.23.0**), `## Architectural constraints` (embedded), and the threat-model are exactly what would have bumped this finding's severity. **Fix direction: severity triage (in `code-review` / the reviewer + `/pm-audit`'s finding-walk) must consult `## Operational limits & budgets` + `## Architectural constraints` + the threat-model when assessing resource/availability findings** — closing the loop from the nfr-slice (the budget section) back into review severity. (The fix itself is near-free here: `size` is already in `GET_FIELDS`, so compare `meta["size"]` to a limit before `get_media` — cheaper than "mirroring the export guard", the data is in hand.)
- **Same model, different prompt → a better catch — undercuts "need a different MODEL" for this class.** Both reviewers (the original Pass-2 and the fresh uber-review) are the **same underlying model, only different prompts/rules**. The better assessment came from the prompt/rules that supplied **deployment context**, not from a different brain. *Consequence — sharpens the "cross-model review for high-risk" backlog item:* distinguish a blind spot from **missing CONTEXT** (fix cheaply by enriching the same model's prompt/rules — deployment target, op-limits, accepted-history) from a blind spot from **shared model bias** (fix with cross-model — different blind spots). **Most of these misses were the former.** So: context-enrichment first (cheap, often sufficient); cross-model reserved for the residual shared-bias blind spots that survive good prompting.
- Ties: nfr-operational-limits-prompt (the budget section → review severity input); the seam-completeness pass (#226 — the folder-weakens-curated contract seam); the cross-model item (context-vs-bias split). PM-supplied 2026-06-05.

## Review-typology engine selection: the code-review-orchestrator is the whole-codebase engine (refines the deny + the EPIC) — 2026-06-05 (this repo)

The irony that resolved it: the DriveBox uber-review that produced all the sharp findings above was run **by `wb-development:code-review-orchestrator`** — the very skill the protocol blocks (`deny-review-orchestrator`, v2.25.1). It is a multi-aspect whole-codebase reviewer — i.e. exactly the engine the review-typology EPIC needs for its **smell / architectural / functional whole-codebase sweeps**. So the resolution is **not "unblock everything"** but a **context split**:

- **Per-diff Pass-2** (the review loop, every change) → the protocol's own `/code-review`. ## Comment-restraint + documentation-minimalism: the over-documentation "noodles" (deterministic subset + AI convention) — 2026-06-05 (this repo)

Source: PM, off the AI-texture analysis. AI-generated code over-documents — justifying comments in paragraphs, per-function docstrings on trivial functions, inline rule-ID citations (`# SC1`, `# AC6`), glossary terms repeated verbatim across files. This is rationale leaking *into* the code; it carries overhead, drifts (a `# 3.10+` comment becomes a lie), and is the loudest "machine texture" tell. PM asks: standards + tools for minimal-but-quality docs, auto-checkable?

- **Standards exist but with a trap.** "Comment WHY not WHAT", self-documenting code (names > comments), DRY-for-docs (don't restate the code), PEP257 (format). **Trap:** most documentation *linters* enforce docstring **presence** (pydocstyle / ruff `D` / pylint missing-docstring / interrogate / darglint) — the **opposite** direction for over-documenting AI; wiring them would amplify the noodles. **Do NOT wire docstring-presence linters.**
- **Deterministic subset (gate-able, no model):** **ban inline rule-IDs in code** (`# SC1` / `# AC6`) via Semgrep/ast-grep — it is the protocol's own reference-don't-duplicate (the ID's home is `docs/` + the test; inline is a drift surface); **flag a docstring on a trivial / short / `_private` function** (Semgrep: "docstring on a function ≤N lines"); a crude **comment-density / comment-to-code ratio** gate (catches paragraph-justifications); **Vale / proselint / write-good** for the *prose* docs (wordiness/passive/weasel — prose quality, not "is this comment needed").
- **Semantic core → AI + a comment-restraint CONVENTION (not a tool).** "Justifies-not-describes", redundant rationale, uniform over-polish are value-vs-noise judgements a tool can't reliably make. Add a **rule of comment restraint** to `### Code conventions`: comment **why-when-non-obvious only**; **rationale lives in the plan / arch / contract / TEST, not inline** (a test like `test_token_written_0600` documents the SC1 invariant better than a `# SC1-safe` comment); no docstrings on trivial functions; don't repeat glossary terms. The AI per-diff/smell review checks it.
- **The clean structural fix (no magic "over-comment detector"):** over-documentation is **rationale leaked into code** → the protocol's own reference-don't-duplicate moves it out: "why" → plan/arch/contract/test (the **failure-inventory** #226 already moves the "why" of edge cases into the plan); code stays minimal. Close the loop with: the comment-restraint convention + the Semgrep inline-rule-ID ban + the "don't wire docstring-presence linters" rule.
- **Same split as everywhere:** deterministic (inline-ID ban, trivial-docstring flag, density gate) → rule; semantic (over-justify, uniform-polish) → AI convention. **Fold the doc-minimalism tooling (Vale + the doc-density/inline-ID Semgrep rules) into the same `/pm-research` on the deterministic gate** (#227). PM-supplied 2026-06-05.
