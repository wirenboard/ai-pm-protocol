# state-model-section — design notes

## Context

EPIC cross-document-consistency auditor, slice 4 — the **third whole-system-property gap** the downstream review named (siblings: invariants-index slice 1 v2.21.0, taxonomy-drift-sweep slice 2 v2.22.0, nfr-operational-limits-prompt slice 3 v2.23.0, all shipped). The gap: a state/event model that is implemented and justified but **embedded in prose** — status values, transitions, recompute/transition triggers, debounce written as paragraph-form "why" — forces a reader to reverse-engineer "the state machine" from narrative. The expert flagged this one **conditional** ("less critical for a Matter bridge than for a state-heavy messenger"), which points directly at the form: a conditional section, not a blanket requirement.

This slice adds (1) a **conditional `## State model` section** to `docs/architecture.md` — a states×transitions table (+ triggers/debounce) that **references** the existing prose and the status taxonomy in `## Behavioral contract`, never re-derives them; and (2) a **conditional `/pm-plan` State-model check** that fires when the orchestrator judges the feature's core **is** a state machine / protocol. It is the direct structural sibling of the nfr-prompt slice (conditional prompt + conditional architecture section) and reuses that pattern verbatim.

Prose-spec / meta-feature on the template repo (software-kind, no-user-facing-contract exception — every scenario subject is `/pm-plan` / the planner / `pm-architect` / a template, non-human). No runtime, no shared state. "Adjacent implementations" are the protocol's own established patterns, mapped below.

Per the brief, I **settle by nfr sibling precedent** everything slices 1–3 already fixed (A2-walk/A4-exclude, born `N/A`, `[?]`-not-invented, post-coding handoff, conditional-prompt-beside-the-others placement, no hook, no hard gate) and spend the variant work on what is genuinely new: **Q1** (placement + the reference relationship to `## Behavioral contract`, and whether a states×transitions table is an inherent partial restatement of the enum) — the real fork. Q2 (detection signal) and Q3 (boundary with the deferred journeys-diagram slice) are judgement calls I settle without a variant fork, mirroring the nfr note's Q2/Q3 treatment.

## Adjacent patterns (protocol patterns of the same shape)

1. **The conditional pre-draft checks in `/pm-plan`** (Stack-component, Interaction-scenario, Security-surface, NFR/operational-limits) — all judgment-triggered, silent when their trigger is absent, recording into existing-owner homes, never a new mandatory section. The **NFR/operational-limits check (slice 3)** is the immediate twin: same shape, same post-coding "Docs to update" handoff to `pm-architect`, same no-hook reality. The new State-model check sits beside it as a same-shaped conditional. This is the settled placement — no fork.

2. **The conditional-architecture-section discipline** — `architecture.md.tmpl` `## Integration contract`, `## Behavioral contract` (~line 72), `### System invariants` index (~line 89), `## Operational limits & budgets` (~line 48). Every conditional section is born `N/A — <reason>`, `[?]` when unknown, never invented to fill, walked by A2, **excluded from the A4 cross-check set** (authored domain content with no external artifact to diverge from). The new `## State model` section is a direct sibling of this shape.

3. **The move-not-copy / reference-don't-restate discipline** — the spine of the whole EPIC. The Behavioral contract states each identifier **once**; `docs/user-journeys.md` and the contracts reference it by section name; the `### System invariants` index points *out* at each home (`SCn` ID / journey name) and keeps **no back-link**, duplicating no rule text. Q1 below turns entirely on applying this discipline to a *table that names states* — the one place the State model risks a partial restatement.

4. **The soft-enforced + no-hook family** — when the trigger is a **semantic judgement a regex cannot make**, no `PreToolUse` hook is added and no hard gate is imposed; the prompt degrades silently if misjudged, and the home keeps a recorded model recoverable at the next refresh. "Is this feature's core a state machine / protocol?" is exactly such a judgement, so this slice's no-hook / no-hard-gate choices are family-consistent — settled, not re-argued (the nfr note's Adjacent pattern 4 and Q3 carry the full argument; this slice inherits it).

## Behavioral risks in this area

Not event-driven code — no subscription/feedback surface. The risks are **protocol-design** risks: (a) the states×transitions table partially restates the status enum and drifts against it — the Q1 drift concern, the one risk genuinely new to this slice; (b) a detection signal that over- or under-fires (Q2); (c) the home accidentally needs the deferred journeys-diagram auditor half to be coherent (Q3). The slice's value is a single, referenceable, drift-bounded home for the transitions; the risks are all about keeping that home from becoming a *second* copy of the state names.

## Settled by nfr precedent (no variant work — stated briefly)

- **A2-walk / A4-exclude:** the new `## State model` section is added to `pm-architect` Section A's A2 walk-list and ownership/handoff, walked `N/A — <reason>` when empty, and **excluded from the A4 cross-check set** (which stays exactly three pairings: File layout ↔ tree, Release flow ↔ CI, Integration contract ↔ README). A states×transitions table is authored domain content with no source-tree / CI / README artifact to diverge from — identical to the Behavioral-contract / System-invariants / Operational-limits exclusions. Get this exact in coding: A2 yes, A4 no.
- **Born `N/A`:** the template section ships `N/A — <reason>` default; the protocol tooling's own `doc/architecture.md` carries it `N/A` (the tooling is not a state machine / protocol).
- **`[?]`-not-invented:** a transition / trigger the orchestrator cannot determine without real system knowledge is recorded `[?]` for the PM — never a guessed transition. The "never reverse-engineer / never invent" line (plan Out-of-scope) is load-bearing here exactly as it was for budgets.
- **Conditional-prompt-beside-the-others:** the State-model check joins the existing conditional pre-draft checks after the NFR/operational-limits check — same judgment-triggered, proportional, no-hook shape.
- **No hook, no hard gate:** family-consistent (Adjacent pattern 4); the prompt degrades silently if misjudged; the home makes a recorded model auditable. A did-it-right auditor check is a deferred slice-N candidate (see Q3), not this slice.
- **Already-captured guard:** inherited verbatim from the NFR system-budget guard — once the section captures the machine (a non-`N/A`/`[?]` table), the check is silent on already-modeled states and fires only for a *new* state / transition / trigger. No re-litigation.

## The real fork — Q1: where the `## State model` section lives and how it references the status enum without restating it

The plan places the section **after `## Behavioral contract`, referencing the enum, never restating it**. The brief asks whether that is right, or whether the state model should live **inside** the Behavioral contract (as a sub-block like `### System invariants`) or **stand fully separate** — and, underneath placement, whether a states×transitions table that *names* the states is an inherent partial restatement of the enum (a drift risk), and if so how that drift is bounded.

These are two coupled questions — placement and the reference relationship — so I treat them as one fork with two variants on placement, and settle the drift-bounding rule (which holds under either placement) explicitly.

### The drift-bounding rule (holds under every variant — settle this first)

A states×transitions table **does** name the states, so it **does** touch the same value set as the enum. The bound that keeps this from being a restatement is a **column-level split of ownership**, the same one-fact-one-owner discipline the threat → constraint and journey → Behavioral-contract splits already use:

- **The value set (which states exist, what each means) is owned by the `## Behavioral contract` status enum** — the State model does **not** redefine it. The table's state column is a *reference into* that enum (the enum is the source of truth for "is `failed` a legal state?"), not an independent declaration.
- **The transitions (which state moves to which, on what trigger, with what debounce) are the State model's own, new datum** — they exist **nowhere else**, so the table is not a copy of anything; it adds the edges to the enum's nodes.

So the table is **not** a partial restatement *if and only if* it is disciplined to add edges, not re-declare nodes: the state names appear as row/column labels that point at the enum, and the meaning/legality of each state stays in the enum. The drift risk ("6 states in the enum, 7 in the table") is bounded by making the enum the **single source of truth for the value set** and the table a consumer of it — exactly the move-not-copy contract the journeys already follow against the same enum. Note the enum template at `architecture.md.tmpl:78` *already* carries an "Allowed next" column; the State model's value-add over that is the **triggers/debounce** (the *why/when* of each edge), which the enum's terse "Allowed next" does not capture — that is the non-overlapping datum that earns the section its keep.

This rule must land as the section's one-line guidance and in `pm-architect`'s section guidance: *the state column references the `## Behavioral contract` status enum (source of truth for the value set); the table's new datum is the transitions and their triggers — never re-declare or re-mean a state here.*

### Variant A: a new top-level `## State model` section, placed immediately after `## Behavioral contract` (plan's choice — recommended)

- **Where:** a new conditional `##` section in `architecture.md.tmpl`, **immediately after `## Behavioral contract (taxonomies & invariants)`** (and after its `### System invariants` sub-block), `N/A — <reason>` default; added to `pm-architect` A2 walk-list + Section A ownership/handoff, A4-excluded.
- **Relation to adjacent:** symmetric with `## Operational limits & budgets` (slice 3) and `## Behavioral contract` — a conditional, born-`N/A`, A2-walked, A4-excluded, authored-on-handoff section. A *states×transitions table + triggers* is a distinct content shape (directed edges with conditions) with a distinct reader (someone tracing the lifecycle), warranting its own labelled home the way taxonomies and budgets earned theirs. Placement *immediately after* the Behavioral contract makes the reference relationship physically obvious: the enum (nodes) is the section directly above, the transitions (edges) the section directly below.
- **Pros:**
  1. **A real, greppable pointer target** — the `/pm-plan` check and any future did-it-right auditor check can reference `## State model` by section name (the same reason slice 3 gave the budget its own section rather than a bullet). A sub-block heading inside another section is a weaker, less stable pointer target.
  2. **Clean A2/A4 inheritance** — born `N/A`, walked, A4-excluded, exactly mirroring the two sibling conditional sections; no new discipline to invent.
  3. **The drift-bound is visually enforced** — the enum sits one section up; "reference the enum, add the edges" is the natural reading order, lowering the chance a coder re-declares states.
  4. **Proportional** — a non-state-bearing project carries one `N/A` line, the same modest cost the other conditional sections already pay.
- **Cons:** one more section on the A2 walk-list and one more `N/A` line every non-state-bearing project carries — the genuine cost the brief flags, identical to the cost slice 3 accepted. Mild and born-`N/A`, so the walk is one line.
- **Risks:** none structural. The one discipline to keep exact: A2-walked but **A4-excluded** (a self-inflicted A4 finding on every authored state model otherwise), and the drift-bounding rule above carried into the guidance line so the table references the enum rather than re-declaring it.

### Variant B: a `### State model` sub-block inside `## Behavioral contract`, beside `### System invariants`

- **Where:** no new top-level section; a `### State model` subsection under `## Behavioral contract`, adjacent to the existing `### System invariants` index, holding the states×transitions table + triggers.
- **Relation to adjacent:** plausible on its face — the state *names* already live in this section's enum, so the transitions between them are "more of the same domain machine fact", and co-locating edges with nodes keeps the whole state machine in one section.
- **Pros:** zero new top-level section, zero A2 walk-list growth (the Behavioral contract is already walked), the enum and its transitions sit in one place so the value set and the edges are never more than a scroll apart; the drift-bound is even tighter (same section).
- **Cons:**
  1. **Weaker pointer target** — the `/pm-plan` check (and the deferred did-it-right auditor check) must reference "the `### State model` sub-block of `## Behavioral contract`" rather than a top-level section; a sub-heading is a less stable, harder-to-grep anchor than a `##` section, undermining the "reference the home by name" requirement (sc3) the same way slice 3's Variant-B bullet did.
  2. **Conflates two `###` roles** — `### System invariants` is an *index that points out* (by `SCn` / journey name, no content of its own); a `### State model` would be a *content table* living beside it. Two structurally different sub-block kinds under one `##` reads as a grab-bag and blurs the Behavioral contract's clean "taxonomies + an out-pointing index" charter.
  3. **No conditional-section `N/A` contract at the sub-block level** — the Behavioral contract is itself conditional, but adding a born-`N/A` *sub-block* means a project with taxonomies but no state machine must write `### State model: N/A` *inside* a non-`N/A` section, an awkward shape the top-level conditional sections avoid by being whole sections.
  4. **Breaks the slice-3 precedent** — the nfr slice deliberately chose a new top-level section over folding into an existing one for these exact pointer-target / shape-mixing reasons; Variant B re-opens a fork slice 3 already settled the other way.
- **Risks:** the state model hides as a sub-block and the EPIC's home-then-check cadence has a weaker anchor to attach the deferred journeys-diagram / did-it-right check to.

### Variant considered and rejected: fully separate top-level section unrelated to `## Behavioral contract`

Placing `## State model` elsewhere (e.g. after `## Architectural constraints`, divorced from the enum) is **not viable**: it severs the table's state column from its source of truth, making the drift-bound a remote cross-reference instead of an adjacent one and inviting the very enum-restatement the discipline exists to prevent. The state names' single home is the Behavioral-contract enum; the State model must sit where that adjacency is obvious. Adjacency-to-the-enum is the binding constraint, which is why Variant A places it *immediately after* the Behavioral contract, not merely "somewhere top-level".

## Recommendation

**Variant A — a new conditional top-level `## State model` section placed immediately after `## Behavioral contract (taxonomies & invariants)`, as the plan proposes.** A states×transitions-plus-triggers table is a distinct content shape with a distinct reader (someone tracing a lifecycle), and giving it a named, conditional, born-`N/A` top-level home buys the same three things slice 3's section bought: a real `##` pointer target for the `/pm-plan` check's "reference by name" requirement (and for the deferred did-it-right check), clean A2-walk/A4-exclude inheritance, and a placement whose adjacency to the enum makes the move-not-copy reading order natural. The drift concern is real but **bounded by column-level ownership** — the enum owns the value set (the table references it), the table owns only the transitions and their triggers (which exist nowhere else) — and that bound holds regardless of placement, so it is not an argument for folding the table inside the Behavioral contract. The cost (one A2 line, one `N/A` per non-state-bearing project) is the same modest, born-`N/A` cost the sibling conditional sections already pay. Keep it A2-walked but A4-excluded.

## Settling Q2 — the detection signal ("state-machine / protocol-bearing feature/project")

The brief asks whether state-bearing has a durable **presence-signal** (analogous to "`threat-model.md` present ⇒ security-bearing"), or is, like the NFR trigger, irreducibly a per-feature/per-project judgement. The two candidate durable signals the brief names: (a) the Behavioral contract already declaring a status enum *with lifecycle*, or (b) the `## State model` section itself once non-`N/A`.

**Settled: irreducibly a judgement at fire time; the only durable signal is the already-captured guard, and it is a *narrowing* signal, not the trigger.** Mirroring the nfr Q2 analysis:

- A **status enum in the Behavioral contract** is **not** a sound trigger. Many projects carry a status enum (the template's own example does) without their *feature core* being a state machine — an enum is a value set, not evidence that *this feature* introduces or alters a lifecycle. Keying on "enum present" would **over-fire** on every project that has any status field. The enum is what the state model *references*, not what *triggers* it.
- The **`## State model` section being non-`N/A`** is a real durable signal, but — exactly like the NFR system-budget guard — it is the **already-captured guard**, not the fire condition. It answers "is this machine already modeled?" (so the check stays silent on already-modeled states), **not** "is this feature's core a state machine?" (the question that decides whether the check fires *at all* on a fresh feature). It cannot bootstrap the first fire, because before the first state-bearing feature the section is `N/A`.

So the trigger is, like the NFR trigger, an **irreducible semantic judgement with no presence shortcut**: *is this feature's core a state machine / protocol* — a status ladder with defined transitions, a named-state lifecycle plus the events that move between states, or a wire/connection protocol with message/connection states? Re-judged each plan from the feature description. There is no per-feature artifact to key on, and no project-level marker file should be manufactured (that would be a new mandatory artifact, contradicting the slice's "no new mandatory document" remark and the no-hook family).

**Soundest formulation:** a single judgement against a **concrete state-machine/protocol list** (status ladder + transitions / named-state lifecycle + triggering events / wire-protocol message-or-connection states), **narrowed by the already-captured guard** (non-`N/A` `## State model` section ⇒ silent on already-modeled states, fire only for a *new* state/transition/trigger). The list must stay concrete in the spec so "state machine" is not read loosely as "any feature with a status field".

**Over/under-fire risk for the PM (named, not a defect):**
- *Under-fire (accepted residual):* the orchestrator judges a genuinely state-bearing feature as non-state-bearing → no prompt → the transitions go unwritten. This is the silent-degradation the slice accepts (no hard gate); the home makes a later-captured model recoverable at the next refresh.
- *Over-fire:* an ordinary feature with a flat status field (no transitions worth modeling) is treated as a state machine and surfaces a State-model question that resolves to `N/A` or a one-row table. Low-cost (one silent-resolving prompt), but worth keeping the trigger list concrete so a mere enum does not read as "state machine".

This is the **same conclusion the nfr note reached for its trigger** — no durable presence-signal, irreducible judgement, gate the read on concrete criteria — with one project-specific addition the nfr trigger lacked: the `## State model` non-`N/A` state *is* a real durable signal, but it serves only the already-captured guard (narrowing), never the initial fire decision.

## Settling Q3 — boundary with the deferred "state-machine ↔ journeys single-diagram" auditor sub-class

The plan scopes **this** slice to "establish the single home + the prompt that fills it" and defers the **state-machine ↔ journeys single-diagram** auditor check (one canonical diagram referenced, not redrawn per journey) to a later slice. The brief asks whether that boundary is clean, and — critically — whether this slice **accidentally needs the auditor half to be coherent**.

**Settled: the boundary is clean, and this slice is self-coherent without the auditor half.**

- **This slice = home + prompt.** It creates the `## State model` section (the single home for the transitions) and the `/pm-plan` check that fills it. Its correctness is entirely internal: the section exists, is born `N/A`, A2-walked/A4-excluded, references the enum without restating it (Q1), and the prompt fires on the judgement (Q2). None of these depend on any *other* document referencing the section.
- **The later slice = the drift check.** "Do `docs/user-journeys.md` (and other docs) *reference* the single state diagram rather than *redraw* it?" is a `pm-auditor` drift sweep — the direct sibling of slice 2's journeys identifier-restatement check and the threat → constraint dangling-`SCn` check. It validates that consumers point at the home rather than copying it. That check has **no corpus to run against until this slice ships and a state-bearing downstream project records a real model** — the same premature-tuning trap the nfr Q3 named for its did-it-right check. It is correctly deferred.

**Does this slice accidentally need the auditor half?** No — and the reason is structural, mirroring the nfr Q3 "home-first / check-next cadence" exactly:

1. The home is **self-justifying**: a single labelled place for the transitions is valuable on its own (a reader finds the state machine in one section instead of reverse-engineering prose), independent of whether any check polices consumers. Slices 1 and 3 both shipped their home before their check; this slice follows the same rhythm.
2. The **move-not-copy discipline this slice already carries is inward-facing** — it governs how the State-model table references the *Behavioral-contract enum* (Q1's drift-bound), which is fully within this slice. The *outward-facing* discipline (how journeys reference the State model) is the deferred slice's concern and does not bear on this slice's coherence. The two are different edges of the reference graph: this slice owns `State model → enum`; the later slice owns `journeys → State model`.
3. There is **no forward dependency**: nothing in the `/pm-plan` check or the section guidance references or presumes the deferred auditor check. The plan's Out-of-scope (lines 67–68) draws the line explicitly and the home is coherent standing alone.

One boundary note worth carrying for the PM: the deferred slice's name ("single-diagram") implies the State-model home may eventually hold (or reference) a *diagram*, while this slice's section is a *table + triggers list*. That is consistent — a table is a textual state diagram, and the deferred check will police "reference the home" regardless of whether the home renders as a table or a diagram. This slice should **not** pre-commit the section to a diagram format to "prepare for" the later slice; a table is the right shape now (greppable, drift-boundable against the enum, no rendering dependency), and the deferred check keys on *reference vs redraw*, not on the home's internal format. Keeping the home a table now does not constrain or pre-empt the later slice.

## Risks for the PM

1. **The states×transitions table is the one place this slice risks a partial restatement of the enum — bound it by column-level ownership (Q1, the one new discipline).** The enum (`## Behavioral contract`) owns the value set; the table references it and owns only the transitions + triggers (which exist nowhere else). The section's guidance line and `pm-architect`'s section guidance must say this explicitly: *reference the status enum, add the edges, never re-declare or re-mean a state here.* Without that line, a coder writing the table is one slip away from a second copy of the state names that drifts — the exact failure the EPIC exists to prevent.

2. **Keep the section A2-walked but A4-excluded (the same exact-in-coding point as slice 3).** A states×transitions table is authored domain content with no source-tree / CI / README artifact to diverge from — so it is added to `pm-architect`'s A2 walk-list (walked, born `N/A`) but **must not** enter the A4 cross-check set (which stays exactly three pairings). Mirror the Behavioral-contract / System-invariants / Operational-limits exclusions; getting it wrong manufactures a self-inflicted A4 finding on every authored state model.

3. **The trigger is irreducibly a judgement — and a status enum is *not* the trigger (Q2).** There is no durable "file present ⇒ state-bearing" signal. Do not key the fire on "the Behavioral contract has a status enum" — many projects carry an enum without their feature core being a state machine; that over-fires. The only durable signal is the `## State model` section being non-`N/A`, and it serves only the **already-captured guard** (silent on already-modeled states), never the initial fire. Keep the state-machine/protocol trigger list **concrete** in the spec so "state machine" is not read as "any feature with a status field". Accept the under-fire (a misjudged trigger drops the prompt silently — recoverable at the next refresh).

4. **No hard gate this slice — and a did-it-at-all gate would be incoherent, not merely over-built (inherited from nfr Q3).** A gate "did the State-model check fire when it should have?" would have to re-derive the semantic trigger no checker can evaluate. A narrow **did-it-right** check ("if a `## State model` section is non-`N/A`, does it reference the enum rather than restate it, and do journeys reference it rather than redraw it") *is* coherent and is the natural later-slice candidate — but it has **no non-`N/A` corpus to tune against until this slice ships** (the premature-tuning trap), and it is precisely the deferred journeys-single-diagram sub-class. Keep it in Out of scope, not pulled forward.

5. **`[?]`-not-invented for transitions (inherited, load-bearing here).** A transition / trigger / debounce the orchestrator cannot determine without real system knowledge is `[?]` for the PM — **never a guessed edge.** A confabulated transition reads as authoritative and is worse than `[?]`; never reverse-engineer the model from code (plan Out-of-scope). The Behavioral-contract "never invent to fill" discipline applies verbatim; carry it into the section's guidance line.
