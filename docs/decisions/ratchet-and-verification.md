# Decision — regression ratchet + plan-level verification scenario

**Question:** how should the protocol require regression-ratchet tests and plan-level verification scenarios, so "compiled = working" false-greens stop recurring? (Field evidence: ad-md-editor, 2026-06-13 — four bugs of one class: code compiled, review passed, user-visible behaviour broken; one bug returned after a refactor because no test pinned it. Backlog: "ratchet-test + verification scenario gap".)

**Answer (researched 2026-06-13, confidence: high on the evidence, the floor-vs-module fork is the Operator's):**

1. **Ratchet rule, broadened and promoted to the Builder/Reviewer floor.** The rule exists today only as a test-methodology module row scoped to "a gate-caught bug" — and the module WAS on in the downstream (`kind: code` ⇒ depth `rich`) yet missed all four bugs, because they were *user*-caught, not gate-caught. The defect is the scoping, not the toggle. The industry form is unconditional: a defect fix carries the test that pins it, failing before the fix — whatever caught the defect. Untestable fixes get an explicit recorded waiver ("deferred: reason"), never silence.
2. **Verification scenario as a floor plan field.** Minimal viable acceptance criterion for a 1-person AI loop: **one** concrete path per user-facing change — `trigger → action → observable result` — that a person could perform on the built artifact right now. This is Given/When/Then collapsed to its three clauses, no Gherkin tooling, no story ceremony. The Reviewer's existing floor item "Plan compliance — every named scenario implemented and tested" then picks it up with zero reviewer edit.
3. **Scenario-walk tie at review.** The Runtime verification ladder gains one clause: when the plan names a verification scenario, the rung evidence states whether THAT scenario was walked — "exercised" claimed on some other path while the named scenario went unwalked is the hallucinated-compliance failure the ladder already names.

## Evidence

- **"No bugfix without a regression test" is standard OSS policy, stated as mandatory.** Django's contribution checklist: *"Is there a proper regression test (the test should fail before the fix is applied)?"* — fail-before-fix explicit, no formal exception carved out (docs.djangoproject.com, submitting-patches; verified 2026-06-13). LLVM Developer Policy: *"Developers are required to create test cases for any bugs fixed and any new features added"* — likewise no formal carve-out (llvm.org/docs/DeveloperPolicy.html; verified 2026-06-13). Where projects do allow exceptions, the working pattern is a recorded waiver with a reason at review time, not silent omission — the form adopted here.
- **Google SRE: every reported bug becomes a test case.** The SRE book (Testing for Reliability) recommends documenting *all reported bugs* as test cases — each fails before the fix, passes after — building a regression suite described as a "gallery of rogue bugs", so that *engineers refactoring the codebase don't accidentally reintroduce bugs already paid for* (sre.google/sre-book/testing-reliability; verified 2026-06-13). That refactor-reintroduction is verbatim the ad-md-editor DOM-race recurrence.
- **Test-first order matters, not just test presence.** Ousterhout (A Philosophy of Software Design) and practitioner consensus: write the failing test before the fix — a test written after may never trigger the bug, so it proves nothing (dev.to/le0nidas; verraes.net "How to Fix a Bug"). Hence RED-before-fix in the rule, not "a test exists somewhere".
- **"Ratchet" in CI practice = a monotonic quality bound.** Notion's ESLint ratchet file (violation counts may only fall; CI blocks any rise), qntm's "Ratchets in software development", LeadDev's quality ratchets (notion.com/blog, qntm.org/ratchet, leaddev.com). The protocol analog: the floor already forbids weakening a test; adding "every defect fix adds its pin" makes the suite self-ratcheting — regression coverage becomes self-generating (the SRE pattern above).
- **Minimal acceptance-criteria form.** Given/When/Then (codified in BDD practice; Ranorex, Atlassian, parallelhq guides) maps context → action → expected outcome; the consensus is its value is the falsifiable outcome clause, not the syntax. For a solo AI loop the minimum is one scenario per user-facing change, human-performable — executable specs (Cucumber-class) are scrum-team overhead here. The downstream's own fix-candidate phrasing ("trigger → action → observable result, performable right now") matches this and is adopted as-is.

## Concrete protocol changes this grounds

Each change names its single home (invariant 6 — promoted rules MOVE out of module fragments, never duplicate):

1. **`src/agents/builder.md` `## Build`** — new bullet: a change that fixes a defect carries the test that pins it — RED on the buggy code before the fix — whatever caught the defect (a gate, a review finding, a user report); where the project's test layer cannot reach the defect class, the deferral is recorded with its reason (plan progress note or commit body), never silent.
2. **`src/agents/builder.md` `## Plan`** — new checklist item after Product questions: **Verification scenario** (user-facing only) — one concrete path `trigger → action → observable result` a person could perform on the built artifact right now; recorded or consciously descoped with a reason.
3. **`src/agents/reviewer.md` `## Check`, Tests item** — extend "added, not weakened": a defect fix without its pinning test or recorded deferral is a finding.
4. **`src/agents/reviewer.md` `## Verdict`, Runtime verification line** — one clause: when the plan names a verification scenario, the rung evidence states whether that scenario was walked; a rung claimed on a different path does not cover it.
5. **`src/modules/test-methodology/{builder,reviewer}.md`** — delete the now-floor `[light]` rows "Test-first ratchet" / "Ratchet present" (supersede, one home); the fragments keep what stays deeper than floor (unreachable layers, full-composition, UI exercised, app-bug vs test-drift).
6. **`src/modules/ui-ux/builder.md`** — re-anchor the User-flow check intro to "deepens the plan's Verification scenario item" (pointer fix; the ≥3-step enumeration stays the GUI deepening of the one-scenario floor).

**Fork for the Operator (not silently taken):** floor promotion (recommended — the deficit is unfelt: the recurrence proves the agent does not feel a missing pin, and the backlog's META rule says unfelt ⇒ always-on organ; the floor cost is two bullets) vs. the lighter alternative — keep the rule in the test-methodology module and only broaden "gate-caught" to "any defect fix" (cheaper, but `kind: docs`/toggled-off projects lose it, and a module row already failed to hold in the field once through narrow scoping).

## What else is missing nearby (prosthesis findings, same field reports)

- **Assertion over silent return** — the DOM-race bug shipped twice through `if (!containerEl) return`: a silent guard on a should-not-happen state converts a crash into an invisible wrong state. Candidate: one Builder `## Build` bullet (fail loudly — assert/throw/log — or record why silence is correct) + one Reviewer Hygiene row (a silent early-return on an impossible state is a finding). Grounded in the field evidence; standard fail-fast practice.
- **Minimum-rung expectation + desktop/native smoke gap — RESOLVED** (5.11.4, `docs/decisions/verification-floor.md`). These two were one deficit: the runtime-exercise expectation was both unconditional-`NOT RUN` and web-only. Resolved by the **write-always / run-on-offer** split — the plan always names the scenario + its primary integration layer (floor, free), the real-layer run is offered + Operator-confirmed on a user-facing change or batched at audit (never automatic); the web-only module rows were superseded, the GUI browser-drive stays as a ui-ux deepening. See that doc.
- **Why five hand-written reports were needed at all** — the structural hole is that a user-reported post-ship bug has no mandatory hook: 8D is offered, the ratchet (until now) fired only on gate-caught bugs. With changes 1–4 the chain closes without human authorship: the bug's fix demands its pin (felt at build), the plan demands its scenario (felt at plan), the review demands the walk (felt at verdict).
