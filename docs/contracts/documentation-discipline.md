# Contract: documentation-discipline

**Every project carries a maintained documentation set — and the protocol holds itself to the same discipline (dogfood)** — docs are bootstrapped at project start and kept current by their owning roles as features land.

Documentation written once and never updated is worse than none — it lies. The doc set is a living, owned thing: every project (greenfield, legacy, or the protocol itself) gets a doc set at the start, and each document has an owning role that refreshes it when a landed feature changes it. The PM never authors or chases documentation. The protocol proves the discipline by developing itself under it.

## Must work

- A project's documentation set is bootstrapped at start — born populated where the source allows, with genuine unknowns marked for the PM, never invented.
- Each document has a single owner; when a landed feature changes a document, the plan names the doc update and the owning role makes it.
- The protocol develops itself under its own protocol — protocol changes go through the same loop a downstream uses, with no privileged shortcut.

## Must not break

- Each doc's single home holds — a fact lives in one place; point to it, don't restate it (invariant 6).
- A value the agent cannot determine without real knowledge is marked for the PM, never invented.
- The protocol's own dogfooding stays real — its docs go through the same loop, not a privileged path.
