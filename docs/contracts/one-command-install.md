# Contract: one-command-install

**One command installs the protocol into a target project** — it vendors the shared adapter, wires the active platform (its deny hooks, agents, commands, and any plugin), and lays down the structure (the constitution, the role agents, the quality-layer shape, the capability modules, the doc templates, and a config slot) — **idempotently**, so re-running is always safe.

The single home for what each step does is `src/adapter/INSTALL.md`; this contract names only the guarantee.

## Must work

- One command (`node src/adapter/install.mjs <target-dir> [--platform claude|opencode]`) installs the protocol into the target: the shared adapter is vendored into the target's tooling location, the active platform is wired (deny hooks, the spawnable agents, the setup command, and the plugin where the platform has one), and the load-instructions surface imports the constitution.
- The core the downstream needs lands in the target — the constitution at `.ai-dev/PROTOCOL.md`, the quality-runner shape at `.ai-dev/quality/` (the format, not this repo's own tool rows), the role agents and capability modules vendored under `.ai-dev/tooling/`. Doc templates are NOT copied to the project root; they live in `.ai-dev/tooling/src/templates/` and are laid down on demand by product discovery / doc bootstrap.
- **The runtime-read bodies get a READ-ALLOWED deploy** — a body a role reads on-demand at run time lands at a readable `.ai-dev/` path, not only under the vendored `.ai-dev/tooling/` (which invariant 2 reads-denies unconditionally, so the tooling copy alone would be unreadable by the very role whose trigger names it): the on-demand procedure bodies at `.ai-dev/procedures/<x>.md`, and the runtime-read capability-module files (the elicitation catalog) at `.ai-dev/modules/<rel>`. Each is byte-identical to its `src/` source (drift-guarded), the stable read path in both dogfood and downstream.
- The active platform is resolved from the `--platform` flag, else the target's `.ai-dev/config.json` `platform`, else a clear error — never a silent guess.
- **The upgrade channel's mechanical half** (`docs/decisions/upgrade-migration.md`): every run stamps the installed protocol version at `.ai-dev/VERSION` and lays the per-version migration notes at `.ai-dev/upgrades.md`; a run whose version differs from the prior stamp (or finds a pre-stamp `.ai-dev/` tree) writes the transient `.ai-dev/UPGRADING.md` handoff marker and prints a restart notice.
- **The vendored installer is self-sufficient** — `node .ai-dev/tooling/src/adapter/install.mjs <target>` (the documented upgrade re-run) performs the same install: the tooling carries every source the installer reads, including its own version.
- **The inactive platform gets a breadcrumb load-surface** — a minimal marker-delimited block in its load-instructions file, pointing an unwired session at the installer and the platform switch; merged in (never clobbering a real file) and replaced by the full wiring when that platform becomes active.
- **Self-host (dogfood) mode** — `--dogfood` wires the protocol's OWN source repo to its `src/` tree (the committed source-mode state) instead of a vendored copy, and skips vendoring + version-stamping; a reinstall of the source repo is therefore idempotent against its committed tree (`git status` clean), never churning the tracked wiring to the downstream layout. Valid ONLY when the target is the source repo. Detail: `src/adapter/INSTALL.md`.
- The command finishes by reporting what it did and the next step (run `/dev-setup` to configure the project).

## Must not break

- **Idempotent** — re-running yields the same wired state: no duplicated deny hook, no duplicated constitution import, no duplicated or corrupted file. A second run is safe. A protocol hook group whose command changed is *replaced* (recognised by the shim's stable path tail), never accumulated; foreign hook groups are preserved untouched.
- **Never touches the project's own docs** — the installer writes nothing to the target's `docs/` or `src/` directories; those belong to the downstream project. Doc templates are accessible under `.ai-dev/tooling/src/templates/` and laid down on demand by product discovery / doc bootstrap.
- **Stays inside the target root** — every write lands beneath the resolved target directory; the installer never writes outside it (the project-boundary floor, enforced mechanically by the deny layer regardless).
- **No platform name leaks into the neutral core** — the installer is adapter-layer code, platform-specific by nature; the concrete platform wiring it performs never adds a platform primitive to `PROTOCOL.md`, the role bodies, or `docs/architecture.md` prose.
- **Self-host mode is fail-closed both ways** — `--dogfood` against a non-source target, and a default (no-flag) install against the source repo, each refuse (exit 1) rather than wire the wrong layout; the source repo can never be silently churned to the downstream layout, and `--dogfood` can never dangle source paths into a real downstream.
