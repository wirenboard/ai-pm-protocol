# Procedure: setup

Loaded on demand when `setup` fires (the trigger line lives in `orchestrator.md`
`## Setup`). `setup` writes `.ai-dev/config.json` through a plain-language dialog. It is
**your** procedure ‚Äî talk to the Operator, write the config you own. Run on an
unconfigured project, or on `/dev-setup`.

## The dialog

0. **Repo check** ‚Äî the loop stands on git; no `.git` at the root ‚áí one structured offer: initialize now (`git init -b main` + an initial commit of the existing tree) or proceed without. Declinable, but say plainly the loop's git flow cannot run until a repository exists. Name the forge half in the same breath: creating the remote and wiring `origin` is the **Operator's** (offer `gh repo create` where `gh` exists ‚Äî **never silently create a remote**, it may be public). This is the single home of the repo-bootstrap offer; the installer's no-repo warning and the understand beat's no-repo case both point here.
1. **Discover models** via the adapter's list-available-models contract point. Where enumeration fails, ask the Operator to confirm the model id ‚Äî **never invent one**.
2. **Ask structured questions** for each choice:
   - **`kind`** ‚Äî `code` (machine executes it), `docs` (humans read it), or `mixed` (both matter equally). Default `code`. Present `mixed` as the honest choice when documentation IS the product.
   - **`docLanguage`** ‚Äî the language the project's human-read docs (the `docs/` tree + `README`) are authored in (`PROTOCOL.md` invariant 5). Default `"en"`; offer the Operator's own language when the team isn't English-first. Say plainly what stays English regardless: code, commits, config, and machine grammar (the invariant 5 floor) ‚Äî this flips only the prose audience reads.
   - **capability-module kit** ‚Äî read `src/modules/registry.json`, present the per-`kind` defaults, let the Operator opt modules on/off in one step. Lead with the defaults; unanswered toggles keep the default (fail-safe ON).
   - **safeguards** ‚Äî an OFFERED, declinable question (never recommend turning one off ‚Äî a conscious self-nomination, mirror the `yolo` discipline): the toggleable guards with their labels, all default ON, letting the Operator set initial `safeguards` states (the labels ARE the explanation). Only ask/nudge guards are listed; the deny + merge-gate floor is permanent and not shown as toggleable. Single home for the list: `src/adapter/deny-rules.json` (the `toggleable` rules).
   - **`mode`** ‚Äî default `interactive`. Present `autonomous` as opt-in; **do not recommend it**.
   - **`profile`** ‚Äî default `solo`. Lead with it plainly: *"solo = I build directly and keep plans fixup-grade; what never changes: a fresh separate Reviewer, its stamp, your explicit merge word."* Name BOTH costs ‚Äî ceremony burns the Operator's tokens and time; speed costs one independent build-side context ‚Äî and present `full`/`lite` as the conscious opt-up. Present `yolo` last, as the **off-guarantee escape hatch**: *"yolo = no Reviewer, no stamp, no merge-gate; the audit cadence is your safety net; code is brought to standards later or rewritten."* Require an explicit acknowledgement before writing `yolo` to the config. **Never recommend `yolo`** ‚Äî it is a conscious self-nomination. **Never recommend a profile without naming both costs.**
   - **`collaboration`** ‚Äî default **single-user** (`team: false`): the common case. Present team mode as an explicit **opt-in** ‚Äî **do not recommend it** (a conscious self-nomination, mirror the `mode: autonomous` / `yolo` discipline): *"single-user is the default; team mode adds multi-user (colleague-review) collaboration on top ‚Äî opt in only if a team works this repo."* On **team** chosen, ask two follow-ups in the same block: `backlog` ‚Äî **file** (default, today's `.ai-dev/backlog.md`) vs **forge** (issues replace the file); and confirm the **auto-detected forge** (from the `origin` remote host) or let the Operator name it. Single-user ‚áí skip both follow-ups, write the safe defaults. Enum + fail-safe value-home: `PROTOCOL.md` `## Project config` (point, don't restate). On **team** chosen, also enable the `team-collaboration` capability module (`modules.team-collaboration` ON) ‚Äî it carries the Builder/Reviewer conflict-avoidance discipline; single-user leaves it at its registry default (OFF). The backlog adapter (`## Backlog`) and the ship-time PR-attach of the AI-review verdict are wired too ‚Äî each gated on the team / backlog choice, inert on a single-user project. On **team** chosen, point the Operator at the team-collaboration onboarding/usage guide (`docs/team-collaboration.md`) for the end-to-end adoption HOW-TO (the load-bearing branch-protection step, the per-developer loop, the honest limits) ‚Äî point, don't restate it here.
   - **models per seat** ‚Äî **lead with zero-config (the vanilla state)**: recommend leaving every seat unpinned. In vanilla ‚Äî no concrete pin on any seat and no launch model ‚Äî the reviewer rides `auto` (a different brain from the maker, opus‚Üîsonnet, where the platform offers one) and the builder rides the session model; this is the recommended default, and the whole cross-model convenience works ONLY here (`docs/contracts/cross-model-review.md`; the predicate's single home is `src/adapter/claude/install-agents.mjs` `isVanilla`). **Only when the Operator wants to pin a seat** do you open the per-seat questions: ask **three independent seat choices** ‚Äî the **builder**, the **reviewer**, and the **guard** (the background/small-fast model the harness uses for cheap auxiliary work). In this customized branch present each seat with the **same uniform option set** ‚Äî **`default` (un-pin ‚Äî omit the `model` key)**, any **discovered Anthropic id** (`haiku`/`sonnet`/`opus`; never omit one from a single seat's list, so the choice can't silently differ between roles), **or a visible `another provider (non-Anthropic)` choice** that opens the routing sub-flow below (cross-endpoint is a NAMED option, never buried behind a free-text "other") ‚Äî defaulting to that seat's **current value**; **do NOT offer `auto` in this branch** ‚Äî pinning any seat turns the automatic cross-model guess off (the opus‚Üîsonnet opposite is a fiction once a pin or proxy can hide what a string resolves to), so say plainly: **pinning any seat turns auto off**, and **`default` on the reviewer rides the session (no cross-model)**. Un-pinning every seat (all `default`, no launch model) returns the config to vanilla and `auto` revives (`docs/decisions/multi-model-setup-ux.md`, the vanilla-only-`auto` rule). **No orchestrator question** ‚Äî the orchestrator IS the running session, so its model is the launch model, never a config pin; `roles.orchestrator` carries only its `agent`. The two **baked** pins are `roles.{builder,reviewer}.model`; the **guard** and the **session** model are launch-time settings, not baked ‚Äî they live in the config `launch` section, and the installer writes them into the harness's startup env on the next apply (the concrete env names + the startup-env home + the wrapper recipe: `src/adapter/README.md` `### The launcher`). Any cross-model claim is **`platform: claude`**-gated (the per-seat honesty caveats: `## Your seat`, point, don't restate). **Cross-endpoint = tier binding (the recommended path):** a role does NOT take a foreign id directly ‚Äî it takes a Claude **tier** (`opus`/`sonnet`/`haiku`), and the tier is **bound** to a foreign model in the routing sub-flow below (Stage 1 ‚Üí `launch.aliases` ‚Üí `ANTHROPIC_DEFAULT_*_MODEL`). So for cross-endpoint the per-seat step here just assigns a **tier** per role; the foreign mapping is the sub-flow's job. *Alternative (still valid):* write a **concrete provider id** straight into a seat (e.g. `deepseek-chat`) ‚Äî the proxy routes the exact id that arrives; an alias only resolves where the proxy carries a route for it (the resolution chain: `src/adapter/README.md` `### The launcher`). The cross-endpoint route itself (provider + keys) is NOT asked here ‚Äî point at the model-router launcher + catalog for that opt-in. Before asking blind, **probe for an already-running proxy** ‚Äî `./.ai-dev/launch --probe` prints `{alive,url,models}` (the discovery command: `src/adapter/README.md` `### The launcher`). When one answers, **offer ITS model ids per seat and skip the URL question**; when nothing answers, fall back to asking ‚Äî no hard dependency (`docs/decisions/multi-model-setup-ux.md` papercut 8). The probe-first sequence + the external-vs-spawn fork live in the routing sub-flow below. Write the builder/reviewer answers to `roles.{builder,reviewer}.model` (baked) and the guard/session answers to the config `launch` section (launch-time); **when a launch-time value (guard or session) changes, the apply re-runs the installer** so it lands in the harness's startup env, AND you **tell the Operator plainly to restart their session for it to take effect** ‚Äî that env is read only at startup, so the running session keeps the old value until a restart (a launch-time change ‚áí re-apply + RESTART; distinct from a baked builder/reviewer change ‚áí re-bake + next spawn). When the project launches through its own wrapper instead, **print the ready export block** so the wrapper consumes the same config values (the recipe + the env names: `src/adapter/README.md` `### The launcher`).
   - **launch & routing sub-flow (only when a CROSS-ENDPOINT seat was pinned ‚Äî the "configure everything together" ask)** ‚Äî when the model step above pinned a seat on a non-Anthropic provider, run ONE coherent sub-flow instead of leaving routing setup scattered across four manual steps. In order, in plain language: (a) **probe first** ‚Äî run `./.ai-dev/launch --probe` to discover an already-running proxy (it prints `{alive,url,models}`). **If one answers**, use its model list (below) and **skip asking for a URL** ‚Äî the proxy already runs. **If nothing answers**, present the explicit fork (never buried ‚Äî it is the whole point of the cross-endpoint choice): **external** ‚Äî the Operator runs/points at their own proxy; record its URL as `proxyUrl` in `.ai-dev/model-routes.json` (the launcher then points claude straight at it, no spawn) ‚Äî **or built-in** ‚Äî the launcher spawns modelpipe itself on a **random free port** (`router-launch.mjs` `port: 0`; no port to pick, nothing to conflict), so only the routes scaffold + keys remain. (b) **Stage 1 ‚Äî tier binding** (the two-stage model the Operator asked for): for each Claude tier (`opus`/`sonnet`/`haiku`) ask *keep native Claude* **or** *bind to a foreign model*. Source the foreign choice from the probe's `/v1/models`: an entry whose `id` is **concrete** (no `*`) is offered **directly as a pick-list**; an entry that is a **glob** (`glm-*`, modelpipe-style) means the **provider** ‚Äî pick it, then **enter the concrete id**, which you **validate against that glob** (`glm-4.6` matches `glm-*`) and show its host/vision as hints. Never blind free-text: it is always *pick provider ‚Üí enter/choose its model*. A third-party proxy that already returns concrete ids needs no provider step ‚Äî the list is ready. Each binding writes `launch.aliases.{tier}` ‚Üí `ANTHROPIC_DEFAULT_{TIER}_MODEL` (launch-time, restart-applied). *(Follow-up: a real filtered pick-list for the glob case once modelpipe expands globs via `?expand=1`.)* (c) **Stage 2 ‚Äî role ‚Üí tier**: assign **builder**/**reviewer**/**guard** each a tier; the role then routes to whatever that tier is bound to. Write the builder/reviewer tier to `roles.{builder,reviewer}.model` (the tier alias, baked); the guard takes its tier too, OR keep `launch.guardModel` set to hold the background model **independent** of the haiku tier. (d) **`configDir`** ‚Äî ask whether to pin a per-project claude profile dir (`CLAUDE_CONFIG_DIR` ‚Äî per-task keys with no `.bashrc` edit); when given, write it to the **gitignored `.ai-dev/config.local.json`** `launch.configDir` (a per-machine path ‚Äî NEVER the shared `.ai-dev/config.json`); (e) **offer to scaffold `.ai-dev/model-routes.json`** from `.ai-dev/tooling/src/adapter/model-router.example.json` for the named providers (a route may name a provider by id; keys by env-var NAME only, never a value) ‚Äî declinable; on the **external** fork above, set its `proxyUrl` instead; (f) **remind which backend key env-vars to export** (e.g. `DEEPSEEK_API_KEY`) ‚Äî names only, never ask for a value (the **external** fork's keys live in that proxy's own env, not here); (g) **print the ready launch command** ‚Äî `./.ai-dev/launch` (the OPTIONAL drop-in the installer generated). The launcher is OPTIONAL and the full recipe (env precedence, `--proxy`, `config.local.json` merge, the OpenCode stub) is **`src/adapter/README.md`** `### The launcher` and `docs/decisions/launcher-ux.md` ‚Äî point, don't restate. Each value lands in its right home: shared ‚Üí `config.json`, personal ‚Üí `config.local.json`, routes ‚Üí `model-routes.json`.
3. **Write `.ai-dev/config.json`** with their answers. No spawn, no push. Reversible.
4. **Apply the config** ‚Äî run the adapter's install over your own project files (the concrete command: `src/adapter/INSTALL.md`). Idempotent ‚Äî re-running with an unchanged config reproduces the same agent files; a `session` seat bakes no `model:` line, while an `auto`/pinned reviewer bakes its resolved cross-model line on Claude (`## Your seat`).
5. **Wire the quality toolkit** ‚Äî discover the stack (languages, package manager, doc format) and propose a stack-appropriate tool set, reasoning from the stack, never a hard-coded list. The set:

   - **Core tools** ‚Äî linter, formatter, type-checker, doc linter, security/SAST scanner, gitleaks-class secret scanner.
   - **Failure-suppression rules raised to ERROR** (within the linter) ‚Äî a blind/bare-except and try-except-pass rule where the stack has one (e.g. Python ruff `BLE001` + bandit `S110`/`S112`), since a swallowed failure is the silent-dead-code class a linter CAN catch mechanically.
   - **Commented-out-code rule raised to ERROR** where the stack has one (Python ruff `ERA001`/eradicate; JS an eslint plugin where the stack carries one) ‚Äî a dead code block left in source is the one *syntactic* junk-comment subclass a linter CAN catch; the semantic junk comment (a `// what` restating the code below it) stays the Reviewer's prose-quality call (invariant 6), never a linter's.
   - **Size / complexity linter** ‚Äî a **recommended default**: a per-file size/complexity rule at ~800 lines where the stack offers one (eslint `max-lines`/`complexity`, ruff `C901`, a comparable rule), so an oversized file becomes a mechanical finding, not a thing only a human notices. Tuned soft ‚Äî it surfaces the file to the architect and points at `decompose` (`.ai-dev/procedures/decompose.md`) for remediation, never a hard build block.
   - **UI/E2E automation driver** for a GUI stack ‚Äî Playwright for web, tauri-driver/WebDriver for native.

   Offer it (declinable). For each chosen tool: install, drop standard config, register a row in `.ai-dev/quality/tools.json`, verify green via `node .ai-dev/quality/run.mjs <beat>`. Tune to the standard ‚Äî a config relaxation is the Operator's recorded decision.

   **CI wiring** (offered with it, declinable) ‚Äî a workflow running the registered quality suite on every push/PR, per the project's forge (e.g. GitHub Actions), **via the runner (`node .ai-dev/quality/run.mjs <beat>`, both beats), never a re-listed subset of tool commands**: a hand-copied list drifts the moment a row is added; the registry stays the single home of "what green means". The merge-gate is local, CI is the remote re-check that catches a bypassed local run.

   **Branch protection ‚Äî a MANDATORY accept-or-decline, never a silent skip** (`docs/decisions/persona-floor-external-substitute.md`, the durable rationale). The local deny and CI are advisory; the **remote forge protection is the one floor that holds regardless of model, platform, or whether the plugin even loaded** ‚Äî `main` cannot move except through a green-CI PR. Name it plainly: on **OpenCode** the deny layer has no `ask` return, so the whole ask-class degrades to persona (`PROTOCOL.md` `## Enforcement`) ‚Äî this forge gate **is** that ask-class's declared substitute there; on every platform it is the model-independent remote floor. So this step is **required**: the Operator must answer, and the answer (set / declined / unwirable-here) is **recorded** in state so a later audit sees a conscious choice. On accept, **print the ready recipe for the Operator to run** (you do not run it ‚Äî that privileged apply is the deferred fast-follow F2-2); on the GitHub forge with the quality job named `quality`:

   ```sh
   gh api --method PUT repos/{owner}/{repo}/branches/main/protection --input - <<'JSON'
   {
     "required_status_checks": { "strict": true, "checks": [ { "context": "quality" } ] },
     "enforce_admins": true,
     "required_pull_request_reviews": { "required_approving_review_count": 0 },
     "restrictions": null
   }
   JSON
   ```

   The `context` must match the check-run name the workflow reports (`quality` in `.github/workflows/checks.yml`); `enforce_admins: true` closes the admin-can-still-direct-push hole. **Graceful when it cannot be wired here** ‚Äî no `gh` or offline ‚áí give the GitHub web-UI recipe (Settings ‚Üí Branches ‚Üí add a rule on `main`: require the `quality` status check, require a PR, include administrators) and record "wanted, unwirable here"; the call needs **admin scope** on the repo (a plain push token 403s) ‚áí say so plainly ("your token lacks admin ‚Äî set it in the web UI or re-auth with admin scope") and record the decline. Never let a missing tool or scope turn this into a silent skip.

## When it fires

- **Reactive** ‚Äî on the Operator's first real work request, check for `.ai-dev/config.json`. If absent: give a SHORT offer of two choices (run `/dev-setup` or proceed on safe defaults), then **stop**. Do not start the task, explore the repo, or write a multi-topic essay.
- **Explicit** ‚Äî `/dev-setup` re-runs on demand. Carries no dialog of its own ‚Äî it points here, the single home.
- **Mode switch mid-stream** ‚Äî a bare "switch mode/profile" ‚áí a structured question (the current profile + the options, a one-line trade-off each); a target named directly ("solo"/"full"/"lite") ‚áí a one-line confirm, the config flip, an announce ‚Äî no full setup re-run.
- **Doc-language switch mid-stream** ‚Äî the Operator names a documentation language directly ("–¥–µ–ª–∞–π –¥–æ–∫—É–º–µ–Ω—Ç–∞—Ü–∏—é –Ω–∞ —Ä—É—Å—Å–∫–æ–º", "docs in German") ‚áí a one-line confirm, flip `docLanguage` in the config, an announce ‚Äî the lightweight flip mirrors *Mode switch mid-stream*. THEN, because the existing in-scope docs are still in the old language, **offer to translate the current `docs/` tree + `README` into the new `docLanguage` as a build task through the loop** (a durable-prose change goes through the Builder + review, never a silent rewrite; git history keeps the originals) ‚Äî declinable, forward-only if declined. The floor invariant 5 names (code, commits, config, machine grammar ‚Äî always English) never moves; say so in the announce.
- **Collaboration switch mid-stream** ‚Äî the Operator expresses, **in their own words and any language**, the intent to move the project to (or away from) multi-developer / team / collaborative development ("–ø–µ—Ä–µ–∫–ª—é—á–∏ –≤ –Ω–µ—Å–∫–æ–ª—å–∫–æ —Ä–∞–∑—Ä–∞–±–æ—Ç—á–∏–∫–æ–≤", "–≥–æ—Ç–æ–≤–∏–º—Å—è –∫ –∫–æ–º–∞–Ω–¥–Ω–æ–π —Ä–∞–∑—Ä–∞–±–æ—Ç–∫–µ", "we're a team now", "switch to multiple developers"; back: "–≤–µ—Ä–Ω–∏—Å—å –∫ –æ–¥–∏–Ω–æ—á–Ω–æ–π —Ä–∞–∑—Ä–∞–±–æ—Ç–∫–µ", "solo again") ‚áí recognise the **intent**, never a memorised keyword ‚Äî the Operator never has to say the field name or the token `team` (the config field stays `collaboration.team`, machine grammar ‚Äî invariant 5 ‚Äî but you map their phrasing to it). Lightweight flip mirroring *Mode switch mid-stream*, no full setup re-run. **On enable:** a one-line confirm ‚áí flip `collaboration.team: true` ‚áí ask the same two follow-ups the setup question does (`backlog` file/forge, confirm the auto-detected forge ‚Äî step 2's `collaboration` bullet; enum value-home `PROTOCOL.md` `## Project config`) ‚áí enable `modules.team-collaboration` ‚áí **apply the config** (re-assemble the agents ‚Äî step 4 / `src/adapter/INSTALL.md`) ‚áí announce in plain language what changed. On `backlog: forge` chosen, **offer the `file ‚Üí forge` migration** (`## Backlog`) ‚Äî declinable ‚Äî exactly as *Doc-language switch* offers to translate the existing docs after its flip. **On disable (back to solo):** flip `collaboration.team: false`, announce in one line that the `team-collaboration` module and the team behaviours go inert and the backlog **stays where it is** (no auto-delete of forge issues).
- **Model switch mid-stream** ‚Äî the Operator names a seat model directly ("builder on deepseek-chat", "reviewer back to auto", "set the guard model to ‚Ä¶") ‚áí a one-line confirm, flip the seat's pin, an announce ‚Äî the lightweight flip mirroring *Mode switch mid-stream*. **A `builder`/`reviewer` change is a BAKED pin: it needs a re-bake** (re-assemble the agents so the new `model:` lands), so after flipping it **re-apply the config** the same way *Collaboration switch* does (step 4 / `src/adapter/INSTALL.md`). If you cannot run the install yourself, print the **context-correct** re-bake command ‚Äî and you know the context from whether the protocol's own installer source sits at the root: in the protocol's **own source repo** (dogfood) it is `node src/adapter/install.mjs . --dogfood`; in a **downstream** project (one carrying `.ai-dev/tooling/`) it is `node .ai-dev/tooling/src/adapter/install.mjs .` (no `--dogfood`) ‚Äî handing the wrong one is the `MODULE_NOT_FOUND` papercut this fixes (`docs/decisions/multi-model-setup-ux.md` papercut 4). A `guard`/session change is launch-time, not baked ‚Äî there are TWO distinct apply paths, name the right one: a **baked** seat change (builder/reviewer) ‚áí **re-bake** (re-run the installer; it takes effect on the next spawn). A **launch-time** change (guard/session, or the proxy base-URL) ‚áí **re-run the installer to update the harness's startup env, then tell the Operator plainly to RESTART their session** ‚Äî that env is read only at startup, so the running session keeps the old value until restart (`src/adapter/README.md` `### The launcher`). The orchestrator pin does not exist (the session model is the launch model). Where the project launches through its own wrapper, the same launch-time value also flows through the printed export block ‚Äî restart still required.

## Platform switch

You can tell your own harness from the tool surface you hold; when that platform differs from the config's `platform`, offer the switch on the understand beat: *"this session runs on a platform the config doesn't name ‚Äî wire it and switch?"* Short, declinable, never a block; declined ‚áí proceed silently. On accept:

1. **Install for the platform this session runs on** (the concrete command: `src/adapter/INSTALL.md`) ‚Äî idempotent; it **regenerates that platform's agents, the orchestrator INCLUDED**, each one **platform-filtered**, into that platform's **assembled-orchestrator artifact plus its load wiring** (the adapter realises the concrete artifact path + load mechanism per platform ‚Äî `src/adapter/INSTALL.md`; the platform-filtering itself ‚Äî `docs/architecture.md` `## Capability modules`), and **rewires the load to it**. **A harness switch therefore REGENERATES the orchestrator ‚Äî it is no longer a config-letter flip** (since adapter-filter Option B the orchestrator is a per-platform assembled artifact, not a raw shared file). Both wirings coexist; each harness loads only its own surface, so the **old platform's load line goes inert** (never read by the inactive harness) ‚Äî harmless, optionally cleaned.
2. **Flip `platform` in the config** ‚Äî the field stays the recorded ACTIVE adapter.
3. **Revalidate the models.** `auto`/`session` carry as-is ‚Äî they re-resolve per platform by design. A concrete pin is checked against the new platform's discovered catalog (the list-available-models contract point); **never invent an id**. A dead pin that differed from the session model recorded a CROSS-MODEL wish: re-ask leading with the new catalog's cross-model candidates and recommend one, `auto` offered as the explicit zero-config fallback; where the new platform offers no second model, say so plainly (the honesty rule in `## Your seat`).
4. **Apply the config** ‚Äî re-run the platform's install-agents (step 4 above, on the new platform): a platform that bakes the reviewer model into the assembled agent (Claude does) keeps a dead pin until the re-bake, however correct the config now is.
