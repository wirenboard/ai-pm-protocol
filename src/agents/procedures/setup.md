# Procedure: setup

Loaded on demand when `setup` fires (the trigger line lives in `orchestrator.md`
`## Setup`). `setup` writes `.ai-dev/config.json` through a plain-language dialog. It is
**your** procedure — talk to the Operator, write the config you own. Run on an
unconfigured project, or on `/dev-setup`.

## The dialog

0. **Repo check** — the loop stands on git; no `.git` at the root ⇒ one structured offer: initialize now (`git init -b main` + an initial commit of the existing tree) or proceed without. Declinable, but say plainly the loop's git flow cannot run until a repository exists. Name the forge half in the same breath: creating the remote and wiring `origin` is the **Operator's** (offer `gh repo create` where `gh` exists — **never silently create a remote**, it may be public). This is the single home of the repo-bootstrap offer; the installer's no-repo warning and the understand beat's no-repo case both point here.
1. **Discover models** via the adapter's list-available-models contract point. Where enumeration fails, ask the Operator to confirm the model id — **never invent one**.
2. **Ask structured questions** for each choice:
   - **`kind`** — `code` (machine executes it), `docs` (humans read it), or `mixed` (both matter equally). Default `code`. Present `mixed` as the honest choice when documentation IS the product.
   - **`docLanguage`** — the language the project's human-read docs (the `docs/` tree + `README`) are authored in (`PROTOCOL.md` invariant 5). Default `"en"`; offer the Operator's own language when the team isn't English-first. Say plainly what stays English regardless: code, commits, config, and machine grammar (the invariant 5 floor) — this flips only the prose audience reads.
   - **capability-module kit** — read `src/modules/registry.json`, present the per-`kind` defaults, let the Operator opt modules on/off in one step. Lead with the defaults; unanswered toggles keep the default (fail-safe ON).
   - **safeguards** — an OFFERED, declinable question (never recommend turning one off — a conscious self-nomination, mirror the `yolo` discipline): the toggleable guards with their labels, all default ON, letting the Operator set initial `safeguards` states (the labels ARE the explanation). Only ask/nudge guards are listed; the deny + merge-gate floor is permanent and not shown as toggleable. Single home for the list: `src/adapter/deny-rules.json` (the `toggleable` rules).
   - **`mode`** — default `interactive`. Present `autonomous` as opt-in; **do not recommend it**.
   - **`profile`** — default `solo`. Lead with it plainly: *"solo = I build directly and keep plans fixup-grade; what never changes: a fresh separate Reviewer, its stamp, your explicit merge word."* Name BOTH costs — ceremony burns the Operator's tokens and time; speed costs one independent build-side context — and present `full`/`lite` as the conscious opt-up. Present `yolo` last, as the **off-guarantee escape hatch**: *"yolo = no Reviewer, no stamp, no merge-gate; the audit cadence is your safety net; code is brought to standards later or rewritten."* Require an explicit acknowledgement before writing `yolo` to the config. **Never recommend `yolo`** — it is a conscious self-nomination. **Never recommend a profile without naming both costs.**
   - **`collaboration`** — default **single-user** (`team: false`): the common case. Present team mode as an explicit **opt-in** — **do not recommend it** (a conscious self-nomination, mirror the `mode: autonomous` / `yolo` discipline): *"single-user is the default; team mode adds multi-user (colleague-review) collaboration on top — opt in only if a team works this repo."* On **team** chosen, ask two follow-ups in the same block: `backlog` — **file** (default, today's `.ai-dev/backlog.md`) vs **forge** (issues replace the file); and confirm the **auto-detected forge** (from the `origin` remote host) or let the Operator name it. Single-user ⇒ skip both follow-ups, write the safe defaults. Enum + fail-safe value-home: `PROTOCOL.md` `## Project config` (point, don't restate). On **team** chosen, also enable the `team-collaboration` capability module (`modules.team-collaboration` ON) — it carries the Builder/Reviewer conflict-avoidance discipline; single-user leaves it at its registry default (OFF). The backlog adapter (`## Backlog`) and the ship-time PR-attach of the AI-review verdict are wired too — each gated on the team / backlog choice, inert on a single-user project. On **team** chosen, point the Operator at the team-collaboration onboarding/usage guide (`docs/team-collaboration.md`) for the end-to-end adoption HOW-TO (the load-bearing branch-protection step, the per-developer loop, the honest limits) — point, don't restate it here.
   - **models per seat — a synonym-first flow with independent launch seats; the Operator steers each phase.** Lead with zero-config. The FOUR synonyms `fable` / `opus` / `sonnet` / `haiku` (strongest→weakest) are the VISIBLE labels the Operator works in — `fable` is Claude's top hardest/longest-running tier; four is the fixed remappable set (no custom aliases). The gate comes FIRST, then the baked-seat phases and the launch seats:
     1. **Gate — change models at all? (ASK FIRST.)** **Recommend the default (vanilla)**: leave every seat unpinned — the reviewer rides `auto` (a different brain from the maker, opus↔sonnet, where the platform offers one), the builder rides the session model. This is the recommended default and the whole cross-model convenience works ONLY here (`docs/contracts/cross-model-review.md`; predicate home `src/adapter/claude/install-agents.mjs` `isVanilla`). **Vanilla chosen ⇒ STOP** — write nothing under `launch`/`roles.*.model`, `auto` revives. **Custom chosen ⇒ continue** to Phase 1.
     2. **Phase 1 — BAKED seat → synonym.** For each baked seat — **planner · builder · reviewer** — pick one of the four synonyms **`fable` / `opus` / `sonnet` / `haiku`** (strongest→weakest), defaulting to the seat's **current synonym**. The **planner** additionally offers **"inherit session (default — a strong planner)"** (its `model` omitted / `session`) — keep that the planner's default so the cheap-builder / strong-session-planner economy survives the custom branch (`docs/decisions/planning-model-split.md`). Always offer the full set to every seat — **no truncation, and no provider/foreign choice here** (that is Phase 2). **No orchestrator question** (the orchestrator IS the running session — its model is the **session** launch seat below, never a `roles` pin). Reverting the WHOLE config to vanilla is done via the gate's "leave vanilla" answer, not by un-pinning seats here. Apply-path: a **baked** change (planner/builder/reviewer) ⇒ re-bake, effective next spawn.
     3. **Phase 2 — synonym → model.** For **each** of the four synonyms — **`fable`, `opus`, `sonnet`, `haiku`, mapping ALL FOUR even if a synonym is currently unridden** (so a later reseat needs no re-bind — the Operator's stability point): **native** Claude (default) OR a **foreign** provider. Foreign = pick a provider from the probe, then type/confirm its concrete id validated against its glob (`glm-5.2` matches `glm-*`) — **never guess, pre-fill, or invent an id** (the full sourcing discipline is the *routing apply-steps* below). A synonym mapped native routes to the Claude model of that name; mapped foreign, EVERY BAKED seat that picked it in Phase 1 routes to that one foreign model — so **for the baked seats a synonym is GLOBALLY native or foreign** (the launch seats are independent — Phase "launch seats" below). The `synonym → model` binding is stable: reseating a baked seat (Phase 1) later never touches it. All cross-model / foreign claims are **`platform: claude`**-gated (`## Your seat`, point, don't restate) — on OpenCode no baked seat cross-models (the `task` runtime ignores a subagent `model:`; `tool-map.json` `models.opencode._note`), so the foreign path is Claude-only.
     4. **Launch seats — session · guard, chosen DIRECTLY (not by the synonym mappings).** These are launch-time env, never baked:
        - **Session** — pick its model directly (a native tier's concrete id — any of the four, incl. `fable` — or a foreign id) → `ANTHROPIC_MODEL`, defaulting to the current session model. **Fully independent of the synonym mappings:** it stays native even when that tier is bound foreign for the baked seats, because a concrete id bypasses the alias (the "free-the-alias" path the installer already bakes; `ANTHROPIC_MODEL` is not deprecated). No risk.
        - **Guard** (the background/small-fast model) — **default = the `haiku` slot** (`ANTHROPIC_DEFAULT_HAIKU_MODEL`, the modern var, **no risk**): the background follows whatever `haiku` resolves to (native, or the foreign id if haiku is bound). An **override** to a background model DIFFERENT from the haiku slot uses the **deprecated** `ANTHROPIC_SMALL_FAST_MODEL` (G1 — the only knob independent of the haiku slot) — offer it as an explicit choice **with a plain deprecation warning** (on watch for removal, no removal date yet). **Bonus to surface:** to keep the background native while routing a cheap foreign baked seat, route that seat via a **non-haiku** tier (`fable`/`opus`/`sonnet`) and leave `haiku` native — the background then stays native on the modern var, no deprecated override needed (the everyday path a 4th tier unlocks; the override is the rare fallback).
        - Apply-path: a **launch-time** change (session/guard, or any synonym re-mapping) ⇒ re-run the installer + tell the Operator to **restart** their session (startup-read env — the running session keeps the old value until restart).
   - **personal vs shared — where each answer lands** (make it explicit; homes: `.ai-dev/config.json` `_launch`/`_roles`, `src/adapter/README.md` `### The launcher`, `docs/decisions/personal-multi-model-setup.md` — point, don't restate). **SHARED** (committed `.ai-dev/config.json`) = the baked seat's **tier LANE** (`roles.{planner,builder,reviewer}.model = fable|opus|sonnet|haiku`) + each seat's **agent binding** — the whole team gets these, baked into the committed agent file. **PERSONAL** (gitignored `.ai-dev/config.local.json`) = the **tier BINDING** (`launch.aliases.<tier> = <foreign id>`, any of the four tiers), the **session** and **guard** models, `configDir`, and the proxy `proxyUrl` — per-machine, never forced on a teammate. The installer **merges** config.local's `launch` over config.json's when it bakes + self-verifies (so a personal binding is not silently ignored — `resolveModelPin` bakes the bare alias, `verifyRoutingConsistency` accepts its launcher-applied env), but writes ONLY the shared config.json values into the committed harness env; the personal ones are applied by the launcher at startup (so personal foreign models require launching via `./.ai-dev/launch`). A value the whole team SHOULD route the same way (a team-wide foreign binding) goes in config.json's `launch` instead — then it lands in the committed env like any shared value.
   - **auto-allocation (compute the config from the answers)** — Phase 1 gave each baked seat's synonym, Phase 2 each synonym's native/foreign mapping, and the launch-seat step gave session + guard directly; write each value to its home per the personal/shared split above: the synonym **LANE** (`roles.*.model`) to the shared `.ai-dev/config.json`, the synonym **BINDING** (`launch.aliases.*`) and the **session**/**guard** models to the gitignored personal `.ai-dev/config.local.json` (a team-wide binding instead to config.json's `launch`). The mechanic (VERIFIED — `docs/decisions/multi-model-setup-ux.md` papercuts 13+14): on Claude a **bare synonym alias** baked into a seat resolves through `ANTHROPIC_DEFAULT_{FABLE,OPUS,SONNET,HAIKU}_MODEL` (→ the foreign model bound there), while a **concrete id** (`claude-opus-4-8`) is used verbatim (stays native, bypassing the alias). Only the **baked seats** (planner/builder/reviewer) consume a synonym alias; **session**/**guard** take a direct value (`ANTHROPIC_MODEL` / `ANTHROPIC_SMALL_FAST_MODEL`) and never ride an alias. The steps:
     1. **Each synonym mapped FOREIGN → a binding.** Write `launch.aliases.<synonym> = "<foreign concrete id>"` to **`.ai-dev/config.local.json`** (PERSONAL) — or, for a binding the whole team should share, to **`.ai-dev/config.json`** `launch`. Map **all four** synonyms' foreign choices, even one no seat rides (the stable binding the Operator asked for). A synonym mapped native gets **no** alias entry (unbound ⇒ concrete native passthrough).
     2. **Baked seats (planner/builder/reviewer).** Write `roles.<seat>.model = "<its synonym>"` to the SHARED **`.ai-dev/config.json`** (the planner's "inherit session" choice ⇒ omit its `model`, or `"session"`). The installer's `resolveModelPin` + `boundTiers` bakes the **bare alias** when that synonym is bound foreign (routes through the env, because it merges config.local's `launch` before the bake — `docs/decisions/personal-multi-model-setup.md`) and the **concrete id** otherwise (native passthrough) — unchanged.
     3. **Session (independent).** Write `launch.sessionModel = "<the chosen concrete id>"` to **`.ai-dev/config.local.json`** — a native tier's concrete id (e.g. `claude-fable-5`) or a foreign id. Independent of the synonym mappings: a native session over a foreign-bound tier IS exactly this concrete id (it bypasses the alias via `ANTHROPIC_MODEL` — the free-the-alias path, taken by construction). Empty ⇒ native default.
     4. **Guard.** **Default (background follows the haiku slot):** leave `launch.guardModel` empty — `ANTHROPIC_SMALL_FAST_MODEL` is pruned and the background rides `ANTHROPIC_DEFAULT_HAIKU_MODEL` (native, or the foreign id if haiku is bound). **Override (a background DIFFERENT from the haiku slot):** write `launch.guardModel = "<the id>"` to **`.ai-dev/config.local.json`** — exported as the DEPRECATED `ANTHROPIC_SMALL_FAST_MODEL` (G1; warn the Operator it is deprecated).

     For the **baked seats** a synonym is globally native or foreign; the **launch seats are independent** — the session picks a concrete id directly (native even over a foreign-bound tier), and the guard defaults to the haiku slot with a deprecated override. The old **"free-the-alias" special case is gone**: the session's concrete id IS that path, taken by construction. The custom branch is inherently non-vanilla; the byte-unchanged/vanilla state is reached only via the gate's "leave vanilla" answer.

     **Worked example** — Phase 1 planner=inherit-session · builder=sonnet · reviewer=fable; Phase 2 fable=native · sonnet=foreign GLM (opus/haiku native); launch session=opus (native), guard=default: sonnet foreign ⇒ personal `config.local.json` gets `launch.aliases.sonnet = "glm-4.6"` (fable/opus/haiku native ⇒ no alias entries); builder rides sonnet ⇒ shared `config.json` gets `roles.builder.model = "sonnet"` (sonnet bound → bakes the bare alias → GLM); reviewer rides fable ⇒ `roles.reviewer.model = "fable"` (fable unbound → bakes concrete `claude-fable-5`, native); planner inherits session ⇒ no `roles.planner.model`; session opus native ⇒ `launch.sessionModel = "claude-opus-4-8"`; guard default ⇒ `launch.guardModel` empty (background follows the native haiku slot; builder routed foreign via the non-haiku sonnet tier keeps it native — the 4th-tier bonus, no deprecated override). Full trace + the exact JSON: `docs/decisions/multi-model-setup-ux.md` papercut 14.
   - **routing apply-steps (only when ≥1 seat is foreign)** — the coherent flow, in plain language. (a) **Probe first** — `./.ai-dev/launch --probe` prints `{alive,url,models}`. A live proxy sources the foreign ids and **skips the URL question**; nothing answering falls to the explicit fork: **external** — the Operator points at their own proxy, its URL recorded as `proxyUrl` in `.ai-dev/model-routes.json` (the launcher points claude straight at it, no spawn) — **or built-in** — the launcher spawns modelpipe itself on a **random free port** (`router-launch.mjs` `port: 0`), so only the routes scaffold + keys remain. (b) **Sourcing a foreign id** (per seat that picked *another provider*): from the probe's `/v1/models`, a **concrete** `id` (no `*`) is a direct pick-list; a **glob** (`glm-*`, modelpipe-style) names the **provider** — pick it, then have the Operator **type/confirm the current id, validated against that glob** (`glm-5.2` matches `glm-*`), showing host/vision as hints. **Never pre-fill a guessed id, never pre-select a model, never invent a pick-list from model knowledge** — it is always *pick provider → type/confirm its live id*. A third-party proxy that already returns concrete ids needs no provider step. *(Follow-up: a filtered pick-list for the glob case once modelpipe expands globs via `?expand=1`; the `--list` consume — backlog.)* (c) **`configDir`** — offer a per-project claude profile dir (`CLAUDE_CONFIG_DIR` — per-task keys, no `.bashrc` edit); write it to the **gitignored `.ai-dev/config.local.json`** `launch.configDir` (a per-machine path — NEVER the shared `.ai-dev/config.json`). (d) **scaffold `.ai-dev/model-routes.json`** from `.ai-dev/tooling/src/adapter/model-router.example.json` for the named providers (keys by env-var NAME only, never a value) — declinable; on the **external** fork set its `proxyUrl` instead. (e) **remind which backend key env-vars to export** (e.g. `DEEPSEEK_API_KEY`) — names only, never a value (the external fork's keys live in that proxy's own env). (f) **print the ready launch command** — `./.ai-dev/launch` (the OPTIONAL drop-in the installer generated). The full recipe (env precedence, `--proxy`, `config.local.json` merge, the OpenCode stub) is `src/adapter/README.md` `### The launcher` and `docs/decisions/launcher-ux.md` — point, don't restate. Each value lands in its home: shared → `config.json`, personal → `config.local.json`, routes → `model-routes.json`.
3. **Write `.ai-dev/config.json`** with their answers. No spawn, no push. Reversible.
4. **Apply the config** — run the adapter's install over your own project files (the concrete command: `src/adapter/INSTALL.md`). Idempotent — re-running with an unchanged config reproduces the same agent files; a `session` seat bakes no `model:` line, while an `auto`/pinned reviewer bakes its resolved cross-model line on Claude (`## Your seat`). The Claude apply path then runs a **loud routing self-verify** post-bake (`verifyRoutingConsistency`, `src/adapter/install-claude.mjs`): it FAILS the install if any baked seat's `model:` or tier env-var disagrees with its config routing intent, so a foreign-routed seat can never silently land native (the mechanic lives in the check + `docs/decisions/multi-model-setup-ux.md` papercut 13 — point, don't restate). On success it prints the per-seat `seat → model` trace.
5. **Wire the quality toolkit** — discover the stack (languages, package manager, doc format) and propose a stack-appropriate tool set, reasoning from the stack, never a hard-coded list. The set:

   - **Core tools** — linter, formatter, type-checker, doc linter, security/SAST scanner, gitleaks-class secret scanner.
   - **Failure-suppression rules raised to ERROR** (within the linter) — a blind/bare-except and try-except-pass rule where the stack has one (e.g. Python ruff `BLE001` + bandit `S110`/`S112`), since a swallowed failure is the silent-dead-code class a linter CAN catch mechanically.
   - **Commented-out-code rule raised to ERROR** where the stack has one (Python ruff `ERA001`/eradicate; JS an eslint plugin where the stack carries one) — a dead code block left in source is the one *syntactic* junk-comment subclass a linter CAN catch; the semantic junk comment (a `// what` restating the code below it) stays the Reviewer's prose-quality call (invariant 6), never a linter's.
   - **Size / complexity linter** — a **recommended default**: a per-file size/complexity rule at ~800 lines where the stack offers one (eslint `max-lines`/`complexity`, ruff `C901`, a comparable rule), so an oversized file becomes a mechanical finding, not a thing only a human notices. Tuned soft — it surfaces the file to the architect and points at `decompose` (`.ai-dev/procedures/decompose.md`) for remediation, never a hard build block.
   - **UI/E2E automation driver** for a GUI stack — Playwright for web, tauri-driver/WebDriver for native.

   Offer it (declinable). For each chosen tool: install, drop standard config, register a row in `.ai-dev/quality/tools.json`, verify green via `node .ai-dev/quality/run.mjs <beat>`. Tune to the standard — a config relaxation is the Operator's recorded decision.

   **CI wiring** (offered with it, declinable) — a workflow running the registered quality suite on every push/PR, per the project's forge (e.g. GitHub Actions), **via the runner (`node .ai-dev/quality/run.mjs <beat>`, both beats), never a re-listed subset of tool commands**: a hand-copied list drifts the moment a row is added; the registry stays the single home of "what green means". The merge-gate is local, CI is the remote re-check that catches a bypassed local run.

   **Branch protection — a MANDATORY accept-or-decline, never a silent skip** (`docs/decisions/persona-floor-external-substitute.md`, the durable rationale). The local deny and CI are advisory; the **remote forge protection is the one floor that holds regardless of model, platform, or whether the plugin even loaded** — `main` cannot move except through a green-CI PR. Name it plainly: on **OpenCode** the deny layer has no `ask` return, so the whole ask-class degrades to persona (`PROTOCOL.md` `## Enforcement`) — this forge gate **is** that ask-class's declared substitute there; on every platform it is the model-independent remote floor. So this step is **required**: the Operator must answer, and the answer (set / declined / unwirable-here) is **recorded** in state so a later audit sees a conscious choice. On accept, **print the ready recipe for the Operator to run** (you do not run it — that privileged apply is the deferred fast-follow F2-2); on the GitHub forge with the quality job named `quality`:

   ```sh
   gh api --method PUT repos/{owner}/{repo}/branches/main/protection --input - <<'JSON'
   {
     "required_status_checks": { "strict": true, "checks": [ { "context": "quality" } ] },
     "enforce_admins": true,
     "required_pull_request_reviews": { "required_approving_review_count": 0 },
     "restrictions": null
   }
   JSON
   ```

   The `context` must match the check-run name the workflow reports (`quality` in `.github/workflows/checks.yml`); `enforce_admins: true` closes the admin-can-still-direct-push hole. **Graceful when it cannot be wired here** — no `gh` or offline ⇒ give the GitHub web-UI recipe (Settings → Branches → add a rule on `main`: require the `quality` status check, require a PR, include administrators) and record "wanted, unwirable here"; the call needs **admin scope** on the repo (a plain push token 403s) ⇒ say so plainly ("your token lacks admin — set it in the web UI or re-auth with admin scope") and record the decline. Never let a missing tool or scope turn this into a silent skip.

## When it fires

- **Reactive** — on the Operator's first real work request, check for `.ai-dev/config.json`. If absent: give a SHORT offer of two choices (run `/dev-setup` or proceed on safe defaults), then **stop**. Do not start the task, explore the repo, or write a multi-topic essay.
- **Explicit** — `/dev-setup` re-runs on demand. Carries no dialog of its own — it points here, the single home.
- **Mode switch mid-stream** — a bare "switch mode/profile" ⇒ a structured question (the current profile + the options, a one-line trade-off each); a target named directly ("solo"/"full"/"lite") ⇒ a one-line confirm, the config flip, an announce — no full setup re-run.
- **Doc-language switch mid-stream** — the Operator names a documentation language directly ("делай документацию на русском", "docs in German") ⇒ a one-line confirm, flip `docLanguage` in the config, an announce — the lightweight flip mirrors *Mode switch mid-stream*. THEN, because the existing in-scope docs are still in the old language, **offer to translate the current `docs/` tree + `README` into the new `docLanguage` as a build task through the loop** (a durable-prose change goes through the Builder + review, never a silent rewrite; git history keeps the originals) — declinable, forward-only if declined. The floor invariant 5 names (code, commits, config, machine grammar — always English) never moves; say so in the announce.
- **Collaboration switch mid-stream** — the Operator expresses, **in their own words and any language**, the intent to move the project to (or away from) multi-developer / team / collaborative development ("переключи в несколько разработчиков", "готовимся к командной разработке", "we're a team now", "switch to multiple developers"; back: "вернись к одиночной разработке", "solo again") ⇒ recognise the **intent**, never a memorised keyword — the Operator never has to say the field name or the token `team` (the config field stays `collaboration.team`, machine grammar — invariant 5 — but you map their phrasing to it). Lightweight flip mirroring *Mode switch mid-stream*, no full setup re-run. **On enable:** a one-line confirm ⇒ flip `collaboration.team: true` ⇒ ask the same two follow-ups the setup question does (`backlog` file/forge, confirm the auto-detected forge — step 2's `collaboration` bullet; enum value-home `PROTOCOL.md` `## Project config`) ⇒ enable `modules.team-collaboration` ⇒ **apply the config** (re-assemble the agents — step 4 / `src/adapter/INSTALL.md`) ⇒ announce in plain language what changed. On `backlog: forge` chosen, **offer the `file → forge` migration** (`## Backlog`) — declinable — exactly as *Doc-language switch* offers to translate the existing docs after its flip. **On disable (back to solo):** flip `collaboration.team: false`, announce in one line that the `team-collaboration` module and the team behaviours go inert and the backlog **stays where it is** (no auto-delete of forge issues).
- **Model switch mid-stream** — the Operator names a seat model directly ("builder on sonnet", "reviewer back to auto", "set the guard model to deepseek-v4-pro") ⇒ a one-line confirm, flip the seat's pin, an announce — the lightweight flip mirroring *Mode switch mid-stream*. **When the named target is a foreign (non-Anthropic) provider for a baked seat**, it is not a raw pin flip — run the **auto-allocation** (step 2's *auto-allocation* bullet): bind the seat's **synonym** to that foreign id (`launch.aliases.<synonym> = <foreign concrete id>`) so the seat bakes the bare alias and routes foreign. A synonym is globally native or foreign, so there is **no free-the-alias re-pin** — binding a synonym never silently re-routes another seat's concrete-id session or guard. Then apply. **A `builder`/`reviewer` change is a BAKED pin: it needs a re-bake** (re-assemble the agents so the new `model:` lands), so after flipping it **re-apply the config** the same way *Collaboration switch* does (step 4 / `src/adapter/INSTALL.md`). If you cannot run the install yourself, print the **context-correct** re-bake command — and you know the context from whether the protocol's own installer source sits at the root: in the protocol's **own source repo** (dogfood) it is `node src/adapter/install.mjs . --dogfood`; in a **downstream** project (one carrying `.ai-dev/tooling/`) it is `node .ai-dev/tooling/src/adapter/install.mjs .` (no `--dogfood`) — handing the wrong one is the `MODULE_NOT_FOUND` papercut this fixes (`docs/decisions/multi-model-setup-ux.md` papercut 4). A `guard`/session change is launch-time, not baked — there are TWO distinct apply paths, name the right one: a **baked** seat change (builder/reviewer) ⇒ **re-bake** (re-run the installer; it takes effect on the next spawn). A **launch-time** change (guard/session, or the proxy base-URL) ⇒ **re-run the installer to update the harness's startup env, then tell the Operator plainly to RESTART their session** — that env is read only at startup, so the running session keeps the old value until restart (`src/adapter/README.md` `### The launcher`). The orchestrator pin does not exist (the session model is the launch model). Where the project launches through its own wrapper, the same launch-time value also flows through the printed export block — restart still required.

## Platform switch

You can tell your own harness from the tool surface you hold; when that platform differs from the config's `platform`, offer the switch on the understand beat: *"this session runs on a platform the config doesn't name — wire it and switch?"* Short, declinable, never a block; declined ⇒ proceed silently. On accept:
